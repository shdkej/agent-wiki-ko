# 100k_concurrent_server

- 원본: `source/shdkej-content/blog/100k_concurrent_server.md`
- 상태: baseline 이후 점진 심화
- 역할: 만다라트 `blog > 100k_concurrent_server` 카테고리 노드의 1:1 대응 페이지

## 한눈에 보는 요약
이 노트는 `10만 동접이 가능한가`를 자극적인 숫자로 증명하려는 글이 아니다. 블로그 층위에서 공개된 실험 기록이지만, 실제 내용은 **한 대의 서버가 어디서 먼저 막히는지 몸으로 확인하는 운영 탐구 노드**에 가깝다. 원문은 소켓/포트/커넥션 개념 혼선에서 출발해 `ulimit`, `TIME_WAIT`, 클라이언트 포트 고갈, 네트워크 대역폭, CPU, 메모리, nginx, Kubernetes ingress, websocket 유지 연결까지 차례로 부딪힌다. 결론도 `한 줄 정답`이 아니라, **동시 접속이라는 숫자는 서버 코드만이 아니라 테스트 토폴로지와 운영 경계가 함께 만든 결과**라는 쪽에 가깝다.

## 만다라트 위치에서 읽는 해석
이 페이지가 `Deep Knowledge/Network`나 `Infra`가 아니라 `blog` 아래에 놓여 있다는 점이 중요하다. 여기서 100k 동접은 네트워크 이론 정리보다, **실제로 해 보다가 어디서 막혔고 어떤 가설을 세우고 어떤 관측 명령으로 다시 확인했는가**를 외부에 설명하는 공개형 운영 실험 기록이다. 즉 이 노드는 기술 백과보다 `실험의 사고 경로`를 보존하는 블로그형 카테고리로 읽는 편이 맞다.

## 이 노드의 중심 질문
- 한 대의 PC로 10만 동시 접속을 유지할 수 있는가?
- 소켓, 포트, 커넥션은 실제 병목과 어떻게 연결되는가?
- 한계는 서버 코드, 프로세스 FD, 클라이언트 포트, 네트워크, 프록시, ingress 중 어디에서 먼저 드러나는가?
- `짧은 요청 처리`, `유지 연결`, `웹소켓`, `쿠버네티스` 실험은 왜 서로 다른 숫자를 보여 주는가?

## 원문 발췌 구조
- `왜 10만인가`: 해외 축구 경기 동접 규모를 보고 현실적 목표치를 잡는다.
- `소켓/포트/커넥션 정리`: 용어 정의가 흔들리는 상태에서 주소 공간과 식별 구조를 붙잡으려 한다.
- `메모리/대역폭 계산`: `tcp_mem`, `wmem`, `rmem`, 랜카드, `iperf`를 통해 자원 감각을 세운다.
- `100,000 user load test`: Go echo 서버/클라이언트 코드로 짧은 요청 테스트를 시작한다.
- `상황1`: `too many open files`, `i/o timeout`, `connection reset`, `cannot assign requested address` 같은 오류를 따라 병목을 좁힌다.
- `상황2`: 10초 유지 연결·websocket 테스트로 `열린 연결을 붙잡는 비용`을 따로 본다.
- `상황3`: HLS/스트리밍 서버 쪽으로 질문을 확장한다.
- `추가 벤치마크와 OS 파라미터`: 동접과 처리량을 분리해 다시 보려는 후속 실험 큐가 이어진다.

## 이 노드를 읽는 4개 층위

### 1. 개념 모델을 세우는 층위
원문 초반은 꽤 오래 `소켓`, `포트`, `커넥션`에 머문다. 깔끔하게 정리된 정의를 바로 얻지는 못하지만, 이 구간은 헛도는 서론이 아니다. 저자는 `65535` 같은 숫자를 외우기보다, **누가 주소 공간을 점유하고 누가 FD를 소모하며 누가 TIME_WAIT를 남기는가**를 이해하려 한다. 뒤의 실험 실패를 다시 읽게 만드는 기반이 여기 있다.

### 2. 관측 가능한 실험으로 바꾸는 층위
원문에는 `watch ss -s`, `/proc/net/tcp`, `netstat`, `iperf`, `ulimit`, `sysctl` 같은 메모가 많다. 이것은 팁 수집이라기보다, `안 된다`를 넘어서 **어떤 상태가 실제로 늘고 줄고 막히는가**를 보려는 관측 습관이다. 이 노트의 핵심 자산은 숫자 자체보다 `병목을 보이게 만든 계기`다.

### 3. 테스트 토폴로지 함정을 알아차리는 층위
원문이 특히 살아나는 지점은 로컬에서 3만~6만 부근에 걸리며 `이게 진짜 서버 한계인가?`를 의심하는 부분이다. 클라이언트와 서버가 한 머신에 있으면 서버만 테스트하는 것이 아니라 **클라이언트 포트 범위, 로컬 네트워크 경계, 가상화 계층, 외부 IP 여부**까지 같이 소모한다. 여러 VM, vagrant, 외부 IP, 다중 IP 할당 실험은 이 함정을 스스로 드러내는 과정이다.

### 4. 단일 서버 튜닝에서 운영 구조 질문으로 넘어가는 층위
후반으로 갈수록 질문은 `Go 서버를 더 빠르게 만들까`보다 `프로세스를 늘릴까`, `nginx를 앞단에 둘까`, `ingress를 어떻게 볼까`, `websocket은 어떤 비용 구조인가`로 이동한다. 즉 이 노트의 진짜 확장은 최적화 팁보다 **하나의 숫자 문제를 운영 구조 문제로 번역하는 과정**에 있다.

## 핵심 논지

### 동시 접속은 하나의 절대 숫자가 아니라 먼저 고갈되는 층의 최소값이다
원문 전반에서 반복되는 것은 단일 원인 찾기 실패다. 처음에는 포트 상한처럼 보이지만, 곧 FD 한계, TIME_WAIT, ephemeral port 범위, `i/o timeout`, CPU 과부하, 네트워크 unreachable, ingress 병목이 연달아 등장한다. 그래서 이 노트는 `10만 가능/불가`보다 **어느 자원 층이 먼저 바닥나는가**를 기록한 문서로 읽어야 한다.

### 짧은 요청 처리와 유지 연결은 같은 테스트가 아니다
원문은 echo HTTP, 10초 유지 연결, websocket 유지 연결, ingress 뒤 replica 테스트를 모두 같은 질문 아래 적어 두지만, 실제로는 서로 다른 성질을 가진다.
- echo HTTP: 연결 자체를 얼마나 빨리 열고 닫고 응답하는가
- 유지 연결: 열린 상태를 얼마나 오래 붙들 수 있는가
- websocket: 연결 수와 상태 유지 비용이 메모리로 얼마나 직접 나타나는가
- Kubernetes: 서버 코드 밖의 ingress/controller 계층이 어디서 막히는가

이 구분을 세워야 수치 혼선이 줄어든다.

### 로컬 숫자는 서버 성능표가 아니라 실험 환경의 결과다
원문에서 `localhost는 되는데 외부 IP는 금방 에러`, `vagrant에서는 1만도 삐걱`, `로컬에서만 하면 왜 3만이 최대 같지?` 같은 대목이 반복된다. 이 흐름은 중요하다. 저자는 결국 **로컬 벤치마크 숫자를 서버 본체의 절대 능력으로 읽으면 해석이 틀어진다**는 사실에 가까워진다.

### multi-process 발견은 팁이 아니라 구조 전환점이다
`충격적 발견`으로 적힌 프로세스당 FD 한계 메모는 이 노트의 전환점이다. 여기서 질문은 `한 프로세스로 얼마나 받느냐`에서 `한 머신에서 프로세스/포트/IP를 어떻게 나눠 받을 것이냐`로 이동한다. 즉 이 노트는 단순 벤치마크를 넘어 **운영 구조를 다시 설계하는 감각**으로 넘어간다.

## 실험 흐름에서 남는 구조적 교훈

### 1. 에러 메시지는 서로 다른 층을 가리킨다
- `too many open files`: 프로세스 자원 한계
- `cannot assign requested address`: 클라이언트 쪽 주소/포트 고갈 의심
- `connection reset by peer`: 서버 측 연결 종료 또는 재사용 충돌 의심
- `i/o timeout`: 처리 지연, 네트워크, 과부하가 섞인 대기 문제
- `network is unreachable`: 경로/가상화/네트워크 계층 문제

원문은 이 오류들을 하나로 뭉개지 않고, 어떤 층을 먼저 의심해야 하는지 배우는 과정으로 읽힌다. 이 점은 [[Meta/Troubleshooting]]의 `증상 → 상태 확인 → 수정 → 재검증` 루프와 정확히 맞닿아 있다.

### 2. 관측 명령이 있어야 실험이 경험담에서 운영 자산으로 바뀐다
원문에서 눈에 띄는 것은 숫자보다 `무엇을 보며 판단했는가`다. `watch ss -s`, `/proc/net/tcp`, `netstat`, `iperf`, `sysctl` 메모는 모두 `느낌상 느리다`를 넘어서 **소켓 수, TIME_WAIT, 대역폭, 커널 파라미터**를 상태로 만들려는 시도다. 이 태도는 [[Deep Knowledge/Monitoring]]의 상태 가시화 감각과도 이어진다.

### 3. 네트워크 개념 이해와 운영 실험이 분리되지 않는다
초반의 TCP/포트/keep-alive/window scaling 질문과 후반의 실제 실험은 따로 놀지 않는다. 원문은 개념을 공부한 뒤 나중에 실험하는 방식보다, **실험에 막혀서 개념 모델을 다시 보강하는 방식**으로 진행된다. 그래서 [[Fundamental/Network]]는 이 노드의 배경지식이자, 이 노트가 계속 되묻는 설명 층으로 연결된다.

### 4. 한계 숫자보다 안정 운영 숫자가 따로 있다
원문은 이론적으로 더 가능해 보여도 `실제 사용 시에는 최대 5만 정도만 안정적으로` 같은 식의 판단으로 돌아온다. 즉 저자에게 중요한 것은 최고 기록 경신보다, **CPU·메모리·네트워크 안정성을 함께 고려한 현실적 상한**이다. 블로그 글이지만 성향은 성능 자랑보다 운영 보수주의에 가깝다.

## 원문에서 드러나는 저자의 선호
- 추상 이론보다 직접 열어 보고 깨져 보는 실험 로그를 더 신뢰한다.
- `몇 만이 됐는가`보다 왜 거기서 멈췄는지, 어느 층이 먼저 고갈됐는지를 더 오래 붙든다.
- 명령어, 파라미터, 에러 메시지를 남겨 다음 실험 진입 비용을 낮추려 한다.
- 완결된 결론보다 미해결 질문과 후속 테스트 큐를 남겨 둔다.
- 성능 최적화를 서버 코드 단독 문제가 아니라 프록시·ingress·멀티프로세스·관측 체계까지 포함한 운영 구조 문제로 본다.
