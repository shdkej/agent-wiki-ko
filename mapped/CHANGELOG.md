# CHANGELOG

- 원본: `source/shdkej-content/CHANGELOG.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 루트 카테고리 페이지

## 한눈에 보는 요약
이 노드는 단순한 배포 기록이 아니라, 개인 웹사이트를 오래 굴리면서 무엇에 반복적으로 시간을 썼는지 보여주는 운영 이력이다. 긴 커밋 목록을 따라가면 몇 가지 축이 남는다. 공개 발행 환경 유지, 검색 노출과 유입 실험, 광고와 수익화 시도, Gatsby 생태계 대응, 그리고 계속되는 의존성 유지보수다. 핵심은 `새로 추가한 것`보다 `계속 다시 맞춰야 했던 것`이 더 많이 보인다는 점이다.

## 핵심 논지
이 CHANGELOG의 장기 가치는 "무엇을 한 번 만들었는가"보다 "어떤 종류의 일이 계속 다시 생겼는가"에 있다. 이 프로젝트는 글을 올리는 공간이면서 동시에 프레임워크, 플러그인, 빌드, 배포, 검색, 광고 설정까지 함께 관리해야 하는 기술 시스템이었다. 따라서 이 노드는 기능 연혁보다, 개인 웹 실천이 어떤 유지비와 반복 판단 위에서 성립하는지 보여주는 루트 수준의 운영 노드로 읽는 편이 맞다.

## 왜 이 노드가 중요한가
- [[Portfolio]]와 함께 프로젝트의 기술적 자기 이해를 가장 직접적으로 보여주는 루트 노드다.
- 블로그와 사이트 운영이 콘텐츠 생산만으로 끝나지 않고, 검색 노출, 수익화, 빌드 안정성, 생태계 대응을 함께 요구한다는 점을 드러낸다.
- 이후 [[Communication/Blogging]], [[Meta/My_space]], [[Meta/About_Development]], [[Deep Knowledge/Web]] 같은 노드를 읽을 때 실제 유지 비용과 운영 스타일을 해석하는 기준점이 된다.

## 원본에서 먼저 잡히는 구조

### 1. 큰 릴리스보다 잦은 소규모 갱신이 중심이다
원본은 소수의 굵직한 발표보다 `update 20-10-26`, `UPDATE 21-02-07` 같은 짧은 갱신 기록이 훨씬 많이 보인다.
- 작업 방식이 대형 개편 중심이 아니라 지속적 손질 중심이라는 뜻이다.
- 사이트를 완성품보다 살아 있는 작업물로 다뤘다는 신호다.
- 변경의 의미가 늘 사용자 기능 추가에만 있지 않고, 유지 가능성 확보에도 있었다.

### 2. 콘텐츠 공간과 기술 시스템이 한 파일 안에 섞여 있다
원본의 항목들은 성격이 제각각이다.
- 검색, sitemap, SEO 추가
- Adsense, Kakao adfit 같은 광고 실험
- Gatsby starter, theme, plugin, dependency 갱신
- 보안 패치와 빌드 관련 수정
- 블로그 포스트, 레이아웃, 팝업, 404 페이지 등의 사용자 노출면 수정

이 혼합은 우연이 아니다. 저자에게 사이트는 단순 게시판이 아니라 발행 실험과 개발 실험이 겹쳐 있는 운영 대상이었다.

## 원본 섹션 대응 압축

원본을 그대로 따라 읽으면 섹션명은 단순하지만, 실제 내용은 아래 다섯 층이 겹쳐 있다. 이번 패스에서는 이 다섯 층을 `누가 보는 표면인가` 기준으로 다시 읽는 틀을 앞세운다.

| 층위 | 원문에서 직접 보이는 대표 항목 | 누가 먼저 체감하는가 | 읽는 기준 |
| --- | --- | --- | --- |
| 짧은 운영 반영 | `update 21-04-20`, `🎸 update 21-04-23` | 운영자 자신 | 큰 릴리스보다 생활형 손질 리듬이 앞선다. |
| 발행면 확장 | `add an seo component`, `add sitemap, google custom search`, `ADD adsense script`, `ADD kakao adfit` | 검색 유입자·광고 실험 대상 | 글을 올리는 것과 노출/유입/수익 실험이 분리되지 않는다. |
| 화면·표현 손질 | `update index with grid`, `update 404 page`, `Update popup color and position` | 실제 방문자 | 사용자가 만지는 표면도 잦은 미세 조정 대상으로 다룬다. |
| 기반 유지·상속 노동 | `update gatsby monorepo`, starter/plugin/dependency/security fix 계열 | 운영자와 도구 체인 | 개인 웹도 upstream 변화 비용을 계속 떠안는 기술 시스템이다. |
| 기록 형식의 껍데기 | `standard-version`, compare 링크, `0.1.1`, `0.1.2` | 미래의 자신·협업자 | 개인 웹을 일기보다 배포 가능한 산출물처럼 다루려는 태도를 보여준다. |

### 0.1.2는 짧은 운영 반영 구간이다
원본의 마지막 버전은 사실상 `update 21-04-20`, `🎸 update 21-04-23` 두 줄에 가깝다.
- 마지막 릴리스조차 큰 기능 묶음보다, 이미 운영 중인 사이트를 짧게 손보고 다시 내보내는 리듬이 앞에 온다.
- 이모지까지 섞인 짧은 문장은 changelog가 엄격한 제품 서사라기보다 생활형 운영 메모에 더 가깝다는 점을 보여준다.

### 0.1.1의 `Features`는 기능보다 발행면 확장에 가깝다
원본의 `Features` 아래에는 `seo component`, `sitemap`, `google custom search`, `adsense`, `kakao adfit`, `grid`, 각종 `update`가 함께 들어 있다.
- 형식상으로는 기능 추가지만, 실제로는 콘텐츠 표면 확장, 검색 진입면 설계, 수익 실험, 외형 정돈이 한꺼번에 묶여 있다.
- 즉 이 구간은 제품 기능 목록보다 `개인 웹을 공개 발행 공간으로 키우는 초기 손질`로 읽는 편이 정확하다.

### 0.1.1의 `Bug Fixes`는 버그 수정이라기보다 기반 유지 층이다
`Bug Fixes` 아래에는 package 정리, fragment 제거, 404 수정, dependency 보안 패치, Gatsby monorepo와 starter 업데이트가 대량으로 쌓여 있다.
- 사용자 눈에는 사소하거나 보이지 않는 항목이 많지만, 실제 운영 노동의 중심은 이쪽에 더 가깝다.
- 따라서 원본의 섹션 제목을 그대로 믿기보다, `표면 확장`과 `기반 유지`라는 두 층으로 다시 읽어야 전체 비중이 맞춰진다.

## 원본에서 드러나는 지속 패턴

### 1. 발행 자체보다 발행 환경 유지가 큰 비중을 차지한다
글이나 화면에 직접 닿는 변경도 있지만, 원본의 큰 비중은 환경 유지에 있다.
- Gatsby 버전 업
- starter 및 plugin 정리
- dependency 보안 패치
- 호환성 수정
- 빌드 안정화

즉 이 노드는 "무엇을 만들었는가"의 기록이면서 동시에 "계속 굴리기 위해 무엇을 감당했는가"의 기록이다.

### 2. 가시성과 유입은 초기에 중요한 관심사였다
검색과 광고 관련 항목이 비교적 이르게 반복 등장한다.
- sitemap
- google custom search
- SEO component
- Adsense
- Kakao adfit

이는 프로젝트가 단순한 개인 메모장이 아니라 발견되고 읽히며 경우에 따라 수익화될 수 있는 공개 공간으로 상상되었음을 보여준다.

### 3. 생태계 드리프트를 감당하는 태도가 강하다
원본에는 Gatsby monorepo 업데이트와 관련 dependency 수정이 압도적으로 많다.
- 프레임워크를 쓰는 순간 생기는 외부 변화 비용을 회피하지 않았다.
- 유지보수성은 별도의 프로젝트가 아니라 운영의 일부로 편입되어 있었다.
- 작은 개인 사이트도 툴체인 위에 올라가면 지속적인 환경 변화에 노출된다는 사실이 선명하다.

### 4. 수정은 누적되지만 방향은 몇 축으로 수렴한다
긴 커밋 목록은 산만해 보이지만 실제로는 몇 가지 질문으로 압축된다.
- 사이트가 더 잘 발견되는가?
- 광고나 외부 플랫폼 연동이 자연스럽게 붙는가?
- 프레임워크 변화에도 사이트가 계속 동작하는가?
- 빌드와 배포가 덜 깨지고 덜 귀찮아지는가?

이 반복 질문이 이 파일을 단순 커밋 보관함이 아니라 운영 철학의 흔적으로 만든다.

## 커밋 문장 형태가 보여주는 작업 리듬

### `update`라는 이름의 반복
원문에는 내용이 자세히 설명되지 않은 `update` 커밋이 매우 많다.
- 작업이 늘 완결된 기능 단위로만 이뤄지지 않았다는 뜻이다.
- 작은 수정, 점검, 반영, 정돈, 재배포가 운영의 일상 리듬이었다.
- 저자는 변화의 의미를 매번 거창하게 포장하기보다, 자주 손보는 실천 자체를 자연스럽게 받아들이는 편으로 보인다.

### 외부 생태계 커밋이 섞여 있는 이유
Gatsby 관련 upstream성 커밋까지 같이 보이는 것은 개인 사이트가 독립 섬이 아니었음을 보여준다.
- 자신의 발행 공간도 오픈소스 생태계 변화의 영향을 직접 받는다.
- 개인 작업과 외부 생태계 유지보수 감각이 분리되지 않는다.
- 결과적으로 CHANGELOG는 개인 작업 이력인 동시에 "어떤 기술 세계에 몸담고 있었는가"를 보여주는 간접 기록이 된다.

## 해석을 돕는 재구성

### 제품·콘텐츠 표면에서의 변화
- 검색성과 유입 개선
- 광고 및 수익화 실험
- 블로그/페이지 표현 수정
- 사용자 노출면의 사소하지만 반복적인 정돈

### 플랫폼·도구 체인에서의 변화
- Gatsby 및 starter 업데이트
- 플러그인 호환성 수정
- dependency 및 보안 패치
- 빌드, 배포, 설정 안정화

### 작업 스타일 신호
- 한 번의 대규모 재작성보다 작은 수정의 누적을 선호한다.
- 문제를 방치하기보다 자주 손본다.
- 공개 공간의 미세한 완성도와 기술 기반 유지보수를 분리하지 않는다.

## 이 노드가 보여주는 숨은 작업 감각

### 사이트를 "표현 공간 + 유지해야 할 시스템"으로 본다
이 CHANGELOG에서 콘텐츠와 인프라가 분리되지 않는 이유는 실제 운영에서 둘이 분리되지 않기 때문이다.
- 글을 쓰는 일은 검색 노출, URL, sitemap, SEO와 연결된다.
- 광고 실험은 레이아웃, 스크립트, 사용자 경험과 연결된다.
- 프레임워크 업데이트는 결국 공개 발행 지속성과 연결된다.

즉 개인 사이트 운영은 표현 행위이면서 시스템 유지 행위다.

### 유지보수는 부차 업무가 아니라 정체성의 일부다
원본을 보면 개선의 상당수는 눈에 띄는 신기능이 아니다.
- 깨지는 부분을 줄인다.
- 오래된 설정을 맞춘다.
- 의존성 변화를 따라간다.
- 검색성과 배포 품질을 손본다.

이런 기록은 화려하진 않지만, "계속 운영하는 사람"의 리듬을 보여준다.

## 커밋 분류 체계 자체가 말해 주는 것

### conventional changelog 형식과 실제 작업 의미가 어긋난다
원본은 `Features`와 `Bug Fixes`로 정리되어 있지만, 실제 커밋들의 성격은 더 복합적이다.
- `Features` 아래에도 광고 삽입, SEO 보강, 검색 추가, 단순 업데이트, 의존성 반영이 함께 섞여 있다.
- `Bug Fixes` 역시 단순 버그 수정만이 아니라 보안 패치, upstream 변경 대응, 환경 정리, UI 자잘한 손질까지 포함한다.
- 즉 이 프로젝트에서 중요한 분류 기준은 기능/버그보다 `운영 지속성에 어떤 종류의 압력이 들어왔는가`에 더 가깝다.

이 점은 저자가 실제로 다룬 문제의 중심이 제품 기능 정의보다 운영 환경 조정이었다는 사실을 드러낸다.

### release note보다 maintenance ledger에 가깝다
일반적인 changelog는 사용자가 체감할 변화를 알려주는 문서에 가깝지만, 이 원본은 운영자가 감당한 변화 목록에 더 가깝다.
- 외부 의존성 업데이트
- 검색/광고/SEO 실험
- 레이아웃 및 페이지 정리
- 사소하지만 반복적인 설정 수정

그래서 이 노드를 읽을 때는 "사용자에게 무엇이 달라졌는가"보다 "운영자가 무엇을 계속 감당했는가"를 보는 편이 더 정확하다.

## 개인 웹 운영의 숨은 비용 구조

### 작은 사이트도 upstream에 깊게 묶인다
원본의 Gatsby monorepo, starter, plugin, dependency 업데이트 비중은 개인 웹이 결코 완전히 가벼운 취미 프로젝트가 아니라는 점을 보여준다.
- 공개 발행을 프레임워크 위에서 시작하는 순간 upstream의 변화 주기를 함께 떠안게 된다.
- 사이트 규모가 작아도 도구 체인 결합도는 작지 않을 수 있다.
- 창작물의 양과 유지보수 부담이 반드시 비례하지 않는다는 점이 여기서 드러난다.

### 운영 비용은 서버비보다 판단 비용에 더 가깝다
이 원본에서 반복되는 부담은 금전만이 아니다.
- 언제 업데이트할지
- 어떤 플러그인을 유지할지
- 광고와 사용자 경험의 균형을 어떻게 잡을지
- 검색성과 관리 복잡도 중 어디까지 받아들일지

즉 CHANGELOG는 개인 웹의 숨은 비용을 서버 요금이 아니라, 반복 판단과 재조정의 비용으로 보여준다.

## 이 원본을 카테고리 노드로 읽을 때의 주의점

### 세부 커밋 사실보다 반복 압력의 지도가 더 중요하다
이 문서는 개별 커밋의 완전한 맥락을 제공하지 않는다.
- 많은 항목이 `update`처럼 압축돼 있어 사건 단위 복원이 어렵다.
- upstream 변경 로그와 개인 작업 흔적이 한 파일 안에 섞여 있다.
- 무엇이 실험이고 무엇이 실제 운영 반영인지 항상 선명하지는 않다.

그래서 이 노드를 읽을 때는 각 커밋을 세밀한 사실 기록으로 과신하기보다, 어떤 압력이 반복적으로 프로젝트를 움직였는지 보는 편이 더 정확하다.

### 기능 목록으로 환원하면 이 노드의 가치가 줄어든다
이 원본을 `SEO 추가`, `광고 추가`, `의존성 업데이트` 같은 체크리스트로만 요약하면 중요한 것이 빠진다.
- 왜 이 일들이 반복해서 다시 생겼는가
- 어떤 종류의 관리 부담이 개인 웹 운영을 구성했는가
- 표현 공간이 어떻게 운영 시스템으로 변했는가

즉 이 노드는 기능 연혁보다 운영 패턴의 지도에 가깝다.

### 사용자의 직접 수정이 들어오면 해석보다 선호 변화 추적이 우선이다
이 노드는 루트 카테고리 성격이 강해 사용자가 직접 문장을 고치면, 단순 표현 수정이 아니라 프로젝트 자기이해의 변화일 수 있다.
- 광고나 SEO를 덜 강조하도록 바꿀 수 있다.
- 기술 스택보다 창작 맥락을 더 전면에 둘 수 있다.
- 유지보수의 의미를 다르게 해석하기 시작할 수 있다.

그 경우 에이전트는 기존 해석을 평탄화해 덮어쓰기보다, `human-reviews/`에 복사하고 무엇을 더 중요하게 보기 시작했는지 선호 변화로 기록하는 편이 맞다.

## 운영 질문별로 다시 묶어 보면
원문의 긴 목록은 연대기 그대로 읽으면 산만하지만, 실제로는 몇 가지 운영 질문으로 다시 묶을 수 있다.

### 1. 사람들이 이 사이트를 더 쉽게 발견할 수 있는가
여기에는 sitemap, google custom search, SEO component, grid/index 정리 같은 항목이 묶인다.
- 발행은 곧 노출 설계라는 감각이 강하다.
- 콘텐츠 생산과 검색 진입면 관리가 분리되지 않는다.
- 개인 웹도 배포만으로 끝나지 않고 `찾아오게 만드는 구조`를 필요로 한다.

### 2. 공개 공간에 수익 실험을 어느 정도까지 붙일 수 있는가
Adsense, Kakao adfit, 스크립트 삽입 관련 변경은 이 질문 아래 읽는 편이 자연스럽다.
- 수익화가 프로젝트 중심을 대체하지는 않는다.
- 하지만 공개 공간을 운영하는 이상 현실적 옵션으로 계속 시험해 본다.
- 이때 중요한 것은 광고 추가 사실 자체보다, 사용자 경험과 운영 복잡도 사이의 균형을 계속 조정했다는 점이다.

### 3. 도구 체인이 계속 변해도 사이트를 굴릴 수 있는가
가장 큰 비중을 차지하는 Gatsby monorepo, starter, plugin, dependency 업데이트는 모두 이 질문으로 수렴한다.
- 원본에서 반복적으로 되돌아오는 진짜 압력은 새 기능보다 호환성 유지다.
- 개인 웹이더라도 프레임워크 위에 올라가면 upstream 변화가 곧 운영 일정이 된다.
- 따라서 이 노드의 핵심은 기능 추가 내역보다 `지속 운영을 위해 감당한 외부 변화`를 추적하는 데 있다.

### 4. 자잘한 불편과 깨짐을 얼마나 자주 손보는가
404 수정, popup 조정, fragment 제거, package 정리, 불필요 문서 삭제, 사소한 업데이트 커밋들은 이 질문으로 읽힌다.
- 운영 감각은 거대한 개편보다 작은 마찰을 그냥 두지 않는 습관에서 더 잘 드러난다.
- `update`가 많은 것은 설명 부족의 신호이기도 하지만, 동시에 상시 손질 리듬의 증거이기도 하다.
- 이 프로젝트는 완성품을 발표하는 모델보다, 살아 있는 표면을 계속 만지는 모델에 가깝다.

## 릴리스 단위로 압축하면 더 선명해지는 흐름

### 0.1.1은 사이트의 운영 표면을 넓히는 시기다
첫 정식 버전 구간에는 SEO component, sitemap, google custom search, Adsense, Kakao adfit, 404 페이지 수정, popup 조정, grid 정리 같은 항목이 뒤섞여 있다.
- 공개 발행면을 더 잘 보이게 하고, 더 찾기 쉽게 만들고, 수익 실험까지 붙여 보는 초기 확장기 성격이 강하다.
- 동시에 Gatsby starter, plugin, dependency 업데이트가 이미 크게 섞여 있어, 표면 확장과 기반 유지가 처음부터 분리되지 않았음을 보여준다.
- 즉 사이트를 키우는 일과 사이트를 계속 안 깨지게 두는 일이 같은 시기부터 함께 시작된다.

### 0.1.2는 큰 전환보다 짧은 운영 반영의 지속을 보여준다
마지막 릴리스 구간은 항목 수가 적지만, `update 21-04-20`, `update 21-04-23`처럼 짧은 반영이 이어진다.
- changelog의 끝이 거대한 선언으로 닫히지 않고, 여전히 자잘한 운영 리듬으로 마감된다는 점이 중요하다.
- 이 파일 전체를 관통하는 핵심이 대형 버전 서사보다 `계속 손보는 운영 습관`이라는 사실을 다시 확인시켜 준다.

## 원본에서 직접 보이는 증거 묶음

간단히 빈도를 세어 보면 이 페이지의 해석이 왜 `신기능 목록`보다 `운영 압력 지도`에 가까운지 더 분명해진다.
- `update`: 233회
- `add`: 32회
- `remove` + `delete`: 9회
- `gatsby monorepo`: 96회
- `sitemap`: 4회, `google custom search`: 4회
- `adsense`: 3회, `adfit`: 3회

빈도만으로는 부족해서, 이번 패스에서는 대표 커밋 묶음을 바로 위 표에 붙여 `반복 압력`과 `실제 사례`가 같이 보이도록 맞췄다.

| 반복 표현 | 이 페이지에서의 해석 |
| --- | --- |
| `update`가 `add`보다 압도적으로 많다 | 이 changelog의 기본 리듬은 확장보다 재조정·유지에 가깝다. |
| `gatsby monorepo`가 매우 자주 나온다 | 개인 웹도 upstream 변화 비용을 지속적으로 떠안는다는 사실이 전면에 드러난다. |
| `sitemap`, `google custom search`, `seo component` | 발행은 게시만이 아니라 검색 진입면 설계까지 포함된다고 본다. |
| `adsense`, `adfit`, `script` | 공개 공간에 수익 실험을 붙이되, 운영과 사용자 경험의 균형을 계속 조정한다. |
| `remove`, `delete`, 각종 `fix` | 큰 기능 발표보다 작은 마찰 제거와 상시 유지보수가 실제 작업 리듬에 더 가깝다. |

이 표는 세부 사실을 완전히 복원하려는 것이 아니라, 긴 목록 전체에서 무엇이 가장 자주 되돌아오는지 빠르게 붙잡기 위한 압축 지도다.

### 형식적 분류보다 실제 압력의 묶음이 더 중요하다
원문은 `Features`와 `Bug Fixes`로 나뉘지만, 실제 내용은 그보다 훨씬 운영 혼합적이다.
- SEO나 광고 삽입은 기능 추가이면서 동시에 발행 전략 조정이다.
- 의존성 업데이트는 버그 수정이면서 동시에 생태계 드리프트 대응이다.
- 페이지 정리나 문서 삭제는 UI 수정이면서 관리 비용 절감이기도 하다.

그래서 이 노드는 changelog의 표면 분류를 그대로 따르기보다, 실제로 어떤 운영 압력이 비슷한 판단을 요구했는지 기준으로 읽는 편이 더 정확하다.

## 이 changelog에는 상속된 이력 층도 섞여 있다

### 개인 사이트 변경 기록만은 아니다
원본에는 개인 사이트에서 직접 했다고 보기 어려운 Gatsby upstream성 항목이 매우 길게 섞여 있다.
- `gatsby monorepo`, `starter`, `plugin`, `www`, `blog`, `docs` 관련 항목이 대량으로 등장한다.
- 이슈 번호와 커밋 설명 형식도 개인 메모보다 오픈소스 릴리스 로그에 가깝다.
- 즉 이 파일은 저자의 순수한 수기 운영 일지라기보다, starter나 의존성 계보에서 상속된 changelog 층과 개인 운영 변경 층이 겹쳐진 기록일 가능성이 높다.

이 점을 잡고 읽으면 원문이 왜 유난히 길고, 왜 개인 사이트의 의도와 직접 연결되지 않는 항목까지 함께 보이는지 설명된다.

### 그래서 더 중요해지는 것은 `사실 하나하나`보다 `선택한 기반의 무게`다
상속된 이력이 섞였다고 해서 이 노드 가치가 줄어드는 것은 아니다. 오히려 다음이 더 선명해진다.
- 저자는 개인 웹을 완전한 맨바닥 구현보다 Gatsby 생태계 위에서 운영했다.
- 그 선택은 빠른 시작과 풍부한 기능을 주는 대신, upstream 변화와 릴리스 계보를 함께 떠안게 만든다.
- 따라서 CHANGELOG는 저자가 직접 한 변화만 기록하는 문서가 아니라, `어떤 기반 위에서 개인 웹을 굴렸는가`까지 드러내는 운영 흔적이 된다.

즉 이 파일의 혼합성 자체가 중요하다. 개인 발행 공간이 독립된 창작물인 동시에, 외부 프레임워크 역사와 결합된 기술 표면이었다는 사실이 여기 드러난다.

## 왜 upstream 항목을 이 노드에서 지우지 않는가
이 원본을 개인 변경만 남기도록 다시 정리하고 싶어질 수 있다. 하지만 이 노드에서는 오히려 upstream성 항목이 남아 있어야, 개인 웹 운영이 실제로 어떤 기반 선택의 무게 위에 놓여 있었는지 보인다.
- Gatsby starter를 선택했다는 사실은 단순 초기 세팅 취향이 아니라, 이후 유지 리듬까지 함께 선택했다는 뜻이다.
- 따라서 upstream 로그는 `잡음`이면서 동시에 `기반 비용의 증거`다.
- mapped 노트의 역할은 그 잡음을 그대로 복제하는 것이 아니라, 왜 이 잡음이 구조적으로 중요했는지 해석해 주는 데 있다.

이 기준을 넣으면 원본의 긴 목록을 정리할 때도 `무관한 항목 삭제`보다 `개인 운영과 기반 상속이 만나는 접점 표시`가 더 중요해진다.

## 버전 번호가 말해 주는 또 다른 층위
원본은 `0.1.1`, `0.1.2`처럼 semantic versioning 껍데기를 갖고 있지만, 내부 내용은 전형적인 제품 릴리스 서사와는 다르다.
- 버전 숫자는 정돈된 배포 단위를 암시하지만, 실제 커밋 흐름은 수많은 `update`, `fix`, upstream 반영으로 이루어져 있다.
- 즉 숫자는 안정적 제품 단계의 선언이라기보다, changelog 도구와 배포 형식을 빌려 개인 웹 운영을 정리한 외피에 가깝다.
- 이 간극은 중요하다. 저자가 실제로 다룬 것은 `버전된 제품 출시`보다 `계속 굴러가게 만드는 운영 리듬`이었기 때문이다.

### release discipline은 있지만 product milestone 서사는 약하다
이 노드에는 분명 release discipline이 있다.
- conventional changelog 형식을 쓴다.
- compare 링크와 버전 번호를 남긴다.
- 기능/버그 수정을 일정한 틀에 넣는다.

하지만 그 discipline이 곧 제품 기획 중심 서사를 뜻하지는 않는다.
- 기능 묶음이 큰 제품 마일스톤으로 설명되기보다 자잘한 운영 반영이 누적된다.
- 버전 경계도 사용자 경험의 큰 전환점보다 운영 편의와 유지보수 묶음에 더 가깝다.
- 따라서 이 원본은 `릴리스 체계를 갖춘 개인 운영 기록`으로 읽는 편이 맞다.

이 관점을 넣으면, semantic versioning 형식과 실제 개인 운영 리듬이 어긋나는 점 자체가 하나의 의미가 된다. 저자는 개인 웹에도 배포 형식을 적용했지만, 그 안에 담긴 실제 노동은 여전히 미세 조정과 유지보수 중심이었다.

### package 생태계의 릴리스 문법을 개인 웹 운영에 빌려온다
원본의 compare 링크, 버전 번호, `standard-version` 문구는 이 changelog가 블로그 일기 형식이 아니라 npm 패키지나 라이브러리 릴리스 문법을 일부 가져왔다는 점을 보여준다.
- 개인 웹도 하나의 배포 산출물처럼 다루고 싶어 했다.
- 창작물 업데이트를 감상문이 아니라 변경 이력으로 관리하려 했다.
- 그래서 이 파일은 `개인 발행 공간`과 `패키지형 릴리스 문화`가 만나는 접점이 된다.
- 즉 저자는 사이트를 단순히 `글 쓰는 곳`으로 두지 않고, 버전과 diff가 남는 작은 제품 표면처럼 다루려 했다.

이 층위를 보강하면, 왜 이 문서가 개인 프로젝트치고는 다소 공학적 형식을 띠는지도 설명된다. 저자는 사이트를 단순 표현 공간으로만 두지 않고, 버전과 배포 흔적이 남는 운영 표면으로 다뤘다.

## 시간 흐름 자체가 말해 주는 운영 리듬
이 원본은 사건 설명이 성긴 대신, 날짜 간격과 커밋 군집으로 운영 리듬을 보여준다.
- 초반에는 sitemap, google custom search, adsense, adfit, SEO component처럼 `발견 가능성·수익화·표면 정리`가 한 묶음으로 자주 나온다.
- 중간으로 갈수록 `gatsby monorepo`, starter, plugin, dependency 계열이 길게 이어지며, 공개 표면 확장보다 기반 유지가 더 큰 시간을 먹는다는 사실이 전면으로 올라온다.
- 후반 `0.1.2`는 `update 21-04-20`, `🎸 update 21-04-23`처럼 짧은 메모형 반영으로 닫혀, 거대한 전환보다 이미 운영 중인 표면을 계속 다시 만지는 리듬을 남긴다.

| 시간대 압력 | 대표 흔적 | 읽기 포인트 |
| --- | --- | --- |
| 초반 | `sitemap`, `google custom search`, `SEO component`, 광고 스크립트 | 먼저 `보이게 만들기`와 `붙여 보기`가 앞선다. |
| 중반 | `gatsby monorepo`, starter/plugin/dependency 갱신 다수 | 선택한 기반이 스스로 유지 일정을 만들어 낸다. |
| 후반 | 날짜형 `update`, 이모지 섞인 짧은 커밋 | 운영이 제품 출시보다 생활 리듬에 더 가까워진다. |

즉 이 문서는 단발성 성과 기록보다 `발행면 확장 → 기반 유지의 장기화 → 생활형 손질의 지속`이라는 운영 시간축을 보여주는 지도이기도 하다.

## 루트 인접 노드와의 역할 분담
이 노드는 루트 인접 노드들 사이에서 `발행면을 실제로 계속 살려 둔 기록`을 맡는다.

| 인접 노드 | 그 노드가 주로 붙드는 것 | CHANGELOG와의 경계 |
| --- | --- | --- |
| [[Portfolio]] | 팀·플랫폼 규모의 문제 해결, 자동화, 구조 변경의 설명 가능성 | `Portfolio`가 성과와 역할 자기소개라면, CHANGELOG는 개인 웹에서 그 감각이 얼마나 잦은 유지 노동으로 나타났는지 보여준다. |
| [[Communication/Blogging]] | 왜 쓰는가, 무엇을 발행할 것인가, 내부 메모를 외부 글로 바꾸는 편집 문법 | CHANGELOG는 그 발행 판단이 실제 웹 표면 유지 작업으로 얼마나 자주 되돌아왔는지 보여준다. |
| [[Meta/My_space]] | 자기 소유 운영 환경, 배포·관측·복구를 한 제어면에서 다루는 감각 | CHANGELOG는 그 큰 운영 욕구가 개인 웹 단위에서 이미 어떤 반복 정비 습관으로 나타났는지 보여준다. |
| [[Communication/Document]] | 재진입 가능한 설명면과 독자별 문서 표면 | CHANGELOG는 설명 문서가 아니라 실제 운영 diff의 축적물이라는 점에서 다르다. |

그래서 이 노드는 `무엇을 말할 것인가`보다 `그 말을 싣는 웹 표면을 어떻게 계속 살려 두었는가`를 붙드는 루트 카테고리 노드로 남는다.

## 이 기록을 덜 평평하게 읽기 위한 3층 분해
원문은 전부 같은 `change`처럼 보이지만, 실제로는 서로 다른 성격의 일이 한 파일에 겹쳐 있다.

### 1. 직접 설계한 발행면 조정
Adsense, Kakao adfit, sitemap, google custom search, SEO component, grid/index 정리, popup 위치 수정 같은 항목은 저자가 자기 웹 표면을 어떻게 노출하고 다듬고 싶은지 비교적 직접적으로 보여준다.
- 무엇을 더 발견 가능하게 만들지
- 어디까지 수익 실험을 붙일지
- 화면 표면을 얼마나 자주 손볼지
- 검색 진입면을 직접 관리할지

### 2. 기반 선택이 요구한 상속 노동
Gatsby monorepo, starter, plugin, dependency 업데이트는 새 방향 제시라기보다, 이미 선택한 도구 체인 위에서 계속 살아남기 위해 흡수한 변화인 경우가 많다.
- 하고 싶어서 한 일이라기보다 계속 굴리기 위해 필요한 일에 가깝다.
- 그래서 이 층은 `무엇을 만들고 싶었는가`보다 `무엇을 계속 감당해야 했는가`를 더 잘 보여준다.

### 3. 생활형 운영 박동
`update 20-10-26`, `UPDATE 21-02-07`, `🎸 update 21-04-23` 같은 짧은 메모형 커밋은 대형 기능 서사보다 `상시 손질`의 리듬을 남긴다.
- 대소문자, 날짜 표기, 이모지까지 흔들린다는 점 자체가 생활형 운영의 박동을 보여준다.
- 이 층 덕분에 changelog는 release note이면서 동시에 운영 맥박 기록이 된다.

이 세 층을 분리하면 같은 `update`라도 의도적 발행 실험인지, 기반 상속 비용인지, 즉시 반영 메모인지 더 덜 평평하게 읽을 수 있다.

## 세 역할이 실제로 반복한 운영 루프
위의 세 층을 시간 순서가 아니라 행동 순서로 다시 묶으면, 이 파일은 개인 웹 운영의 반복 루프를 비교적 또렷하게 남긴다.

1. `보이게 만들기` — SEO, sitemap, search, grid처럼 유입과 표면을 손본다.
2. `붙여 보기` — adsense, adfit, script처럼 현실적 실험을 얹어 본다.
3. `버티게 만들기` — Gatsby, starter, plugin, dependency 업데이트로 기반을 계속 살린다.
4. `다시 만지기` — 날짜형 `update`, popup, 404, fragment 제거처럼 작은 마찰을 계속 걷어 낸다.

이 순환은 [[Communication/Blogging]]이 말하는 `발행 판단`과 [[Meta/My_space]]가 말하는 `자기 소유 운영면` 사이를 실제 작업 리듬으로 이어 준다. 즉 CHANGELOG의 핵심은 개별 기능 사실보다, `발행 → 실험 → 유지 → 미세조정`이 반복되는 생활형 웹 운영 루프를 남긴 데 있다.

## CHANGELOG가 실제로 기록하는 노동의 성격
원문을 끝까지 따라가면 사용자가 체감하는 변화보다 운영자가 감당한 변화가 훨씬 많다.
- 검색, 광고, 레이아웃, 페이지 수정은 보이는 표면이다.
- 의존성 업데이트, 호환성 복구, 설정 정리, 빌드 안정화는 보이지 않는 바닥이다.
- 그래서 이 파일은 `보이는 결과`보다 `보이지 않는 유지 노동` 비중이 큰 프로젝트였다는 증거가 된다.

동시에 창작 리듬과 유지보수 리듬도 분리되지 않는다.
- 블로그, 검색, 광고, 레이아웃 수정이 같은 파일에 기록된다.
- Gatsby와 plugin, dependency 갱신도 같은 파일 안에서 이어진다.
- 결국 `웹사이트를 운영한다`는 일은 글 작성, 노출 설계, 스크립트 삽입, 빌드 유지, 배포 복구를 함께 감당하는 복합 노동으로 남는다.

## 이 노드가 8x8 루트 카테고리로 남아야 하는 이유
이 파일은 특정 기능 묶음의 changelog가 아니라, 개인 웹 운영이라는 루트 주제의 압축 표면이다.
- 검색과 광고는 `사람이 들어오는 면`을 보여준다.
- 레이아웃과 페이지 수정은 `사람이 실제로 만지는 면`을 보여준다.
- Gatsby, starter, dependency 업데이트는 `그 표면을 계속 살려 두는 바닥 면`을 보여준다.
- 짧은 `update` 메모는 `운영 리듬과 생활 박동`을 보여준다.

그래서 이 노드는 하위 사례를 설명하는 보조 문서가 아니라, 만다라트 루트에서 `개인 웹을 어떻게 굴렸는가`를 한눈에 붙드는 카테고리 노드로 읽어야 한다.

## changelog를 남긴다는 행위 자체가 보여주는 것
원본은 소규모 수정과 유지보수의 누적이지만, 형식상으로는 standard-version 기반 changelog를 끝까지 유지한다.
- 자잘한 수정도 버전과 릴리스 문맥 안에 넣어 기록한다.
- 즉 작업이 아무리 작아도 `배포 가능한 변화`로 다루려는 습관이 있다.
- 핵심은 semantic versioning의 엄밀함보다, 개인 프로젝트에도 `변화를 남기고 되짚을 수 있는 형식`을 붙이려 했다는 데 있다.

많은 커밋이 `update` 수준으로 짧게 끝나 서술 밀도는 낮지만, 운영 기억은 남는다.
- 날짜와 군집은 어떤 시기에 운영 에너지가 어디로 쏠렸는지 보여준다.
- 반복 명사는 무엇을 자주 만졌는지 알려준다.
- 버전 경계는 `어느 시점까지를 한 덩어리로 보았는가`를 남긴다.

즉 이 changelog는 회고 문서로는 성기지만, 루트 운영 흔적 문서로는 충분한 밀도를 가진다.

## 원문 발췌 구조 메모
원문을 다시 펼치면 이 노드가 왜 `기능 목록`보다 `운영 압력 분포`에 가까운지 더 직접 보인다.

### 0.1.2는 사실상 생활형 update 묶음이다
`🎸 update 21-04-23`, `update 21-04-20` 두 줄이 중심이라, 버전 번호보다 이미 굴러가는 사이트를 짧게 다시 맞춘 흔적이 더 강하게 남는다.

### 0.1.1 Features도 순수 기능 목록은 아니다
같은 섹션 안에 `ADD adsense script`, `ADD kakao adfit`, `add sitemap, google custom search`, `add an seo component`, 각종 `update`, Gatsby starter 상속 항목이 함께 있다.

즉 `Features`는 신기능 발표라기보다 발행면 확장, 수익 실험, 검색 진입면 설계, 생활형 갱신, 기반 상속이 한 번에 겹친 층이다.

### 0.1.1 Bug Fixes도 실제로는 유지 노동 묶음이다
`remove unnecessary fragment`, `update 404 page`, `Update popup color and position`처럼 직접 만진 표면 수정과, dependency security fix·`gatsby monorepo` 반복 업데이트·blog/docs/www/starters 계열 upstream 수정이 같은 절 아래 뒤섞여 있다.

즉 이 파일의 `Bug Fixes`는 단순 버그 수정 목록이 아니라, `내가 바로 고친 표면`과 `선택한 기반이 끌고 온 상속 노동`이 같은 버전 껍데기 안에서 평평해지는 운영 장부에 가깝다.

## 이 파일이 실제로 남기는 자기서사
표면상으로는 changelog지만, 실제로는 `나는 웹을 어떤 태도로 운영했는가`에 대한 간접 자기서사도 남긴다. 이번 패스에서는 특히 같은 목록 안의 항목이 다음 세 역할로 번갈아 나타난다는 점을 더 분명히 본다.

| 역할 | 대표 흔적 | 이 노드에서의 의미 |
| --- | --- | --- |
| 발행 운영자 | SEO, sitemap, adsense, adfit, grid, popup | 보이게 만들고, 붙여 보고, 다듬는 사람 |
| 기반 관리자 | gatsby monorepo, starter, plugin, dependency, security fix | 선택한 기술 기반의 무게를 계속 흡수하는 사람 |
| 생활형 유지자 | 날짜형 `update`, 짧은 `fix`, `remove` | 큰 선언 없이도 표면을 자주 다시 맞추는 사람 |

그래서 `add`보다 `update`가 더 많이 보인다는 사실은 단순 서술 부족이 아니라, 이 프로젝트의 기본 리듬이 확장보다 재조정에 가까웠다는 증거이기도 하다. [[Portfolio]]가 팀·서비스 규모의 운영 역량을 보여주고 [[Communication/Blogging]]이 발행 판단의 문법을 보여준다면, 이 CHANGELOG는 그 둘 사이를 실제 유지 노동으로 이어 주는 일상적 증거다.

## 보존해야 할 긴장
- 개인 표현과 프레임워크 유지 비용 사이의 긴장
- 빠른 발행과 장기 안정성 사이의 긴장
- 수익화 실험과 사용자 경험 사이의 긴장
- 가시성 확대와 운영 복잡도 증가 사이의 긴장
- 작은 사이트 감각과 실제 툴체인 부담 사이의 긴장

이 긴장을 남겨두는 것이 중요하다. 원본은 단순히 꾸준한 업데이트 기록이 아니라, 자기 소유의 웹 공간을 유지하는 데 드는 실제 비용과 우선순위를 보여주기 때문이다.
