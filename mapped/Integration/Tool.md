# Tool

- 원본: `source/shdkej-content/Integration/Tool.md`
- 성격: 만다라트 `Integration` 축의 1:1 대응 카테고리 노드
- 이번 상태: 기존 기준선은 유지하되, 넓게 흩어진 도구 메모를 `개인 인터페이스`, `원격 운영`, `재설치 가능 환경`, `확장되는 시스템 관심사`의 흐름으로 더 촘촘하게 재정리함

## 한눈에 보는 핵심
이 노드의 `Tool`은 베스트 앱 추천서가 아니다. 원문은 맥 도구, 터미널 도구, 업무 툴, 인프라 툴, Tailscale, OpenClaw, Linux, Git, Terraform, gRPC, 스트리밍 메모까지 한 파일에 놓는다. 겉으로는 산만하지만 실제 중심은 분명하다.

1. 도구는 취향 수집보다 `작업 환경 전체를 조립하고 다시 세팅할 수 있게 하는 부품`이다.
2. 좋은 툴 메모는 기능 소개보다 `검색`, `제어`, `자동화`, `권한`, `복구`를 더 오래 남긴다.
3. 로컬 생산성 앱, 협업 SaaS, 서버 운영 툴, 인프라 도구는 분리된 세계가 아니라 하나의 작업 시스템의 다른 표면이다.
4. 이 노드는 완성된 정답집보다, 계속 붙이고 다시 찾아오는 `개인 운영체제의 부품 창고`에 가깝다.

## 이 노드가 실제로 답하려는 질문
- 어떤 도구 조합이 일을 더 잘하게 해 주는가가 아니라, 어떤 조합이 마찰을 줄이고 다시 시작 가능하게 만드는가?
- 개인용 앱과 서버 운영 툴은 왜 한 카테고리 안에서 함께 읽혀야 하는가?
- 툴을 많이 안다는 것과 툴을 엮어 시스템을 만든다는 것은 어떻게 다른가?
- 설치 성공보다 운영 중 문제 해결과 복구 절차가 왜 더 오래 기록되는가?
- 새 도구를 볼 때 저자가 실제로 먼저 보는 것은 기능인가, 경계 조건과 편입 비용인가?

## 핵심 논지
원문은 `tool`을 프로그램 목록이 아니라, 하루의 입력부터 서버 운영까지 이어지는 작업 표면 전체로 다룬다. 카라비너, 레이캐스트, fzf, ripgrep, Obsidian, GitHub, Kubernetes, Grafana, Tailscale, OpenClaw, DNS, Terraform, Vault, Kafka, gRPC가 한 노트에 공존하는 이유도 여기에 있다.

즉 이 노드에서 툴은 다음 전체를 뜻한다.
- 손의 움직임과 화면 전환을 줄이는 개인 인터페이스
- 반복 작업을 외부 기억으로 옮기는 자동화 장치
- 원격 환경을 다루고 상태를 확인하는 운영 손잡이
- 꼬였을 때 다시 세팅하고 복구할 수 있게 하는 기억 저장소
- 점점 더 큰 시스템으로 넘어갈 때 필요한 연결 도구

## 원문을 읽을 때 먼저 보이는 큰 층위

### 1. 상단 목록은 추천 앱이 아니라 `내가 만지는 작업 표면 지도`다
Traefik, Signoz, Grafana, GitHub Actions, Jenkins, Kubernetes, k3s, ArgoCD, Next.js, GA4, Pencil이 한 묶음으로 적힌다. 카테고리가 뒤섞인 것처럼 보여도 실제로는 관측, 배포, 구현, 디자인, 분석, 문서화가 한 사람이 연속해서 만지는 표면이라는 뜻이다.

### 2. `블로그 소프트웨어 탑 텐`은 취향 리스트보다 인터페이스 층 분해다
맥 도구, 터미널 도구, 소프트웨어, 개발도구, 업무도구, 개발내도구, 크롬 익스텐션, 앱이라는 8개 축은 예쁜 taxonomy보다 실제 생활의 접점 순서에 가깝다. 입력, 검색, 작성, 협업, 배포, 소비가 하루 안에서 어떻게 나뉘는지 보여 준다.

### 3. 서비스 목록은 편한 앱 모음이 아니라 `디지털 의존 지형`이다
회사와 개인 서비스를 나눠 적은 부분은 단순 계정 목록이 아니다. Google Drive, Asana, Confluence, Telegram, Gmail, GitHub, Analytics, Cloudflare, Netlify, Pocket 같은 항목이 함께 놓인다는 것은 기록, 협업, 배포, 커뮤니케이션, 저장이 여러 플랫폼에 분산돼 있다는 뜻이다.

### 4. 최신 툴 메모는 채택 후기보다 `운영 편입 과정`을 보여 준다
Tailscale 메모는 설치 찬양보다 `--ssh`, 키 만료, 재인증을 적는다. OpenClaw 메모도 기능 자랑보다 모니터링, cron, 브라우저 제약, GitHub 노트 접근, dashboard 포트포워딩, 모델 설정, 스킬 설치처럼 실제 편입 지점을 남긴다. 새 도구를 볼 때도 저자는 가장 먼저 `어디서 막히는가`를 적는다.

### 5. 뒤쪽의 Linux, Git, ETC 확장은 잡학 과시가 아니라 `복구 가능한 운영 기억`의 축적이다
명령어, 파일 경로, 설정 파일, 옵션 조합, 재시작 순서가 길게 이어지는 이유는 이 노트가 이론 요약보다 재진입용 기억 저장소이기 때문이다. 한 번 해결한 문제를 나중에 다시 꺼내 쓸 수 있게 남기는 방식이다.

## 구조 재정리

### 1. 도구는 기능보다 `작업 인터페이스 설계`의 재료다
카라비너, 레이캐스트, hidden bar, spectacle, Vimium 같은 도구는 모두 입력 마찰과 전환 비용을 줄이는 쪽에 놓여 있다. 이 노드의 초반은 `좋은 앱을 많이 쓴다`보다 `손과 화면의 이동을 얼마나 덜 낭비하느냐`에 가깝다.

여기서 드러나는 기준은 다음과 같다.
- 자주 쓰는 동작을 짧게 만든다.
- 화면과 창의 전환비용을 줄인다.
- 검색과 호출을 빠르게 만든다.
- 반복되는 선택을 기본값과 단축키로 밀어 넣는다.

### 2. 터미널 도구는 지식보다 `검색과 조합의 속도`를 담당한다
vim, zshrc, UltiSnips, Rclone, fzf, ripgrep, jq, Claude Code 같은 도구가 함께 등장하는 것은 우연이 아니다. 이들은 모두 더 빨리 찾고, 더 빨리 조합하고, 더 빨리 수정하기 위한 층이다.

즉 이 노드에서 CLI 도구의 가치는 다음에 있다.
- 많은 것을 보기보다 정확히 찾아내기
- 복잡한 UI보다 재실행 가능한 텍스트 흐름 남기기
- 사람 기억을 줄이고 검색성과 스크립트 가능성을 높이기

### 3. 협업 도구와 개인 도구는 분리되지 않고 하나의 운영 루프를 이룬다
Asana, Slack, Confluence, Google Drive 같은 업무 도구와 Obsidian, ChatGPT, 메일, 브라우저, 메신저 앱이 한 문맥에 놓인다. 이는 저자에게 생산성이 개인 정리와 협업 정리가 분리된 문제가 아니라는 뜻이다.

개인 정리와 협업은 이렇게 이어진다.
- 개인 메모에서 아이디어를 잡는다.
- 협업 도구에서 상태를 공유한다.
- 문서화로 팀 기억을 남긴다.
- GitHub, dashboard, monitoring에서 실행 결과를 다시 확인한다.

### 4. 이 노드의 중요한 축은 `재설치 가능 환경`이다
글로벌 설치 목록을 파일로 모으는 스크립트, dotfiles, startup files, backup, cron, ssh, git without password, known_hosts 처리, dashboard 포트포워딩 메모가 반복해서 나온다. 이는 단순한 사용 팁이 아니라 `내 환경을 다시 만들 수 있는가`를 계속 묻는 방식이다.

여기서 툴 메모는 사용 후기보다 다음 역할을 한다.
- 다시 세팅할 때 빠르게 복원할 기준점이 된다.
- 자주 막히는 권한, 경로, 인증 문제를 외부화한다.
- 도입한 툴이 일회성 실험인지 장기 편입 자산인지 가늠하게 한다.

### 5. OpenClaw와 Tailscale 메모는 `원격 운영의 실제 문턱`을 보여 준다
이 부분은 특히 중요하다. 원문은 OpenClaw를 추상적 AI 도구로 다루지 않고, 모니터링, cron 뉴스 수집, 헤드리스 브라우저 가능 여부, GitHub 노트 접근, dashboard 접근, 모델 변경, 스킬 세팅처럼 구체적 운영 흐름 속에 넣는다. Tailscale도 마찬가지로 접속 성공이 아니라 `SSH가 언제 되는가`, `키 갱신은 어떻게 하는가`, `재인증 후 무엇이 달라지는가`를 적는다.

즉 새 도구는 `신기함`이 아니라 아래 질문을 통과해야 편입된다.
- 원격에서 믿고 쓸 수 있는가?
- 상태 확인이 쉬운가?
- 갱신과 인증 비용은 어느 정도인가?
- 기존 메모, 서버, GitHub 흐름과 이어지는가?

### 6. Linux와 Git 메모는 이 노드의 하부 토대다
후반부의 Linux와 Git 파트는 별도 노트로 떼어도 될 만큼 길지만, 여기 남아 있다는 사실이 중요하다. 상위 도구를 계속 쓰기 위해서는 결국 셸, 권한, 파일 시스템, 네트워크, 백업, 브랜치, 인증, 복구를 직접 다뤄야 하기 때문이다.

이 부분은 다음을 보여 준다.
- 화려한 도구는 기본 운영 감각 위에서만 오래 간다.
- Git은 협업 툴이면서 동시에 개인 복구 장치다.
- Linux 메모는 기능 암기보다 재진입 가능한 작업 기억이다.
- 상위 생산성은 하위 운영 안정성에 기대고 있다.

### 7. ETC 파트는 툴 수집이 아니라 `시스템 조합 능력의 확장`이다
Ansible, Terraform, Serverless, Vault, Nagios, Lambda cronjob, API gateway, Kafka, gRPC, 동영상 스트리밍이 함께 등장한다. 각각 깊이는 다르지만 공통점은 뚜렷하다. 저자는 도구를 개별 기능으로 소비하기보다, 이 도구가 어느 레이어를 맡고 어디와 붙는지를 계속 본다.

그래서 이 파트는 이렇게 읽는 편이 맞다.
- Terraform vs Serverless는 제품 비교가 아니라 책임 배치 문제다.
- Vault는 보안 도구이면서도 운영 편의와 충돌하는 판단 대상이다.
- gRPC와 API gateway는 기술 선택이면서 통신 구조 설계 문제다.
- 스트리밍 메모는 기능 호기심보다 실제 운영 부하와 전달 구조를 읽는 실험이다.

## 이 노드에서 반복되는 판단 축

### 검색성 vs 화려함
ripgrep, jq, fzf, raycast, dashboard, monitoring처럼 빠르게 찾고 바로 조작할 수 있는 도구가 반복된다.

### 편의성 vs 재현 가능성
당장 편한 툴도 쓰지만, 설치 목록, dotfiles, cron, backup 메모를 남기는 걸 보면 재현 가능성을 더 길게 본다.

### 로컬 마찰 감소 vs 원격 운영 안정성
키보드 단축, 브라우저 확장, 앱 추천과 함께 Tailscale, OpenClaw, SSH, dashboard 메모가 놓인다. 둘은 다른 문제가 아니라 같은 생산성 문제다.

### 사용법 암기 vs 복구 가능한 기억
명령어 자체보다 권한, 파일 위치, 재시작 순서, 인증 갱신, 꼬였을 때 복구 루프가 더 오래 남는다.

### 단발성 시도 vs 장기 편입
새 도구를 보면 먼저 써 보는 데서 멈추지 않고, 이 도구가 기존 노트, 서버, 작업 흐름에 안정적으로 편입되는지 확인하려는 성향이 보인다.

## 원문 층위별 압축

### A. 개인 인터페이스 층
- 카라비너, 레이캐스트, hidden bar, spectacle
- Vimium 같은 브라우저 조작 도구
- 메모, 메일, 브라우저, 모바일 앱

### B. 검색과 텍스트 조작 층
- vim, UltiSnips, zshrc
- fzf, ripgrep, jq
- Rclone, Claude Code

### C. 협업과 디지털 서비스 층
- Asana, Slack, Confluence, Google Drive
- GitHub, Analytics, Cloudflare, Netlify
- 개인 서비스와 회사 서비스의 의존 지형

### D. 원격 운영과 관측 층
- Traefik, Signoz, Grafana, Prometheus
- Jenkins, GitHub Actions, Kubernetes, k3s, ArgoCD
- Tailscale, OpenClaw dashboard, 포트포워딩, 모델/스킬 설정

### E. 하부 운영 토대 층
- Linux 조작, shell, backup, cron
- SSH, 권한, 파일 경로, known_hosts
- Git 인증, 브랜치, 복구, 재정리

### F. 시스템 확장 실험 층
- Terraform, Serverless, Ansible
- Vault, Nagios, Lambda cronjob
- API gateway, Kafka, gRPC, 동영상 스트리밍

## 오래 남길 아이디어
- 툴은 앱 이름보다 `마찰을 줄이고 상태를 제어하고 다시 만들 수 있게 하는 손잡이`다.
- 좋은 도구 메모는 기능 요약보다 경로, 권한, 자동화, 인증, 복구 순서를 남긴다.
- 개인 생산성과 서버 운영은 다른 취미가 아니라 같은 작업 시스템의 앞단과 뒷단이다.
- 새 도구는 신기함만으로 편입되지 않고, 원격 접근과 운영 안정성을 통과해야 장기 자산이 된다.
- 이 노트의 산만함은 약점이기도 하지만, 동시에 실제 작업 환경이 얼마나 넓은 표면 위에서 유지되는지 보여 주는 증거이기도 하다.

## 선호 신호
저자는 툴을 화려한 기능 소비보다 `작업 환경을 길들이는 운영 표면`으로 보는 경향이 강하다. 특히 검색성, 단축, 원격 접근, 관측 가능성, 재설치 가능성, 꼬였을 때 복구하는 순서를 중시한다. 또한 앱을 개별 카테고리로 깨끗하게 분리하기보다, 실제 하루의 흐름 안에서 서로 어떻게 이어지는지를 더 중요하게 보는 편이다.

## 다음에 더 볼 질문
- 이 노드 안에서 `개인 인터페이스`, `원격 운영`, `협업 서비스`, `시스템 조합`을 더 명시적 태그 축으로 드러내면 재활용성이 높아지는가?
- OpenClaw, Claude Code, GitHub, Obsidian을 묶어 `AI 보조 작업 환경` 하위 축으로 따로 심화할 가치가 있는가?
- 서비스 목록을 `기록`, `협업`, `배포`, `커뮤니케이션`, `관측`으로 재분류하면 의존 지형이 더 또렷해지는가?
- Linux와 Git 파트가 별도 노드로 이미 살아 있는 만큼, 이 Integration 노드에서는 상위 조합 원리만 더 강하게 남기는 편이 좋은가?

## 드리프트 점검
대응 원본 `source/shdkej-content/Integration/Tool.md`의 수정 시각은 mapped 노트보다 오래되어, 이번 실행에서는 사용자의 최근 직접 수정본으로 판단하지 않았다. 따라서 `human-reviews/` 복사는 생략했다.

## 이번 갱신 메모
- 기존 baseline은 유지한 채, `툴 추천 모음` 인상을 줄이고 `개인 운영체제의 부품 창고`라는 성격이 더 빨리 드러나도록 구조를 보강했다.
- `블로그 소프트웨어 탑 텐`, 서비스 목록, Tailscale/OpenClaw 메모, Linux/Git/ETC 확장을 하나의 연속된 작업 환경으로 읽는 연결 문장을 추가했다.
- 새 도구를 보는 저자의 기준이 기능 자체보다 원격 편입, 권한, 인증, 복구, 재설치 가능성에 있다는 점을 더 선명하게 정리했다.
