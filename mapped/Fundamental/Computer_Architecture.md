# Computer_Architecture

- 원본: `source/shdkej-content/Fundamental/Computer_Architecture.md`
- 상태: 기준선 재구성 완료
- 역할: 만다라트 `Fundamental` 축에서 컴퓨터가 실제로 어떻게 실행되고, 메모리와 운영체제가 어떤 층위에서 맞물리는지 정리하는 대응 노트

## 핵심 논지
이 노트의 중심은 컴퓨터 구조를 교과서식 부품 목록으로 외우는 데 있지 않다. 저자는 계속해서 **"프로그램이 실제로 어디에 올라가서, 무엇이 무엇을 읽고, 운영체제가 그 흐름을 어떻게 조정하는가"**를 붙잡으려 한다. 즉 CPU, 메모리, 캐시, 프로세스, 커널, 디스크, 시그널, 파일시스템은 따로 떨어진 과목이 아니라, 실행이라는 하나의 흐름을 서로 다른 층위에서 설명하는 요소들이다.

## 이 노드가 다루는 질문
- 디스크에 있는 프로그램이 어떻게 실행 중인 프로세스가 되는가
- CPU, RAM, 캐시, 디스크는 각각 어떤 역할을 맡는가
- 프로세스와 스레드는 실행 단위로서 어떻게 구분되는가
- 메모리 관리 문제는 왜 성능과 안정성 문제로 이어지는가
- 운영체제와 커널은 하드웨어와 프로그램 사이에서 무엇을 조정하는가
- 리눅스 디렉터리, 표준 스트림, 시그널 같은 운영 지식은 왜 구조 이해와 이어지는가

## 한눈에 보는 실행 흐름
원문 첫머리의 요약을 가장 압축하면 다음과 같다.

1. 프로그램은 디스크에 저장되어 있다.
2. 실행하면 코드와 데이터가 메모리로 올라간다.
3. 프로세스가 만들어지며 스택과 힙 같은 실행 공간이 잡힌다.
4. CPU는 메모리에서 명령과 데이터를 읽어 처리한다.
5. 운영체제는 여러 실행 단위와 자원을 스케줄링하고 조정한다.

이 흐름 하나가 이 노트 전체를 묶는다. CPU, 메모리, OS, 캐시, IPC 메모는 모두 이 실행 경로의 다른 단면이다.

## 원문 발췌 구조
- 실행 출발점: `디스크 → RAM → CPU → output` 한 줄로 프로그램 실행의 큰 경로를 먼저 잡는다.
- CPU/프로세스 층: 불 대수, 트랜지스터, 어셈블리, 프로세스, 스레드로 이어지며 `무엇이 실제로 실행되는가`를 묻는다.
- 메모리 층: C 기준 메모리 구조, leak, fragmentation, paging, segmentation으로 `어디에 어떻게 올려 두는가`를 본다.
- 속도 보정 층: DRAM/SRAM, 캐시, Redis batch 예시를 통해 `느린 저장과 빠른 접근 사이의 완충층`을 붙든다.
- 조정 메커니즘 층: IPC, semaphore, mutex, GIL, epoll/IOCP/kqueue로 `여러 실행 단위가 어떻게 부딪히지 않고 움직이는가`를 다룬다.
- 운영체제/실무 층: 커널 구조, ARM 메모, 리눅스 디렉터리, rc.local, 파티션, 표준 스트림, 시그널로 `실제 시스템을 만질 때 어디를 봐야 하는가`까지 이어진다.

## 이 노드를 읽는 4개 층위

### 1. 실행 경로 층
가장 바깥 질문은 단순하다. `저장된 프로그램이 어떻게 지금 실행되는가?` 원문은 부품 정의를 길게 하기보다 이 경로를 여러 번 다른 표현으로 되짚는다.

### 2. 메모리 배치 층
실행은 곧 메모리 배치 문제이기도 하다. 코드/데이터/스택/힙, fragmentation, paging 같은 메모는 프로그램이 단순히 `돌아간다`가 아니라 `어떤 자리 배치와 비용 위에서 돈다`는 감각을 만든다.

### 3. 동시성 조정 층
프로세스, 스레드, semaphore, mutex, epoll은 모두 `여럿이 같이 움직일 때 어떤 규칙이 필요한가`라는 같은 질문의 다른 얼굴이다. 이 층을 빼면 앞의 CPU·메모리 메모가 정적 지식으로 굳어 버린다.

### 4. 운영 접점 층
후반의 리눅스 디렉터리, 표준 스트림, 시그널, 부팅 스크립트 메모는 잡학 부록이 아니라 구조 이해가 실제 운영 인터페이스로 내려오는 지점이다. 즉 이 노드는 하드웨어 입문과 서버 운영 입문이 만나는 경계 노드다.

## 구조별 해석

### 1. CPU, 계산 장치라기보다 실행의 현재 위치
원문은 불 대수, 트랜지스터, 플립플롭, IC, CPU로 이어지는 흐름을 적어 두며, 계산이 결국 물리적 스위칭 구조 위에 세워져 있다는 점을 짚는다. 동시에 어셈블리, 기계어, 고급 언어로 이어지는 층위를 붙여 두는데, 이는 "사람이 쓰는 명령이 어떻게 CPU가 이해하는 형태로 내려가는가"를 보려는 메모다.

핵심은 CPU를 스펙 이름으로 보는 것이 아니라, **메모리에 올라온 코드를 실제로 읽고 수행하는 현재 실행 지점**으로 이해하는 데 있다.

- 기계어는 CPU 아키텍처와 강하게 결합된다.
- 어셈블리는 그 구조를 사람이 다룰 수 있게 가까이 옮긴 표현이다.
- 고급 언어는 다시 그 위에 추상화를 쌓는다.
- 결국 실행 순간에는 다시 CPU가 이해하는 명령 형태로 내려온다.

원문의 `CISC - intel`, `RISC - arm` 메모도 같은 문제의식 안에 있다. 저자는 브랜드 비교보다 명령 집합 구조와 실행 모델 차이를 감각적으로 붙잡으려 한다.

### 2. 프로세스, 디스크의 정적 파일이 메모리 안의 동적 상태로 바뀌는 순간
원문에서 가장 중요한 부분 중 하나는 프로그램과 프로세스를 분리해 보려는 시도다.

- 프로그램은 디스크에 저장된 코드와 데이터다.
- 프로세스는 그것이 메모리에 적재되어 실제 실행 상태를 가진 것이다.
- 실행 과정에서 코드, 데이터, 스택, 힙이 각각 자리를 잡는다.

이 구분은 중요하다. 저자는 단순히 "프로세스는 실행 중인 프로그램"이라는 정의를 외우기보다, **정적 저장물과 동적 실행 상태의 차이**를 이해하려고 한다.

### 3. 스레드, 하나의 프로세스 안에서 일을 나누는 실행 단위
원문은 멀티프로세싱과 멀티태스킹의 역사적 감각을 짧게 적어 두며, 사람들이 여러 일을 동시에 하고 싶어 했기 때문에 실행 단위가 세분화되었다는 관점을 드러낸다.

이 노트에서 프로세스와 스레드는 운영체제 시험문제용 정의보다, **"실행을 얼마나 잘게 나누고 어떻게 간섭을 제어할 것인가"**의 문제와 이어진다. 뒤에 나오는 세마포어, mutex, GIL 메모가 그 연장선이다.

### 4. 메모리, 단순 저장소가 아니라 실행 구조의 무대
원문은 C 언어 메모리 구조를 기준점으로 삼는다.

- Code: 기계어 코드 영역
- Data: 전역 변수, 상수 등
- Stack: 지역 변수, 함수 호출 문맥
- Heap: 동적 할당 영역

여기서 중요한 것은 언어별 차이를 인정하는 태도다. 자바는 JVM 위에서, 파이썬과 JavaScript는 인터프리터 위에서 다르게 움직인다고 적어 둔 점은, 저자가 메모리를 단일 공식으로 외우지 않고 **실행 환경에 따라 달라지는 층위**로 보고 있음을 보여준다.

또한 메모리 누수, internal fragmentation, external fragmentation, paging, segmentation을 함께 적어 둔 흐름은 메모리 관리가 단순 배치가 아니라 **성능과 사용 가능성의 관리 문제**라는 감각으로 이어진다.

### 5. 페이징·세그멘테이션 메모를 여기 남긴 이유
원문은 정의를 정교하게 풀지는 않지만, 둘을 같은 자리에서 적어 둠으로써 중요한 차이를 붙든다.

- paging은 고정 크기 단위로 메모리를 다루려는 감각에 가깝다.
- segmentation은 의미 있는 영역 단위로 나누려는 감각에 가깝다.

즉 저자가 붙잡으려는 핵심은 세부 시험 답안보다, 메모리 관리는 `남는 공간을 어떻게 줄이고`, `실행 상태를 어떻게 배치하며`, `주소를 어떻게 다룰 것인가`의 문제라는 점이다.

## 캐시를 따로 적어 둔 이유
원문은 메모리와 CPU 캐시의 구조 차이를 분리해 적는다.

- 메모리는 주로 DRAM
- 캐시는 SRAM
- CPU와 메모리의 속도 차이를 완충하기 위해 캐시가 중요해진다

여기서 눈에 띄는 것은 저자가 캐시를 단지 하드웨어 부품으로 보지 않고, Redis 배치 로그 저장 같은 예시와 함께 **"느린 저장소와 빠른 접근 사이의 완충층"**이라는 더 넓은 패턴으로 읽고 있다는 점이다.

즉 이 노드에서 캐시는 두 층위에 걸친다.
- 하드웨어 층위: CPU와 메모리 사이의 속도 차이를 줄이는 장치
- 시스템 설계 층위: DB 앞단에서 쓰기 부담을 완화하는 임시 축적 장치

이런 메모 방식은 저자가 기초 지식과 운영 실무 감각을 자주 연결한다는 신호다.

## IPC와 동시성 제어 메모의 의미
원문은 socket, pipe, message queue, semaphore, shared memory를 한 묶음으로 적고, 이어서 semaphore, mutex, GIL을 메모한다. 이는 운영체제를 정적 구조가 아니라 **여러 실행 단위가 부딪힐 때 필요한 조정 체계**로 이해하려는 흐름이다.

### 세마포어와 mutex
- 세마포어는 접근 가능 수를 세는 신호 장치로 이해한다.
- mutex는 사실상 이진 세마포어처럼 접근을 잠그는 도구로 본다.
- 핵심 관심사는 이론 정의보다 간섭 방지와 deadlock 회피다.

### GIL 메모가 말해 주는 것
파이썬의 GIL을 mutex와 연결해 적은 점은, 언어 런타임도 결국 실행 단위 충돌을 관리하는 선택을 한다는 감각을 드러낸다. 즉 동시성 문제는 OS 과목에만 있는 것이 아니라, 언어와 런타임 설계에도 스며든다.

## I/O와 비동기 이벤트 루프 감각
`IOCP`, `epoll`, `kqueue` 메모는 이 노드에서 매우 중요하다. 저자는 이것을 단순 API 이름으로 적은 것이 아니라, **현대 컴퓨터가 I/O 대기와 다중 처리기를 어떻게 감당하는가**의 힌트로 적어 둔 것으로 보인다.

핵심 흐름은 이렇다.
- 컴퓨터 일은 계산만이 아니라 I/O 대기가 크다.
- 현대 시스템은 여러 요청과 이벤트를 동시에 다뤄야 한다.
- 그래서 운영체제는 비동기 I/O를 효율적으로 처리하는 메커니즘을 제공한다.

이는 [[Network]], [[Software]], [[Troubleshooting]] 쪽 운영 감각과도 이어질 수 있는 지점이다.

## 운영체제, 하드웨어와 프로그램 사이의 조정층
원문은 운영체제 계보, 마이크로커널 대 모놀리식 커널, ARM 메모, 리눅스 디렉터리 구조까지 폭넓게 적어 둔다. 얼핏 산만해 보이지만 실제로는 하나의 질문으로 묶인다.

**"컴퓨터를 실제로 쓰게 만드는 관리 층은 어떻게 생겼는가"**

### 커널 메모의 숨은 축
- 모놀리식 커널은 성능과 직접성을 더 얻기 쉽다.
- 마이크로커널은 역할 분리와 이식성 쪽 감각이 더 강하다.
- 결국 성능, 범용성, 분리도 사이의 trade-off 문제로 본다.

여기서도 저자는 특정 진영을 고르기보다, 구조적 장단점의 긴장을 기억하려 한다.

### ARM 메모의 의미
ARM 관련 메모는 단순 칩 비교가 아니라, 저전력, 소음, 호환성, 성능, big.LITTLE 같은 설계 선택이 실제 기기 경험과 어떻게 이어지는지를 묻는 흔적이다. 즉 아키텍처 차이를 스펙표보다 **전력, 발열, 호환성, 사용자 경험**과 연결해 보려는 시도다.

### 부팅·파일시스템·파티션 메모의 의미
`/etc/rc.local`, 파티션 분할, LVM 메모가 같은 노드에 있는 것도 중요하다. 저자에게 컴퓨터 구조 이해는 CPU 내부 원리에서 멈추지 않고, `켜질 때 무엇이 먼저 올라오는가`, `파일은 어디에 놓이는가`, `용량과 복구는 어떻게 다룰 것인가`까지 포함한다.

- 부팅 메모는 실행의 시작 순서를 묻는 운영 질문이다.
- 디렉터리 구조 메모는 시스템 상태를 찾는 좌표계다.
- 파티션/LVM 메모는 저장 공간도 고정 자원이 아니라 운영 중 재배치 대상임을 보여 준다.

## 리눅스 디렉터리와 시스템 운영 감각
원문 후반의 리눅스 디렉터리, `rc.local`, 파티션, LVM, 표준 스트림, 시그널 메모는 앞의 CPU/메모리 설명과 다른 주제로 보일 수 있다. 하지만 실제로는 컴퓨터 구조 이해가 운영 실무와 만나는 지점이다.

### 왜 이 메모들이 여기 붙어 있는가
- `/etc`, `/var`, `/proc`, `/dev`를 알면 시스템 상태를 읽는 위치가 생긴다.
- 부팅 스크립트와 파티션 구조를 알면 "컴퓨터가 켜질 때 무엇이 먼저 실행되는가"를 이해하게 된다.
- 표준 스트림과 시그널을 알면 프로세스 제어와 종료 처리를 더 명확히 볼 수 있다.

즉 이 노트는 하드웨어 구조에서 끝나지 않고, **운영체제 위에서 실제 컴퓨터를 다루는 감각**까지 한 덩어리로 붙잡고 있다.

## 표준 스트림과 시그널, 실행 제어의 실무 문법
원문은 `stdin`, `stdout`, `stderr`, `SIGTERM`을 따로 적어 둔다. 이 메모는 작지만 중요하다.

- 표준 스트림은 프로그램이 바깥과 연결되는 기본 인터페이스다.
- 시그널은 실행 중인 프로세스에 상태 변화를 요청하는 기본 제어 수단이다.
- `SIGTERM`과 `SIGKILL` 차이를 적은 것은 종료도 단순 중단이 아니라 **정리 가능한 종료와 강제 종료의 차이**가 있음을 의식한 것이다.

이는 훗날 서버 종료, graceful shutdown, 배치 중단, 운영 장애 복구 감각과 직접 연결된다.

## 이 노트에서 반복되는 숨은 패턴

### 1. 구조 암기보다 실행 흐름 이해
CPU, 메모리, 커널, 디렉터리, 시그널이 한 파일에 섞여 있는 이유는, 저자가 주제를 분절된 시험 범위보다 실제 실행 시스템으로 이해하려 하기 때문이다.

### 2. 기초 지식과 운영 감각을 분리하지 않음
캐시에서 Redis 배치 저장으로, 시그널에서 graceful shutdown으로, ARM에서 전력/호환성으로 넘어가는 흐름은 기초와 실무가 한 줄기로 이어진다는 감각을 보여 준다.

### 3. 층위를 자꾸 구분하려는 태도
- 프로그램 vs 프로세스
- 메모리 vs 캐시
- 세마포어 vs mutex
- SIGTERM vs SIGKILL
- 모놀리식 vs 마이크로커널

저자는 같은 영역 안에서도 층위와 역할 차이를 계속 나눠 보려 한다.

### 4. 구조를 알면 운영 복구가 쉬워진다는 믿음
원문 후반의 표준 스트림, 시그널, 디렉터리, LVM 메모는 모두 장애나 변경 상황에서 다시 들어갈 수 있는 손잡이를 만들려는 성격이 강하다. 즉 구조 이해의 목적이 이론 완성보다 `문제가 생겼을 때 어디를 봐야 하는지 아는 상태`에 가깝다.

## 보존해야 할 긴장
- 추상화가 높아질수록 편해지지만, 실행 원리를 모르면 문제 원인을 놓치기 쉽다.
- 성능은 중요하지만, 이식성과 범용성도 무시할 수 없다.
- 동시성을 늘릴수록 처리량은 좋아질 수 있지만, 간섭과 복잡도도 함께 커진다.
- 운영체제 지식은 잡학처럼 보이지만 실제 문제 해결에서는 구조 이해와 바로 이어진다.
