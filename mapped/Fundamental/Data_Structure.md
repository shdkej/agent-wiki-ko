# Data_Structure

- 원본: `source/shdkej-content/Fundamental/Data_Structure.md`
- 상태: 심화 재구성 완료
- 역할: 만다라트 `Fundamental` 축에서 자료구조 선택 기준, 대표 구조, 알고리즘 감각을 함께 묶는 대응 노트

## 핵심 논지
이 노트의 중심은 개별 자료구조를 외우는 데 있지 않다. 저자는 계속해서 **"어떤 입력과 작업 패턴에서 무엇을 선택해야 하는가"**를 묻는다. 즉 자료구조를 이름 목록으로 배우기보다, 접근 방식, 메모리 배치, 삽입/삭제 비용, 탐색 방식, 순서성, 중복 허용 여부, 데이터 분포, 입출력 빈도에 따라 판단하는 감각을 만들려는 노트다.

## 이 노드가 다루는 문제
- 배열, 연결 리스트, 해시 테이블, 트리, 그래프, 스택, 큐, 힙을 언제 구분해서 써야 하는가
- 자료구조와 알고리즘이 실제로 어디서 만나는가
- 메모리 관점의 스택/힙과 자료구조 관점의 스택/힙은 어떻게 다른가
- 탐색, 정렬, 최단경로, 인덱스 같은 실전 문제에서 어떤 구조를 떠올려야 하는가
- "문제를 보면 어떤 구조가 숨어 있는지"를 어떻게 감지할 수 있는가

## 선택 기준
원문 초반의 체크리스트는 이 노트 전체를 관통하는 판단 축이다.

- 접근 방식: 임의 접근이 중요한가, 순차 탐색이 허용되는가
- 메모리 공간: 연속 메모리가 유리한가, 포인터 기반 연결이 허용되는가
- 순서성: 입력 순서나 정렬 상태를 보존해야 하는가
- 동시성: 여러 작업이 동시에 접근할 때 구조적 제약이 있는가
- 중복 허용: 같은 값을 여러 번 담아야 하는가
- 변경 가능성: 값이 자주 바뀌는가, 구조 변경이 잦은가
- 읽기/쓰기 빈도: 조회 중심인가, 삽입/삭제 중심인가
- 크기와 분포: 데이터 크기, 희소성, 키 분포가 어떤가

이 기준을 보면 저자는 자료구조를 "정답 목록"이 아니라 **연산 비용과 데이터 성질의 조합 문제**로 본다.

## 기본 구조 해석

### 1. 배열, 가장 단순하지만 강력한 기준선
배열은 연속 메모리를 바탕으로 인덱스 접근이 빠르다. 그래서 탐색 자체가 아니라 **위치를 이미 알고 있을 때의 접근 속도**가 핵심 장점이다.

- 장점: 인덱스 접근이 빠름, 구조가 단순함
- 약점: 중간 삽입/삭제 시 이동 비용이 큼
- 연결되는 감각: "조회가 많고 위치 계산이 가능하면 배열 쪽이 기본값"

원문은 Python list까지 같이 언급하면서, 언어가 추상화를 제공해도 내부에서는 여전히 메모리 배치와 재배치 비용이 중요하다는 점을 암시한다.

### 2. 연결 리스트, 변경 비용을 줄이기 위한 구조
연결 리스트는 배열의 삽입/삭제 부담을 줄이기 위한 발상으로 등장한다. 대신 원하는 위치를 찾는 탐색이 느리다.

- 장점: 노드 연결만 바꾸면 삽입/삭제가 쉬움
- 약점: 탐색이 느림, n번째 접근이 비효율적임
- 핵심 판단: "구조 변경은 자주 일어나지만 임의 접근은 적은가"

원문이 연결 리스트를 설명하면서 곧바로 트리, 그래프, BFS/DFS로 이동하는 흐름은 중요하다. 저자에게 연결 리스트는 단독 구조라기보다 **노드와 연결이라는 사고방식의 입문**이다.

### 3. 해시 테이블, 검색 속도를 얻기 위한 공간-분포 전략
해시 테이블은 원문에서 반복적으로 궁금증을 불러오는 구조다. 저자는 단순히 "검색 O(1)"을 외우지 않고 왜 그런지, 배열과 뭐가 다른지, 충돌이 생기면 왜 느려지는지, 공간 효율과 어떤 관계가 있는지를 캐묻는다.

- 핵심 장점: 키 기반 탐색이 빠름
- 핵심 변수: 해시 함수 품질, 충돌 처리 방식
- 저자의 질문: "딕셔너리와 해시 테이블은 어떻게 연결되는가", "왜 굳이 해시화하는가"

이 노트에서 해시 테이블은 단순 설명 대상이 아니라, **속도와 공간의 교환**, **주소 계산을 통한 접근**, **현실 구현의 타협**을 이해하는 출발점이다.

### 4. 트리, 그래프, 힙, 우선순위 큐
원문은 각 구조를 엄밀히 분리해 교과서식으로 정리하기보다, 문제 풀이 경험 속에서 서로 연결해 이해한다.

- 트리: 계층 구조와 탐색 구조의 기반
- 그래프: 경로, 연결, 네트워크 문제의 기본 표현
- 힙: 완전 이진 트리 성질을 이용한 우선순위 관리 구조
- 우선순위 큐: "정렬된 상태를 계속 유지해야 할 때"의 실용 구조

특히 우선순위 큐에 대해 "미리 다 받아 한번 정렬하는 것과 언제 다른가"를 묻는 부분이 좋다. 이 질문은 정적 일괄 처리와 동적 삽입 환경을 구분하는 감각으로 이어진다.

## 자료구조와 알고리즘의 접점
이 노트는 자료구조만 말하다가 자연스럽게 정렬, 그래프 탐색, 다익스트라, DP, 분할 정복으로 넘어간다. 이는 저자가 둘을 별개 과목으로 보지 않는다는 뜻이다.

### 탐색과 순회
- BFS: 큐가 필요하다
- DFS: 스택적 사고 또는 재귀와 연결된다
- 그래프/트리 문제는 방문 여부, 현재 위치, 방향성 관리가 핵심이다

원문에 있는 질문들, 예를 들면 "어떻게 현재 위치를 기억하지?", "이미 들렸던 곳을 어떻게 체크하지?"는 알고리즘 공식보다 **상태 표현 방식**을 먼저 고민하는 태도를 보여준다.

### 정렬
- merge sort: 분할 후 병합, 안정적인 사고 훈련의 중심
- quick sort: 피벗 선택이 성능에 중요
- bubble sort, insertion sort: 단순하지만 성능 한계가 뚜렷

여기서 중요한 포인트는 정렬 알고리즘 암기보다 **왜 분할이 잘못되면 성능이 무너지는가**를 감각적으로 붙잡으려는 흐름이다.

### 동적 계획법과 분할 정복
원문은 둘을 비슷해 보이지만 다른 방식으로 구분한다.

- 분할 정복: 문제를 나누고, 각 조각을 독립적으로 푼 뒤 합친다
- 동적 계획법: 중복되는 부분문제를 허용하고 저장해 재사용한다
- 재귀: 문제 축소의 표현 방식이지, 그 자체가 항상 전략은 아니다

이 구분은 훗날 알고리즘 문제 풀이뿐 아니라 시스템 설계에서도 재사용 가능한 사고 틀이다.

## 메모리 관점과 자료구조 관점의 겹침
원문은 [[Computer_Architecture]]를 링크하며 스택과 힙의 의미가 맥락에 따라 달라진다고 적는다. 이 대목은 아주 중요하다.

- 컴퓨터 구조 관점의 스택/힙: 프로그램 메모리 영역과 할당 방식의 이야기
- 자료구조 관점의 스택/큐/힙: 데이터 접근 규칙과 연산 특성의 이야기

같은 단어라도 층위가 다르다는 점을 분리해 이해해야 혼동이 줄어든다. 이 노트는 자료구조 학습이 결국 메모리 모델, 실행 모델과 완전히 분리되지 않는다는 사실을 보여준다.

## 실전 감각으로 번역한 핵심 질문
원문 전반에 흩어진 질문을 실전형 판단으로 압축하면 다음과 같다.

1. 조회가 많은가, 변경이 많은가
2. 순서가 중요한가, 키 기반 탐색이 중요한가
3. 모든 데이터를 한 번에 다 아는가, 중간중간 들어오는가
4. 최단경로/연결성/네트워크 문제인가
5. 방문 여부와 상태 저장이 핵심인가
6. 인덱스로 바로 가는 구조가 필요한가
7. 메모리 사용량과 구현 복잡도를 얼마나 감수할 수 있는가

이 질문들은 자료구조를 외우는 대신 문제를 읽는 체크리스트로 쓸 수 있다.

## 원문에서 반복되는 숨은 패턴

### 1. "이론 이름"보다 "왜 그 구조를 떠올려야 하는가"
저자는 B+트리, trie, priority queue, 인덱스 같은 용어를 적어두지만, 관심은 정의보다 선택 근거에 있다.

### 2. 구현보다 먼저 감지 능력
"이 문제가 트리/그래프 구조라는 걸 어떻게 알아차릴까"라는 질문은 매우 핵심적이다. 이는 코딩 전에 문제를 구조화하는 능력을 중시한다는 뜻이다.

### 3. 자료구조와 문제 풀이를 분리하지 않음
배열, 해시, 트리, 그래프를 설명하다가 바로 BFS, 다익스트라, DP, 인덱스로 연결한다. 즉 자료구조는 문제를 푸는 렌즈이지, 독립된 암기 목록이 아니다.

### 4. 언어별 구현 차이에 대한 감각
Python, Go, Redis, DB 인덱스까지 함께 적어두는 방식은 "추상 개념이 언어와 시스템에서 어떻게 구체화되는가"를 신경 쓴다는 신호다.

## 보존해야 할 긴장
- 단순한 구조가 이해와 구현에 유리하지만, 성능 요구가 커지면 더 복잡한 구조가 필요하다.
- 이론적 시간복잡도는 중요하지만, 실제로는 메모리 배치, 충돌, 입력 형태, 데이터 유입 방식이 함께 작동한다.
- 자료구조를 배우면 알고리즘이 쉬워지지만, 반대로 알고리즘 문제를 많이 풀어야 자료구조 선택 감각도 생긴다.
- 언어가 자료구조를 감춰 주더라도 내부 원리를 모르면 성능과 동작을 오해하기 쉽다.
