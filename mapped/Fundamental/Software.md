# Software

- 원본: `source/shdkej-content/Fundamental/Software.md`
- 상태: 기준선 재구성
- 역할: 만다라트 `Fundamental` 축에서 코딩, 아키텍처, 제품, 팀, 비즈니스로 이어지는 상위 카테고리 노드

## 한눈에 보는 요약
이 노드에서 `소프트웨어`는 단순히 프로그램을 만드는 기술이 아니다. 사람의 흐름을 보조하고, 바꾸기 쉬운 구조를 만들고, 오류와 운영 부담을 견디게 하며, 결국 제품과 비즈니스까지 연결되는 현대의 메타도구다. 원문은 언어, 동시성, 패턴, 문서화, 수익화, 캐시, 객체지향, 플러그인, 대용량 데이터까지 매우 넓게 흩어져 있지만, 실제로는 하나의 질문으로 수렴한다. **사람이 감당할 수 있으면서도 계속 바뀌는 현실을 버티는 소프트웨어를 어떻게 만들 것인가.**

## 핵심 논지
원문을 카테고리 노드로 읽으면, 좋은 소프트웨어는 기능이 많은 소프트웨어가 아니라 다음 조건을 함께 만족하는 소프트웨어에 가깝다.

1. 사람의 기존 흐름을 무리하게 깨지 않으면서도 더 좋은 길을 연다.
2. 변경이 쉬워서 현실의 수정 요청을 감당할 수 있다.
3. 에러, 네트워크 불안정, 데이터 증가, 운영 누락을 견디도록 설계된다.
4. 코드 차원에 머물지 않고 배포, 문서, 수익 모델, 사용자 경험까지 이어진다.
5. 무거운 프레임워크 숭배보다 가벼움, 투명성, 저결합, 복구 가능성을 더 중시한다.

즉 이 노드는 `무엇을 구현할까`보다 `무엇을 오래 운영 가능한 형태로 만들까`를 묻는 기반 노드다.

## 왜 이 노드가 중요한가
- `Coding`보다 한 단계 위에서, 코드를 왜 그런 방식으로 짜야 하는지를 설명하는 기준 노드다.
- `Architecture`, `Product`, `Business`, `Teamwork`, `Document`, `Devops`와 연결되는 허브 역할을 한다.
- 원문의 넓은 메모들을 단순 잡기술 모음으로 읽지 않고, `소프트웨어라는 매체를 어떻게 다룰 것인가`에 대한 장기 운영 철학으로 묶게 해준다.

## 원본에서 먼저 잡히는 큰 구조

### 1. 소프트웨어는 사람 사이에 놓이는 도구다
원문 초반의 `사람 - 소프트웨어 - 사람`, `제작자 - 소프트웨어 - 유저`라는 표현이 이 노드의 출발점이다. 소프트웨어는 기술 자체보다 **사람과 사람, 제작자와 사용자, 요구와 결과 사이를 매개하는 구조물**로 다뤄진다.

### 2. 소프트웨어는 변경을 위해 존재한다
종이보다 소프트웨어가 강한 이유를 `수정 가능성`에서 찾는다. 변경이 자유롭다는 점이 가장 큰 장점이며, 선택 전에 기준을 세우고 기준 안에서는 빠르게 선택하는 태도도 여기서 나온다. 즉 설계의 핵심은 완벽한 최초 구현이 아니라 **변경 비용을 낮추는 기준 설정**이다.

### 3. 기술 메모는 결국 운영 문제로 모인다
동시성, 디자인 패턴, 직렬화, API 문서화, 캐시, 대용량 데이터, 에러 처리, 12 factor app 같은 항목이 길게 이어지지만, 공통 질문은 같다.
- 실패해도 버틸 수 있는가
- 다시 읽고 다시 실행할 수 있는가
- 환경이 달라도 유지되는가
- 데이터가 커져도 감당 가능한가
- 사용자가 체감하는 속도와 안정성을 지킬 수 있는가

### 4. 코드 밖의 수익과 서비스 운영까지 소프트웨어의 일부로 본다
무료와 유료의 경계, 개인 맞춤형 서비스, 잠자는 앱 운영 대행, 사용자 피드백 정리 같은 메모는 이 노드가 개발 기술에 갇혀 있지 않다는 신호다. 소프트웨어는 제작물인 동시에 **운영물, 상품, 관계 유지 장치**다.

## 이 노드의 중심 질문

### 어떤 소프트웨어가 사람의 길을 열어주는가
원문은 `좋은 소프트웨어는 기존의 흐름을 바꾸는 게 아니라 길을 잡아주고 확장된 길을 열어준다`고 적는다. 이는 사용자를 교육해야만 쓸 수 있는 시스템보다, 기존 습관과 필요 위에서 더 넓은 가능성을 열어주는 시스템을 선호한다는 뜻이다.

### 얼마나 오래 관리하지 않아도 버틸 수 있는가
`계속 관리해주지 않아도 유지되는 소프트웨어가 만들어질 수 있을까`라는 질문은 중요하다. 이 노드의 관심은 단기 구현보다 **유지 비용이 낮고 복구 가능한 구조**에 있다.

### 가벼움과 통제력은 어디서 균형을 잡는가
원문 곳곳에서 `Convenience <-> Control`, 의존성 회피, 무거운 툴 회의감, 단순한 기본 + 선택적 확장 선호가 반복된다. 즉 편의성은 원하지만, 그것이 불투명한 종속성과 과한 복잡성으로 바뀌는 순간 경계한다.

### 기술 선택이 제품과 돈으로 이어지는가
이 노드는 기술적 완성도만 묻지 않는다. 사람들이 왜 돈을 안 쓰는지, 무엇을 무료로 두고 무엇을 유료화해야 하는지, 개인 대상 운영 지원이 가능한지까지 묻는다. 즉 소프트웨어는 기술 산출물이 아니라 **가치 교환 구조**다.

## 반복해서 드러나는 판단 축

### 1. 변경 가능성
원문에서 가장 강한 축이다.
- 소프트웨어는 변경을 위해 존재한다.
- 기준을 세우고 그 안에서 빠르게 선택한다.
- 기존 코드 전체를 흔들지 않고 확장 부분만 추가되길 원한다.
- 외부 모듈은 분리하고 인터페이스는 추상적으로 둔다.
- 빌드와 배포는 덮어쓰기보다 재생성 쪽을 선호한다.

이 때문에 이 노드에서 좋은 소프트웨어는 완결품보다 **수정 친화적 구조물**에 가깝다.

### 2. 안정성과 복구 가능성
앱은 계속 껐다 켜도 문제 없어야 하고, API는 멱등성을 가져야 하며, 에러 핸들링과 테스트를 먼저 세팅해야 한다는 문장이 이어진다. 이는 기능 추가보다 먼저 **고장 났을 때 어떻게 버틸지**를 설계해야 한다는 태도다.

### 3. 가벼움과 저결합
원문은 외부 API나 라이브러리를 덜 쓰고 싶어 하고, no dependency, low coupling, lightweight를 반복한다. 이는 폐쇄성이 아니라, 시스템이 커질수록 외부 의존이 숨은 비용과 복잡성을 만든다고 보기 때문이다.

### 4. 투명성과 문서화
`Transparent`, 테스트 코드, OpenAPI/Swagger, encode/decode 구분, 로그와 히스토리 메모는 모두 같은 방향을 가리킨다. 즉 소프트웨어는 돌아가기만 해서는 부족하고, **어떻게 동작하는지 드러나야** 한다.

### 5. 규모 증가 대응
람다 아키텍처, 캐시 규칙, 대용량 데이터 조회, 정렬과 필터, 시간 범위 제한 같은 메모는 이 노드가 초소형 도구에만 머무르지 않음을 보여준다. 사용자가 많아지고 데이터가 커질수록, 단순 구현보다 **조회 비용과 운영 패턴**이 중요해진다.

## 원문을 다시 묶은 6개 층위

### 1. 소프트웨어의 본질: 사람 중심의 메타도구
원문은 소프트웨어를 망치 같은 도구이자 작은 우주라고 표현한다. 공부, 구매, 정보 탐색, 작업 수행을 모두 매개하는 메타도구라는 인식이 깔려 있다. 다만 메타라는 말이 공허한 추상이 되지 않으려면, 결국 사용자에게 직관적으로 작동해야 한다는 현실 감각도 같이 붙어 있다.

### 2. 설계 원칙: 추상화는 멋이 아니라 교체와 보호를 위한 것
객체 내부 구조 숨기기, 인터페이스와 유닛 함수 분리, 인자 수 줄이기, 순수 함수 지향, 큰 객체는 포인터로, 공개 API는 추상적으로 두기 같은 메모는 모두 하나의 방향으로 모인다. **추상화는 구현을 숨겨 시스템을 덜 깨지게 만드는 보호막**이어야 한다.

여기서 객체지향이나 패턴은 교과서 정리보다 운영적 해석이 강하다.
- Strategy는 기능 교체를 국소화하는 방법
- Command는 기능 단위를 분리하는 방법
- Adapter는 외부 라이브러리와 내 구조의 마찰을 줄이는 장치
- Proxy는 무거운 동작을 지연시키는 흐름 제어 장치

### 3. 실행 모델: 동시성과 비동기는 성능보다 흐름 설계 문제
동시성, 병렬성, 블로킹, 논블로킹 설명은 단순 개념 정리 같지만, 실제 관심은 `어디가 제어권을 쥐고 있는가`, `완료 순서가 다를 때 어떤 이득이 생기는가`, `A와 B를 큐로 분리하면 무엇이 좋아지는가` 같은 흐름 설계다. 이 노드는 비동기를 유행어보다 **시스템 책임 분산 방식**으로 본다.

### 4. 서비스 운영: 12 factor, 서버리스, API 문서화, 캐시
후반부 메모는 서비스형 소프트웨어를 오래 굴리기 위한 기준으로 읽힌다.
- 환경 분리와 설정 외부화
- 무상태 프로세스와 로그 스트림화
- 포트 기반 공개와 독립 배포
- OpenAPI로 계약을 드러내기
- 캐시를 명확한 룰로만 허용하기
- EOL과 버저닝 안내하기

즉 원문은 소프트웨어를 코드 파일이 아니라 **배포되고 문서화되고 관측되는 서비스**로 본다.

### 5. 데이터와 규모: 검색, 필터, 정렬, 조회 제한
원문은 정렬 종류, 필터링, 전체 개수, 시간 범위 제한, 대용량 배치까지 매우 실무적인 메모를 남긴다. 이는 좋은 소프트웨어가 내부 구조만 예쁜 것이 아니라, **사용자에게 탐색 가능한 형태로 데이터를 드러내는 시스템**이어야 한다는 감각을 보여준다.

### 6. 가치화: 무료/유료 경계와 운영 지원 서비스
유료화 메모는 단순 사업 아이디어보다 중요하다. 저자는 소프트웨어의 가치가 기능 구현에서 끝나지 않고, 사용자 지원과 피드백 정리, 운영 대행, 개인 맞춤화 같은 서비스 층에서 다시 만들어질 수 있다고 본다. 즉 이 노드의 `software`는 코드 판매보다 **문제 해결 구조 전체**에 가깝다.

## 이 노드에서 특히 보존해야 할 긴장
- 편의성을 높일수록 통제력은 줄 수 있다.
- 추상화를 늘릴수록 코드가 깔끔해지지만 실체를 놓칠 수 있다.
- 라이브러리를 쓰면 빨라지지만 의존성과 불투명성도 커진다.
- 자동화와 문서화는 운영을 돕지만, 제대로 설계하지 않으면 오히려 관리 포인트가 늘어난다.
- 개인 맞춤형 서비스는 유망해 보이지만 개인은 비용 지불에 인색할 수 있다.
- 가벼운 도구를 선호하지만, 규모가 커질수록 체계적 운영 장치도 필요하다.

이 긴장들을 지워버리면 원문은 평범한 소프트웨어 개론처럼 납작해진다. 이 노드의 핵심은 해답보다 **균형 감각**에 있다.

## 원문에서 읽히는 저자의 선호

### 1. 무거운 만능 도구보다 가벼운 기본 구조를 선호한다
기본 기능이 많은 도구보다, 작고 단순하지만 필요할 때 기능을 붙일 수 있는 구조를 더 좋아한다.

### 2. 멋진 추상화보다 실패 지점을 먼저 생각한다
앱 안정성, 멱등성, 테스트, 에러 핸들링, 껐다 켜도 버티는 상태가 반복되는 점이 이를 보여준다.

### 3. 코드를 쓰는 일과 서비스를 운영하는 일을 분리하지 않는다
API 명세, 캐시 정책, 로그, 대용량 조회, 버저닝, 유료화 메모가 한 노트 안에 공존한다. 즉 개발은 배포 이후를 포함한 일이다.

### 4. 소프트웨어를 결국 사람의 삶을 바꾸는 인프라로 본다
메타도구, 개인 맞춤 서비스, 사용자 흐름, 200ms 체감 반응 시간 같은 메모는 저자가 기술보다 사용 경험과 생활 변화 쪽을 함께 보고 있음을 보여준다.

## 재사용 가능한 판단 프레임
이 노드를 실무용 질문으로 압축하면 다음과 같다.

- 이 소프트웨어는 사람의 기존 흐름을 돕는가, 아니면 불필요하게 깨는가?
- 나중에 바뀔 때 수정 범위를 국소화할 수 있는가?
- 앱 재시작, 네트워크 오류, 중복 요청, 데이터 증가를 견디는가?
- 의존성과 복잡성은 얻는 이득에 비해 과하지 않은가?
- 운영자가 상태를 이해하고 문서화하고 복구할 수 있는가?
- 사용자에게 필요한 반응 속도와 탐색 구조를 제공하는가?
- 기능 제공을 넘어 가치 교환 구조까지 설계되어 있는가?

## 인접 노드와의 연결
- [[./Coding]]: 구현 수준에서 변경 용이성과 에러 처리를 다루는 하위 노드.
- [[./Architecture]]: 정책, 경계, 복잡도 통제 측면에서 이 노드의 상위 설계 감각을 확장한다.
- [[../Deep Knowledge/Product]]: 소프트웨어를 실제 사용자 가치와 실험 구조로 연결한다.
- [[../Deep Knowledge/Devops]]: 배포, 환경 분리, 운영 자동화, 서비스 안정성으로 이어진다.
- [[../Deep Knowledge/Data]]: 검색, 저장, 대용량 처리, 조회 비용 문제와 직접 닿아 있다.
- [[../Deep Knowledge/Web]]: 서비스 전달 매체로서의 웹 감각과 만난다.
- [[../Communication/Document]]: 문서화와 명세 공개를 통해 재진입 비용을 줄이는 방향과 연결된다.
- [[../Integration/Business]]: 가치화와 고객 관계, 운영 지원 모델 측면에서 이어진다.

## 이 원본을 카테고리 노드로 읽을 때의 주의점
이 문서는 너무 넓어서 잘못 읽으면 잡다한 개발 메모 모음처럼 보인다. 하지만 만다라트 노드로 읽을 때는 개별 기술 키워드보다 아래 패턴을 먼저 잡는 편이 정확하다.

- `코딩`보다 넓고 `아키텍처`보다 생활 세계에 가깝다.
- 기술 선택과 운영 선택, 제품 선택, 수익화 선택이 한 흐름 안에 있다.
- 추상화와 실용주의가 충돌할 때, 대체로 실용적 안정성과 복구 가능성 쪽으로 기운다.
- 미완성 질문과 탐색 흔적을 일부 남겨두는 것이 이 노드의 성격에 맞다.
