# Web

- 원본: `source/shdkej-content/Deep Knowledge/Web.md`
- 상태: 기준선 재구성
- 역할: 원본 만다라트 노드와 1:1 대응하는 웹 카테고리 페이지

## 한눈에 보는 요약
이 노드는 웹 기술 스택 요약보다, 웹이 어디까지 할 수 있는지, 브라우저와 앱의 경계가 어떻게 바뀌는지, 좋은 웹 경험은 무엇으로 성립하는지, 그리고 프론트엔드 기술 선택이 실제 사용자 경험과 어떻게 연결되는지를 함께 묻는 탐색 노드에 가깝다. 역사, 런타임 구조, UI 편의성, 성능, 접근성, 앱 유도, 시니어 사용성까지 한 파일에 섞여 있는 이유는 저자가 웹을 단순 화면 기술이 아니라 생활 인터페이스로 보고 있기 때문이다.

## 핵심 논지
원본에서 웹은 단순히 "페이지를 그리는 기술"이 아니다. 문서 연결에서 시작해 애플리케이션 성격을 흡수해 왔고, 그 과정에서 플러그인, 자바스크립트, PWA, SSR/CSR, 번들러, 브라우저 엔진 같은 층위가 겹쳐졌다. 동시에 저자는 기술 유행을 따라가는 데서 멈추지 않고, 웹이 실제로 사용자에게 어떤 경험을 제공해야 하는지, 어디서 앱보다 불편해지는지, 어떤 기능은 왜 아직 웹에서 어렵게 느껴지는지까지 계속 확인한다.

## 원본에서 먼저 잡히는 구조

### 1. 웹의 진화사를 기능 확장 관점으로 본다
도입부는 문서 이동, 하이퍼링크, 플러그인, 자바스크립트, 웹어셈블리로 이어지는 흐름을 짧게 훑는다. 핵심은 기술 이름 나열이 아니라, 웹이 원래 못 하던 일을 계속 흡수하며 확장되어 왔다는 감각이다.

- 문서 연결에서 시작했다.
- 플래시와 액티브엑스가 웹 바깥 기능을 메웠다.
- 무거움과 보안 문제가 생겼다.
- 자바스크립트가 이를 대체하며 웹 내부 역량이 커졌다.
- 이제는 웹어셈블리처럼 더 낮은 수준 실행 모델까지 흡수하려 한다.

즉 이 노드에서 웹은 고정된 플랫폼이 아니라, 경계가 계속 이동하는 플랫폼이다.

### 2. 기술 메모가 곧 사용자 경험 메모로 이어진다
번들러, 브라우저 엔진, 이벤트 루프, SSR/CSR 같은 기술 메모 뒤에 바로 다크모드, 접근성, 새 탭, 세션, 뒤로가기, 스크롤 위치, 모바일 뷰 같은 경험 메모가 붙는다. 이는 저자가 기술을 내부 구조와 외부 체감 사이의 연결 문제로 이해한다는 뜻이다.

### 3. 완결된 답보다 미해결 질문이 많이 남아 있다
원본에는 확정형 설명보다 질문형 문장이 자주 나온다.

- 웹어셈블리는 대세가 될까?
- CSR은 검색엔진이 어떻게 읽을까?
- 커뮤니티 사이트에 CSR이 정말 빠를까?
- 비동기가 효율적인 이유를 어떻게 이해해야 할까?
- 앱 설치를 유도하는 이유는 무엇일까?
- 노년층에게 웹 접근을 어떻게 더 쉽게 만들 수 있을까?

이 노드는 지식 정리문이라기보다 웹을 배우고 설계하는 판단 지도를 겸한다.

## 이 노드의 큰 축

### 1. 웹의 가능 범위는 계속 넓어지지만, 사용자 기대도 함께 높아진다
원본의 첫 질문은 사실상 이것이다. 웹은 어디까지 앱을 닮아갈 수 있는가?

- 백그라운드 처리
- 센서 활용
- 푸시 알림
- 실시간 데이터 반영
- 앱 설치 유도와 딥링크 연결

여기서 PWA가 한 번 언급되는 것은 중요하다. 저자는 "웹으로는 안 된다"라고 단정하기보다, 웹이 조금씩 그 경계를 밀고 있다는 사실을 잡아두고 있다. 다만 동시에 웹이 앱처럼 보이려 할수록, 브라우저 제약과 사용자 기대치 차이를 더 정교하게 다뤄야 한다는 긴장도 생긴다.

### 2. 좋은 웹 경험은 기능 추가보다 흐름 보존에서 나온다
원본에서 반복되는 UX 메모를 모으면, 저자가 웹에서 특히 싫어하는 실패 유형이 보인다.

- 새로고침 시 중복 동작이 다시 실행되는 것
- 뒤로가기가 기대대로 작동하지 않는 것
- 팝업과 페이지 이동이 구분되지 않는 것
- SPA에서 스크롤 위치가 복원되지 않는 것
- 모바일과 데스크탑 사이에서 레이아웃 철학이 흔들리는 것
- 반쪽 브라우저 폭 같은 현실적 뷰포트를 무시하는 것

즉 웹 경험의 핵심은 화려한 인터랙션보다, 사용자가 이어서 행동할 수 있게 상태와 맥락을 보존하는 것이다.

### 3. 접근성과 편의성은 부가 기능이 아니라 기본 설계 항목이다
원본의 짧은 체크리스트는 사실 강한 기준을 드러낸다.

- dark mode
- accessibility
- i18n
- 새 탭 이동
- 세션 관리
- 뒤로가기와 undo 감각
- `컨텐츠로 건너뛰기`

이 목록은 웹 기능 체크리스트라기보다 "모든 사람이 무리 없이 쓰는 인터페이스"에 대한 최소 조건처럼 읽힌다. 이 감각은 [[Integration/Design]] 노드의 접근성 메모와도 직접 이어진다.

## 세부 주제별 해석

### 웹의 역사 메모가 의미하는 것
플래시, 액티브엑스, 자바스크립트, 웹어셈블리를 한 줄로 잇는 방식은, 웹의 역사를 표준 경쟁사로 보기보다 "웹이 외부 의존 기능을 내부 표준으로 흡수하는 과정"으로 읽고 있다는 뜻이다.

이 관점은 중요하다.
- 웹은 원래 약해서 플러그인에 의존했다.
- 이후 표준 브라우저 기능과 자바스크립트 생태계가 성장했다.
- 앞으로도 지금 외부처럼 보이는 기능이 나중엔 기본 역량이 될 수 있다.

즉 저자는 웹의 한계를 고정적으로 보지 않는다.

### 번들러 메모가 의미하는 것
번들러 파트는 webpack, vite 같은 도구 비교보다 "현대 웹은 이미 많은 빌드 최적화를 전제로 작동한다"는 감각을 보여준다.

- minify
- tree shaking
- chunk 분리와 lazy loading
- 캐시 무효화용 해시
- asset 처리
- CSS 후처리

이 메모는 프론트엔드 작업이 이제 단순 HTML 작성이 아니라, 빌드 도구가 성능과 배포 전략 일부를 담당하는 구조라는 점을 잡아둔다.

### 브라우저와 Node.js 메모가 의미하는 것
브라우저 엔진, 렌더링 엔진, Web API, 태스크 큐, 싱글 스레드, Node.js 비동기 모델에 대한 메모는 완전히 정돈되진 않았지만 관심 방향이 분명하다. 저자는 언어 문법보다 런타임이 실제로 어떻게 움직이는지 알고 싶어 한다.

특히 중요한 질문은 이것이다.
- 자바스크립트는 왜 싱글 스레드인데도 비동기 작업을 잘 처리할 수 있는가?
- 브라우저와 Node.js는 무엇을 공통으로 갖고 무엇이 다른가?
- 성능 문제를 이해하려면 프레임워크보다 런타임 구조를 먼저 봐야 하지 않는가?

즉 이 노드에서 웹 공부는 프레임워크 API 습득이 아니라 실행 구조 이해로 향한다.

### SSR, CSR, SSG 메모가 의미하는 것
이 부분은 기술 선택 기준이 아주 노골적으로 드러나는 구간이다.

- 첫 진입 속도는 SSR 쪽이 유리할 수 있다.
- 페이지 간 이동 체감은 CSR이 유리할 수 있다.
- SEO는 서버에서 먼저 읽을 수 있는 HTML과 연결된다.
- 커뮤니티처럼 데이터가 계속 바뀌는 서비스는 CSR이 무조건 좋다고 보기 어렵다.
- Next.js는 처음 SSR, 이후 CSR의 절충으로 이해된다.

즉 저자는 SSR/CSR을 진영 문제로 보지 않는다. 핵심은 서비스 성격, 검색 유입, 첫 진입과 재방문, 데이터 갱신 빈도, 사용자 흐름에 맞춰 판단하는 것이다.

### semantic HTML 메모가 의미하는 것
`article`, `section`, `aside`, `nav`, `header`, `footer` 메모는 마크업 문법 암기보다 읽기 모드, 참조 가능성, 스크린리더 집중 경로와 연결되어 있다. 즉 시맨틱 태그는 코드 취향이 아니라 의미 전달 장치다. 이 역시 접근성, 구조화, 읽기 경험을 함께 보려는 태도와 맞닿아 있다.

### 앱 유도 메모가 의미하는 것
원본 후반부에서 앱 설치 유도, 행동 기반 알림, 젊은 사용자 소비 패턴, 웹에서 앱으로 연결하는 서비스가 언급된다. 이는 웹을 독립 완결 플랫폼으로만 보지 않고, 앱과 함께 쓰는 획득 채널 또는 전환 채널로 읽고 있다는 뜻이다.

즉 웹의 목표가 항상 웹 안에서 모든 것을 끝내는 것은 아니다.
- 웹은 발견과 탐색에 강하다.
- 앱은 재방문, 알림, 소비 습관 형성에 강할 수 있다.
- 따라서 둘의 경계 설계가 제품 전략 문제가 된다.

### 노년층 접근 메모가 의미하는 것
노년층이 회원가입, 비밀번호, 사이트 간 이동, 인증 절차를 어려워할 수 있다는 메모는 단순 UX 팁이 아니다. 저자는 디지털 격차를 기능 부족이 아니라 절차 마찰의 문제로 본다.

핵심은 이런 질문이다.
- 기능이 많아도 진입 절차가 복잡하면 아무 소용이 없다.
- 음성인식 같은 기능 추가가 실제 해결책이 아닐 수도 있다.
- 사용자가 원하는 목적지까지 가는 경로를 얼마나 줄이느냐가 더 중요할 수 있다.

이는 웹을 기술 시스템이 아니라 생활 사용성의 문제로 본다는 증거다.

## 이 노드에서 특히 보존해야 할 긴장
- 웹은 앱처럼 더 많은 일을 할 수 있게 되지만, 브라우저 제약은 완전히 사라지지 않는다는 긴장
- CSR은 부드러운 이동을 줄 수 있지만, 첫 진입과 SEO에서는 불리할 수 있다는 긴장
- 풍부한 인터랙션은 좋아 보이지만, 뒤로가기와 스크롤 복원 같은 기본 흐름을 쉽게 깨뜨린다는 긴장
- 최신 도구 체인은 편의를 주지만, 웹이 점점 빌드 시스템 의존적으로 변한다는 긴장
- 접근성을 높이려면 더 많은 설계 판단이 필요하지만, 그 노력은 종종 기능 경쟁에 밀린다는 긴장
- 앱 유도는 비즈니스상 유리할 수 있지만, 웹 자체의 자율성과 즉시성은 약해질 수 있다는 긴장

## 원본에서 드러나는 저자의 선호

### 1. 웹을 생활 인터페이스로 본다
기술 스택보다 실제 사용 흐름, 알림, 뒤로가기, 노년층 접근, 앱 유도, 편의 UI를 함께 고민한다.

### 2. 프레임워크 선택보다 사용자 체감 차이를 더 중시한다
SSR/CSR 논의도 결국 SEO, 첫 진입, 이동 속도, 커뮤니티 특성 같은 체감 문제로 환원된다.

### 3. 접근성과 상태 보존에 민감하다
다크모드, 스킵 링크, 세션, 새 탭, history, 스크롤 위치 같은 메모는 사용자가 길을 잃지 않는 경험을 중요하게 본다는 신호다.

### 4. 아직 덜 정리된 질문을 그대로 보존하는 편이 맞다
이 노트는 완결 교과서보다 학습 중인 판단 노트의 성격이 강하다. 따라서 모호한 질문을 섣불리 정답화하기보다, 이후 [[Integration/Design]], [[Deep Knowledge/Product]], [[Fundamental/Network]]와 연결하면서 점진적으로 다듬는 편이 원문 성격에 더 가깝다.
