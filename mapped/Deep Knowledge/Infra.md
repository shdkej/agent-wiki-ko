# Infra

- 원본: `source/shdkej-content/Deep Knowledge/Infra.md`
- 상태: 기준선 재구성
- 역할: 원본 만다라트 노드와 1:1 대응하는 인프라 카테고리 노드

## 핵심 요약
이 노드는 인프라를 단순한 AWS 서비스 목록으로 보지 않는다. 원문은 인프라를 **서비스 규모에 따라 필요한 구성요소를 조절하고, 클라우드의 편의와 통제력 손실을 같이 감수하며, 서버리스와 서버 기반 방식을 상황에 맞게 섞고, 네트워크·권한·비용·복구 가능성을 함께 판단하는 운영 설계 문제**로 다룬다. 그래서 `정적 페이지 + CDN + S3 + Lambda` 같은 최소 조합에서 시작해, 규모가 커질수록 데이터베이스, 캐시, 오토스케일링, 메시지 큐, VPC, IAM, 관측, 테스트 전략이 붙는다.

## 이 노드가 실제로 답하려는 질문
- 지금 서비스 단계에 필요한 최소 인프라는 어디까지인가?
- 관리 포인트를 줄이기 위해 클라우드와 서버리스를 얼마나 받아들일 것인가?
- 어떤 작업은 Lambda에 맞고, 어떤 작업은 결국 서버나 컨테이너가 더 낫나?
- VPC, NAT, Endpoint, IAM 같은 기반 제약은 비용과 보안에 어떤 영향을 주나?
- 인프라 지식은 서비스 선택 암기보다 어떤 운영 판단으로 축적되어야 하나?

## 핵심 논지
1. **인프라는 처음부터 완전체로 설계하는 것이 아니라, 단계별로 필요한 층을 붙여 가는 구조다.**
   - 원문은 최소 인프라, 안정화를 위한 추가 인프라, 더 탄탄한 인프라를 나눠 적는다.
   - 이는 인프라를 고정 정답보다 성장 단계에 맞춘 증설 문제로 본다는 뜻이다.
2. **클라우드는 관리 포인트를 줄여 주지만, 내부 통제력과 하드웨어 감각 일부를 포기하게 만든다.**
   - 원문은 클라우드의 편리함을 인정하면서도 캐시, CPU, 디스크 최적화 같은 저수준 통제는 줄어든다고 적는다.
   - 즉 편의성은 곧 추상화 수용이며, 그만큼 다른 종류의 판단이 중요해진다.
3. **서버리스는 기본 선호이지만, 지속 연결·낮은 지연·상태 유지가 중요해지면 한계가 선명해진다.**
   - Lambda는 관리 부담이 적고 모듈화에 유리하지만, 콜드스타트와 상태 관리, 비용 급증 문제가 반복된다.
   - 원문은 결국 채팅 같은 실시간 시스템에서는 Fargate, Redis, 별도 데이터 계층이 더 현실적일 수 있다고 본다.
4. **네트워크와 권한은 인프라의 부속 이슈가 아니라 비용, 보안, 접속 가능성 전체를 좌우하는 중심 층이다.**
   - NAT Gateway 비용, VPC Endpoint, Route53, CloudFront, subnet, route table, IAM 권한 지연 등이 길게 남아 있다.
   - 이는 인프라를 컴퓨팅 선택보다 `경계와 연결의 설계`로 이해하고 있음을 보여 준다.
5. **좋은 인프라 메모는 서비스 이름 나열보다 증상, 제약, 수정, 재검증의 운영 기억을 남겨야 한다.**
   - 원문 후반의 EKS, Beanstalk, EC2 OS 패치, RDS 접근, DNS, ingress, CNI 메모는 거의 전부 실제 문제 해결 기록이다.
   - 이 노드의 가치는 이론 요약보다, 어떤 상태를 확인했고 무엇이 병목이었는지에 있다.

## 원문 구조를 다시 읽기

### 1. 규모별 인프라 층위
원문 첫머리는 인프라를 세 단계로 끊어 본다.
- 최소한의 인프라: 정적 페이지, CDN, S3, Lambda, API Gateway
- 규모가 커질 때 추가되는 것: 데이터베이스, 캐시, 오토스케일러
- 더 탄탄한 단계: 메시지 큐

이 구조는 중요하다. 저자는 인프라를 처음부터 거대한 플랫폼으로 상상하지 않고, **서비스 성격과 트래픽 변화에 따라 필요한 구성요소가 달라지는 계단형 구조**로 이해한다.

### 2. 클라우드와 서버리스에 대한 기본 태도
원문은 `모든 것을 위한 클라우드는 없다`고 적는다. 여기서 핵심은 벤더 비교 자체보다 선택 기준이다.
- 비용
- 사용자
- 시간
- 공간
- 회귀 가능한 요소

또한 가능한 한 정적 페이지를 만들고 CDN을 활용하자는 메모가 반복된다. 이는 **상태를 최대한 앞단에서 줄이고, 동적인 부분만 뒤로 밀어 운영 복잡도를 통제하려는 태도**에 가깝다.

### 3. 서버리스 선호와 그 한계 학습
원문은 서버리스를 먼저 고려하는 편이다.
- 관리 포인트가 줄어든다.
- 모듈화와 서비스 연결이 쉽다.
- 가볍고 실행 보장이 좋다.

하지만 동시에 분명한 한계도 적는다.
- 지속 연결이 필요하면 어렵다.
- 빠른 응답이 중요하면 어렵다.
- 콜드스타트가 있다.
- 사용자가 많아질수록 비용이 커질 수 있다.
- 웹소켓 같은 상태 관리가 어렵다.

그래서 이 노드에서 서버리스는 정답이 아니라 **먼저 시도해 볼 수 있는 기본값**이며, 실시간성·상태성·고부하가 커질수록 서버나 컨테이너로 되돌아오는 판단이 중요해진다.

### 4. AWS 기반 지식의 실제 무게중심
원문 중간 이후는 AWS 자격증 요약처럼 보일 수 있지만, 실제로는 서비스 소개보다 운영 포인트가 더 많이 남아 있다.
- EC2 추가 비용 항목: public static IP, bandwidth
- NAT Gateway 비용 폭탄 가능성
- S3, DynamoDB용 VPC Endpoint 활용
- VPC, subnet, route table, IGW, NACL, security group 관계
- IAM 정책과 principal의 구분
- Route53, CloudFront, 캐시 무효화, 지리 제한
- SQS 실패 큐, Lambda 연계

즉 이 노드는 AWS 서비스 백과사전이 아니라, **실제 운영에서 자주 마주치는 경계면과 숨은 비용의 메모 집합**으로 읽는 편이 맞다.

### 5. 핸즈온과 장애 메모의 비중
후반부에는 이론보다 실무 메모가 훨씬 많이 남아 있다.
- Beanstalk를 private VPC에 붙일 때 IGW, NAT, Endpoint를 어떻게 볼지
- EKS subnet, ingress controller, NLB/ALB 전환, external-dns, aws cni 설정
- EC2 OS 패치 중 Docker 재시작 위험과 LB 분리 필요
- RDS private subnet 접근 문제
- IAM 적용 지연
- keyv cronjob, Redis 세션 저장, lifecycle policy, 인증서 캐시 문제

이 부분은 인프라를 `제품 설명`으로 평탄화하면 놓치기 쉽다. 원문은 인프라를 배경지식보다 **실제 운영 중 부딪히는 마찰과 복구 루프의 축적**으로 다루고 있다.

## 반복되는 판단 축

### 최소 구성 vs 운영 안정성
- 시작은 단순할수록 좋다.
- 하지만 데이터, 캐시, 오토스케일링, 큐는 규모가 커질수록 빠르게 필요해진다.
- 즉 단순함은 영구 상태가 아니라 현재 단계에 맞춘 선택이다.

### 서버리스 vs 지속 실행 서버
- 서버리스는 관리가 쉽고 가볍다.
- 하지만 실시간성, 상태 유지, 비용 예측이 어려워지면 불리하다.
- 이 노드는 둘 중 하나를 신념처럼 택하지 않고, 워크로드 성격으로 가른다.

### 관리 편의 vs 세부 통제력
- 클라우드는 하드웨어 운영 고민을 줄여 준다.
- 반대로 내부 최적화, 자원 배치, 네트워크 경계의 감각을 덜 직접 통제하게 만든다.
- 원문은 이 손실을 불만이라기보다 선택의 대가로 이해한다.

### 기능 구현 vs 숨은 비용
- 인스턴스 요금만 보면 싸 보일 수 있다.
- 하지만 공인 IP, 트래픽, NAT, 로깅, 관리성 비용이 뒤늦게 붙는다.
- 저자는 기능보다 비용 누수 지점을 계속 기록해 둔다.

### 빠른 연결 vs 안전한 경계
- 외부 접근을 열면 테스트는 쉬워진다.
- 하지만 subnet, security group, 인증서, DNS, LB 구조가 엮이면 보안과 운영 위험이 커진다.
- 이 노드는 연결 가능성을 항상 경계 설계와 함께 본다.

## 세부 해석

### 1. 정적 우선 사고방식
`스테이트리스한 것은 정적으로`라는 문장은 이 노드 전체를 관통한다. 가능한 한 정적 파일, CDN, 캐시 계층에서 해결하고, 상태가 필요한 지점만 서버로 보내려는 태도다. 이는 인프라 복잡도를 줄이는 동시에 배포와 확장도 단순하게 만든다.

### 2. 채팅 예시가 드러내는 판단법
채팅 시스템 메모는 서버리스 한계를 가장 선명하게 보여 준다.
- 실시간성
- 웹소켓 상태 관리
- 사용자 증가에 따른 비용 급증
- 콜드스타트

그래서 Fargate, Redis pub/sub, DynamoDB, RDS 조합을 검토한다. 중요한 것은 특정 조합이 아니라, **워크로드 특성 때문에 아키텍처가 바뀌는 판단 과정**이 노출된다는 점이다.

### 3. 네트워크는 거의 모든 운영 기억의 교차점이다
VPC, NAT, Endpoint, Route53, CloudFront, ENI, subnet, NLB/ALB 전환, ingress, external-dns 메모는 서로 흩어져 있지만 결국 같은 질문으로 모인다. `어디를 공개하고 어디를 숨기며, 어떤 경로로 연결하고, 그 대가로 무엇을 내는가?` 이 노드에서 네트워크는 별도 하위 과목이 아니라 인프라 전체의 연결 문법이다.

### 4. IAM 메모는 추상 권한론보다 실제 운영 지연과 구조를 중시한다
IAM 설명은 교과서적 정의도 있지만, 핵심은 최소 권한 원칙, 그룹 기반 관리, resource-based policy와 identity-based policy의 차이, 그리고 `적용에는 시간이 걸린다` 같은 실무 감각이다. 즉 권한은 문서상 구조보다 **실제 배포와 운영에서 바로 체감되는 제약 조건**으로 기록되어 있다.

### 5. 테스트 전략도 인프라의 일부로 본다
후반의 stress test, 벤치마크 링크, locust, redis I/O test, curl loop 메모는 인프라를 구축에서 끝나는 것이 아니라 **어느 정도까지 버티는지 확인하는 실험 시스템**까지 포함해서 본다는 신호다. 이는 [[Monitoring]]과도 직접 이어진다.

## 숨은 패턴
- 이 노드는 AWS 서비스 공부 노트처럼 보이지만 실제로는 **운영 판단 로그**에 더 가깝다.
- 저자는 인프라를 화려한 아키텍처보다, 비용 누수와 연결 경계, 권한 구조, 복구 가능성을 다루는 현실적인 조정 작업으로 본다.
- 서버리스 선호가 분명하지만, 현실 제약이 드러나면 기꺼이 서버·컨테이너·캐시·큐로 이동한다.
- 실무 메모가 많은 이유는 이 노드의 가치가 완결된 정답보다 `다음 비슷한 문제를 더 빨리 풀 수 있는 운영 기억`에 있기 때문이다.

- [[Devops]]: 배포, 롤백, CI/CD, 운영 복원력은 인프라 선택과 바로 이어진다.
- [[Monitoring]]: 인프라 계층의 정상 상태와 이상 징후를 어떻게 드러낼지 연결된다.
- [[Container]]: 서버리스 한계를 넘는 순간 컨테이너, ECS, EKS, 이미지 운영 문제가 같이 붙는다.
- [[../../mapped/Fundamental/Network]]: VPC, Route53, LB, 보안그룹, 서브넷 이해의 기초 층을 받쳐 준다.
- [[../../mapped/Fundamental/Architecture]]: 단계적 확장, 저결합, 복구 가능성은 결국 아키텍처 판단과 겹친다.
- [[../../mapped/Meta/Troubleshooting]]: 실제 핸즈온 메모 대부분은 증상 확인, 수정, 재검증 루프로 읽히며 트러블슈팅 노드와 강하게 맞닿는다.

## 이 노드를 읽을 때 놓치지 말아야 할 점
- 서비스 이름 암기보다 어떤 조건에서 어떤 선택이 깨지는지 읽어야 한다.
- AWS 자격증 범위 메모도 시험 대비보다 운영 지형을 넓게 잡아 두려는 흔적으로 보는 편이 맞다.
- 미완 질문과 TODO가 많다는 점은 약점이 아니라, 실제 인프라 학습이 계속 열린 상태였다는 증거다.
- 따라서 이 페이지는 깔끔한 인프라 백과사전보다, 실무에서 반복 재사용할 판단 축과 운영 기억을 보존하는 노드로 유지하는 것이 적절하다.

## 이후 심화 포인트
- 서버리스 판단 기준을 `적합한 작업`, `한계가 선명한 작업`, `대체 조합`으로 더 세분화할 수 있다.
- AWS 실무 메모를 `네트워크`, `권한`, `배포/런타임`, `데이터`, `비용` 축으로 재분류하면 탐색성이 더 좋아진다.
- 후반의 EKS·LB·CNI 메모는 별도 하위 운영 패턴으로 압축해도 가치가 있다.
- 당근 인프라/채팅 사례처럼 외부 레퍼런스가 이미 붙어 있는 부분은 추후 `운영 레퍼런스에서 읽은 기준` 섹션으로 정리할 수 있다.

## 이번 갱신 메모
- 1차 대응 초안을 한국어 기준의 구조화된 기준선 페이지로 전면 재구성했다.
- 원문 전체를 다시 읽고 `규모별 인프라`, `클라우드와 서버리스`, `AWS 운영 포인트`, `핸즈온/장애 메모`의 흐름으로 재배열했다.
- 인프라를 서비스 목록이 아니라 비용, 경계, 복구 가능성, 워크로드 적합성을 함께 다루는 카테고리 노드로 읽히게 보강했다.
