# Data

- 원본: `source/shdkej-content/Deep Knowledge/Data.md`
- 상태: 심화 정리
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 한눈에 보는 요약
이 노드에서 `Data`는 단순히 데이터베이스 종류를 외우는 주제가 아니다. 저자는 데이터를 저장, 검색, 복제, 분석, 기억, 탐색, 판단의 문제로 함께 다룬다. 즉 `Data`는 저장소 설계 노드이면서, 대용량 처리 노드이고, 검색 경험 노드이며, 인간이 정보를 놓치지 않으려는 심리까지 걸쳐 있는 운영 감각의 축이다.

## 핵심 논지
- 데이터는 쌓이는 속도가 처리 속도보다 빠르기 때문에, 저장보다 조회와 후처리의 비용이 더 빨리 문제를 만든다.
- 데이터 기술 선택은 RDB, NoSQL, Redis, Elasticsearch, ClickHouse 같은 제품 비교가 아니라, 일관성, 검색성, 메모리, 증분 갱신, 분석 용도 분리의 판단 문제다.
- 좋은 데이터 구조는 중첩을 늘리는 구조보다 연결을 드러내고 다시 찾기 쉬운 구조에 가깝다.
- 정보 축적은 기술 시스템의 문제이면서 동시에 인간 기억의 불안, 탐색 욕구, 기록 의존의 문제이기도 하다.

## 데이터 노드의 재해석

### 데이터는 "어디에 저장하느냐"보다 "어떻게 다시 꺼내고 감당하느냐"의 문제다
원문은 초반에 데이터베이스 기본 용어를 적어두지만, 실제 무게중심은 조회 비용과 운영 비용으로 계속 이동한다. 대량 select, 메모리 점유, 검색 인덱스 재구성, 분석용 저장소 분리 같은 메모가 반복되는 것을 보면, 저자는 데이터를 정적 자산보다 계속 읽히고 이동하고 변환되는 흐름으로 보고 있다.

### 데이터 기술의 핵심 차이는 저장 모델보다 실패 방식에 있다
- RDB는 관계와 일관성에 강하지만, 대량 조회와 확장 시 병목이 도드라진다.
- NoSQL은 문서형, 분산형 맥락에서 유용하지만, CAP과 모델링 선택을 함께 감당해야 한다.
- Redis는 빠르지만 메모리 기반 구조의 한계와 자료구조별 비용을 계속 의식해야 한다.
- Elasticsearch는 검색 경험을 크게 개선하지만, 스키마 수정과 재색인 비용이 크다.
- ClickHouse나 Athena는 분석에 유리하지만, 트랜잭션 저장소를 대체하기보다 읽기 계층을 분리하는 쪽에 가깝다.

즉 이 노드의 중심 질문은 "어떤 DB가 최고인가"가 아니라 "어떤 종류의 실패와 복잡성을 내가 감당할 것인가"에 더 가깝다.

## 세부 해석

### 1. Database, 설계, 쿼리

#### ACID와 CAP를 함께 적어둔 이유
원문은 관계형 데이터베이스의 ACID와 NoSQL의 CAP를 나란히 적는다. 이는 특정 이론을 공부했다는 표시라기보다, 저장소 선택 시 무엇을 우선 보장할지 계속 고민하고 있음을 보여준다. 일관성, 가용성, 분산 허용성은 기능 목록이 아니라 시스템 성격을 정하는 축이다.

#### 인덱스와 데이터 양은 초기 설계의 핵심 감각이다
`데이터 양`, `인덱스`를 따로 적어둔 점은 중요하다. 저자는 스키마를 예쁘게 만드는 것보다, 실제 양이 커졌을 때 검색과 정렬이 무너지지 않는지 먼저 본다. 이는 `[[Monitoring]]`의 병목 감각과도 연결된다.

#### 느린 쿼리와 N+1은 "나중에 보이는 문제"가 아니다
비싼 쿼리를 어떻게 찾을지, N+1 문제를 어떻게 피할지는 구현 후 최적화 이슈가 아니라 처음부터 관측 가능한 구조를 만들라는 신호다. 데이터 설계와 애플리케이션 설계가 분리되지 않는다는 뜻이기도 하다.

### 2. 대용량 처리와 메모리

#### 한 번에 많이 읽지 말라는 원칙
원문은 대량 검색에서 paging, iterator, stream을 반복해서 언급한다. 이는 단순 팁이 아니라 데이터 노드의 운영 원칙이다.
- 데이터를 많이 저장하는 것과 한 번에 많이 꺼내는 것은 다르다.
- 시스템이 버티는 구조와 개발자가 다루기 쉬운 구조는 다를 수 있다.
- 조회 단위와 처리 단위를 분리하지 않으면 메모리 문제가 곧 장애로 이어진다.

#### 메모리 문제는 코드가 아니라 데이터 접근 방식에서 생긴다
2차 캐시가 old 영역 메모리를 계속 잡고 GC가 회수하지 못한 경험은 중요한 신호다. 저자는 성능 최적화를 위해 캐시를 켰다가, 오히려 운영 안정성을 해칠 수 있다는 점을 배운 것으로 보인다. 이 감각은 `[[Container]]`, `[[Infra]]`처럼 재현성과 운영 복잡성을 중시하는 노드와 맞닿아 있다.

### 3. Redis, 빠른 구조의 이면

#### Redis는 "빠른 저장소"가 아니라 자료구조 선택의 훈련장이다
set, hash, string, bitmap, scan, rehash 메모를 함께 남긴 점을 보면, 저자는 Redis를 캐시 제품 하나로 보지 않는다. 어떤 자료구조가 어떤 비용을 만드는지 체감하는 도구로 보고 있다.

#### 운영 기준
- `keys`, `flushall`보다 `scan`을 선호한다.
- `maxmemory`를 확인한다.
- swap이 발생하면 속도가 급격히 떨어질 수 있다.
- 작은 저장 단위와 자료구조별 표현 방식을 고민한다.

이 메모들은 모두 "빠른 도구일수록 더 조심해서 써야 한다"는 감각으로 묶인다.

### 4. Elasticsearch, 검색은 별도 시스템이다

#### 검색은 단순 조회의 확장이 아니다
원문에서 Elasticsearch 메모는 꽤 구체적이다. 인덱스, 문서, 매핑, tokenizer, analyzer, synonym, user dictionary, reindexing, shard, routing 같은 키워드가 이어진다. 이는 검색을 DB 부가기능이 아니라 별도 시스템으로 이해하고 있다는 뜻이다.

#### 검색 시스템에서 중요한 판단
- 인덱스 수정은 어렵고 재색인이 비싸다.
- 사용자 사전과 동의어 사전은 검색 품질과 운영 비용을 동시에 바꾼다.
- 기본값을 모르면 전체 검색조차 기대와 다르게 동작한다.
- 검색 품질은 데이터 저장보다 분석기 설정, 쿼리 구성, 용어 정규화에 더 크게 좌우된다.

#### 검색 경험에 대한 관심
후반부의 permutations, 유의어 검색, `-키워드`, `+키워드` 같은 메모를 보면, 저자는 검색 엔진 내부보다 사용자가 "어떻게 찾을 수 있게 할 것인가"를 계속 의식한다. 데이터 구조와 UX가 붙어 있는 노드다.

### 5. ClickHouse, Athena, CDC

#### 트랜잭션 저장소와 분석 저장소를 분리하려는 감각
Postgres에서 ClickHouse로 복제하는 흐름, 실시간 복제와 배치 복제를 구분하는 메모는 중요하다. 이는 데이터를 한 곳에 다 욱여넣기보다, 쓰는 목적에 따라 읽기 경로를 분리하려는 발상이다.

#### 여기서 드러나는 판단 기준
- 운영계 DB는 정확한 기록과 서비스 처리를 담당한다.
- 분석계 저장소는 대량 집계와 탐색을 담당한다.
- CDC, Kafka, Airflow 같은 도구는 데이터를 이동시키는 연결 계층이다.
- Athena는 배치성 질의에는 편하지만, 실시간성이 중요하면 다른 선택이 나을 수 있다.

이 축은 `[[Monitoring]]`의 지표 수집, `[[Infra]]`의 시스템 분리와도 자연스럽게 이어진다.

## 인간의 기억과 데이터 구조가 만나는 지점
이 노드의 후반부는 기술 메모에서 곧바로 기억, 기록, 정보 과잉, 웹서핑, 인박스, 맞춤형 서비스, 연상력으로 이동한다. 얼핏 산만해 보이지만 실제로는 같은 질문을 반복한다.

> 쌓이는 것을 어떻게 잃지 않고, 다시 찾고, 과부하 없이 다룰 것인가?

이 질문은 데이터 엔지니어링의 질문이면서 개인 지식 관리의 질문이기도 하다.

### 반복해서 보이는 긴장
- 많이 모을수록 미래에 도움 될 것 같지만, 처리 지연이 답답함을 만든다.
- 다 기록하면 안심되지만, 기록 의존이 기억 훈련을 약화시킬 수 있다.
- 더 많은 정보를 알고 싶지만, 모두 가지려는 욕심은 초조함을 만든다.
- 개인화 서비스는 편리하지만 감시와 편향의 문제를 동반한다.

즉 `Data` 노드는 기술 스택 노드이면서 동시에 정보 욕망을 조절하는 노드다.

## 이 노드에서 드러나는 저자의 선호
- 데이터를 예쁘게 모델링하는 것보다, 나중에 다시 찾고 가공하고 운영할 수 있는 구조를 더 중시한다.
- 대용량 상황에서의 메모리, 페이지네이션, 스트리밍, 캐시 부작용에 민감하다.
- 검색 품질을 위해 사전, 동의어, 쿼리 문법 같은 세부 조정을 아끼지 않는 편이다.
- 단일 시스템 만능주의보다 목적별 저장소 분리와 연결 구조를 선호한다.
- 정보 수집 욕구가 강하지만, 동시에 과잉 수집과 지연 처리의 스트레스도 크게 느끼는 편이다.
