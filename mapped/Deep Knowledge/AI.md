# AI

- 원본: `source/shdkej-content/Deep Knowledge/AI.md`
- 상태: 구조 재정리 완료, 이후 점진 심화 대상
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 논지
이 노드의 `AI`는 단순한 기술 요약이 아니다. 저자는 AI를 세 갈래로 함께 다루고 있다. 첫째, **AI · 머신러닝 · 딥러닝 · LLM의 개념 지도**. 둘째, **LLM 이후 일과 제품 설계가 어떻게 바뀌는가**. 셋째, **AI를 실제로 잘 쓰기 위한 문서화, 프롬프팅, 운영, 검증 습관**이다. 즉 이 페이지는 이론 노트이면서도, 동시에 개인용 AI 작업 시스템의 설계 메모다.

## 원본 구조 재정리
원본은 크게 다음 층위로 읽힌다.

1. AI, 머신러닝, 딥러닝, LLM의 기본 구분
2. LLM 이후 추천, 생성형 AI, 에이전트, 도구 사용의 변화
3. AI를 함께 일하는 존재로 다루는 사용법
4. 문서 관리, 메모리 관리, AI 친화적 개발 방식
5. 프롬프트 설계와 모델 선택 노하우
6. 컨텍스트 길이, 벡터 스토어, RAG에 대한 이해
7. 머신러닝, 트랜스포머, MLOps, 데이터 분석의 기초 메모
8. AI 서비스와 AI 제품을 만들 때의 운영 고려사항

구성은 넓고 메모적이지만, 중심 질문은 의외로 일관된다.

> AI를 단순히 "질문에 답하는 모델"이 아니라, 문서, 도구, 검증, 운영 체계 안에 어떻게 넣어야 실제로 쓸모 있게 되는가?

## 개념 축, AI > ML > DL > LLM
원본 첫머리의 `AI > ML > DL` 구도는 이 노트의 기본 좌표계다.

### AI
가장 넓은 범주다. 지능적 문제 해결 전반을 가리킨다.

### 머신러닝
규칙을 사람이 일일이 짜는 대신, 데이터와 결과를 통해 모델을 학습시키는 접근이다. 원본의 `Programming`과 `ML` 대비는 이를 직관적으로 보여준다.
- 전통적 프로그래밍: `Algorithm + Data => Result`
- 머신러닝: `Result + Data => Model`, 그리고 `Data => Predictions`

### 딥러닝
신경망을 여러 층으로 쌓아 표현 학습을 강화하는 방식이다. 지도학습, 비지도학습, 강화학습과 결합해 더 복잡한 패턴을 처리한다.

### LLM
언어를 다루는 대규모 모델 계열이다. 이 노트에서 LLM은 단지 한 모델 유형이 아니라, 이후의 RAG, Vector Store, MCP, Agent 같은 작업 구조를 촉발하는 중심 축으로 등장한다.

## LLM 이후의 변화, 반응형에서 의도 이해로
원본에서 가장 또렷한 변화 인식은 추천 시스템에 대한 문장에 있다.

- 과거: 패턴 매칭, 반응형 추천
- 이후: 의도 이해, 맥락 이해, 선제적 제안

이 변화 인식은 단순히 "LLM이 더 똑똑하다"는 말이 아니다. 저자는 AI의 가치를 **출력 생성 능력**보다 **사용자 의도와 상황을 더 잘 읽는 능력**에서 본다. 그래서 이 노트는 모델 성능 자체보다도, 맥락 전달과 작업 구조 설계에 더 많은 비중을 둔다.

## 생성형 AI의 진화 방향, 인간 개입 감소
원본은 생성형 AI의 진화를 다음처럼 적어 둔다.
- 어시스턴트: 규칙 기반
- 에이전트: 단일 목표, 광범위한 작업
- 에이전틱: 완전 자율, 다중 에이전트

여기서 중요한 점은 자율성이 높아질수록 능력만 커지는 것이 아니라, 설계 책임도 커진다는 감각이다. 같은 문단 주변에 레이턴시, 비용, 불필요한 호출, ReAct, 메모리, 툴 이야기가 붙어 있는 이유도 여기에 있다. 즉 이 노트에서 에이전트는 마법이 아니라, **행동과 관찰을 반복하는 비용 있는 시스템**이다.

## 에이전트는 행동 시스템이다
원본은 에이전트를 매우 짧게 정의한다.
- 액션을 하고 관찰을 한다
- 메모리와 툴이 필요하다

이 짧은 문장이 사실상 전체 노트의 운영 철학을 압축한다.

### 에이전트 설계에서 반복되는 판단 축
- 초기 레이턴시를 감당할 수 있는가
- 비용이 과도하게 불어나지 않는가
- 불필요한 호출을 줄였는가
- 생각과 행동을 분리할 것인가
- 메모리와 툴 사용을 어디까지 허용할 것인가

특히 `ReAct (Reasoning + Action)` 언급은, 저자가 에이전트를 단순 생성기가 아니라 단계적 작업자로 이해하고 있음을 보여준다.

## AI 사용법, 맡길 것과 체화할 것을 나누기
이 노트에서 가장 실용적인 부분 중 하나다. 저자는 AI 사용의 핵심을 "많이 시키는 법"이 아니라 **무엇을 직접 판단하고 무엇을 위임할지 나누는 법**으로 본다.

### 사람이 붙들어야 하는 것
- 디자인 레퍼런스처럼 직접 보고 느껴야 하는 감각
- 최종 go / no-go 판단
- 경험을 무엇으로 남길지의 기준
- 빠짐을 막는 책임

### AI에게 위임할 수 있는 것
- 초안 작성
- 정리와 구조화
- 역할별 시뮬레이션
- 자료 탐색과 빠른 비교
- 누락 점검

즉 저자에게 AI는 대체자가 아니라, **결정을 더 쉽게 하도록 구조를 만들어 주는 인턴 혹은 작업자**에 가깝다.

## 문서가 곧 AI의 기억이다
원본 전반에 반복되는 핵심 패턴은 "AI를 잘 쓰려면 결국 문서화해야 한다"는 믿음이다.

### 문서 관리에 대한 명시적 기준
- 100줄: 파일 열기 전 확인 가능한 양
- 300줄: 최적, 100줄 이상이면 목차 필요
- 500줄: 최대
- 500개 파일: 인덱스 파일로 관리
- 1,000개 이상: RAG 같은 의미 기반 탐색 고려

이 기준은 단순 파일 관리 팁이 아니다. 저자는 문서 길이와 파일 수를 **AI가 다룰 수 있는 컨텍스트 구조**와 연결해서 본다.

### 메모리 관리 신호
- 스킬 문서는 description까지 읽는다
- `memory.md`
- `claude.md`

즉 기억은 모델 내부에만 기대는 것이 아니라, 외부 문서 구조로 설계해야 한다는 입장이다.

## AI 친화적 개발, 코드보다 기준점을 먼저 둔다
원본의 `ai 친화적 개발`과 `LLM이 쓰는 문서` 부분은 실제 개발 운영 철학을 보여준다.

### 기준 문서 우선
- PRD와 API 문서를 기준점으로 둔다
- 수정은 이 문서를 먼저 바꾸고 코드와 연계한다
- 테스트 코드도 PRD 변경과 연결되어야 한다
- 요구사항은 `REQ-ACTION-001` 같은 식별자를 가진다

### LLM 작업 환경의 구성 요소
- rules: 시스템 지침
- agents.md: 기본 프로젝트 안내서
- plan: 작업 기준 문서
- hooks: 응답 후 실행 규칙
- agents: 서브 에이전트 분리
- slash commands: 반복 지시의 커맨드화
- skills: 외부 기능 확장

여기서 AI는 코드 생성기라기보다, **문서로 유도되는 작업 엔진**으로 이해된다.

## 프롬프트 노하우, 잘 시키는 법은 잘 설명하는 법이다
원본의 프롬프트 메모는 흩어져 있지만, 몇 가지 원칙으로 묶을 수 있다.

### 1. 아는 것과 모르는 것을 구분해 설명한다
모델이 이미 아는 일반 지식은 간결하게, 프로젝트 특수 맥락은 구체적으로 전달한다.

### 2. 역할과 평가 기준을 같이 준다
- 3명의 전문가 토론
- 로드맵 제시
- 초등학생도 이해하게 설명
- 아래 프롬프트를 개선하기 위해 질문부터 하게 만들기

### 3. 적극성, 깊이, 길이를 조절한다
원본은 AI가 너무 알아서 하게 둘지, 이것만 하게 할지를 제어해야 한다고 적는다. 즉 좋은 프롬프트는 내용만이 아니라 **행동 성향의 제어**도 포함한다.

### 4. 사고를 외부화하게 한다
단계별 사고, 특정 전문가의 사고 연쇄, 검증 과정 출력을 요청하는 이유는 단지 설명을 듣기 위해서가 아니라, 작업의 빈틈을 줄이기 위해서다.

### 5. 결국 프롬프트는 자기 이해를 요구한다
`프롬프트를 쓰면서 오히려 더 배우게 된다`는 메모는 중요하다. 저자는 프롬프트 작성이 단순 명령 기술이 아니라, 자신의 직관과 판단 기준을 재언어화하는 과정이라고 본다.

## RAG, 벡터 스토어, 컨텍스트에 대한 현실적 감각
이 노트는 RAG를 과대평가하지 않는다.

### 컨텍스트 길이에 대한 감각
- 모델마다 컨텍스트 한계가 다르다
- 최대치까지 넣는다고 성능이 그대로 유지되지는 않는다
- 토큰이 늘면 비용도 늘어난다

### 벡터 스토어와 RAG에 대한 태도
- 유사한 정보를 찾는 데는 유용하다
- 청킹 방식이 중요하다
- 그러나 RAG는 문맥 전체를 진짜로 이해하게 만드는 장치는 아니다

이 부분은 저자가 최신 AI 시스템을 도입할 때도, 마케팅 문구보다 실제 한계와 비용 구조를 더 중시한다는 신호다.

## 머신러닝과 트랜스포머, 넓지만 실용 중심의 기초 메모
원본 하반부는 머신러닝, 트랜스포머, 데이터 분석, MLOps 메모가 넓게 펼쳐져 있다. 깊은 이론 정리라기보다, 나중에 다시 연결해 쓸 수 있는 기초 재료 모음에 가깝다.

### 머신러닝 기초
- 지도학습, 비지도학습, 강화학습
- 회귀와 분류
- 데이터 클리닝, 정규화, 데이터 분리, 교차검증
- 평가지표와 평가방법

### 트랜스포머 메모
- 다음 토큰 예측이 기본 동작
- attention이 문맥적 의미 조정을 돕는다
- 임베딩, 파라미터, 언임베딩, temperature 같은 개념이 붙는다
- 역전파와 가중치 조정이 핵심 학습 메커니즘이다

### MLOps 및 서빙
- continuous training / deploy
- 모델 빌드와 서빙 분리
- TensorFlow Serving
- `.h5`, `pb`, weight, optimizer 같은 저장 단위 문제

즉 이 노트의 머신러닝 파트는 연구 노트라기보다, **실제 구현과 운영에 닿아 있는 입문-실무 브리지 메모**에 가깝다.

## AI 제품을 만들 때 반복되는 걱정
원본 전반의 실무 메모를 묶으면, 저자가 AI 제품을 볼 때 반복적으로 확인하는 항목이 드러난다.

- 결과가 다음 작업에 누적되는가
- 실패 시 재시도 가능한가
- 할루시네이션을 어떻게 제어하는가
- 긴 작업을 비동기로 처리하는가
- 서비스가 죽었다 살아나도 이어서 처리 가능한가
- 사용자가 프롬프트를 볼 수 있는가
- 복잡한 레이아웃, 정밀 제어, 이미지 처리에서 어디까지 믿을 수 있는가
- 디버깅 가능성을 확보했는가

즉 이 노트는 AI를 "잘 생성하는가"보다, **서비스로 굴릴 수 있는가**의 관점으로 자주 돌아온다.

## 이 노드에서 드러나는 저자의 AI관
이 노트를 관통하는 저자의 기본 태도는 다음처럼 요약할 수 있다.

1. AI는 강력하지만, 문서와 기준 없이 쓰면 쉽게 흐트러진다.
2. 좋은 AI 활용은 모델 선택보다 작업 구조 설계에 더 크게 좌우된다.
3. AI는 사람의 판단을 대체하기보다, 판단 가능한 상태를 빨리 만들어야 한다.
4. RAG, 에이전트, 긴 컨텍스트 같은 확장 장치도 한계를 인정한 채 써야 한다.
5. 결국 중요한 것은 "잘 물어보는 법"이 아니라 "어떤 시스템 안에서 물을 것인가"다.
