# Cloud

- 원본: `source/shdkej-content/Deep Knowledge/Cloud.md`
- 상태: 점진 심화, 원본 공란 상태 재확인
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 논지
이 노드는 아직 원본 본문이 비어 있다. 그래서 현재의 `Cloud`는 클라우드 일반론을 적어 넣는 자리보다, 저자가 `Deep Knowledge` 안에서 따로 확보해 둔 **상위 카테고리 경계**를 보존하는 노드로 읽는 편이 정확하다. 이번 보강의 목적도 내용을 임의로 채우는 것이 아니라, 공란 자체가 말해 주는 범위와 주변 노드에 이미 흩어진 클라우드 흔적을 함께 묶어 두는 데 있다.

특히 이 노드는 `AWS 서비스 모음`이나 `클라우드 입문 치트시트`로 평탄화되기 쉽다. 하지만 현재 저장소 맥락에서는 그런 일반 지식보다, **추상화를 어디까지 받아들이는지**, **운영 통제를 어디서 내려놓는지**, **비용·속도·복구 가능성을 어떤 순서로 비교하는지** 같은 상위 판단 질문을 담는 그릇으로 보는 편이 더 잘 맞는다.

## 원본 상태
- 원본 파일 전체를 다시 확인한 결과 본문이 비어 있다.
- 따라서 직접 발췌할 문장, 소제목, 사례, 주장 목록은 현재 없다.
- 이번 정리는 빈 소스를 억지로 채우는 작업이 아니라, 추후 사용자가 직접 내용을 넣었을 때 어떤 변화가 들어왔는지 더 정확히 비교·흡수할 수 있도록 해석 기준을 다듬는 작업이다.

## 만다라트 안에서의 위치 의미
이 저장소에서는 각 markdown 파일을 독립 문서가 아니라 8x8 만다라트의 **카테고리 노드**로 읽어야 한다. 그런 기준에서 `Cloud`의 공란은 단순 누락이라기보다, 아직 세부 메모가 흩어져 있더라도 이 주제가 `Infra`, `Container`, `Devops`, `Web`과는 다른 이름으로 따로 보존되어야 한다는 신호에 가깝다.

즉 지금 단계의 핵심은 `Cloud`를 임의의 상식 문서로 풍성하게 만드는 것이 아니라, 앞으로 사용자가 어떤 내용을 넣더라도 그 내용을 받을 **주제 경계와 흡수 기준**을 흔들지 않는 데 있다. 이번 페이지의 역할은 지식을 대신 채우는 것이 아니라, 나중에 실제 원문이 들어왔을 때 그 내용을 어느 층위의 판단으로 읽어야 하는지 미리 고정해 두는 데 있다.

## 이 빈 노드가 시사하는 범위
현재 원문은 비어 있지만, 인접 노드에는 이미 클라우드와 직접 맞닿는 흔적이 흩어져 있다. 특히 `Infra` 원문에는 다음과 같은 문장 덩어리가 이미 보인다.
- `모든것을 위한 클라우드는 없다`
- `정적페이지로 최대한 만들어서 cdn 이용하기`
- `클라우드는 편리하지만 내부를 통제할 수 없다`
- `서버리스를 선호`와 그 한계에 대한 메모
- AWS 비용 항목, NAT gateway 비용, VPC Endpoint 같은 운영 메모

또한 `Devops`와 `Container` 원문을 함께 보면, 현재 저장소에서 클라우드는 단순한 호스팅 장소가 아니라 다음 층위와 접속한다.
- 개발자가 빠르고 안전하게 배포할 수 있게 하는 **운영 표면**
- 컨테이너·서버리스·관리형 서비스 중 무엇을 맡기고 무엇을 직접 쥘지 정하는 **책임 분배면**
- 비용·성능·복구 가능성의 균형을 잡는 **선택 아키텍처**

이 신호를 보면 `Cloud`는 단순히 AWS/GCP/Azure 기능을 모으는 페이지보다, 적어도 아래 층위를 다룰 가능성이 높다.
- **추상화 수용 범위**: 하드웨어 최적화 같은 내부 통제를 어디까지 포기하고 어떤 편의성을 얻을 것인가
- **구조 선택 기준**: 정적 페이지, CDN, Lambda, Fargate, DB, MQ 같은 요소를 어떤 조합 논리로 택할 것인가
- **비용 감각**: 인스턴스 요금표보다 숨은 네트워크·게이트웨이·운영 비용까지 포함해 읽는가
- **서버리스 경계선**: 관리 포인트 감소의 이점과 콜드스타트·실시간성·지속 연결 한계를 어디서 가르는가
- **책임 위임의 한계**: 관리형 서비스가 편리해질수록 무엇이 보이지 않게 되고, 그 대가를 어디서 다시 치르는가

다만 위 항목들은 현재 `Cloud` 원문에 적혀 있는 내용이 아니라, 주변 source 노드에서 읽히는 범위 가설이다. 이후 원본이 채워지면 반드시 그 직접 서술을 우선한다.

## 인접 source가 이미 던지고 있는 질문
현재 `Cloud` 원문은 비어 있어도, 관련 source를 다시 읽으면 이 노드가 장차 어떤 질문을 받아야 하는지는 어느 정도 보인다.

| 인접 source | 실제로 보이는 메모 | `Cloud` 쪽에서 받아야 할 상위 질문 |
| --- | --- | --- |
| `Infra` | `모든것을 위한 클라우드는 없다`, `클라우드는 편리하지만 내부를 통제할 수 없다` | 편의와 통제의 교환비를 어디서 받아들일 것인가 |
| `Infra` | 정적 페이지 + CDN, Lambda, Fargate, NAT gateway 비용, VPC Endpoint | 구조 선택이 비용·복구성·운영 난이도에 어떤 차이를 만드는가 |
| `Infra` | 서버리스를 선호하지만 콜드스타트·지속 연결·실시간성 한계를 의식 | 서버리스 선호가 어디서 멈추고 다른 운영 모델로 넘어가는가 |
| `Infra` | IAM shared responsibility, 계정 분리, CloudWatch/Chatbot, VPC Flow Logs | 관리형 서비스를 써도 무엇을 직접 보이고 통제해야 하는가 |
| `Devops` | 개발자가 빠르고 안전하게 배포하고 쉽게 롤백해야 한다는 목표 | 클라우드 선택이 배포 속도와 복구 가능성에 어떤 기본값을 제공하는가 |
| `Container` | EKS·Fargate·Lambda 비교, AWS CNI·IP 한계, ingress/NLB 메모 | 실행 추상화가 높아질수록 어떤 네트워크·운영 세부가 다시 돌아오는가 |

이 표의 핵심은 `Cloud`를 채우기 위해 외부 일반론을 더하는 것이 아니라, 이미 저장소 안에 흩어진 질문을 한 단계 위로 끌어올려 이 노드의 역할을 더 선명하게 만드는 데 있다.

## 지금 이 노드에서 보존해야 할 최소 구조
원문이 비어 있어도 다음 구조는 미리 고정해 두는 편이 유용하다.
1. `Cloud`는 서비스 이름 목록이 아니라 **선택 기준을 다루는 상위 노드**일 가능성이 높다.
2. 세부 구현 메모는 [[../../mapped/Deep Knowledge/Infra]], [[../../mapped/Deep Knowledge/Container]], [[../../mapped/Deep Knowledge/Devops]], [[../../mapped/Deep Knowledge/Web]]로 흩어질 수 있다.
3. 따라서 이 페이지는 지금 당장 상세 지식을 쌓기보다, 나중에 원문이 생겼을 때 그것이 **철학·비용·구조·운영 모델 중 어디를 건드리는지** 분류하는 관문 역할을 해야 한다.

### 현재 단계에서 먼저 잡아 둘 분류축
- **플랫폼 선택**: 특정 벤더 기능보다, 관리형 서비스를 어디까지 받아들일지의 선택.
- **책임 분배**: 인프라팀/클라우드/매니지드 서비스/애플리케이션이 각각 무엇을 책임지는지.
- **비용의 보이는 층과 숨은 층**: 컴퓨팅 요금뿐 아니라 네트워크, 게이트웨이, 운영 복잡도, 복구 비용까지 포함하는가.
- **가시성·권한 경계**: CloudWatch, Flow Logs, IAM, 계정 분리처럼 무엇을 누가 보고 열 수 있는지까지 운영 설계에 포함하는가.
- **회귀 가능성**: 한번 선택한 구조에서 다른 형태로 돌아오거나 옮겨 갈 수 있는가.

### 현재 단계에서 적지 않는 편이 나은 것
- 원문에 없는 벤더 기능 백과사전식 요약.
- `Cloud`를 `Infra`의 AWS 세부 메모 복사본으로 만드는 확장.
- 최신 업계 정답처럼 보이는 일반론을 덧씌워 원래 노드의 빈 경계를 지워 버리는 정리.

## 인접 노드와의 관계
### [[../../mapped/Deep Knowledge/Infra]]
가장 가까운 이웃 노드다. 실제로 현재 클라우드 관련 메모 대부분은 `Infra` 쪽에 흩어져 있다. 그러나 그 메모들은 VPC, NAT gateway, 서버리스, AWS 비용처럼 실행 레벨이 강하다. `Cloud`는 그보다 한 단계 위에서, 왜 클라우드를 택하는지와 어떤 추상화를 받아들일지를 묻는 노드가 될 가능성이 높다.

### [[../../mapped/Deep Knowledge/Container]]
컨테이너는 클라우드 위에서 워크로드를 어떻게 포장하고 이동시키는지의 문제에 가깝다. `Cloud`가 플랫폼과 책임 분배의 선택이라면, `Container`는 그 선택 위에서 배포 단위와 실행 환경을 어떻게 다룰지를 더 세부적으로 파고드는 노드다.

### [[../../mapped/Deep Knowledge/Devops]]
`Devops` 원문은 빠르고 안전한 배포, 협업 문화, 피드백 루프를 중심으로 전개된다. 따라서 `Cloud`가 자원·서비스·관리 모델의 선택이라면, `Devops`는 그 기반 위에서 조직과 배포 흐름을 어떻게 운영할지의 문제다. 둘은 겹치지만 초점이 다르다.

### [[../../mapped/Deep Knowledge/Web]]
정적 배포, CDN, API, 인증, 저장소 같은 웹 실행 환경은 클라우드와 직접 맞닿는다. 그러나 `Web`이 사용자에게 보이는 서비스 구조에 더 가깝다면, `Cloud`는 그 구조를 어떤 운영 모델 위에 올릴지 결정하는 배경 레이어다.

## 이 노드에 들어오면 좋은 내용 / 다른 노드에 남겨둘 내용
### `Cloud`에 들어오면 좋은 것
- 특정 벤더 서비스 자체보다, **왜 관리형/서버리스/컨테이너/가상서버를 고르는지**에 대한 선택 이유
- 비용표보다, **숨은 비용과 운영 포인트가 어느 선택에서 튀어나오는지**를 비교하는 메모
- `편리함을 얻는 대신 통제를 어디까지 포기할 수 있는가` 같은 상위 질문
- 관측·권한·계정 분리처럼 **운영 가시성과 책임 경계**를 어디서 설계할지에 대한 기준
- 나중에 구조를 바꿔야 할 때 **회귀 가능성**을 판단하는 기준

### 다른 노드에 남겨둘 내용
- VPC, IAM, Route53, CloudFront 세부 항목처럼 이미 `Infra`가 더 잘 받는 실행 메모
- Dockerfile, compose, Kubernetes 리소스처럼 `Container`가 더 잘 받는 배포 단위 메모
- CI/CD, 테스트 자동화, 조직 문화처럼 `Devops`가 더 잘 받는 팀 운영 메모
- 웹 렌더링, 프론트 흐름, 사용자 인터페이스처럼 `Web`이 더 잘 받는 서비스 표면 메모

이 구분은 현재 공란 노드를 지키기 위한 임시 경계다. 하지만 바로 이 경계 덕분에, 이후 사용자가 `Cloud` 원문을 직접 채웠을 때 그 문장을 어디에 흡수해야 하는지 더 빠르게 판별할 수 있다.

## 지금 시점의 해석
이 노드의 공란은 단순 누락일 수도 있지만, 다른 노트에 흩어진 클라우드 관련 생각이 아직 하나의 상위 판단 노드로 수렴되지 않았다는 뜻일 가능성도 크다. 지금 보이는 주변 흔적을 기준으로 하면 `Cloud`는 장기적으로 다음 역할 중 하나로 발전할 수 있다.

- 클라우드 선택 기준을 모으는 상위 판단 노드
- AWS, GCP, Azure 같은 벤더 지식을 비교하는 노드
- 서버리스, 네트워크, 데이터, 비용, 통제 가능성의 교환관계를 다루는 운영 철학 노드

현재 단계에서는 이 가능성을 억지로 확정하지 않고, 원본이 들어올 때 어느 방향으로 실제 문장이 기울어지는지 보는 편이 맞다.

## 원본이 채워질 때 가장 먼저 볼 질문
- 클라우드를 편의성의 문제로 보는가, 운영 전략의 문제로 보는가
- 비용 최적화와 개발 속도 중 무엇을 더 자주 우선하는가
- 서버리스 선호가 어디까지 유효하고, 어디서 서버·컨테이너로 되돌아오는가
- 관측, 권한, 계정 분리, shared responsibility를 얼마나 상위 판단으로 다루는가
- 특정 벤더 지식보다 공통 원리와 회귀 가능한 구조를 더 중시하는가
- `Infra`에 이미 적힌 실행 메모를 반복하는가, 아니면 그 위의 선택 기준을 새로 여는가

## 원본 갱신이 들어오면 우선 볼 델타
- 새로 생긴 문장이 벤더 비교인지, 운영 철학인지, 비용/구조 기준인지 먼저 분류한다.
- `Infra`에 있는 AWS·서버리스·네트워크 메모와 겹친다면 반복인지 상위 재정리인지 구분한다.
- 사용자가 직접 쓴 표현이 현재 mapped의 가설보다 훨씬 좁거나 넓다면, 기존 해석을 유지하지 말고 `human-reviews/`에 복사 후 선호 변화로 기록한다.
- 특히 `모든것을 위한 클라우드는 없다` 같은 선택 기준 문장이 실제 `Cloud` 원문에 들어오는지, 아니면 전혀 다른 관심사로 기울어지는지를 먼저 확인한다.

## 유지 원칙
- 원본이 비어 있는 동안에는 과도한 내용 확장을 하지 않는다.
- 다만 관련 source 노드에 이미 흩어진 같은 주제 흔적은 읽어 두고, 이 페이지의 경계와 향후 흡수 기준을 더 선명하게 만드는 데만 제한적으로 활용한다.
- 사용자가 원본을 직접 채우면 그 버전을 우선 읽고 필요 시 `human-reviews/`에 복사해 선호 변화로 기록한다.
- 이후 심화는 이 페이지를 새로 쓰기보다, 실제 원문이 생긴 뒤 구조·논지·링크를 점진적으로 보강하는 방식으로 진행한다.
- 빈 노드일수록 `무엇이 아직 비어 있는지`와 `어디까지가 현재 가설인지`를 분리해 적는다.

## 이번 갱신 메모
- 원문 공란 상태를 다시 확인한 뒤, `Cloud`를 채워 넣어야 할 문서가 아니라 상위 선택 기준을 받을 카테고리 노드로 더 분명하게 다듬었다.
- `Infra`뿐 아니라 `Devops`, `Container`와의 차이도 다시 읽어, 이 노드를 `호스팅 기술`보다 `책임 위임·운영 표면·선택 아키텍처`를 다루는 상위 경계로 보강했다.
- 이번 패스에서는 인접 source의 실제 문장을 바탕으로 `Cloud`가 받아야 할 상위 질문을 표로 압축해, 공란 노드라도 주변 메모와의 연결 근거가 더 빠르게 보이도록 다듬었다.
- 이번 런에서는 특히 IAM shared responsibility, 계정 분리, CloudWatch/Flow Logs, EKS·Fargate·Lambda 비교 메모를 다시 읽어, `Cloud`가 단순 자원 임대가 아니라 `무엇을 맡기고 무엇을 계속 보여야 하는가`를 다루는 운영 경계 노드라는 점을 더 선명하게 보강했다.
- 무엇을 이 노드에 넣고 무엇을 인접 노드에 남겨둘지의 포함/제외 기준을 추가해, 이후 직접 원문이 들어왔을 때 드리프트 없이 흡수하기 쉽게 만들었다.
