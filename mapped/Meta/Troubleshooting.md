# Troubleshooting

- 원본: `source/shdkej-content/Meta/Troubleshooting.md`
- 성격: 만다라트 `Meta` 축의 1:1 대응 카테고리 노드
- 이번 상태: 얇은 1차 대응 노트에서 기준선 수준의 구조화 페이지로 재구성

## 한눈에 보는 핵심
이 노트는 개별 오류 해결법 모음이지만, 그냥 잡다한 메모 더미는 아니다. 원문은 Jupyter, Docker, WSL, Git, Go, Kubernetes, OAuth, Linux 부팅 문제처럼 범위가 넓지만, 반복해서 드러나는 중심은 꽤 일관적이다.

1. 문제 해결은 영감이 아니라 **조건 단순화, 관련 정보 연결, 근거 없는 추측 배제**로 진행해야 한다.
2. 좋은 트러블슈팅은 명령어 암기보다 **증상 분리, 원인 후보 축소, 상태 관측, 복구 순서 확보**에 가깝다.
3. 이 노트의 핵심 자산은 특정 툴 팁보다 **한 번 겪은 장애를 다음엔 더 빨리 복구하는 운영 기억**이다.
4. 특히 Kubernetes, Docker, Linux 계열 메모에서는 화려한 해결보다 **재설치, 초기화, 권한, 네트워크, 리소스 상태 확인** 같은 기초 복원 절차가 반복해서 중요하게 나온다.

## 핵심 논지
원본의 `Trouble Shooting`은 단순한 오류 사전이 아니다. 더 정확히는, 다양한 개발 환경과 운영 환경에서 발생한 실패를 어떻게 다뤄야 하는지에 대한 개인 운영 매뉴얼에 가깝다. 첫머리의 원칙 세 줄, 즉 `조건을 단순하고 명확하게 정리`, `조건과 관련된 정보를 찾아 연결`, `근거없는 무엇이 개입되지 않아야 함`이 이 노드 전체를 지배한다. 뒤에 붙는 수많은 사례는 이 원칙을 도구별로 반복 훈련한 기록으로 읽는 편이 맞다.

## 원본에서 먼저 잡히는 큰 구조

### 1. 시작점은 해결책보다 문제 정의 방식이다
원문은 곧바로 `Ctrl+F로 찾으라`고 말하고, 바로 아래에 문제 해결 원칙 세 줄을 둔다. 즉 이 노드는 특정 기술 스택보다 먼저 `어떻게 문제를 쪼개고 확인할 것인가`를 다룬다.

### 2. 대부분의 메모는 증상 중심으로 기록되어 있다
`not working`, `connection refuse`, `cannot access`, `pending`, `broken`, `not ready`, `fatal`, `404` 같은 표현이 매우 자주 나온다. 이는 이 노트가 이론 정리보다 실제로 마주친 실패 신호를 중심으로 축적되었다는 뜻이다.

### 3. 해결은 자주 재설치, 리셋, 권한 수정, 설정 보정으로 수렴한다
원문 전반에서 많이 반복되는 해결 패턴은 아래와 같다.
- 패키지 재설치
- 캐시 제거
- 설정 파일 수정
- 경로와 권한 확인
- 서비스 재시작
- kubeadm reset, minikube delete 같은 초기화
- DNS, 포트, 네트워크 확인
- 로그와 상태 확인

즉 저자는 문제를 추상 교훈으로 덮기보다, 실제 복구 절차를 남겨 다음 진입 비용을 낮추려 한다.

### 4. 후반부로 갈수록 운영 장애 비중이 커진다
초반은 로컬 개발환경, 에디터, 패키지 설치, GitHub Pages, WSL 같은 개발 환경 이슈가 많고, 중반 이후는 Kubernetes, k3s, 리소스 부족, 노드 상태, 저장소, 네트워크 연결, 재부팅 후 장애처럼 운영 이슈가 두드러진다. 즉 이 노드는 개발자 개인 환경 문제집이면서 동시에 소규모 인프라 운영 장애 기록이기도 하다.

## 이 노드가 실제로 다루는 질문
- 문제를 만났을 때 무엇부터 확인해야 하는가?
- 증상, 원인, 해결책을 어떻게 분리해서 기록해야 다음에도 재사용할 수 있는가?
- 재설치와 초기화는 언제 빠른 해결이고, 언제 근본 원인 은폐가 되는가?
- 로그, 메트릭, 시스템 상태, 권한, 네트워크를 어떤 순서로 확인해야 하는가?
- 개발 환경 오류와 운영 환경 장애를 같은 메모 방식으로 다뤄도 되는가?
- 문서화된 트러블슈팅은 개인 기억을 어떻게 운영 자산으로 바꾸는가?

## 구조 재정리

### 1. 이 노드의 핵심은 정답 목록보다 진단 태도다
원문 첫 세 줄은 짧지만 중요하다.
- 조건을 단순하고 명확하게 정리
- 조건과 관련된 정보를 찾아 연결
- 근거 없는 무엇이 개입되지 않아야 함

이 세 문장은 결국 트러블슈팅을 다음처럼 정의한다.
- 문제를 재현 가능한 단위로 줄인다.
- 관측된 사실과 설정 정보를 연결한다.
- 추측이나 공포가 아니라 확인된 단서로 다음 행동을 정한다.

즉 이 노드는 문제 해결을 감각적 요령보다 증거 기반 정리 작업으로 본다.

### 2. 트러블슈팅은 `증상 → 상태 확인 → 수정 → 재검증`의 반복 루프다
원문 대부분의 사례는 아주 짧게 적혀 있지만 구조는 비슷하다.
- 증상: 설치 실패, 연결 거부, 포트 접근 불가, 이미지 이상, 한글 깨짐, NotReady, pending
- 상태 확인: 로그, 환경변수, 버전, 설정 파일, 리소스 상태, describe, journalctl, syslog
- 수정: 재설치, 설정 수정, reset, 서비스 재시작, 권한 변경, 경로 수정
- 재검증: 다시 접속, 다시 빌드, 다시 apply, 다시 init

이 점에서 이 노드는 해결법 모음보다 운영 루프 기록에 가깝다.

### 3. 원문은 문제의 성격을 네 가지 층위로 반복한다

#### 개발 환경 층위
Jupyter, Vimwiki, Eclipse, npm 권한, chsh, WSL, Docker Desktop 연결, 키보드 설정 같은 문제는 개인 작업 환경의 마찰을 줄이는 층위다.

#### 빌드·런타임 층위
Go 테스트, gRPC 코드 생성, Gatsby SSR, React 접근, lambda 인코딩, alpine 바이너리 실행 문제는 코드가 실제 실행되는 경계의 문제다.

#### 인프라·오케스트레이션 층위
Terraform, Ansible, Jenkins, AWS API Gateway, Grafana/Prometheus, Kubernetes, k3s는 서비스 구성과 배포, 운영 중 장애 대응의 층위다.

#### 시스템 복구 층위
리눅스 부팅 불가, Wi-Fi 하드블록, 도커 하이퍼바이저, 메모리 부족, 노드 NotReady, 재부팅 후 swap 문제는 시스템 자체를 다시 살리는 복구 층위다.

이 층위를 구분해 읽으면 원문의 산만함이 줄고, 같은 종류의 실패를 더 빨리 묶어볼 수 있다.

### 4. Kubernetes 메모는 이 노드의 운영 철학을 가장 잘 드러낸다
원문 중 가장 밀도 높은 구간은 Kubernetes다. 여기서 반복되는 패턴은 분명하다.
- 접속 불가면 API 서버 자체 문제인지, 노드 문제인지, 리소스 부족인지 먼저 분리한다.
- OOM, kubelet, coredns, flannel, weave, selector, label, volume, swapoff 같은 기본 상태를 먼저 본다.
- 잘못 꼬였으면 `reset`, `delete`, `killall` 같은 초기화 절차를 과감히 사용한다.
- 다만 초기화만으로 끝내지 않고 `top`, `describe`, `journalctl`, `syslog`로 원인을 추적하려 한다.

즉 작성자는 쿠버네티스를 마법의 플랫폼보다, 상태가 꼬이면 기본 자원과 연결 상태부터 다시 살펴야 하는 복잡한 운영 시스템으로 이해한다.

### 5. 재설치는 패배가 아니라 복구 전략의 일부다
원문에는 `reset`, `delete`, `reconfigure`, `reinstall`, `prune`이 많이 나온다. 이는 근본 원인 분석을 포기했다는 뜻보다는, 시스템을 빨리 정상 상태로 되돌린 다음 다시 관찰 가능한 상태를 확보하려는 태도에 가깝다. 즉 이 노드에서 재설치는 무식한 마지막 수단이 아니라, 복구 가능성을 확보하는 실무적 선택지다.

### 6. 많은 문제는 결국 권한, 경로, 버전, 네트워크, 자원으로 돌아온다
서로 다른 사례를 묶어 보면 실제 반복 범주는 제한적이다.
- 권한: sudo, 파일 소유권, runAsUser, npm global permission
- 경로: config 위치, mount path, go package path, kubeconfig, CNAME
- 버전: tensorflow-numpy, helm v3, 이미지 태그, 플러그인 버전 충돌
- 네트워크: DNS, 포트포워딩, API endpoint, ingress pending, docker host
- 자원: OOM, prometheus 메모리, node unknown, disk/volume 상태

이 노트는 결국 도구별 팁보다 `어느 축이 문제인지 먼저 가르는 습관`을 훈련시킨다.

## 반복되는 판단 축

### 추측 vs 관측
원문은 근거 없는 개입을 경계한다. 즉 감으로 건드리기보다 현재 상태와 로그를 먼저 확인하는 쪽이 기본이다.

### 국소 수정 vs 초기화
설정 한 줄 수정으로 끝낼지, reset과 재설치로 복구할지 계속 판단한다. 작은 수정이 빠를 때도 있지만, 상태가 너무 꼬였으면 초기화가 더 싸다.

### 빠른 복구 vs 근본 원인 추적
당장 서비스나 개발환경을 살리는 것이 먼저일 때가 많지만, 이후 같은 일이 반복되지 않게 하려면 OOM 원인, DNS 누락, selector 불일치 같은 구조적 원인을 다시 확인해야 한다.

### 도구 문제 vs 시스템 문제
애플리케이션이 안 뜨는 것처럼 보여도 실제론 포트, 볼륨, 권한, 노드 상태, swap, cgroup 설정 문제일 수 있다. 원문은 이 경계 혼동을 자주 겪었고, 그래서 시스템 층위 확인이 중요해진다.

### 단발성 팁 vs 재사용 가능한 운영 기억
각 사례는 짧은 팁처럼 보여도, 축적되면 다음 장애에서 확인 순서를 빠르게 제공한다. 이 노트의 가치는 개별 팁보다 축적된 점검 루틴에 있다.

## 원문에서 특히 보존해야 할 부분

### 첫 세 줄의 문제 해결 원칙
이 부분은 짧지만 사실상 전체 노트의 헌법이다. 이후 수백 줄의 사례는 모두 이 세 줄을 구체적인 도구 세계로 옮긴 것에 가깝다.

### `Ctrl+F` 전제
원문이 `specific trouble`을 찾으라고 명시하는 점은 중요하다. 즉 이 노트는 교과서보다 색인형 현장 메모다. 따라서 mapped에서도 모든 사례를 납작한 요약으로 덮기보다, 증상 중심 접근을 보존해야 한다.

### Kubernetes OOM과 NotReady 메모
이 구간은 단순 해결책 기록이 아니라, 관측, 리소스 확인, 로그 확인, killall, restart, top, describe로 이어지는 운영식 문제 해결 흐름을 잘 보여준다.

### 재설치와 reset 메모들
`kubeadm reset`, `minikube delete`, `terraform init -reconfigure`, 캐시 삭제, 재설치 같은 메모는 실패 은폐가 아니라 복구 루틴의 일부라는 감각을 드러낸다.

## 오래 남길 아이디어
- 좋은 트러블슈팅은 똑똑한 추측보다 조건 분리와 관측 가능한 사실 정리에서 시작한다.
- 문제 기록은 `증상`, `확인한 상태`, `수정한 것`, `결과`가 함께 있어야 다음에 재사용된다.
- 재설치와 초기화는 무지의 신호가 아니라, 상태를 다시 통제 가능한 범위로 되돌리는 전략일 수 있다.
- 서로 다른 장애도 실제로는 권한, 경로, 버전, 네트워크, 자원 다섯 축으로 자주 환원된다.
- 운영 환경의 문제는 애플리케이션 오류처럼 보이더라도 시스템 자원과 오케스트레이션 상태에서 시작할 때가 많다.
- 트러블슈팅 문서는 문서 노드 중에서도 특히 `재진입 속도`를 높이는 문서다.

## 정리 메모
- 얇은 1차 대응 노트를 한국어 기준의 구조화된 트러블슈팅 운영 노드로 전면 재구성했다.
- 원문 사례집을 단순 도구 목록으로 납작하게 만들지 않고, 증상 중심 기록, 상태 관측, 리셋/복구, 운영 재진입이라는 흐름으로 다시 묶었다.
- 특히 Kubernetes, Docker, 로컬 개발환경 메모를 함께 읽어 이 노드의 중심이 `문제 해결법` 자체에 있다는 점을 더 선명하게 드러냈다.
