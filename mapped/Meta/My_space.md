# My_space

- 원본: `source/shdkej-content/Meta/My_space.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 요약
이 노드는 `내가 만든 서비스들을 쉽게 띄우고, 더러워지면 다시 세우고, 비용 안에서 오래 운영할 수 있는 개인 공간`을 만들기 위한 메타 운영 설계 노드다. 핵심은 서버 한 대를 갖는 것이 아니라, **IaC, GitOps 성향의 배포, 관측 가능성, 재구축 가능성, 비용 감각을 함께 갖춘 개인 운영 환경**을 만드는 데 있다.

## 이 노드가 실제로 답하려는 질문
- 개인이 감당 가능한 예산 안에서 어디까지 플랫폼 같은 운영 환경을 만들 수 있는가?
- 앱 개발과 배포 책임을 어떻게 분리해야 서버가 수작업으로 오염되지 않는가?
- Kubernetes는 개인 공간에서도 실제로 가치가 있는가, 아니면 학습비와 운영비가 과한가?
- 모니터링, 로그, 알림, 롤백, GitOps까지 갖춰야 비로소 `내 공간`이라 부를 수 있는가?
- 2025년의 비용 좌절과 2026년의 Oracle ARM 재시도는 무엇이 달라졌기 때문에 가능해졌는가?

## 핵심 논지
1. **이 노드의 목표는 개인 서버 소유보다 복원 가능한 운영 상태 확보다.**
   - 원문은 `원할 때 접속`, `배포가 쉬워야 한다`, `재구축이 바로 되어야 한다`, `지저분해지면 원하는 상태로 바로 만들 수 있어야 한다`를 반복한다.
   - 즉 `My_space`는 물리적 공간보다 원하는 상태를 다시 만들 수 있는 시스템에 가깝다.
2. **배포는 앱 저장소가 아니라 인프라 계층이 맡아야 한다.**
   - `앱은 이미지만 만들고 배포는 인프라에서 처리한다`는 문장이 이 노드의 중심축이다.
   - 이는 수작업 서버 운영에서 벗어나, 동일한 방식으로 여러 앱을 다시 올릴 수 있는 공용 제어면을 만들겠다는 뜻이다.
3. **쿠버네티스는 기능 목록이 아니라 운영 훈련장으로 읽힌다.**
   - 롤링 업데이트, 롤백, 카나리, 오토스케일링, 페일오버, MLOps 서버, gRPC 서버 배포 메모는 모두 운영 역량을 검증하는 과제처럼 적혀 있다.
   - 그래서 이 노드는 `쿠버네티스를 쓴다`보다 `운영을 몸으로 이해한다`에 더 가깝다.
4. **현실의 가장 큰 제약은 기술적 이상보다 비용과 리소스다.**
   - DigitalOcean 관리형 Kubernetes, AWS, GCP 비교도 결국 개인이 감당 가능한 월 비용과 메모리 한계로 수렴한다.
   - `opentelemetry 좀 설정하려니까 리소스 부족`, `k3s를 돌리는 것만으로도 0.9G 메모리` 같은 메모는 이 노드가 추상론보다 운영 현실에 붙어 있음을 보여 준다.
5. **개인 환경이라도 관측, 알림, 복구 루프가 있어야 운영 공간이 된다.**
   - 로그 수집, 모니터링, 상태 추적, 오버로드 알림, 롤백, 버전 업데이트 확인이 반복되는 이유는, 띄우는 것보다 망가졌을 때 다룰 수 있어야 진짜 시스템이 되기 때문이다.
6. **2026 메모는 `개인 서버`에서 `작은 플랫폼 제어면`으로 시야가 이동한 흔적이다.**
   - Oracle ARM 인스턴스의 역할 분리, ArgoCD auto sync, `project repo CI -> space repo 태그 수정 -> sync` 흐름은 이제 단순 구축보다 운영면 설계가 앞선다는 뜻이다.
   - 같은 `My_space`라도 2021~2025가 비용과 막힘의 실험장이라면, 2026은 실제로 굴러가는 제어 구조를 세우는 단계에 더 가깝다.

## 원문 구조를 다시 읽기
원문은 시기별 메모와 실험 메모가 섞여 있지만, 실제로는 아래 흐름으로 읽힌다.

### 1. 개인 운영 공간의 요구사항 정의
- IaC로 관리되어야 한다.
- 접속과 배포가 쉬워야 한다.
- 지저분해지면 원하는 상태로 빠르게 재구축되어야 한다.
- 쿠버네티스 안에서 에러를 한눈에 볼 수 있어야 한다.
- 앱은 이미지만 만들고, 배포는 인프라가 맡아야 한다.

이 부분은 단순 희망사항이 아니라 이후 모든 시행착오의 평가 기준이다.

### 2. 호스팅 선택의 진행 이력
1. Cloud 상의 Kubernetes 호스팅 시도
   - 월 10만원 이상 비용 문제로 무산
2. DigitalOcean 관리형 Kubernetes 시도
   - 저사양에서 서비스가 제대로 돌지 않아 무산
3. Oracle Cloud 무료 ARM 고사양 서버로 재시도
   - k3s와 IaC 기반으로 다시 진행

이 흐름은 벤더 비교라기보다 `개인 예산으로 감당 가능한 운영 환경의 하한선`을 찾는 과정이다.

### 3. 필요한 최소 인프라의 정의
- 3노드 Kubernetes
- 로드밸런서
- 로그 수집
- 모니터링
- API 서버
- feedback
- CI/CD 도구

즉 목표는 장난감 클러스터가 아니라, 작은 규모라도 서비스 운영에 필요한 제어면을 갖춘 공간이다.

### 4. 실험 과제 목록
- 롤링 업데이트, 롤백 테스트
- 카나리 배포 테스트
- 스트레스 테스트와 오토스케일링 검증
- 페일오버, 이중화 확인
- MLOps 서버와 gRPC 서버 배포
- 버전 업데이트 추적

이 노드는 `구축 완료` 문서가 아니라 `무엇을 검증해야 운영 체계라고 부를 수 있는가`를 적어 둔 학습 과제 노트이기도 하다.

### 5. 2025의 좌절과 2026의 재출발
- 2025년에는 DigitalOcean 비용과 리소스 부족 때문에 중단과 흔들림이 많다.
- 2026년에는 Oracle ARM 인스턴스를 기반으로 역할이 나뉜 서버와 ArgoCD 자동 동기화까지 적어 두며, 보다 실제 운영 구조에 가까운 메모가 등장한다.
- 특히 `argocd/apps` 변경이 곧 배포 반영으로 이어지는 흐름은, 생각이 단순 인프라 실험에서 GitOps적 운영 설계로 옮겨갔음을 보여 준다.

### 6. 실제 운영 워크로드의 등장
- 텔레그램 봇, SNS, SQS, Lambda, note-reminder가 이미 이 공간 안에서 흐른다.
- 즉 `My_space`는 미래 계획만 적은 빈 설계 노트가 아니라, 이미 알림·자동화·배포 실험이 지나가는 운영 허브다.
- 그래서 이 노드의 핵심은 `무엇을 깔까`보다 `여러 워크로드를 같은 규칙으로 다룰 수 있나`에 있다.

## 원문 발췌 구조
- 목표 선언: `확장 가능하고 변경이 쉬운 환경`, `앱은 이미지만 만들고 배포는 인프라에서 처리`
- 요구사항: IaC, 쉬운 접속/배포, 빠른 재구축, 한눈에 보는 에러 상태
- 진행 이력: 관리형 Kubernetes 비용 무산 -> DigitalOcean 저사양 한계 -> Oracle ARM + k3s 재시도
- 검증 과제: 롤링 업데이트, 롤백, 카나리, 오토스케일링, 페일오버, MLOps/gRPC 배포
- 최소 인프라: 3노드 Kubernetes, 로드밸런서, 로그, 모니터링, API, feedback, CI/CD
- 실제 워크로드: 텔레그램 메시징 파이프라인, note-reminder 연동
- 2026 운영면: Oracle 인스턴스 역할 분리, ArgoCD auto sync, CI 템플릿 기반 태그 갱신

## 이 노드에서 반복되는 판단 축

### 확장성 vs 개인 예산
- 확장 가능하고 변경이 쉬운 환경을 원한다.
- 하지만 관리형 Kubernetes, 로드밸런서, 관측 도구는 모두 비용을 만든다.
- 그래서 이 노드는 기술 선택보다 `개인이 감당 가능한 운영비의 경계`를 계속 묻는다.

### 재현성 vs 즉흥 수정
- 당장 서버에 들어가 고치면 빠를 수 있다.
- 하지만 원문은 점점 IaC와 배포 분리 쪽으로 이동한다.
- 즉 편의보다 재현성과 복구 가능성을 더 높은 기준으로 삼는다.

### 학습 욕구 vs 운영 현실
- Kubernetes를 직접 써 보며 깊이 이해하고 싶다.
- 동시에 작은 리소스에서는 관측 도구조차 무겁다.
- 그래서 이 노드는 이상적인 학습 경로와 현실적인 운영 제약이 부딪히는 장면을 그대로 보존한다.

### 구축 자체 vs 운영 루프 완성
- 클러스터를 띄우는 것만으로는 끝이 아니다.
- 로그, 모니터링, 알림, 버전 추적, 롤백, GitOps가 이어져야 비로소 `공간`이 된다.
- 저자에게 운영은 설치보다 `지속적으로 다룰 수 있음`에 더 가깝다.

### 단일 서버 운영 vs 공용 제어면 구축
- 초반에는 `개인 서버를 갖고 싶다`는 욕구가 앞에 있다.
- 하지만 후반으로 갈수록 핵심은 서버 보유가 아니라 여러 앱과 자동화를 동일한 절차로 싱크시키는 공용 제어면에 가까워진다.
- 그래서 이 노드는 홈랩 취미보다 `작게 시작한 플랫폼 엔지니어링 연습장`으로 읽는 편이 더 정확하다.

## 세부 해석

### 1. `My_space`는 서버가 아니라 운영 상태다
이 노드에서 말하는 공간은 집에 있는 장비나 특정 클라우드 계정 자체가 아니다. 원하는 서비스를 일정한 규칙으로 올리고, 오류를 관찰하고, 필요하면 다시 세울 수 있는 상태를 뜻한다. 그래서 `재구축`, `원하는 상태`, `배포 분리`가 계속 반복된다.

### 2. `앱은 이미지만 만들고 배포는 인프라에서 처리한다`의 무게
이 문장은 개발과 운영의 경계를 어떻게 그릴지에 대한 선언이다.

- 애플리케이션 저장소는 코드와 이미지 생성에 집중한다.
- 실제 반영은 Terraform, ArgoCD, 클러스터 설정 같은 인프라 층에서 담당한다.
- 이렇게 해야 여러 앱이 같은 규칙을 따르고, 서버가 수동 편집으로 오염되지 않는다.

즉 개인 인프라를 `플랫폼처럼 다루고 싶다`는 의도가 분명하다.

### 3. 메시징 시스템 메모가 보여 주는 것
원문에는 텔레그램 봇, SNS, SQS, Lambda, note-reminder가 등장한다. 이 부분은 단순 부가 기능이 아니라, 개인 공간이 정적 호스팅이 아니라 **실제 자동화 워크로드가 흐르는 운영 허브**가 되기를 바란다는 신호다.

### 4. DigitalOcean 메모는 실패담이 아니라 운영 감각 축적이다
- provider 버전 차이
- `.terraform` 상태 오염
- SSH permission 문제
- ingress, DNS, Traefik, Grafana 연결 문제
- 리소스 부족으로 인한 컴포넌트 불안정

이런 메모는 지저분해 보여도 중요하다. 저자는 단순 성공 사례보다, 실제로 어디서 운영이 막히는지를 기억해 두려는 편이다. 그래서 이 노드는 기술 소개보다 `어디서 삐끗하는가`를 보존하는 가치가 크다.

### 5. 2026 메모는 운영 구조가 더 선명해진 시점이다
2026 섹션에서는 Oracle 인스턴스의 역할 분리와 CI/CD, ArgoCD auto sync 흐름이 비교적 간결하게 정리돼 있다. 이는 이전의 탐색과 시행착오가 단순 실험에서 끝나지 않고, `프로젝트 레포의 CI → space 레포의 argocd/apps 태그 갱신 → 자동 동기화` 같은 운영 구조로 응집되고 있음을 보여 준다.

### 6. 2026의 핵심 변화는 `클러스터를 띄움`이 아니라 `상태를 전달하는 경로`다
2026 메모에서 중요한 것은 서버 스펙 자체보다 역할 분리와 동기화 경로다.

- `worker / system / 실험용`처럼 역할을 나눠 둔다.
- 앱 배포는 ArgoCD application 설정과 auto sync로 묶는다.
- 프로젝트 레포의 CI는 결과 이미지를 만들고, space 쪽에서는 태그 변경 커밋으로 반영한다.

이 구조는 `개발 -> 이미지 -> 배포 반영 -> 클러스터 상태`가 한 흐름으로 이어져야 한다는 생각을 보여 준다. 즉 저자에게 `내 공간`은 서버 묶음이 아니라 **상태가 전달되는 운영 경로**다.

## 운영 루프로 다시 압축하면
1. 앱 레포에서 코드와 이미지를 만든다.
2. 인프라/space 레포에서 배포 대상을 선언하고 태그를 갱신한다.
3. ArgoCD가 자동 동기화해 실제 클러스터 상태를 맞춘다.
4. 로그·모니터링·알림으로 이상 징후를 본다.
5. 롤백, 재구축, 리소스 조정으로 다시 안정 상태를 만든다.

이 압축은 원문의 흩어진 메모들이 사실 하나의 `개인 운영 루프`를 가리키고 있음을 더 빠르게 보여 준다.

## 숨은 패턴
- 이 노드는 개인 인프라 메모처럼 보이지만 실제로는 **작은 규모에서 플랫폼 엔지니어링 감각을 익히는 메타 노드**다.
- 비용 계산, 리소스 예측, 배포 분리, 관측 도구, 롤백 실험, 버전 모니터링이 모두 같은 방향을 가리킨다.
- 저자는 `최신 기술을 쓴다`보다 `다시 만들 수 있고, 상태를 알 수 있고, 여러 서비스를 일관되게 다룰 수 있다`를 더 중요하게 여긴다.
- 따라서 핵심은 Kubernetes 그 자체보다 **복원 가능한 개인 운영 시스템**이다.

## 인접 노드와의 연결
- [[../blog/My_space]]: 블로그 노드가 비용과 시행착오를 외부 공유용 서사로 더 압축했다면, 이 Meta 노드는 장기 운영 기준과 최근 세팅 이력을 더 직접적으로 담는다.
- [[../Deep Knowledge/Devops]]: 작은 팀의 자율성과 복구 가능성, 공통 플랫폼 설계라는 관점이 그대로 이어진다.
- [[../Deep Knowledge/Container]]: 앱은 이미지로 만들고 배포 책임은 인프라가 갖는다는 분리가 직접 연결된다.
- [[../Deep Knowledge/Infra]]: 필요한 구성 요소를 묻는 노드라면, 이 노드는 그것을 내 환경에서 어떻게 실제로 운영할지를 묻는다.
- [[../Meta/AGENTS]]: 공통 문서와 역할 분화를 통해 운영 가능한 시스템을 만든다는 메타 감각이 닮아 있다.
- [[../Integration/Tool]]: Tailscale, n8n, 원격 접속, 개인 운영 표면을 어떤 인터페이스로 붙일지 고민하는 축이 이어진다.
- [[../Portfolio]]: 기술 스택보다 병목 제거, 운영 단순화, 재구축 가능성을 성과로 보는 서술 습관이 이어진다.

## 이 노드를 읽을 때 놓치지 말아야 할 점
- 완성된 인프라 문서가 아니라 비용, 실험, 실패, 재시도, 운영 기준이 겹쳐 있는 메타 설계 노드다.
- `내 공간`은 서버 소유가 아니라 재구축 가능성과 배포 일관성의 다른 이름이다.
- Kubernetes 채택 여부보다, 무엇을 자동화하고 무엇을 관찰해야 하는가의 기준이 더 중요하다.
- 2025와 2026의 차이는 단순 벤더 변경이 아니라, 시행착오 메모가 점점 운영 구조 메모로 응집된다는 데 있다.

## 이후 심화 포인트
- 2025와 2026 메모를 대비시켜 `비용 기반 탐색기`에서 `실제 운영 루프 설계`로의 이동을 더 선명하게 정리할 수 있다.
- 메시징 시스템, GitOps 배포, 버전 모니터링을 하나의 운영 루프로 묶어 압축할 수 있다.
- `개인 서버`와 `개인 플랫폼`의 차이를 다른 Meta 노드와 연결해 더 분명하게 만들 수 있다.

## 이번 갱신 메모
- 기존 baseline은 유지한 채 `원문 발췌 구조`를 추가해 목표 선언, 검증 과제, 실제 워크로드, 2026 운영면의 배치를 빠르게 다시 볼 수 있게 했다.
- 2026 Oracle/ArgoCD 메모를 `클러스터 띄우기`가 아니라 `상태 전달 경로 설계`로 해석하는 단락과 `운영 루프` 압축을 추가해 페이지의 재진입성을 높였다.
- `My_space`를 서버 보유 욕구보다 공용 제어면과 복원 가능한 운영 상태를 만드는 메타 노드로 더 선명하게 다듬었다.
