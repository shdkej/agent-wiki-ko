# 100k_concurrent_server

- 원본: `source/shdkej-content/blog/100k_concurrent_server.md`
- 상태: baseline 이후 점진 심화
- 역할: 만다라트 `blog > 100k_concurrent_server` 카테고리 노드의 1:1 대응 페이지

## 한눈에 보는 요약
이 노트는 `10만 동접 달성기`가 아니라, **한 대의 서버가 어디서 먼저 막히는지를 실험으로 추적하는 블로그형 운영 기록**이다. 원문은 소켓·포트·커넥션 개념 혼선에서 출발해 `ulimit`, `TIME_WAIT`, 클라이언트 포트 고갈, CPU, 메모리, 네트워크 대역폭, nginx, Kubernetes ingress, websocket 유지 연결까지 차례대로 부딪힌다. 핵심 결론도 단일 숫자가 아니다. **동시 접속 수는 서버 코드만의 능력이 아니라 테스트 토폴로지, 커널 한계, 클라이언트 자원, 앞단 프록시 구조까지 합쳐진 결과**라는 쪽에 더 가깝다.

## 만다라트 위치에서 읽는 해석
이 페이지가 `Fundamental/Network`나 `Deep Knowledge/Infra`가 아니라 `blog` 아래에 놓여 있다는 점이 중요하다. 여기서 100k 동접은 교과서형 네트워크 지식 정리가 아니라, **직접 열어 보고 깨져 보고 다시 가설을 세우는 공개형 실험 로그**다. 즉 이 노드는 `정답 설명서`보다 `실험 사고 경로`를 남기는 카테고리로 읽는 편이 맞다.

## 이 노드의 중심 질문
- 한 대의 PC로 10만 동시 접속을 유지할 수 있는가?
- 소켓, 포트, 커넥션은 실제 병목과 어떻게 연결되는가?
- 한계는 서버 코드, 프로세스 FD, 클라이언트 포트, 네트워크, 프록시, ingress 중 어디에서 먼저 드러나는가?
- `짧은 요청 처리`, `유지 연결`, `websocket`, `Kubernetes` 실험은 왜 서로 다른 숫자를 보여 주는가?

## 원문 발췌 구조
- `왜 10만인가`: 실제 스포츠 중계 동접 규모를 기준 삼아 목표 숫자를 잡는다.
- `소켓/포트/커넥션`: 65535라는 숫자가 무엇의 상한인지 계속 다시 묻는다.
- `메모리/대역폭 계산`: `tcp_mem`, `wmem`, `rmem`, 랜카드, `iperf`를 통해 자원 감각을 세운다.
- `100,000 user load test`: Go echo 서버/클라이언트로 짧은 요청 테스트를 시작한다.
- `상황1`: `too many open files`, `i/o timeout`, `connection reset`, `cannot assign requested address`를 따라 병목을 좁힌다.
- `상황2`: 10초 유지 연결·websocket 테스트로 `열린 연결을 붙드는 비용`을 따로 본다.
- `상황3`: HLS/스트리밍 쪽으로 질문을 넓힌다.
- `추가 벤치마크와 OS 파라미터`: 동접과 처리량을 분리해 다시 보려는 후속 실험 큐가 이어진다.

## 이 노드를 읽는 5개 층위

### 1. 개념을 바로 외우기보다 병목과 연결해 이해하는 층위
원문 초반은 `소켓`, `포트`, `커넥션` 정의에서 여러 번 흔들린다. 하지만 이 구간은 헛도는 서론이 아니다. 저자는 용어 사전 하나를 얻으려는 것이 아니라, **누가 주소 공간을 점유하고 누가 FD를 소모하며 누가 TIME_WAIT를 남기는가**를 실제 병목과 연결해 이해하려 한다.

### 2. 숫자보다 먼저 관측 표면을 세우는 층위
원문에는 `watch ss -s`, `/proc/net/tcp`, `netstat`, `iperf`, `ulimit`, `sysctl` 메모가 반복된다. 이것은 팁 모음이 아니라 `안 된다`를 **상태 변화로 보이게 만드는 관측 습관**이다. 이 노트의 자산은 몇 만을 찍었는지보다 `무엇을 보며 다음 가설을 세웠는가`에 있다.

### 3. 서버 한계와 테스트 환경 한계를 분리하는 층위
이 노트에서 특히 중요한 지점은 `로컬에서는 왜 여기서 막히지?`, `vagrant에서는 왜 더 일찍 무너지지?`, `외부 IP로 하면 왜 금방 실패하지?` 같은 되물음이다. 저자는 점차 **클라이언트와 서버가 한 머신에 있으면 서버만 테스트하는 것이 아니라 클라이언트 포트 범위와 로컬 네트워크 경계도 함께 테스트하게 된다**는 사실에 가까워진다.

### 4. 단일 서버 튜닝에서 운영 구조 질문으로 넘어가는 층위
후반부로 갈수록 관심은 `Go 서버를 더 빠르게 만들까`보다 `nginx는 왜 더 버티지`, `프로세스를 나누면 어떻게 되지`, `다중 IP를 쓰면`, `ingress를 확장하면`으로 옮겨 간다. 즉 질문이 코드 미세 최적화에서 **운영 구조 설계**로 확장된다.

### 5. 최고치보다 안정 운영 상한을 찾는 층위
원문은 10만을 향해 달리지만, 중간중간 `실제 운영에서는 5만 정도만 안정적으로 받는 편이 낫겠다`는 식의 판단으로 돌아온다. 그래서 이 노트는 성능 자랑이 아니라, **최대치와 안정치가 다르다는 운영 감각**을 배우는 기록이기도 하다.

## 실험군을 분리해서 읽는 최소 프레임
| 실험군 | 원문에서 실제로 본 것 | 먼저 드러난 병목/질문 |
| --- | --- | --- |
| 짧은 HTTP echo 요청 | Go 기본 서버, nginx, docker 안팎 비교, 5천→1만→2만→3만 이상 상승 시도 | FD 한계, TIME_WAIT, CPU 과부하, `i/o timeout` |
| 유지 연결 테스트 | 10초간 연결 유지, 소켓 수 관찰, 응답보다 유지 비용 확인 | 열린 소켓 수, 메모리 점유, 클라이언트 포트 고갈 |
| websocket 실험 | 5만 수준 유지 성공, 51,000 이후 정지, 59,980 근처 한계 관찰 | 메모리 사용량, ephemeral port 범위, 주소 할당 실패 |
| Kubernetes 실험 | ingress/traefik 뒤 replica 분산, controller 확장 | 앱 바깥 앞단 병목, ingress 계층의 메모리·분산 한계 |
| 후속 스트리밍/HLS 질문 | HLS 서버, gRPC streaming, throughput 질문으로 확장 | `동접 수`와 `처리량`을 같은 척도로 볼 수 있는가 |

이 표를 세워 두면 원문 속 숫자들이 서로 충돌하는 것이 아니라, **서로 다른 실험군이 서로 다른 병목 층을 보여 준 결과**로 다시 읽힌다.

## 핵심 논지

### 동시 접속 수는 하나의 절대 성능이 아니라 먼저 고갈되는 층의 최소값이다
원문 전반에서 반복되는 것은 단일 원인 찾기의 실패다. 처음에는 포트 상한처럼 보이지만, 곧 FD 한계, TIME_WAIT, ephemeral port 범위, `i/o timeout`, CPU 과부하, 네트워크 unreachable, ingress 병목이 연달아 등장한다. 그래서 이 노트는 `10만 가능/불가` 문서가 아니라 **어느 자원 층이 먼저 바닥나는가**를 기록한 문서로 읽어야 한다.

### 짧은 요청 처리와 유지 연결은 같은 테스트가 아니다
원문은 echo HTTP, 10초 유지 연결, websocket 유지 연결, ingress 뒤 replica 테스트를 모두 같은 질문 아래 적어 두지만, 실제로는 서로 다른 성질을 가진다.
- echo HTTP: 연결을 얼마나 빨리 열고 닫고 응답하는가
- 유지 연결: 열린 상태를 얼마나 오래 붙들 수 있는가
- websocket: 상태 유지 비용이 메모리와 포트 한계에 어떻게 드러나는가
- Kubernetes: 서버 코드 밖 계층이 어디서 막히는가

이 구분을 세워야 숫자 혼선이 줄어든다.

### 로컬 벤치마크 숫자는 서버 본체의 순수 능력표가 아니다
`localhost에서는 되는데 외부 IP는 금방 실패`, `vagrant는 더 약함`, `한 머신에서 클라이언트와 서버를 같이 돌리니 포트 범위에 걸림` 같은 대목은 중요하다. 저자는 결국 **로컬 수치를 그대로 서버 단독 성능으로 읽으면 해석이 틀어진다**는 지점에 도달한다.

### 멀티 프로세스 발견은 팁이 아니라 구조 전환점이다
`프로세스당 FD 한계가 ulimit`이라는 발견은 이 노트의 중요한 전환점이다. 여기서 질문은 `한 프로세스로 얼마나 받나`에서 `한 머신에서 프로세스·포트·IP를 어떻게 나눠 받나`로 이동한다. 즉 이 노트는 벤치마크 메모를 넘어 **운영 구조를 다시 짜는 감각**으로 넘어간다.

## 병목 층을 읽는 실전 지도

### 1. 프로세스/OS 자원 층
- `too many open files`
- `ulimit -n`
- 파일 디스크립터와 소켓 수의 관계
- `sysctl`로 만지는 커널 TCP 파라미터

이 층은 애플리케이션 코드가 아니라 **프로세스와 커널이 허용하는 기본 수용량**을 보여 준다.

### 2. 클라이언트 주소 공간 층
- `cannot assign requested address`
- `1024 - 61000` 범위 관찰
- 로컬 테스트 시 클라이언트 포트 고갈
- VM, 외부 IP, 다중 IP 분산 필요성

이 층은 서버가 못 받는 것이 아니라 **테스트를 거는 쪽이 먼저 주소를 다 써 버리는 문제**를 보여 준다.

### 3. 네트워크/대역폭 층
- `network is unreachable`
- `iperf` 측정
- localhost와 외부 IP의 결과 차이
- 가상화 계층이 하나 늘 때 체감되는 성능 변화

이 층은 연결 가능 여부와 대역폭이 단순 CPU·메모리 문제와 별개로 **전달 경로 자체의 상태**라는 점을 드러낸다.

### 4. 애플리케이션 처리 층
- 3만 부근 CPU 풀 사용
- Go 기본 서버와 nginx 비교
- `fasthttp` 실험
- 응답시간과 TPS를 분리해서 봐야 한다는 후반 질문

이 층에서는 단순히 연결 수만이 아니라 **응답 속도, 요청 처리량, 구현체 차이**가 본격적으로 등장한다.

### 5. 앞단 운영 구조 층
- nginx를 앞에 둘 때의 변화
- Kubernetes ingress/traefik 병목
- replica 확장 효과
- HPA 실험 계획

이 층에 오면 문제는 서버 함수 최적화가 아니라 **부하를 어디서 흡수하고 어떻게 분산할 것인가**로 이동한다.

## 원문이 남기는 구조적 교훈

### 에러 메시지는 서로 다른 층을 가리킨다
- `too many open files`: 프로세스 자원 한계
- `cannot assign requested address`: 클라이언트 주소/포트 고갈 의심
- `connection reset by peer`: 서버 측 연결 종료 또는 재사용 충돌 의심
- `i/o timeout`: 과부하·대기·네트워크 지연이 섞인 문제
- `network is unreachable`: 경로/가상화/네트워크 계층 문제

원문은 이 오류들을 하나로 뭉개지 않고, **다음에 어느 층을 먼저 의심할지** 배우는 과정으로 읽힌다. 이 점은 [[mapped/Meta/Troubleshooting]]의 `증상 → 상태 확인 → 수정 → 재검증` 루프와 정확히 맞닿아 있다.

### 관측 명령이 있어야 경험담이 운영 자산으로 바뀐다
이 노트에서 가치 있는 것은 `몇 만 성공`보다 `무엇을 보며 판단했는가`다. `ss -s`, `/proc/net/tcp`, `netstat`, `iperf`, CPU/메모리 확인 메모는 모두 `느낌상 느리다`를 넘어서 **병목을 상태로 만들기 위한 관측 인터페이스**다. 이 감각은 [[mapped/Deep Knowledge/Monitoring]]과도 연결된다.

### 네트워크 개념 학습과 운영 실험은 분리되지 않는다
초반의 TCP·포트·keep-alive·window scaling 질문과 후반의 실험 실패는 따로 놀지 않는다. 원문은 개념을 외운 뒤 실험하는 구조가 아니라, **실험에 막혀서 개념 모델을 다시 보강하는 구조**로 움직인다. 그래서 [[mapped/Fundamental/Network]]는 이 노드의 배경지식이면서 동시에 계속 되묻는 해설 층이다.

### 최고 기록보다 안정 운영 숫자가 따로 있다
원문은 이론적으로 더 가능해 보여도, 실제 서비스에서는 `5만 정도만 안정적으로` 같은 판단으로 돌아온다. 이는 최고치보다 **CPU·메모리·네트워크·운영 여유를 함께 고려한 현실적 상한**을 더 중요하게 본다는 뜻이다.

## 이 노드를 다시 사용할 때의 읽기 순서
1. **증상부터 모은다**: `open files`, `reset`, `timeout`, `cannot assign`, `unreachable`.
2. **관측 표면을 확인한다**: 소켓 수, TIME_WAIT, CPU, 메모리, 대역폭, 포트 범위.
3. **지금 테스트 토폴로지를 의심한다**: 로컬/외부 IP/VM/컨테이너/다중 프로세스 여부.
4. **실험군을 분리한다**: 짧은 요청, 유지 연결, websocket, ingress, 스트리밍.
5. **운영 판단으로 회수한다**: 최고치보다 안정적으로 몇 만을 어디서 어떻게 나눠 받을지 결정한다.

이 순서를 세워 두면 원문의 긴 의식 흐름이 `대규모 동접이라는 현상을 해부하는 재진입 가능한 루프`로 바뀐다.

## 원문에서 드러나는 저자의 선호
- 추상 이론보다 직접 열어 보고 깨져 보는 실험 로그를 더 신뢰한다.
- `몇 만이 됐는가`보다 왜 거기서 멈췄는지, 어느 층이 먼저 고갈됐는지를 더 오래 붙든다.
- 명령어, 파라미터, 에러 메시지를 남겨 다음 실험 진입 비용을 낮추려 한다.
- 완결된 결론보다 미해결 질문과 후속 테스트 큐를 남겨 둔다.
- 성능 최적화를 서버 코드 단독 문제가 아니라 프록시·ingress·멀티프로세스·관측 체계까지 포함한 운영 구조 문제로 본다.
