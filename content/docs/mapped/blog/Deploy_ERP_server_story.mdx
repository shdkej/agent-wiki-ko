# Deploy_ERP_server_story

- 원본: `source/shdkej-content/blog/Deploy_ERP_server_story.md`
- 상태: 기준선 재구성
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 요약
이 노드는 Odoo 기반 ERP 서버를 Docker Compose로 빠르게 올리고, 수정 가능한 개발 상태와 재현 가능한 배포 상태 사이의 간극을 직접 겪으며 정리한 **초기 운영 설계 실습 노드**다. 표면적으로는 ERP 서버 구축기처럼 보이지만, 실제 핵심은 `컨테이너로 빨리 띄우기`에서 끝나지 않고, `수정 내용이 왜 사라지는지`, `볼륨과 이미지의 역할을 어떻게 나눌지`, `모니터링과 SSL을 어디에 붙일지`, `다른 컴퓨터로 어떻게 옮길지`까지 한 번에 건드리는 데 있다.

## 이 노드가 실제로 답하려는 질문
- 회사 내부 전산 작업을 위해 ERP를 가장 빠르게 어떻게 띄울 수 있는가?
- 오픈소스 ERP 프레임워크를 쓰면 어디까지 빨라지고, 어디서부터 다시 직접 설계해야 하는가?
- `docker exec`로 컨테이너 안을 고쳐서 해결한 내용은 왜 재실행하면 사라지는가?
- 개발 편의와 배포 재현성을 동시에 얻으려면 Compose, Dockerfile, volume을 어떻게 나눠 써야 하는가?
- 최소 운영 구성을 만든 뒤 모니터링과 SSL은 어떤 식으로 덧붙여야 하는가?

## 핵심 논지
1. **이 노드의 출발점은 ERP 기능 구현보다 내부 업무를 빠르게 전산화할 실행 기반 확보였다.**
   - 목표는 데이터를 한 곳에 모으고, 추적하고, 관리하는 것이다.
   - 따라서 기술 선택도 이상적인 최신 스택보다 빨리 띄우고 관리 가능한 조합에 맞춰진다.
2. **초기 구성은 Odoo + PostgreSQL + Nginx + Nagios라는 운영 최소단위로 압축된다.**
   - Odoo는 비즈니스 로직 프레임워크,
   - PostgreSQL은 상태 저장,
   - Nginx는 외부 진입점,
   - Nagios는 최소 감시 기능을 맡는다.
3. **가장 큰 학습은 컨테이너 내부 수정보다 이미지와 볼륨 설계가 중요하다는 점이다.**
   - 처음에는 컨테이너 안에 직접 들어가 addon 경로를 바꾸고 파일을 수정한다.
   - 하지만 `docker-compose down` 이후 그 변경이 사라지면서, 임시 수정과 재현 가능한 수정의 차이가 드러난다.
4. **그래서 이 노드는 결국 `한 파일로 끝내고 싶던 희망`에서 `조금 더 복잡하지만 확장 가능한 구조`로 이동한다.**
   - `docker-compose.yml` 하나로 끝내려던 계획은 깨진다.
   - 대신 Dockerfile을 추가해 설정 변경을 이미지 빌드 단계로 끌어올린다.
5. **운영 관심사는 시작부터 배포 이후까지 이미 넓다.**
   - 데이터 볼륨 이동,
   - 다른 컴퓨터로 이관,
   - Nagios 기반 모니터링,
   - SSL 설정,
   - CI/CD와 HAProxy 같은 다음 단계 과제가 한 노드 안에 함께 등장한다.

## 원문 구조를 다시 읽으면 보이는 흐름

### 1. 목표 설정
원문 첫머리는 기술 취향보다 먼저 업무 목적을 분명히 둔다.

- 회사 내부 전산 작업화
- 데이터 집약, 추적, 관리
- `docker-compose.yml` 하나로 구성하고 싶은 욕구

즉 이 노드는 학습용 튜토리얼보다, **업무 문제를 빠르게 처리하기 위한 실전 출발 메모**다.

### 2. 왜 Odoo와 PostgreSQL인가
원문은 여러 언어 기반 ERP 프레임워크 중에서 Python 기반 Odoo를 택한다.

- Python을 써보고 싶었다.
- MVC 구조와 모듈 구성이 잘 되어 있다.
- 지원 기능이 많다.
- Docker 이미지가 이미 있다.

DB는 MySQL, MSSQL, PostgreSQL 중에서 PostgreSQL을 골랐다.

- Docker 볼륨 상태를 그대로 옮기기 좋다고 판단했다.

여기서 중요한 점은 기술 순수주의가 아니라 **당장 만들 수 있고 옮기기 쉬운 것**을 기준으로 잡는다는 것이다.

### 3. Compose로 최소 실행 기반 만들기
초기 `docker-compose.yml`은 매우 얇다.

- `erp: odoo:10.0`
- `db: postgres:9.4`

그 다음 포트, 환경변수, 볼륨, 작업 디렉토리를 붙이며 실제 운영 가능한 형태로 다듬는다.

이 단계의 의미는 단순 설정 추가가 아니다. 원문은 Compose를 통해 다음 문제를 한 번에 묶으려 한다.

- 서비스 간 실행 순서
- 포트 개방
- DB 연결 정보
- 데이터 보존 볼륨
- 외부 addon 연결

즉 Compose는 이 노드에서 **여러 운영 경계를 한 파일에서 묶어 관리하려는 첫 번째 제어판**으로 쓰인다.

## 이 노드의 진짜 학습 포인트: 수정이 왜 사라지는가
원문에서 가장 중요한 전환은 `docker exec` 이후에 온다.

### 컨테이너 안 직접 수정
- 컨테이너에 root로 들어간다.
- vim을 설치한다.
- `project` 모듈을 외부 addons 디렉토리로 옮긴다.
- `odoo.conf`의 `addons_path`를 수정한다.
- XML view를 고친 뒤 컨테이너를 재시작한다.

이 단계에서 작성자는 실제로 화면 변화를 확인한다. 즉, 빠른 실험은 성공한다.

### 그런데 down/up 하면 초기화된다
`docker-compose down` 후 다시 올리면, 컨테이너 안에서 직접 수정했던 내용이 사라진다.

이 경험이 이 노드의 핵심이다. 여기서 다음 사실이 몸으로 확인된다.

- 실행 중 컨테이너는 수정 가능하지만,
- 그 수정은 이미지에 기록되지 않으며,
- 재생성 시 쉽게 증발한다.

그래서 원문은 곧바로 실무 판단으로 넘어간다.

- 이 작업을 자동화해야 한다.
- 결국 `docker-compose.yml` 하나만으로는 부족하다.
- 대신 구조는 더 확장 가능해졌다.

이 문장은 짧지만 중요하다. 작성자는 단순 실패로 받아들이지 않고, **재현 가능한 구조로 한 단계 올라가기 위한 신호**로 읽는다.

## 구조 전환: Compose-only에서 Dockerfile 포함 구조로
원문은 Dockerfile을 도입해 다음 작업을 이미지 빌드 단계로 이동시킨다.

- vim 설치
- `odoo.conf`의 addons 경로 변경
- 기본 project addon 제거

그리고 `image: odoo:10.0` 대신 `build: ./`로 바꾼다.

이 변화의 의미는 분명하다.

### 이전 상태
- 컨테이너 실행 후 수작업 수정
- 빠르지만 재현 불가

### 이후 상태
- 빌드 정의로 수정 내역 고정
- 다시 띄워도 동일 상태 복원 가능

즉 이 노드는 개발 메모이지만, 핵심 배움은 **운영 변경을 런타임이 아니라 빌드 타임으로 끌어올려야 한다**는 것이다.

## 데이터 이동과 다른 컴퓨터로의 이관
원문 중반 이후에는 볼륨을 검사하고, 실제 Docker volume 경로를 찾아 데이터를 복사해 다른 컴퓨터로 옮기는 절차가 나온다.

- `docker volume ls`
- `docker volume inspect`
- 실제 `/var/lib/docker/volumes/.../_data` 경로 복사
- 새 환경에서 해당 위치로 다시 복원

이 부분은 단순 백업 팁이 아니다. 작성자는 ERP를 장난감 서버가 아니라 **상태를 품은 업무 시스템**으로 보고 있다.

따라서 중요한 질문도 자연스럽게 바뀐다.

- 서비스를 띄울 수 있는가?
- 수정이 남는가?
- 데이터가 보존되는가?
- 다른 컴퓨터로 옮길 수 있는가?

이 흐름은 `[[../Deep Knowledge/Container]]` 노드의 `개발은 볼륨, 배포는 이미지` 감각과 직접 연결된다. 여기서는 그 일반론이 실제 ERP 사례 안에서 먼저 몸으로 등장한다.

## 최소 운영 확장: Nagios와 SSL
원문 후반은 자세한 설정을 길게 적지 않지만, 두 가지를 바로 붙인다.

### Nagios
- Docker socket을 마운트해 컨테이너 프로세스를 관찰한다.
- 즉, 서비스를 띄우는 것만으로 끝내지 않고 최소한의 감시 체계를 함께 둔다.

### SSL을 위한 Nginx
- 외부 공개를 염두에 두고 reverse proxy와 SSL 설정을 별도 서비스로 분리한다.

이 두 요소는 이 노드의 성격을 더 분명히 보여준다. 작성자는 단순 로컬 개발 환경이 아니라, **실제로 써야 하는 내부 서비스 운영 구도**를 생각하고 있다.

## 반복되는 판단 축

### 1. 빠른 시작 vs 재현 가능한 운영
- Compose만으로 시작하면 빠르다.
- 하지만 수정이 런타임 수작업에 머물면 재실행 시 날아간다.
- 그래서 Dockerfile과 볼륨 설계가 필요해진다.

### 2. 오픈소스 활용 vs 커스터마이징 책임
- Odoo 덕분에 ERP 기반은 빨리 얻는다.
- 하지만 실제 회사 업무에 맞추려면 addon, 설정 파일, view 수정이 필요하다.
- 즉 프레임워크 선택은 시작을 줄여주지만, 운영 책임을 없애주지는 않는다.

### 3. 단순 구성 vs 확장성
- 처음엔 `docker-compose.yml` 하나로 끝내고 싶다.
- 그러나 데이터 보존, 설정 변경, 모니터링, SSL, 이관 요구가 생기면서 파일과 레이어가 늘어난다.
- 이 복잡성 증가는 실패가 아니라 운영 현실의 반영이다.

### 4. 로컬 실험 vs 실제 서비스 감각
- `localhost:8069`에서 시작하지만,
- 곧 데이터 이관, SSL, Nagios, CI/CD, HAProxy까지 생각한다.
- 즉 이 노드는 단순 입문기보다 **로컬 실험이 운영 사고로 확장되는 전환점**을 보여준다.

## 이 노드를 카테고리 노드로 읽을 때 중요한 점
이 문서를 단순히 `Odoo 설치 방법`으로만 읽으면 핵심이 많이 사라진다. 실제로 더 중요한 것은 다음 세 가지다.

1. **초기 서비스 구축에서 가장 먼저 부딪히는 운영 경계가 무엇인지 보여준다.**
   - 코드 수정 경로
   - 상태 저장 방식
   - 빌드와 실행의 분리
   - 서비스 이관 방법
2. **수정의 지속성 문제를 통해 이미지와 볼륨의 역할 차이를 체감하게 한다.**
3. **작은 내부 시스템도 곧바로 모니터링, SSL, 배포 자동화 질문으로 연결된다는 점을 보여준다.**

그래서 이 노드는 특정 ERP 제품 메모라기보다, **업무 시스템을 처음 서비스화할 때 생기는 운영 설계 감각의 사례 노드**로 읽는 편이 더 정확하다.

## 남겨진 과제와 미해결 질문
원문 마지막의 아쉬운 점은 이 노드의 다음 단계를 잘 보여준다.

- CI/CD
- Test Code
- HAProxy
- Vagrant에서의 IP 세팅 자동화
- repository split
- docker build to image
- announcing update

즉 현재 수준은 `직접 띄우고 수정하고 옮길 수 있는 상태`까지 왔고, 다음 관심사는 **자동화, 검증, 고가용성, 배포 체계화**로 이동한다.

## 이번 재구성에서 강화한 점
- 매우 얇은 1차 대응 초안을 한국어 기준의 구조화된 페이지로 전면 재구성했다.
- 원문 전체를 다시 읽고, 이 노드를 단순 ERP 설치기가 아니라 `빠른 구축에서 재현 가능한 운영으로 넘어가는 초기 사례`로 재해석했다.
- `컨테이너 내부 수정의 한계`, `Compose-only 구조의 깨짐`, `Dockerfile 도입의 의미`, `데이터 이관`, `Nagios/SSL 확장`의 흐름을 중심으로 구조를 다시 묶었다.
- 관련 mapped 노트와의 연결을 보강하되, 실제 작성 변경은 이 페이지 하나에만 제한했다.
