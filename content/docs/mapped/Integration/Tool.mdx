# Tool

- 원본: `source/shdkej-content/Integration/Tool.md`
- 성격: 만다라트 `Integration` 축의 1:1 대응 카테고리 노드
- 이번 상태: 기존 baseline은 유지하되, 이 노드가 만다라트 `Integration` 안에서 맡는 역할을 더 분명히 하고, 흩어진 도구 메모를 `작업 표면 지도`, `편입 문턱`, `복구 가능한 운영 기억`, `시스템 증설 감각`의 사다리로 다시 조여 읽히게 다듬었다.

## 한눈에 보는 핵심
이 노드의 `Tool`은 추천 앱 모음이 아니다. 맥 유틸, 터미널 툴, 업무 서비스, Tailscale, OpenClaw, Linux, Git, Terraform, gRPC, 스트리밍 메모가 한 파일에 섞여 있는 이유는, 저자가 툴을 `하루의 입력·검색·협업·원격 운영·복구·확장`을 한 몸처럼 잇는 작업 시스템으로 보기 때문이다.

1. 도구는 기능보다 `작업 마찰을 줄이고 다시 세팅할 수 있게 하는 손잡이`다.
2. 좋은 툴 메모는 사용 후기보다 `권한·경로·인증·자동화·복구`를 오래 남긴다.
3. 로컬 생산성 앱, 협업 SaaS, 서버 운영 툴, IaC 도구는 서로 다른 취미가 아니라 같은 운영체제의 다른 면이다.
4. 이 노드는 정답집보다 `개인 운영체제의 부품 창고`이자 `다시 들어가기 위한 재진입 지도`에 가깝다.

## 이 노드가 실제로 답하려는 질문
- 어떤 도구가 멋진가가 아니라, 어떤 도구 조합이 내 작업 흐름을 덜 끊기게 만드는가?
- 개인 앱과 서버 운영 툴이 왜 한 카테고리에서 함께 읽혀야 하는가?
- 새 툴을 볼 때 기능보다 먼저 확인해야 하는 편입 문턱은 무엇인가?
- 한 번 해결한 문제를 나중에도 다시 쓸 수 있게 남기려면 어떤 식으로 기록해야 하는가?
- 툴을 많이 아는 것과 툴을 엮어 자기 시스템을 만드는 것은 어떻게 다른가?

## 핵심 논지
원문에서 `tool`은 프로그램 이름 목록이 아니다. 손의 움직임을 줄이는 개인 인터페이스부터, 검색과 조합을 빠르게 하는 CLI, 협업 서비스, 원격 운영 도구, Linux/Git 기반 복구 기억, Terraform·Vault·gRPC 같은 시스템 확장 실험까지 한 카테고리 안에 넣는다.

즉 이 노드에서 툴은 다음 전체를 뜻한다.
- 입력과 화면 전환을 줄이는 개인 인터페이스
- 검색·조합·스크립트화를 담당하는 텍스트 기반 작업면
- 기록·협업·배포·관측을 분산된 서비스 위에서 이어 붙이는 운영 루프
- 권한, 인증, 경로, 재시작 순서를 외부화하는 복구 기억
- 더 큰 시스템으로 넘어갈 때 필요한 연결 부품과 판단 실험

## 만다라트 위치에서 읽기
이 페이지가 `Integration/Tool`에 놓여 있다는 점이 중요하다. 같은 `Tool`이라도 [[../Fundamental/Tool]]이 셸, 권한, 파일 경로, 복구 절차 같은 하부 감각을 붙드는 노드라면, 여기의 `Tool`은 그 하부 감각 위에 여러 표면을 실제 생활과 업무 흐름으로 엮는 통합 노드에 가깝다.

- `Fundamental/Tool`이 손에 도구를 쥐는 법이라면, 이 노드는 여러 도구를 한 작업면으로 편성하는 법에 가깝다.
- `Meta/AGENTS`가 역할·문서·협업 토폴로지를 다룬다면, 이 노드는 그 구조를 실제로 굴리게 하는 로컬/원격/협업 도구층을 다룬다.
- `Integration/Work`가 일의 흐름과 우선순위를 다룬다면, 이 노드는 그 흐름이 덜 끊기게 만드는 인터페이스와 제어면을 다룬다.

그래서 원문이 앱 추천, CLI, 원격 접속, Linux, Git, IaC, 스트리밍까지 넓게 흩어져 보여도, 실제로는 `서로 다른 도구층을 한 사람이 감당 가능한 운영 표면으로 묶는 법`을 기록한 카테고리로 읽는 편이 맞다.

## 먼저 잡아야 하는 네 개의 큰 층

### 1. 상단 목록은 앱 추천이 아니라 `내가 직접 만지는 작업 표면 지도`다
Traefik, Signoz, Grafana, GitHub Actions, GitHub, Next.js, Pencil, Jenkins, Kubernetes, k3s, ArgoCD가 한 덩어리로 등장한다. 분류가 매끈하지 않은 대신, 구현·배포·관측·디자인·분석을 한 사람이 연속해서 만지는 현실 표면이 드러난다.

### 2. `블로그 소프트웨어 탑 텐`은 취향 리스트보다 `인터페이스 층 분해`에 가깝다
맥 도구, 터미널 도구, 소프트웨어, 개발도구, 업무도구, 개발내도구, 크롬 익스텐션, 앱이라는 8개 묶음은 예쁜 taxonomy보다 실제 접점 순서를 보여 준다. 손, 터미널, 작성, 협업, 배포, 모바일 일상까지가 한 흐름이다.

### 3. 서비스 목록은 편한 앱 모음이 아니라 `디지털 의존 지형`이다
회사 서비스와 개인 서비스를 나눠 적는 부분은 단순 계정 목록이 아니다. 기록, 협업, 배포, 커뮤니케이션, 저장, 분석이 여러 외부 표면에 나뉘어 있고, 이 분산을 어떻게 감당할지가 실제 운영 문제라는 점을 보여 준다.

### 4. Linux·Git·ETC 확장은 잡학 과시가 아니라 `복구 가능한 운영 기억`의 저장소다
명령어, 경로, 재시작 순서, 옵션, 인증, 충돌 처리, 네트워크, 백업 메모가 길게 이어지는 이유는 이 노트가 이론 설명서가 아니라 `다시 들어가서 손을 움직일 수 있게 하는 기억 저장소`이기 때문이다.

## 구조 재정리

### 1. 도구의 첫 기준은 화려함보다 `입력 마찰 절감`이다
카라비너, 레이캐스트, hidden bar, spectacle, scroll reverser, Vimium 같은 도구는 모두 손의 이동과 화면 전환 비용을 낮추는 쪽에 있다. 이 노드의 출발점은 좋은 앱을 많이 쓰는 일이 아니라, 자주 하는 동작을 짧게 만드는 일이다.

이 층에서 반복되는 기준은 다음과 같다.
- 자주 쓰는 호출을 단축한다.
- 화면·창 전환 비용을 낮춘다.
- 검색과 접근을 기본 동작으로 끌어올린다.
- 반복 선택을 설정과 단축키로 밀어 넣는다.

### 2. 터미널 도구는 지식보다 `검색과 조합의 속도`를 책임진다
vim, zshrc, UltiSnips, Rclone, fzf, ripgrep, jq, Claude Code가 한 묶음으로 보이는 이유는 모두 더 빨리 찾고, 더 빨리 조합하고, 더 빨리 수정하기 위한 층이기 때문이다.

여기서 CLI 도구의 가치는 다음에 있다.
- 많이 보기보다 정확히 찾기
- 클릭 기록보다 재실행 가능한 텍스트 흐름 남기기
- 사람 기억을 줄이고 검색성과 스크립트 가능성 높이기
- 작은 조합으로 큰 반복 작업을 줄이기

### 3. 협업 도구와 개인 도구는 분리되지 않고 하나의 루프를 만든다
Asana, Slack, Confluence, Google Drive 같은 업무 도구와 Obsidian, 메일, 브라우저, 메신저, GitHub, Analytics, Cloudflare, Netlify가 같은 문맥에 있다. 이는 생산성이 개인 정리와 협업 정리로 쪼개지지 않는다는 뜻이다.

이 루프는 대체로 이렇게 이어진다.
1. 개인 메모와 검색 도구로 생각을 붙잡는다.
2. 협업 도구에서 상태를 공유한다.
3. GitHub나 배포 표면에서 실행 결과를 확인한다.
4. 모니터링과 대시보드에서 상태를 다시 읽는다.
5. 문제가 생기면 터미널과 운영 메모로 돌아가 복구한다.

### 4. 이 노드의 중심축 중 하나는 `재설치 가능 환경`이다
글로벌 설치 목록을 파일로 모으는 스크립트, dotfiles, startup files, backup, cron, ssh, git without password, known_hosts 처리, 포트포워딩 메모가 반복해서 나온다. 이것은 사용 후기보다 `내 환경을 다시 만들 수 있는가`라는 질문이 더 중요하다는 뜻이다.

즉 툴 메모는 다음 역할을 한다.
- 재설치 때 빠르게 복구할 기준점 제공
- 자주 막히는 권한·경로·인증 문제 외부화
- 실험 도구와 장기 편입 자산 구분
- 환경 이동 비용 감소

### 5. Tailscale과 OpenClaw 메모는 `새 도구의 실제 편입 문턱`을 보여 준다
이 노드에서 특히 살아 있는 부분이다. Tailscale 메모는 설치 성공담보다 `--ssh`, 키 만료, 재인증 같은 운영 문턱을 적는다. OpenClaw 메모도 기능 자랑보다 모니터링, cron, 브라우저 제약, GitHub 노트 접근, dashboard 포트포워딩, 모델 설정, 스킬 설치처럼 실제 편입 지점을 남긴다.

즉 새 도구를 볼 때 먼저 통과해야 하는 질문은 다음이다.
- 원격에서 믿고 쓸 수 있는가?
- 상태 확인이 쉬운가?
- 인증과 갱신 비용은 어느 정도인가?
- 기존 노트·서버·GitHub 흐름과 연결되는가?
- 막혔을 때 우회로를 만들 수 있는가?

### 6. Linux와 Git 메모는 상위 툴들을 떠받치는 `하부 운영 토대`다
후반의 Linux와 Git 파트는 양이 많지만 주변부가 아니다. 상위 생산성 앱과 관측 툴을 오래 쓰려면 결국 셸, 권한, 파일 시스템, 네트워크, 백업, 브랜치, 인증, 복구를 직접 다뤄야 하기 때문이다.

이 파트가 보여 주는 것은 다음이다.
- 화려한 도구는 기본 운영 감각 위에서만 오래 간다.
- Git은 협업 툴이면서 동시에 개인 복구 장치다.
- Linux 메모는 이론보다 재진입 가능한 작업 기억이다.
- 상위 생산성은 하위 안정성에 기대고 있다.

### 7. ETC 파트는 수집벽이 아니라 `시스템 조합 능력의 확장`이다
Ansible, Terraform, Serverless, Vault, Nagios, Lambda cronjob, API gateway, Kafka, gRPC, 동영상 스트리밍 메모는 깊이가 제각각이어도 공통점이 있다. 저자는 개별 툴 사용법보다 `이 도구가 어느 레이어를 맡고 어디와 붙는가`를 계속 본다.

그래서 이 파트는 이렇게 읽는 편이 맞다.
- Terraform vs Serverless는 제품 비교가 아니라 책임 배치 문제다.
- Vault는 보안 도구이면서 운영 편의와 충돌하는 판단 대상이다.
- gRPC와 API gateway는 기능 선택이 아니라 통신 구조 설계 문제다.
- 스트리밍 메모는 호기심보다 실제 운영 부하와 전달 구조를 읽는 실험이다.

## 이 노드에서 보이는 도구 편입 사다리
원문 전체를 한 번 더 묶어 보면, 도구를 받아들이는 순서가 대체로 아래처럼 반복된다.

### 1. 먼저 손과 화면의 마찰을 줄인다
Karabiner, Raycast, Vimium, hidden bar, spectacle처럼 손이 자주 부딪히는 인터페이스부터 다듬는다. 시작점은 거대한 시스템이 아니라 `지금 당장 매일 반복되는 동작`이다.

### 2. 그다음 검색·기록·조합 속도를 끌어올린다
vim, zshrc, ripgrep, jq, fzf, Obsidian, 메일, 브라우저, GitHub가 이 층에 들어간다. 여기서는 `어떻게 더 많이 하느냐`보다 `어떻게 더 빨리 찾고 다시 꺼내느냐`가 중요하다.

### 3. 그 위에 협업과 배포의 공용 표면을 얹는다
Asana, Slack, Confluence, Google Drive, GitHub Actions, Jenkins, Netlify, Cloudflare처럼 혼자서 끝나지 않는 표면들이 나온다. 이때 도구는 개인 취향이 아니라 `여러 사람이 같은 상태를 볼 수 있게 하는 인터페이스`가 된다.

### 4. 원격 접속과 복구 수단을 붙여 작업면을 늘린다
Tailscale, OpenClaw dashboard 포워딩, SSH, backup, cron, known_hosts, git without password 메모가 여기에 걸친다. 즉 도구 편입은 설치 완료가 아니라 `다른 장소와 다른 기기에서도 계속 이어지는가`까지 가야 끝난다.

### 5. 마지막으로 시스템 증설 실험으로 넘어간다
Terraform, Vault, gRPC, Kafka, streaming 같은 후반 메모는 새로운 기능 수집이 아니라, 앞에서 만든 작업면을 어디까지 넓힐 수 있는지 시험하는 단계다. 이 노드의 넓은 범위는 산만함이 아니라 `도구 편입이 결국 시스템 설계로 이어진다`는 증거다.

## 원문 발췌 구조

### A. 작업 표면 초벌 목록
- Traefik, Signoz, Grafana, Prometheus
- GitHub, GitHub Actions, Jenkins
- Next.js, Kubernetes, k3s, ArgoCD
- Pencil, GA4

초반부터 구현·배포·관측·분석·디자인이 한 줄기처럼 붙어 있다.

### B. `블로그 소프트웨어 탑 텐`의 8개 층
1. 맥 도구: 카라비너, 레이캐스트, 상태창 관리, hidden bar, spectacle
2. 터미널 도구: vim, zshrc, Ultisnips, Rclone, Fzf, Claude code, ripgrep, jq
3. 소프트웨어: 옵시디언, ChatGPT, 크롬, 메일, iterm
4. 개발도구: vscode, 해머스푼
5. 업무도구: 아사나, 구글드라이브, 슬랙, 컨플루언스, flow
6. 개발내도구: 도커, 쿠버네티스, 깃, terraform, minikube
7. 크롬 익스텐션: vimium, onetab, color picker, translate, grammarly
8. 앱: 토스, 지도, 기본툴

이 8층 자체가 `하루의 접점 분해도`처럼 읽힌다.

### C. 서비스 지형 메모
- 회사: Drive, Asana, Confluence, Joplin, Axure
- 개인: Telegram, Gmail, GitHub, Analytics, Keep, Cloudflare, Netlify, Pocket, rsync 등
- 회원가입 서비스: Google, Naver, Facebook, Apple, PayPal

서비스 사용이 많다는 사실보다, 삶과 일이 여러 외부 표면에 분산되어 있다는 점이 더 중요하다.

### D. 최신 툴 편입 메모
- Tailscale: SSH 옵션, 키 만료, 재인증
- OpenClaw: 모니터링, cron, 헤드리스 브라우저, GitHub 노트 접근, dashboard 포워딩, 모델 변경, 스킬 세팅

새 도구를 `광고 문구`가 아니라 `운영 문턱 체크리스트`로 읽는 대목이다.

### E. 하부 운영 기억
- bash cheatsheet, startup files, 한글 입력기, distro 경험
- sudo, backup, cron, iptables, ssh, DB, DNS, storage, NFS, WSL
- Git 설정, rebase/merge 충돌, stash, hooks, 대용량 파일 정리

문제 해결 후 다시 손을 대야 하는 항목들이 길게 축적되어 있다.

### F. 시스템 확장 실험
- ansible, terraform, serverless, vault
- nagios, lambda cronjob, cloud infra
- kafka, grpc, api gateway, streaming

도구를 `기능`이 아니라 `시스템 책임의 배치`로 읽는 방향이 드러난다.

## 이 노드에서 반복되는 판단 축

### 검색성 vs 화려함
ripgrep, jq, fzf, raycast, dashboard, monitoring처럼 빠르게 찾고 바로 조작할 수 있는 도구가 계속 남는다.

### 편의성 vs 재현 가능성
당장 편한 것만이 아니라 dotfiles, 설치 스크립트, cron, backup 메모가 반복되는 걸 보면 재현 가능성을 더 길게 본다.

### 로컬 마찰 감소 vs 원격 운영 안정성
키보드 단축, 브라우저 확장, 앱 추천과 함께 Tailscale, OpenClaw, SSH, dashboard 메모가 놓인다. 둘은 다른 관심사가 아니라 같은 생산성 문제다.

### 사용법 암기 vs 복구 가능한 기억
명령어 자체보다 권한, 파일 위치, 재시작 순서, 인증 갱신, 꼬였을 때 복구 루프가 더 오래 남는다.

### 단발성 시도 vs 장기 편입
새 도구는 일단 써 보는 것에서 끝나지 않고, 기존 메모·서버·협업 흐름에 안정적으로 편입되는지 시험받는다.

## 오래 남길 아이디어
- 툴은 앱 이름보다 `마찰을 줄이고 상태를 제어하고 다시 만들 수 있게 하는 손잡이`다.
- 좋은 도구 메모는 기능 설명보다 경로, 권한, 인증, 자동화, 복구 순서를 남긴다.
- 개인 생산성과 서버 운영은 다른 취미가 아니라 같은 작업 시스템의 앞면과 뒷면이다.
- 새 도구는 신기함만으로 편입되지 않고, 원격 접근·상태 확인·갱신 비용을 통과해야 장기 자산이 된다.
- 이 노드의 산만함은 약점이지만, 동시에 실제 작업 환경이 얼마나 넓은 표면 위에서 유지되는지 보여 주는 증거이기도 하다.

- [[../Meta/My_space]]: 개인 서버, 원격 접근, 재구축 가능한 작업 공간이라는 관심사가 만난다.

## 선호 신호
저자는 툴을 화려한 기능 소비보다 `작업 환경을 길들이는 운영 표면`으로 보는 경향이 강하다. 특히 검색성, 단축, 원격 접근, 관측 가능성, 재설치 가능성, 꼬였을 때 복구하는 순서를 중시한다. 또한 앱을 예쁘게 분류하는 일보다, 실제 하루의 입력-검색-공유-운영-복구가 어떻게 이어지는지를 더 중요하게 보는 편이다.

추가로 이 노드에서는 `새 도구를 얼마나 빨리 써보는가`보다 `기존 노트·서버·협업 흐름에 얼마나 무리 없이 편입되는가`를 더 오래 본다. 그래서 기능 감탄보다 포트포워딩, 재인증, 설치 스크립트, 대시보드 접근, 백업, 권한 처리 메모가 더 진하게 남는다.
