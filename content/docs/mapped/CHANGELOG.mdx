# CHANGELOG

- 원본: `source/shdkej-content/CHANGELOG.md`
- 상태: 점진 심화
- 역할: 루트 만다라트에서 `개인 웹을 어떻게 계속 굴렸는가`를 붙드는 카테고리 노드

## 한눈에 보는 요약
이 노드는 기능 연혁보다 운영 압력 지도에 가깝다. 원문에는 검색 노출, 광고 실험, 화면 손질, Gatsby 기반 유지, 짧은 `update` 메모가 한 파일 안에 겹쳐 있다. 따라서 이 CHANGELOG는 `무엇을 만들었는가`보다 `무엇을 계속 다시 맞춰야 했는가`를 보여 준다.

## 핵심 해석
개인 웹은 여기서 단순한 글 저장소가 아니라 다음 네 일을 동시에 요구하는 표면으로 나타난다.

1. **보이게 만들기** — SEO, sitemap, 검색 진입면 정리
2. **붙여 보기** — Adsense, Kakao adfit 같은 현실적 수익 실험
3. **버티게 만들기** — Gatsby monorepo, starter, plugin, dependency 대응
4. **다시 만지기** — 404, popup, fragment, 자잘한 `update`와 `fix`

즉 이 파일의 중심은 제품 출시 서사보다 `발행 → 실험 → 유지 → 미세조정`의 반복 루프다.

## 왜 이 노드가 중요한가
- [[Portfolio]]가 팀 규모의 운영 능력을 보여 준다면, 이 노드는 개인 웹 표면에서 그 감각이 얼마나 생활형 유지 노동으로 나타났는지 보여 준다.
- [[Communication/Blogging]]이 `무엇을 발행할 것인가`를 붙든다면, 이 노드는 `그 발행면을 계속 살아 있게 만드는 비용`을 붙든다.
- [[Meta/My_space]]가 나중의 자기 소유 제어면 상상이라면, 이 CHANGELOG는 그보다 이른 단계의 `생활형 운영 박동`이다.

## 원문 구조를 읽는 기본 틀
원문은 `Features`와 `Bug Fixes`로 나뉘지만, 실제로는 아래 다섯 층이 겹친다.

| 층위 | 대표 흔적 | 실제 의미 |
| --- | --- | --- |
| 생활형 반영 | `update 21-04-20`, `🎸 update 21-04-23` | 큰 기능보다 자주 손보는 운영 리듬 |
| 발행면 확장 | `seo component`, `sitemap`, `google custom search` | 글을 `발견 가능하게` 만드는 작업 |
| 수익 실험 | `adsense`, `adfit`, script 삽입 | 공개 공간에 현실적 옵션을 붙여 보는 단계 |
| 표면 손질 | `update index with grid`, `update 404 page`, popup 조정 | 방문자가 직접 만지는 화면 조정 |
| 기반 상속 노동 | `gatsby monorepo`, starter/plugin/dependency 업데이트 | 선택한 프레임워크가 끌고 오는 장기 유지비 |

이 다섯 층을 먼저 잡아야, 원문이 왜 기능 목록보다 운영 장부처럼 보이는지 이해된다.

- `Bug Fixes` 항목: 215개
- `update`: 229회
- `gatsby monorepo`: 96회
- `sitemap`: 4회
- `google custom search`: 4회
- `adsense`: 3회

| 반복 표현 | 읽기 포인트 |

| `update`가 압도적으로 많다 | 확장보다 재조정과 유지가 기본 리듬이다. |

### `Features` / `Bug Fixes` 이름보다 더 중요한 역전
형식상으로는 기능과 버그 수정을 나누지만, 실제 분포는 거의 반대로 읽힌다.

- `Features`에도 날짜형 `update`, 광고 삽입, 검색 노출 정리, starter 상속 항목이 섞여 있다.
- `Bug Fixes`에는 단순 결함 수정만이 아니라 dependency 보안 반영, monorepo 추적, 화면 자잘한 보정이 대량으로 들어 있다.
- 그래서 이 파일의 진짜 중심은 `무엇을 새로 냈는가`보다 **얼마나 자주 다시 맞췄는가**에 있다.

### 날짜형 `update`가 몰린 구간
`update` 표기는 고르게 흩어지지 않고 몇 구간에 뭉친다.

| 구간 | 횟수 | 읽기 포인트 |
| --- | --- | --- |
| 2020-10 | 9회 | 개인 웹 표면을 자주 다시 열어 보는 초기 박동 |
| 2020-11 | 11회 | 발행면과 기반 조정이 함께 잦아지는 구간 |
| 2020-12 | 7회 | 연말에도 멈추지 않는 생활형 유지 |
| 2021-02 | 11회 | upstream 반영과 직접 손질이 다시 겹친 구간 |
| 2021-03 | 5회 | notes theme, grid, index 정리가 섞이는 전환기 |
| 2021-04 | 6회 | 큰 릴리스보다 짧은 재방문으로 끝맺는 후반 리듬 |

이 표는 `업데이트가 많다`는 인상만 주는 대신, **짧은 재방문이 실제로 언제 몰렸는지**를 다시 보게 해 준다.

## 버전별로 압축해서 읽으면 보이는 것

### 0.1.1은 `표면 확장 + 기반 유지`가 동시에 시작되는 구간이다
원문 `Features`에는 아래가 한꺼번에 섞여 있다.
- `add an seo component`
- `add sitemap, google custom search`
- `ADD adsense script`
- `ADD kakao adfit`
- `update index with grid`
- Gatsby starter 상속 항목들

겉으로는 기능 추가지만, 실제로는 `발견 가능성`, `수익 실험`, `표면 정리`, `기반 상속`이 동시에 움직이는 초반 확장기다.

### 0.1.1의 `Bug Fixes`는 실제로 유지 노동 묶음이다
여기에는 아래가 함께 들어 있다.
- `remove unnecessary fragment`
- `update 404 page`
- `Update popup color and position`
- dependency security fix
- 반복적인 `gatsby monorepo` 업데이트

즉 이 절은 단순 버그 수정 목록이 아니라, `직접 만진 표면`과 `기반이 요구한 상속 노동`이 같은 버전 껍데기 안에서 평평해지는 구간이다.

### 0.1.2는 큰 전환보다 생활형 update의 지속을 보여 준다
마지막 버전은 사실상 `update 21-04-20`, `🎸 update 21-04-23` 같은 짧은 반영이 중심이다.
- changelog의 끝이 거대한 선언으로 닫히지 않는다.
- 이미 운영 중인 웹 표면을 계속 다시 맞추는 리듬이 유지된다.
- 이 파일의 핵심이 대형 마일스톤보다 `지속적 손질`임을 다시 확인시킨다.

## 대표 커밋 묶음으로 보면 더 선명해지는 것

### 1. 유입면을 먼저 여는 묶음
- `add an seo component that can be used between pages/templates`
- `add sitemap, google custom search`

이 묶음은 블로그나 개인 웹을 `써 두면 누가 보겠지` 수준으로 두지 않는 태도를 보여 준다. 먼저 해야 할 일은 화면 장식보다 **검색엔진과 방문자가 들어올 진입면을 정리하는 것**이다.

### 2. 현실 수익을 시험하는 묶음
- `ADD adsense script`
- `ADD kakao adfit`

광고 스크립트는 부수 기능이 아니라, 공개 공간을 취미 표면에만 두지 않고 **현실 경제성과 함께 시험해 보는 운영 선택**이다. 다만 반복 횟수는 높지 않아, 수익화 자체가 주제의 전부가 아니라 `붙여 보고 판단하는 보조 실험`으로 읽는 편이 맞다.

### 3. 사람이 바로 체감하는 표면 손질 묶음
- `update index with grid`
- `update 404 page`
- `Update popup color and position`

이 항목들은 작은 수정처럼 보이지만, CHANGELOG가 백엔드 유지 일지에만 머물지 않는다는 증거다. 저자는 **들어오는 경험, 찾지 못했을 때의 경험, 방해받는 경험**까지 운영 범위로 본다.

### 4. 기반 선택이 만든 상속 노동 묶음
- `update gatsby monorepo`
- `update starters and examples`
- 각종 starter/plugin/dependency 업데이트

이 묶음은 가장 반복적이고, 그래서 가장 중요하다. 원문의 많은 줄은 `내가 만들고 싶은 기능`보다 **선택한 기반이 계속 요구하는 따라가기 노동**으로 채워져 있다. 개인 웹이라도 framework를 택한 순간, 운영자는 창작자이면서 동시에 upstream 추적자가 된다.

## CHANGELOG 안의 두 종류의 시간

| 시간 종류 | 원문에서 보이는 표식 | 읽는 방법 |
| --- | --- | --- |
| 생활 시간 | `update 20-10-26`, `update 21-04-14` 같은 날짜형 메모 | 오늘 다시 열어 손본 생활형 운영 박동 |
| 릴리스 시간 | `0.1.1`, `0.1.2`, compare 링크, `standard-version` 문구 | 변경을 버전 껍데기 안에 묶어 두는 배포 형식 |

이 둘이 어긋난다는 점이 중요하다. 문서는 릴리스 문법을 쓰지만, 실제 본문은 생활 시간의 촘촘한 재방문으로 움직인다. 그래서 이 CHANGELOG를 읽을 때는 `버전이 어떤 기능을 뜻하는가`보다 **버전 형식이 생활형 운영을 어떻게 수습하고 있는가**를 보는 편이 더 정확하다.

## 이 파일에는 서로 다른 출처의 로그가 함께 눌려 있다
원문을 다시 보면 CHANGELOG는 하나의 목소리로만 쓰이지 않는다. 적어도 세 층이 같은 릴리스 껍데기 안에서 섞인다.

| 출처 층 | 대표 항목 | mapped에서 읽는 법 |
| --- | --- | --- |
| 직접 손본 개인 웹 표면 | `update index with grid`, `update 404 page`, `popup color and position` | 방문자가 바로 체감하는 조정 기록 |
| 개인 운영 실험 | `adsense`, `adfit`, `seo component`, `google custom search` | 공개 공간을 어떻게 노출·수익화·검색화할지 시험한 흔적 |
| Gatsby 생태계 상속 로그 | `gatsby monorepo`, starter/plugin/docs/www 항목 다수 | 내가 고른 기반이 계속 밀어 넣는 유지 노동 |

이 구분이 필요한 이유는, 원문을 그대로 따라가면 `내 사이트를 위해 직접 한 일`과 `기반을 따라가느라 흡수한 일`이 모두 같은 `Features/Bug Fixes` 이름 아래 평평해지기 때문이다. 이 노드의 가치 중 하나는 바로 **그 평평해진 표면 아래의 책임 출처를 다시 분리해 보여 주는 것**이다.

## 이 changelog를 덜 평평하게 읽는 3층 분해

### 1. 직접 설계한 발행면 조정
SEO, sitemap, 검색, 광고, grid, popup 같은 항목은 저자가 웹 표면을 어떻게 보여 주고 싶은지 직접 드러낸다.
- 더 잘 발견되게 만들기
- 수익 실험을 어디까지 붙일지 조정하기
- 화면 표면을 계속 다듬기

### 2. 기반 선택이 만든 상속 노동
Gatsby monorepo, starter, plugin, dependency 업데이트는 대개 `하고 싶어서 한 일`보다 `계속 굴리기 위해 필요한 일`에 가깝다.
- 개인 웹도 프레임워크를 선택하는 순간 유지 일정을 함께 떠안는다.
- 그래서 upstream 항목은 잡음이면서 동시에 기반 비용의 증거다.

### 3. 생활형 운영 박동
날짜형 `update`, 짧은 `fix`, `remove`는 완결된 기능 설명보다 운영의 생활 리듬을 남긴다.
- 작은 마찰을 그냥 두지 않는다.
- 큰 선언보다 상시 손질이 익숙하다.
- changelog가 release note이면서 동시에 맥박 기록이 된다.

## 이 파일이 말하는 운영 시간축
원문은 세 시기로 압축해 읽을 수 있다.

| 시간축 | 대표 압력 | 의미 |
| --- | --- | --- |
| 초반 | SEO, sitemap, search, 광고 | 먼저 `보이게 만들기`와 `붙여 보기`가 앞선다. |
| 중반 | Gatsby/starter/plugin/dependency 갱신 다수 | 기반 선택이 장기 유지 일정을 만든다. |
| 후반 | 짧은 `update`, 생활형 반영 | 제품 출시보다 운영 리듬이 더 중요해진다. |

즉 흐름은 대체로 `발행면 확장 → 기반 유지 장기화 → 생활형 손질 지속`으로 읽힌다.

## `update` 묶음이 보여 주는 실제 운영 박동
원문에서 날짜형 `update` 표기는 단순한 빈 커밋명이 아니다. 특히 2020-10~12, 2021-02, 2021-04 구간에 짧은 업데이트가 몰려 있다는 점은 이 사이트가 `가끔 크게 개편하는 프로젝트`라기보다 `계속 열어 보고 계속 조금씩 만지는 표면`이었다는 증거에 가깝다.

- 큰 기능 출시가 없어도 changelog는 계속 움직인다.
- upstream 갱신과 생활형 수정이 같은 시기에 겹친다.
- 따라서 운영 피로는 대형 마이그레이션보다 `짧은 재방문이 끊기지 않는 상태`에서 더 선명하게 드러난다.

이 리듬은 [[Portfolio]]의 팀 규모 자동화와 대비된다. `Portfolio`가 큰 병목을 구조 재설계로 푸는 문서라면, `CHANGELOG`는 작은 마찰을 오래 방치하지 않는 생활형 운영 습관을 보여 준다.

## 이 노드가 보여 주는 숨은 비용 구조

### 작은 사이트도 가볍지 않다
원문에서 가장 무거운 층은 화려한 기능보다 유지보수다.
- 버전 업데이트
- 호환성 맞추기
- 보안 패치
- 빌드 안정화
- 의존성 정리

개인 웹이라고 해서 유지비가 작아지는 것이 아니다. 오히려 콘텐츠 양과 별개로 툴체인 결합도가 유지비를 만든다.

### 비용의 핵심은 돈보다 판단이다
이 파일이 반복해서 묻는 것은 결국 이런 질문들이다.
- 언제 업데이트할까?
- 광고를 어디까지 붙일까?
- 검색성과 복잡도 사이에서 어디까지 감수할까?
- 어떤 기반을 계속 따라갈까?

그래서 이 CHANGELOG는 서버 비용표보다 `반복 판단 비용`의 기록에 더 가깝다.

## 상속된 이력이 섞여 있다는 점이 중요하다
원문에는 개인 사이트의 직접 수정만이 아니라 Gatsby upstream성 항목이 대량으로 섞여 있다.
- `gatsby monorepo`
- starter / plugin / docs / blog / www 계열 항목
- 이슈 번호와 릴리스 문법

이 혼합은 오히려 중요한 정보다.
- 저자는 개인 웹을 완전한 맨바닥 구현보다 Gatsby 생태계 위에서 운영했다.
- 그 선택은 빠른 시작을 줬지만, 동시에 상속 로그와 유지비를 함께 가져왔다.
- 따라서 upstream 항목은 삭제해야 할 잡음이 아니라 `기반 선택의 무게`를 보여 주는 증거다.

## semantic versioning 껍데기와 실제 리듬의 차이
이 파일은 `0.1.1`, `0.1.2`, compare 링크, `standard-version` 문구를 갖고 있다. 형식만 보면 패키지나 제품 릴리스 기록처럼 보인다. 하지만 내부 리듬은 다르다.

- 버전 번호는 정돈된 배포 단위를 암시한다.
- 실제 내용은 수많은 `update`, `fix`, upstream 반영으로 채워져 있다.
- 특히 `starter`·`plugin`·`www`·`docs` 계열 항목은 개인 사이트 changelog라기보다 Gatsby 생태계 갱신 이력이 눌려 들어온 흔적에 가깝다.
- 즉 이 문서는 `제품 마일스톤 서사`보다 `릴리스 형식을 빌린 개인 운영 기록`에 가깝다.

이 간극이 중요하다. 저자는 사이트를 단순 감상문 공간으로 두지 않고, 버전과 diff가 남는 작은 배포 표면처럼 다뤘다.

### 빌려 온 릴리스 문법이 운영 기억을 압축하는 방식
맨 위의 `standard-version` 문구와 compare 링크는 원래 패키지/라이브러리 세계의 릴리스 문법에 더 가깝다. 그런데 원문 본문은 제품 로드맵보다 훨씬 생활형이고, 개인 웹 표면 손질과 Gatsby 상속 업데이트가 섞여 있다. 이 조합이 보여 주는 것은 다음과 같다.
- changelog 형식은 `거창한 버전 전략`의 증거라기보다, 자잘한 운영 반영을 잃지 않게 붙잡는 압축 장치다.
- 직접 한 수정이 짧고 비정형적일수록, 오히려 바깥의 릴리스 형식이 임시 질서를 제공한다.
- 그래서 이 파일의 가치 일부는 `무엇이 릴리스급 변화였는가`보다 `개인 운영의 작은 박동을 어떤 그릇에 담아 두었는가`에 있다.

즉 이 CHANGELOG는 semantic versioning을 엄격하게 수행한 기록이라기보다, **패키지 세계의 문법을 빌려 개인 웹 유지 노동을 붙잡아 둔 기록**으로 읽는 편이 더 정확하다.

이 점에서 [[Meta/My_space]]와도 닿는다. 다만 `My_space`가 처음부터 `복구 가능한 제어면`을 설계하는 노드라면, `CHANGELOG`는 그 이전 단계에서 **이미 생활형 업데이트를 버전과 diff의 형식으로 붙잡아 보던 습관**을 보여 준다. 즉 둘은 단절이 아니라 `생활형 웹 운영 → 자기 소유 운영면 설계`로 이어지는 시간차가 있다.

## 이 노드를 읽을 때 보존해야 할 긴장
- 개인 표현 vs 프레임워크 유지 비용
- 빠른 발행 vs 장기 안정성
- 수익화 실험 vs 사용자 경험
- 검색성 확대 vs 운영 복잡도
- 작은 사이트 감각 vs 실제 툴체인 부담

이 긴장을 지우면, 이 파일은 단순 커밋 목록으로 납작해진다.

## 8x8 루트 카테고리로서의 의미
CHANGELOG는 하위 기능 부록이 아니라 루트 노드다. 이유는 이 한 파일에 개인 웹 운영의 네 면이 모두 들어 있기 때문이다.

- **사람이 들어오는 면**: SEO, sitemap, search
- **현실을 붙여 보는 면**: 광고와 script 실험
- **사람이 실제로 만지는 면**: grid, 404, popup, 세부 화면 조정
- **그 표면을 살려 두는 바닥 면**: Gatsby와 dependency 유지

그래서 이 노드는 `개인 웹을 어떤 태도로 계속 굴렸는가`를 가장 직접적으로 압축해 보여 주는 루트 카테고리 노드로 남는다.

## 이번 패스에서 더 선명해진 점
- `Features` 81개보다 `Bug Fixes` 215개가 더 많다는 점은, 이 파일이 성장 기록보다 유지 기록에 더 가깝다는 해석을 수치로 뒷받침한다.
- 날짜형 `update`가 2020-10, 2020-11, 2021-02에 특히 몰린다는 점을 드러내니, `짧은 재방문이 끊기지 않는 운영 박동`이라는 기존 해석이 더 구체적인 시간축을 갖게 되었다.
- `standard-version`과 compare 링크는 제품급 릴리스 관리의 증거라기보다, 생활형 수정과 상속 로그를 붙잡아 두는 외부 문법으로 읽는 편이 맞다.
- 그래서 이 파일의 중심은 버전 번호 자체보다, `작은 운영 반영을 버리지 않기 위해 어떤 기록 그릇을 빌려 왔는가`에 있다.
- CHANGELOG를 읽을 때는 커밋 내용만이 아니라 `그 내용을 어떤 형식으로 저장하려 했는가`도 함께 봐야 개인 웹 운영 감각이 덜 평평해진다.
