# AI

- 원본: `source/shdkej-content/Deep Knowledge/AI.md`
- 상태: 점진 심화, 2026-04-19 갱신
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 논지
이 노드의 `AI`는 기술 계보 요약에 머물지 않는다. 원문은 AI를 크게 네 층으로 동시에 다룬다. 첫째, **AI · ML · DL · LLM의 개념 지도**. 둘째, **LLM 이후 추천, 생성, 에이전트, RAG가 일과 제품을 어떻게 바꾸는가**. 셋째, **문서, 메모리, 프롬프트, 워크플로를 통해 AI를 실제 작업자로 쓰는 법**. 넷째, **서비스로 굴릴 때 필요한 검증, 비용, 재시도, 비동기, 운영 복원성**이다. 그래서 이 페이지는 이론 노트이면서도 개인 AI 운영체계 설계 노드다.

## 이 노드를 카테고리 노드로 읽는 기준
원문은 범위가 넓고 메모가 촘촘해서 산만해 보이지만, 주제는 흩어지지 않는다. 이 노트는 `AI`를 하나의 모델 설명서로 쓰지 않고 아래 네 질문이 겹치는 상위 카테고리로 쓴다.

1. AI는 무엇이며, ML/DL/LLM과 어떻게 이어지는가?
2. LLM 이후 일의 구조와 제품 구조는 어떻게 바뀌는가?
3. AI를 안정적으로 쓰고 굴리려면 어떤 문서와 운영 루프가 필요한가?
4. 모델 바깥의 기억, 도구, 검증, 서비스 복원력은 어떻게 설계해야 하는가?

즉 이 노드의 중심은 모델 성능 자랑이 아니라, **AI를 이해하고, 위임하고, 통제하고, 운영 가능한 상태로 묶는 법**이다.

### 이 노드를 읽는 4개 층위
- **모델 개념 층**: AI / ML / DL / LLM / Transformer / 학습 방식 같은 바닥 개념
- **작업 위임 층**: 에이전트, MCP, 메모리, 프롬프트, 역할 분리, 문서 기준점
- **서비스 운영 층**: 비용, 레이턴시, 비동기, 재시도, 로그, 가드레일, UX
- **학습·제품화 층**: RAG, 벡터 스토어, 파인튜닝, AI Stack, 데이터 분류와 평가

이 네 층을 같이 잡아야 원문의 넓은 범위가 잡담으로 흐르지 않고, 왜 `LLM 세팅 메모`와 `트랜스포머`, `서비스 체크리스트`가 한 파일 안에 함께 있는지도 자연스럽게 읽힌다.

## 원문 발췌 구조
원문은 깔끔한 교재 순서보다, 실제로 저자가 AI를 배우고 굴리며 부딪힌 층위를 넓게 펼쳐 놓은 형태에 가깝다. 특히 아래 순서를 보존해야 이 노드가 `AI 일반론`으로 평탄화되지 않는다.

1. `AI > ML > DL`과 `Programming vs ML` 비교로 시작하는 개념 지도
2. 추천 시스템, 생성형 AI, 에이전트, ReAct, MCP, 메모리 같은 LLM 이후의 작업 구조 변화
3. 실제 에이전트 세팅 메모와 Slack MCP, 사용량 확인, 완료 확인 자동화 같은 현장 운영 질문
4. `체화해야 할 것`과 `AI에게 맡길 것`을 가르는 사용 원칙
5. 문서 길이, 파일 수, memory.md, claude.md, skill description 같은 외부 기억 설계 기준
6. PRD, API 문서, 요구사항 ID, 테스트 연계, hooks, slash commands, skills 같은 AI 친화적 개발 문법
7. 프롬프트 노하우, 모델 선택, 이미지 프롬프트, 적극성/깊이/길이 조절법
8. 컨텍스트 길이, RAG, 벡터 스토어, 청킹, 파인튜닝, 학습 데이터 준비에 대한 현실적 고민
9. AI 서비스 운영 체크리스트: 재시도, 가드레일, 비동기, 로그, 네트워크, UX, 복구 가능성
10. ML, 트랜스포머, MLOps, 데이터 분석 메모로 내려가는 기초 학습층

## 개념 지도, AI > ML > DL > LLM
원문 첫머리의 `AI > ML > DL` 구도는 이 노트의 기본 좌표계다.

### AI
가장 넓은 범주다. 지능적 문제 해결 전반을 가리킨다.

### 머신러닝
사람이 규칙을 직접 모두 짜는 대신, 데이터와 결과를 통해 모델을 학습시키는 접근이다.
- 전통적 프로그래밍: `Algorithm + Data => Result`
- 머신러닝: `Result + Data => Model`, 이후 `Data => Predictions`

### 딥러닝
신경망을 여러 층으로 쌓아 더 복잡한 패턴을 처리하는 방식이다. 지도학습, 비지도학습, 강화학습과 결합해 표현 학습의 범위를 넓힌다.

### LLM
언어를 다루는 대규모 모델 계열이다. 이 노트에서 LLM은 모델 한 종류라기보다, 이후의 RAG, Vector Store, MCP, Agent, Prompting, Workflow를 다시 설계하게 만든 중심 축으로 등장한다.

## LLM 이후의 변화, 반응형에서 의도 이해로
원문은 추천 시스템 변화에서 중요한 전환점을 짚는다.
- 과거: 패턴 매칭, 반응형 추천
- 이후: 의도 이해, 고객 맥락 이해, 선제적 제안

저자가 보는 AI의 가치는 단순 생성 능력보다 **사용자 의도와 현재 상황을 읽는 능력**에 있다. 그래서 이 노트는 모델 benchmark보다도, 맥락 전달과 작업 구조 설계에 더 큰 비중을 둔다.

## 생성형 AI의 진화 방향, 자율성보다 설계 책임
원문은 생성형 AI의 진화 방향을 다음처럼 적는다.
- 어시스턴트: 규칙 기반
- 에이전트: 단일 목표, 광범위한 작업
- 에이전틱: 완전 자율, 다중 에이전트

여기서 중요한 것은 능력의 확장보다 책임의 확장이다. 같은 문단 주변에 레이턴시, 비용, 불필요한 호출, 메모리, 툴, ReAct가 붙어 있는 이유도 여기에 있다. 즉 자율성이 높아질수록 더 중요해지는 것은 자유가 아니라 **관측, 제한, 분리, 재시도 설계**다.

## 에이전트는 행동 시스템이다
원문은 에이전트를 짧게 정의한다.
- 액션을 하고 관찰을 한다
- 메모리와 툴이 필요하다

이 문장이 이 노트의 운영 철학을 압축한다.

### 에이전트 설계에서 반복되는 판단 축
- 초기 레이턴시를 감당할 수 있는가
- 비용이 과도하게 불어나지 않는가
- 불필요한 호출을 줄였는가
- 생각과 행동을 분리할 것인가
- 메모리와 툴을 어디까지 허용할 것인가
- 완료 확인, 로그 확인, 상태 복원을 어떻게 할 것인가

특히 `ReAct` 언급은 에이전트를 단순 생성기가 아니라 **단계적 추론과 행동을 오가는 작업자**로 보고 있음을 보여준다.

## AI 사용법, 맡길 것과 체화할 것을 나누기
이 노트에서 실전성이 가장 잘 드러나는 부분이다. 저자는 AI 활용의 핵심을 "무엇을 많이 시킬까"보다 **무엇을 직접 붙들고 무엇을 위임할까**로 본다. 여기서 중요한 목적은 단순 위임이 아니라, 사람이 최종 결정만 하면 되는 상태까지 작업면을 정리해 두는 것이다.

### 사람이 붙들어야 하는 것
- 디자인 레퍼런스처럼 직접 보고 느껴야 하는 감각
- 최종 go / no-go 판단
- 경험을 어떤 문서와 기준으로 남길지의 결정
- 빠짐을 막는 책임
- 서비스에 실제로 넣을 요구사항과 금지사항

### AI에게 위임할 수 있는 것
- 초안 작성과 구조화
- 역할별 시뮬레이션
- 자료 탐색과 빠른 비교
- 누락 점검과 체크리스트 생성
- 로그와 문서를 기반으로 한 재정리

즉 저자에게 AI는 대체자보다 **결정 가능한 상태를 빨리 만들어 주는 인턴, 운영자, 공동작업자**에 가깝다. `빠짐을 막게 하라`, `결정만 하면 되는 구조로 가야 된다`는 문장은 이 노드의 위임 철학을 가장 압축해서 보여 준다.

## 문서가 곧 AI의 기억이다
원문 전반에 가장 강하게 반복되는 믿음은 "AI를 잘 쓰려면 결국 문서화해야 한다"는 점이다. 특히 이 노드에서 기억은 추상적인 `long-term memory` 개념이 아니라, 실제로 에이전트가 다시 읽고 이어받을 수 있는 파일군으로 상정된다.

### 문서 관리 기준
- 100줄: 파일 열기 전 확인 가능한 양
- 300줄: 최적, 100줄 이상이면 목차 필요
- 500줄: 최대
- 500개 파일: 인덱스 파일로 관리
- 1,000개 이상: RAG 같은 의미 기반 탐색 고려

이 기준은 단순 정리 팁이 아니다. 저자는 문서 길이와 파일 수를 **AI가 실제로 다룰 수 있는 컨텍스트 구조**와 연결해 본다.

### 메모리 관리 신호
- 스킬 문서는 description까지 읽는다
- `memory.md`
- `claude.md`
- 규칙, 계획, 히스토리 로그를 별도 문서로 둔다
- `TECH_SPEC.md`, `LLM.md`, `CLAUDE.md` 같은 기준 파일을 심볼릭 링크와 워크플로로 연결한다

즉 기억은 모델 내부에 기대는 것이 아니라, 외부 문서 구조와 파일 계층으로 설계해야 한다는 입장이다.

### 원문이 상정하는 최소 공통 문서군
원문에 흩어진 메모를 모으면, 저자가 AI 작업 환경의 바닥으로 상정하는 문서군은 꽤 선명하다.
- **rules**: 반드시 지켜야 하는 전역 지침
- **agents / agents.md**: 역할, 프로젝트, 협업 기본 안내
- **plan**: 지금 무엇을 어떤 순서로 할지 고정하는 작업면
- **spec / PRD / API 문서**: 코드보다 먼저 바꾸는 기준점
- **memory / history / log**: 이전 판단과 진행 상태를 다시 불러오는 외부 기억
- **skills / hooks / slash commands**: 반복 행동을 재사용 가능한 인터페이스로 고정하는 층

이 정리는 중요하다. 원문에서 AI는 대화 상대가 아니라, **문서 세트를 읽고 그 문서 위에서 행동하는 작업자**에 가깝기 때문이다.

## AI 친화적 개발, 코드보다 기준 문서를 먼저 둔다
원문의 `AI 친화적 개발`, `LLM이 쓰는 문서`, `바이브 코딩` 메모는 같은 흐름으로 묶인다. 핵심은 AI가 코드를 직접 만지는 것보다 먼저, 어떤 문서를 기준점으로 읽고 어디에 수정 흔적을 남겨야 하는지를 정하는 데 있다.

특히 이 노드에서는 `개발을 AI에게 맡긴다`는 말이 곧바로 `기준 문서, 테스트, 진행 로그, 완료 지점, 다시 시작 규칙을 먼저 세운다`는 뜻으로 이어진다. 즉 생산성 향상보다 **재시작 가능한 개발 상태**가 먼저다.

### 기준 문서 우선
- PRD와 API 문서를 기준점으로 둔다
- 수정은 이 문서를 먼저 바꾸고 코드와 연계한다
- 테스트 코드도 PRD 변경과 연결되어야 한다
- 요구사항은 `REQ-ACTION-001` 같은 식별자를 가진다

### LLM 작업 환경의 공통 문서군
- rules: 시스템 지침
- agents.md: 기본 프로젝트 안내서
- plan: 작업 기준 문서
- hooks: 응답 후 실행 규칙
- agents: 서브 에이전트 분리
- slash commands: 반복 지시의 커맨드화
- skills: 외부 기능 확장

즉 AI는 코드 생성기라기보다 **문서에 의해 안내되는 작업 엔진**이다. 이 점은 [[../../mapped/Meta/AGENTS]]와 직접 맞닿는다.

## 프롬프트 노하우, 잘 시키는 법은 잘 설명하는 법이다
원문 곳곳의 프롬프트 메모는 몇 가지 운영 원칙으로 다시 묶을 수 있다.

### 1. 아는 것과 모르는 것을 구분해 설명한다
일반 지식은 짧게, 프로젝트 특수 맥락은 길고 구체적으로 준다.

### 2. 역할과 평가 기준을 함께 준다
- 전문가 토론 형식
- 학습 로드맵 제시
- 쉬운 설명 수준 지정
- 프롬프트 개선 전 질문 먼저 받기

### 3. 적극성, 깊이, 길이를 조절한다
저자는 AI가 너무 알아서 하게 둘지, 좁게 제한할지를 제어해야 한다고 본다. 좋은 프롬프트는 내용 요청만이 아니라 **행동 성향 제어**를 포함한다.

### 4. 사고를 외부화하게 한다
단계별 사고, 검증 과정 출력, 체크리스트 요구는 단지 설명을 듣기 위한 것이 아니라, 작업의 빈틈을 줄이기 위한 장치다.

### 5. 프롬프트 작성은 자기 기준을 다시 배우는 일이다
`프롬프트를 쓰면서 오히려 더 배우게 된다`는 메모는 중요하다. 프롬프트 작성은 명령 기술이 아니라, 자신의 직관과 판단 기준을 다시 언어화하는 과정으로 읽힌다.

## AI를 쓸 때의 공통 운영 루프
원문 여러 조각을 합치면 저자가 선호하는 운영 루프가 보인다.

1. 해야 할 판단과 위임 범위를 먼저 나눈다.
2. rules, plan, spec 같은 기준 문서를 준비한다.
3. 작은 태스크 단위로 실행시킨다.
4. 로그와 중간 진행 상황을 보게 한다.
5. 테스트, 검증, 완료 확인을 별도 단계로 둔다.
6. 히스토리와 프롬프트를 다시 문서로 환원한다.

즉 AI 활용은 단발성 질문보다 **문서화된 작업 루프**에 가깝다.

### 작업 루프를 서비스 루프로 확장하는 감각
원문 후반의 메모를 보면 저자는 여기서 멈추지 않는다. 개인 생산성 차원을 넘어, `자연어를 받아들여서 우리의 툴로 연결한다`, `우리의 툴이 자연어에 적합하게 동작하게 한다`는 식으로 AI 프로덕트의 진화 방향도 함께 본다. 즉 좋은 AI 활용은 프롬프트 1회 성공보다, **문서화된 작업 루프가 결국 제품 인터페이스로 자라나는 과정**에 더 가깝다.

## RAG, 벡터 스토어, 긴 컨텍스트에 대한 현실 감각
이 노트는 RAG를 과장하지 않는다. 원문은 개인 노트를 읽는 AI, 학습시킬 수 있는 개인용 AI, S3·벡터DB·명령어 학습 같은 상상을 적어 두지만, 동시에 그 장치들이 어디까지 되고 어디서부터 안 되는지도 계속 되묻는다.

### 컨텍스트 길이에 대한 태도
- 모델마다 컨텍스트 한계가 다르다
- 최대치까지 넣는다고 성능이 유지되지는 않는다
- 성능 포화점은 최대치보다 낮은 지점에 있을 수 있다
- 토큰이 늘면 비용도 늘어난다

### 벡터 스토어와 RAG에 대한 태도
- 유사한 정보를 찾는 데는 유용하다
- 청킹 방식이 중요하다
- 확장 장치가 될 수는 있다
- 그러나 문맥 전체를 진짜로 이해하게 만드는 장치는 아니다

이는 `Data` 노드의 검색 감각과 이어진다. 즉 저자는 RAG를 만능 기억장치보다 **제한 있는 검색 보조 계층**으로 본다.

### 원문이 끝까지 놓지 않는 혼동 지점
후반 메모에는 `많은 파일을 한 번에 읽히게 할 수는 없다`, `학습시킬 때는 많은 데이터가 있어야 하는데 새로운 정보는 많으면 안 된다`, `RAG는 확장장치가 될 순 있지만 문맥을 이해하게 만드는 장치로는 쓸 수 없다`는 식의 혼란과 재정리가 그대로 남아 있다. 이 흔적은 중요하다. 이 노드는 이미 정답을 다 이해한 설명서가 아니라, **긴 컨텍스트·RAG·파인튜닝·체인 설계를 어디까지 믿어야 하는지 계속 시험하는 운영 학습 노드**이기 때문이다.

## AI 제품을 만들 때 반복해서 보는 체크리스트
원문 전반의 실무 메모를 묶으면, 저자가 AI 제품을 볼 때 반복적으로 확인하는 항목이 선명하다.

### 서비스 수준의 질문
- 결과가 다음 호출에 누적되는가
- 실패 시 재시도 가능한가
- 서비스가 죽었다 살아나도 이어서 처리 가능한가
- 긴 작업을 비동기로 처리하는가
- 사용자가 프롬프트와 상태를 볼 수 있는가

### 품질과 통제의 질문
- 할루시네이션을 어떻게 제어하는가
- 필수 규칙 위반을 어떻게 감시하는가
- 어려운 요구사항일수록 검증을 어떻게 붙일 것인가
- 복잡한 레이아웃, 이미지, 정밀 제어를 어디까지 믿을 것인가

### 운영과 비용의 질문
- 호출 비용과 지연 시간이 감당 가능한가
- 로그와 디버깅 경로가 충분한가
- 네트워크 트래픽과 배치 작업을 어떻게 설계할 것인가
- 기존 시스템에 덧붙는 분기 복잡성을 어떻게 억제할 것인가

즉 이 노트에서 AI는 "잘 생성하는 모델"보다 **운영 가능한 서비스 컴포넌트**에 가깝다.

## 머신러닝과 트랜스포머 메모의 위치
원문 하반부는 머신러닝, 트랜스포머, 데이터 분석, MLOps 메모가 넓게 펼쳐져 있다. 이 부분은 깊은 연구 노트라기보다, LLM 실전 감각을 떠받치는 **기초 재료 저장층**으로 읽는 편이 맞다.

### 왜 기초 ML 메모가 이 노드 안에 남아 있는가
저자는 AI를 순수 사용자 도구로만 다루지 않는다. `temperature`, `attention`, `back propagation`, `training / serving`, `h5`, `pb`, `TensorFlow Serving` 같은 메모를 함께 둔 것은, 지금 쓰는 LLM과 AI 서비스가 어떤 종류의 기술적 제약 위에 서 있는지 잊지 않으려는 태도에 가깝다. 즉 이 층은 전문 ML 교재라기보다, **마법화 방지용 바닥 지식**이다.

특히 `스포티파이는 단순 패턴이 아닌 인과관계를 찾는다`, `attention은 문맥에 맞게 단어 방향을 조금씩 이동시킨다`, `training / serving을 분리한다` 같은 메모는 개념 백과사전을 만들려는 의도보다, 지금 쓰는 AI 시스템을 너무 마법처럼 보지 않기 위한 바닥 지식으로 보존되어 있다.

### 남겨둘 기초 축
- 지도학습, 비지도학습, 강화학습
- 회귀와 분류
- 데이터 클리닝, 정규화, 데이터 분리, 교차검증
- attention, embedding, parameter, temperature, back propagation
- training / serving 분리
- TensorFlow Serving, `.h5`, `pb`, weight, optimizer
- 데이터 분석, 시각화, 통계 검정

중요한 점은 이 기초 메모가 독립된 ML 교재를 만들려는 시도라기보다, **나중에 다시 연결할 실무 발판**이라는 점이다.

## AI Stack 메모를 이 노드 안에서 읽는 법
원문 말미의 `AI Stack` 메모는 짧지만 중요하다. 반도체 → 클라우드 → foundation model → application으로 이어지는 층을 따로 적어 둔 것은, AI를 단일 모델 경쟁으로 보지 않고 여러 공급층이 겹친 산업 구조로 본다는 뜻이다.

이 짧은 메모 덕분에 이 노드는 단순 사용 팁 모음을 넘어선다. 즉 저자에게 AI는 프롬프트 몇 줄의 문제가 아니라, 하부 인프라·모델·애플리케이션이 함께 움직이는 스택이며, 자신은 그중 특히 `애플리케이션과 운영 인터페이스` 층에서 무엇을 설계할지를 고민하고 있다.

## 이 노드에서 특히 보존해야 할 긴장

### 자동화 욕망 vs 통제 욕망
AI에게 많이 맡기고 싶어 하지만, 동시에 plan, check, verify로 제어하려 한다.

### 많은 맥락 vs 제한된 컨텍스트
많이 알려줘야 잘하지만, 너무 많이 넣으면 비용과 성능이 무너진다.

### 빠른 생성 vs 검증 비용
만드는 속도는 빠르지만, 어려운 요구일수록 검증이 더 비싸진다.

### 범용 모델 vs 작업 특화 시스템
기성 LLM만으로는 컨벤션 유지가 어렵고, 문서·훅·스킬·메모리 같은 외부 구조가 필요하다.

이 긴장들을 보존해야 이 노드가 단순 도구 예찬으로 납작해지지 않는다.

## 이 노드에서 드러나는 저자의 AI관
1. AI는 강력하지만 문서와 기준 없이 쓰면 쉽게 흐트러진다.
2. 좋은 AI 활용은 모델 선택보다 작업 구조 설계에 더 크게 좌우된다.
3. AI는 사람 판단을 대체하기보다, 판단 가능한 상태를 빨리 만들어야 한다.
4. 에이전트, RAG, 긴 컨텍스트는 유용하지만 모두 한계와 비용을 인정한 채 써야 한다.
5. 결국 중요한 것은 잘 묻는 법보다, **어떤 문서 시스템과 운영 루프 안에서 묻는가**다.

- [[../../mapped/Deep Knowledge/Data]]: 청킹, 벡터 스토어, 검색, 저장-조회 분리 감각과 맞닿는다.
- [[../../mapped/Deep Knowledge/Devops]]: 배포, 운영, 로그, 재시도, 자동화 관점에서 이어진다.

특히 [[../../mapped/Meta/AGENTS]]를 함께 읽으면, 이 AI 노드의 `rules / plan / skills / hooks / agents` 메모가 단순 툴 목록이 아니라 역할 기반 운영체계의 하부 층이라는 점이 더 선명해진다. 또한 [[../../mapped/Deep Knowledge/Data]]와 나란히 보면, 긴 컨텍스트와 RAG를 `무한 기억`으로 상상하기보다 `제한 있는 검색 보조 계층`으로 보는 태도가 두 노드에서 일관되게 반복된다는 점도 확인할 수 있다.

## 선호 신호
저자는 AI를 신기한 모델보다 문서 기반 작업 체계의 일부로 보는 경향이 강하다. 또한 프롬프트 기교만으로 문제를 풀기보다, 기준 문서, 역할 분리, 로그 확인, 테스트, 복원 가능 상태를 함께 설계하려 한다. 기술적으로는 최신 용어와 제품을 빠르게 흡수하지만, 실제 판단은 늘 비용, 검증, 운영 가능성, 맥락 전달로 돌아온다.

특히 이 노드에서는 `많이 시키는 법`보다 `어디까지 맡기고 어디서 사람이 결정권을 회수할지`를 더 중요하게 보는 성향이 강하다. AI를 잘 쓰는 사람의 조건도 모델 지식 그 자체보다, 기준을 문서로 밖에 꺼내고 검증 루프를 붙일 수 있는 운영 감각에 가깝다.

## 정리 메모
- 기존 baseline은 유지하되, 이 노드를 `모델 개념 + 작업 위임 + 서비스 운영 + 학습·제품화`의 4층 카테고리로 더 선명하게 재정리했다.
- 에이전트 설계, 문서 메모리, 공통 문서군, AI 친화적 개발, 프롬프트 운영, 제품 체크리스트를 분리해 읽기 쉽게 보강했다.
- 원문 하반부의 ML/MLOps 메모와 `AI Stack` 메모는 삭제하지 않고, LLM 실전 감각을 떠받치는 기초 재료층과 산업 스택 인식으로 재배치해 해석했다.
