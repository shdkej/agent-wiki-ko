# Container

- 원본: `source/shdkej-content/Deep Knowledge/Container.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 관리 노트

## 핵심 요약
이 노드는 컨테이너를 단순한 패키징 도구로 보지 않는다. 원문은 컨테이너를 **격리된 실행 환경을 표준화하고, 개발·배포·운영·복구 루프를 다시 만들 수 있게 하는 운영 인터페이스**로 다룬다. 관심사는 Docker 명령 암기보다 이미지와 볼륨의 역할 분리, Dockerfile 신뢰성, stdout 중심 로그 수집, 쿠버네티스의 추상화 계층, 그리고 핫픽스·롤백까지 감당하는 배포 체계를 어떻게 설계할지에 모여 있다.

## 이 노드가 실제로 답하려는 질문
- 컨테이너는 왜 VM보다 가볍고 이식 가능한가?
- 개발 단계와 배포 단계에서 이미지와 볼륨을 어떻게 다르게 써야 하는가?
- Dockerfile은 단순 빌드 스크립트가 아니라 어떤 운영 계약을 담아야 하는가?
- 쿠버네티스는 컨테이너 위에 어떤 복잡성을 추가하고, 또 무엇을 단순화하는가?
- 실서비스에서 중요한 것은 서버 직접 수정이 아니라 재빌드·재배포·롤백 가능한 루프 아닌가?

## 핵심 논지
1. **컨테이너의 본질은 격리와 재현성이다.**
   - namespace, cgroups, overlayFS, chroot 같은 리눅스 기능을 조합해 프로세스와 자원을 분리한다.
   - VM처럼 OS 전체를 복제하지 않으면서도 실행 환경을 일정하게 묶는다.
2. **이미지와 볼륨은 같은 저장 단위가 아니라 다른 목적의 계약이다.**
   - 이미지는 배포 가능한 결과물이다.
   - 볼륨은 개발 중 빠른 수정과 실험을 위한 인터페이스다.
   - 그래서 원문의 판단은 반복해서 `개발은 volume, 배포는 image`로 수렴한다.
3. **Dockerfile은 보안·속도·추적 가능성이 교차하는 운영 문서다.**
   - 무엇을 설치했고 무엇이 바뀌었는지 남긴다.
   - 체크섬 검증, non-root 사용자, 레이어 정리, multi-stage 분리는 운영 신뢰성과 직접 연결된다.
4. **쿠버네티스는 컨테이너를 많이 돌리기 위한 제어 계층이다.**
   - 배포, 스케일링, 헬스체크, 서비스 디스커버리, 권한 관리, 롤백을 다룬다.
   - 하지만 복잡성을 제거한다기보다 더 높은 수준의 복잡성으로 재배치한다.
5. **실서비스에서 중요한 것은 서버 수정 편의가 아니라 재배포 가능성이다.**
   - 실서버 직접 수정은 빠르지만 재현성과 신뢰성을 무너뜨린다.
   - 그래서 핫픽스와 롤백이 빠른 파이프라인이 필요하다는 결론으로 이어진다.

## 구조 재정리

### 1. 컨테이너의 기반 원리
원문은 컨테이너를 설명할 때 기술 스택을 아래처럼 쌓아 올린다.

- `chroot`: 루트 파일시스템을 분리한다.
- `namespace`: pid, net, mnt, ipc, uts, user 등 자원을 격리한다.
- `cgroups`: CPU, 메모리, 디스크, 네트워크 사용량을 제한한다.
- `overlayFS`: 이미지 레이어와 컨테이너 변경 레이어를 분리한다.

이 배열은 `컨테이너가 왜 가볍고 portable한가`에 대한 실제 기술적 설명이다. 즉 이 노드는 컨테이너를 마법처럼 보지 않고 **리눅스 커널 기능을 조합한 결과물**로 이해하려고 한다.

### 2. Docker를 보는 관점
원문은 Docker를 편의 도구로만 보지 않는다.

- 장점: isolation, dependency 분리, portability, light-weight
- 한계: 버전 차이, 거대한 단일 프로세스 구조, `docker.sock` 위험
- 대안 흐름: OCI, runC, cri-o, podman, Buildah, skopeo

여기서 중요한 패턴은 특정 도구 충성도가 아니라 **표준화와 역할 분리**다. Docker가 시작점이지만, 관심은 점차 OCI 준수, 런타임 분리, 빌드/실행/배포 역할의 해체 쪽으로 이동한다.

### 3. 개발 환경과 배포 환경의 분리
원문에서 가장 자주 되돌아오는 실무 문제는 이것이다.

- 개발할 때는 코드를 즉시 반영하고 싶다.
- 배포할 때는 고정된 이미지로 안정적으로 올리고 싶다.
- compose와 Dockerfile, volume과 COPY, build target과 runtime target이 여기서 충돌한다.

정리하면 다음 판단으로 읽힌다.

| 상황 | 우선 수단 | 이유 |
|---|---|---|
| 로컬 개발 | volume + compose | 빠른 수정, 즉시 반영 |
| 테스트/빌드 | Dockerfile multi-stage | 재현 가능한 빌드 |
| 배포 | 고정 이미지 + 태그 | 추적 가능, 롤백 가능 |
| 운영 수정 | 서버 직접 수정 지양 | 재현성 붕괴 방지 |

이 노드의 실전 메시지는 분명하다. **개발 편의성과 배포 안정성은 같은 방식으로 얻지 않는다.**

### 4. 로그와 운영 가시성
원문 초반에 stdout, `docker logs`, Fluentd, Elasticsearch, nginx 로그 설정이 먼저 등장하는 점이 중요하다. 이는 컨테이너를 단순 실행 단위가 아니라 **관측 가능한 실행 단위**로 본다는 뜻이다.

- 로그는 파일보다 stdout으로 내보내고 수집하도록 설계한다.
- 애플리케이션과 인프라의 경계를 나누고, 수집은 외부 시스템이 담당한다.
- 쿠버네티스에서도 이 관점은 DaemonSet 로그 수집, metrics-server, Prometheus, Grafana 같은 운영 스택으로 이어진다.

즉 `컨테이너를 띄운다`는 말은 사실상 `로그와 상태를 밖에서 읽을 수 있게 만든다`까지 포함한다.

### 5. Dockerfile을 보는 기준
원문은 Dockerfile을 세세하게 적어 두었는데, 반복되는 기준은 다음과 같다.

- 체크섬 검증으로 신뢰성을 높인다.
- root 대신 일반 사용자를 기본값으로 둔다.
- 잘 변하지 않는 설치 단계는 묶어서 레이어를 줄인다.
- 빌드 산출물과 런타임 산출물을 분리한다.
- `docker.sock` 노출 위험을 경계한다.
- 개발용 target과 배포용 final image를 구분한다.

이 기준은 단순 최적화가 아니라 **보안, 속도, 재현성의 교차점**에 있다. 저자는 Dockerfile을 `이미지 만드는 파일`보다 **운영 원칙이 새겨지는 경계면**으로 다룬다.

## 원문 발췌 구조
원문 전체의 실제 배열을 다시 접어 보면, `Container`는 도구 사용법보다 **실행 환경을 고정하고 다시 배포할 수 있게 만드는 사다리**로 전개된다.

- **도커 기본 성질**: isolation, portability, light-weight 같은 장점과 LXC/runC/OCI 맥락.
- **로그와 실행 관측**: stdout, `docker logs`, Fluentd, Elasticsearch, nginx 로그 출력 방식.
- **컨테이너 내부 원리**: chroot, cgroups, overlayFS, namespace.
- **Dockerfile 규율**: 체크섬, non-root, 레이어 최소화, multi-stage, `docker.sock` 경계.
- **개발/배포 충돌 메모**: volume vs COPY, compose vs image, build target vs final image.
- **배포 루프 설계**: 핫픽스, 롤백, GitHub Actions, ansible, 이미지 버저닝.
- **쿠버네티스 운영 메모**: 리소스, 네트워크, ingress, 권한, 테스트, 멀티클러스터.
- **외부 도구 확장층**: Helm, k3s, ArgoCD, Istio, Spinnaker, GlusterFS, EKS.

이 배열을 보면 원문은 `Docker 소개 → Kubernetes 소개` 식의 교재형 노트가 아니다. 오히려 **개발에서 운영까지 끊기지 않는 재현성 확보 문제**를 도구층마다 다시 메모한 기록에 가깝다.

## 컨테이너를 읽는 운영 루프
이 노드의 실전 감각은 아래 루프로 압축하는 편이 재진입성이 높다.

1. **격리한다** — 실행 환경을 이미지와 네임스페이스로 고정한다.
2. **보이게 한다** — 로그와 상태를 stdout·메트릭·외부 수집 계층으로 뽑아낸다.
3. **나눈다** — 개발용 volume, 배포용 image, 권한용 계정, 네트워크용 service처럼 책임을 분리한다.
4. **되돌릴 수 있게 한다** — 태그, rollout history, hotfix 규칙, 롤백 경로를 남긴다.
5. **증설한다** — 단일 서버에서 시작해 Kubernetes, ingress, autoscaling, multi-cluster로 확장한다.

이 루프를 기준으로 보면 원문 후반의 방대한 쿠버네티스 메모도 산만한 부록이 아니라, `격리된 실행 단위`를 실제 서비스 운영체계로 키우는 다음 단계로 자연스럽게 연결된다.

## 쿠버네티스 파트의 의미
원문 후반의 대부분은 사실상 쿠버네티스 운영 노트다. 여기서 읽히는 핵심은 다음과 같다.

### 쿠버네티스가 제공하는 것
- 배포와 업데이트 자동화
- 스케일 조정
- 서비스 디스커버리
- 롤링 업데이트와 롤백
- Service, Ingress, LoadBalancer 추상화
- RBAC, ServiceAccount 등 권한 관리
- namespace, controller, operator를 통한 구조화

### 쿠버네티스가 새로 만드는 복잡성
- 네트워크 계층 이해 필요 (`Service`, `NodePort`, `Ingress`, `kube-proxy`, `netfilter`)
- 저장소와 볼륨 관리 복잡성 증가
- 보안 정책과 권한 모델이 어려워짐
- 디버깅 경로가 길어짐
- 클러스터 규모가 커질수록 control plane 한계와 운영 부담이 드러남

따라서 이 노드는 쿠버네티스를 `다 해결해 주는 플랫폼`으로 보지 않는다. 오히려 **단일 서버의 수작업 문제를 더 큰 규모에서 관리 가능한 규칙으로 바꾸는 도구**로 본다.

## 쿠버네티스를 이루는 운영 층위
원문에 흩어진 메모를 다시 묶으면, 쿠버네티스는 대략 네 층으로 읽힌다.

### 1. 리소스 층
- Pod, ReplicaSet, Deployment, Service
- ConfigMap, Volume, hostPath
- Role, RoleBinding, ClusterRole, ServiceAccount

이 층은 `무엇을 실행하고 무엇을 노출하며 누가 만질 수 있는가`를 결정한다.

### 2. 네트워크 층
- Pod 간 통신
- Service의 추상화
- NodePort와 외부 노출
- Ingress와 Ingress Controller
- Istio Gateway, VirtualService, DestinationRule

원문은 특히 서비스 네트워크를 `팟은 계속 바뀌므로 고정된 접근 지점을 별도로 둬야 한다`는 문제로 이해한다. 즉 네트워크는 연결 기술이 아니라 **변하는 실행 대상을 안정된 이름으로 묶는 기술**이다.

### 3. 배포 층
- `kubectl apply`
- 이미지 태그 교체
- `kubectl rollout history`
- 카나리, 롤링 업데이트, 롤백
- GitHub Actions, ArgoCD, Spinnaker, Helm

이 층에서 중요한 것은 툴 이름보다 **배포 이력이 남고 되돌릴 수 있으며 사람 손실 없이 반복 가능한가**다.

### 4. 운영 확장 층
- 모니터링
- 로그 수집
- autoscaling
- chaos testing
- multi-cluster
- EKS, k3s, metallb, storage 계열 도구

이 층은 `클러스터를 띄운다` 이후에야 시작되는 실제 운영 부담을 보여 준다. 원문이 k3s, EKS, GlusterFS, metrics-server, cluster autoscaler, chaos tool까지 흩어 적는 이유도 여기에 있다.

## 테스트와 복구를 보는 방식
이 노드에서 테스트는 단순 기능 확인이 아니다. 원문은 계속해서 아래 질문을 던진다.

- 한 pod를 일부러 죽이면 복구 시간은 얼마인가?
- 롤백은 몇 초 만에 가능한가?
- HPA가 붙을 때 연결은 어떻게 유지되는가?
- 특정 노드에 부하가 몰리면 무슨 일이 벌어지는가?
- preStop, readiness, liveness는 실제 서비스 종료 시나리오와 맞물리는가?

즉 여기서 테스트는 `정상 동작 확인`보다 **장애를 넣고 회복 경로를 확인하는 운영 리허설**에 가깝다. 컨테이너는 실행 포맷이고, 진짜 관심은 그 위에 얹히는 복원력이다.

## 반복되는 긴장 관계

### 편의성 vs 재현성
- 볼륨 마운트와 서버 직접 수정은 편하다.
- 하지만 나중에 동일 상태를 다시 만들기 어렵다.

### 추상화 vs 통제력
- 클라우드, 쿠버네티스, 서비스 메시가 많은 것을 감춘다.
- 대신 내부 동작을 이해하지 못하면 장애 원인 추적이 어려워진다.

### 단순 배포 vs 운영 체계
- `docker-compose`로도 시작은 가능하다.
- 그러나 실제 운영으로 가면 이미지 버전, 핫픽스 규칙, 롤백, 권한, 모니터링, 테스트 체계가 필요하다.

### 단일 클러스터 집중 vs 멀티 클러스터 분리
- 한 클러스터에 몰아 넣으면 관리가 쉬워 보인다.
- 그러나 규모가 커질수록 환경 분리와 책임 분리가 필요해진다.

### 도구 도입 vs 운영 흡수 비용
- Helm, Istio, ArgoCD, Spinnaker는 모두 유용해 보인다.
- 하지만 각각 학습 비용, 자원 사용, 디버깅 난도를 함께 가져온다.
- 원문은 도구를 `좋은가`보다 `감당 가능한가`로 평가하려는 성향을 보인다.

## 실전 운영 판단으로 압축하면
원문 전체를 실전 질문으로 축약하면 결국 다음 체크리스트로 읽힌다.

- 지금 상태를 이미지와 설정으로 다시 만들 수 있는가?
- 로그와 메트릭이 밖으로 나오도록 설계했는가?
- 개발용 편의 설정과 배포용 고정 산출물을 분리했는가?
- latest 같은 흐릿한 기준 대신 태그와 이력을 남기고 있는가?
- 장애가 나면 `죽인다 → 확인한다 → 되돌린다 → 다시 배포한다`가 실제로 가능한가?
- 새 도구를 넣을 때 편의뿐 아니라 운영 흡수 비용도 계산했는가?

이 체크리스트는 컨테이너를 기술 소개가 아니라 **복구 가능한 운영 시스템을 만들기 위한 질문 묶음**으로 읽게 해 준다.

## 숨은 패턴
- 이 노드는 컨테이너 자체보다 **배포 체계 설계 노드**에 가깝다.
- 메모의 질문 대부분이 `이 기술이 무엇인가`보다 `실무에서 어디서 꼬이는가`에 집중되어 있다.
- Docker, Kubernetes, Istio, Helm, ArgoCD, Spinnaker, k3s, EKS 같은 키워드는 모두 **재현 가능한 운영 흐름을 만들기 위한 도구 후보군**으로 등장한다.
- 특히 원문은 `docker-compose로 빨리 띄우기`와 `실서버는 재빌드/재배포로 다루기` 사이의 긴장을 여러 번 반복해, 컨테이너를 개발 편의보다 **운영 규율을 만드는 장치**로 더 강하게 읽게 만든다.
- 따라서 이 노드의 핵심 관심사는 기능 설명이 아니라 **운영 판단 능력**이다.

## 인접 노드와의 연결
- [[AI]]: AI 에이전트를 실제 서비스로 운영하려면 결국 컨테이너, 배포, 관측, 롤백 체계가 필요하다.
- [[Cloud]]: 컨테이너는 클라우드 자원 위에서 실행되며, 클라우드는 실행 기반을 제공하지만 운영 단순화의 전부는 아니다.
- [[Infra]]: 인프라는 어떤 구성 요소를 둘지의 문제이고, Container는 그 요소들을 어떻게 패키징하고 운용할지의 문제다.
- [[Devops]]: Container는 DevOps의 핵심 실행 단위이며, Devops 노드는 이 노드를 팀 프로세스와 배포 문화까지 확장한다.
- [[Monitoring]]: stdout 로그, 메트릭, 장애 확인, 복구 시간 측정은 관측성 없이 성립하지 않는다.
- [[Web]]: 실제 웹서비스 운영에서는 ingress, TLS, 로드밸런서, API gateway와 만나 앞단 트래픽 구조로 이어진다.

## 이후 심화 포인트
- Docker / OCI 런타임 / Podman 계열의 역할 차이를 더 선명하게 정리할 수 있다.
- 쿠버네티스 파트를 `네트워크`, `배포`, `권한`, `운영 실수` 축으로 한 번 더 압축할 수 있다.
- 이미지 버저닝, 핫픽스, 롤백 규칙을 이 노드만의 운영 체크리스트로 따로 추출할 수 있다.
- EKS, k3s, Helm, ArgoCD 메모를 `로컬 실험 ↔ 관리형 운영 ↔ GitOps` 비교 축으로 재정리할 수 있다.

## 이번 갱신 메모
- 기존 baseline은 유지하되, 원문 전체 배열을 `도커 기본 성질 → 관측 → Dockerfile 규율 → 개발/배포 충돌 → 쿠버네티스 운영 확장`의 순서로 다시 접는 `원문 발췌 구조`를 추가했다.
- `격리 → 가시화 → 책임 분리 → 롤백 가능성 → 증설`의 `컨테이너 운영 루프`를 새로 세워, 길고 넓은 메모가 실제로는 하나의 운영 문법을 반복한다는 점을 더 빠르게 읽히게 다듬었다.
- `docker-compose로 빨리 띄우기`와 `실서버는 재빌드/재배포로 다루기` 사이의 긴장을 별도 패턴으로 끌어올려, 이 노드가 개발 편의 기술보다 운영 규율 설계 노드에 가깝다는 점을 더 분명하게 보강했다.
