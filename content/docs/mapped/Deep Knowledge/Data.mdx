# Data

- 원본: `source/shdkej-content/Deep Knowledge/Data.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 카테고리 노드와 1:1 대응하는 관리 노트

## 한눈에 보는 요약
이 노드에서 `Data`는 데이터베이스 종류를 외우는 기술 노트가 아니다. 원문은 데이터를 **저장하는 법**, **다시 찾는 법**, **대량으로 다룰 때 망가지지 않는 법**, **분석용으로 분리하는 법**, **개인 기억과 기록 욕망을 감당하는 법**까지 한 묶음으로 다룬다. 즉 `Data`는 저장소 선택 노드이면서 동시에 조회 비용, 검색 경험, 운영 메모리, 정보 과잉 불안을 함께 붙드는 상위 카테고리 노드다.

## 만다라트 위치에서 읽는 해석
이 페이지는 `Deep Knowledge` 안에 놓여 있기 때문에, 단순 DB 입문서보다 **정보를 오래 보존하고 다시 호출하는 기반 구조를 어떻게 설계할 것인가**에 더 무게가 실린다. 같은 `Data`라도 `Fundamental` 축이었다면 자료구조·정규화·기초 이론 쪽이 더 중심이었을 텐데, 여기서는 실제 운영에서 부딪히는 조회 병목, 검색 품질, 분석용 복제, 기록 과잉 불안까지 함께 다룬다. 즉 이 노드는 `데이터 저장 기술`보다 `데이터를 계속 다루는 운영 지성`에 더 가깝다.

## 이 노드가 실제로 답하려는 질문
- 어떤 저장 구조가 지금 문제에 맞는가?
- 데이터는 많이 모으는 것보다 어떻게 다시 꺼내고 감당할지가 왜 더 중요한가?
- 검색 시스템은 왜 단순 DB 조회의 확장이 아니라 별도 설계 대상인가?
- 운영계 저장소와 분석계 저장소는 어디서 갈라져야 하는가?
- 사람의 기억과 노트 시스템 문제를 데이터 구조 문제와 같은 축에서 읽을 수 있는가?

## 핵심 논지
- 데이터 기술 선택의 핵심은 기능 비교보다 **어떤 실패와 복잡성을 감당할지**를 정하는 데 있다.
- 대량 데이터에서는 저장 용량보다 조회 단위, 메모리 점유, 재색인 비용, 복제 경로가 더 빨리 병목이 된다.
- 좋은 데이터 구조는 중첩을 늘리는 구조보다 연결을 드러내고 다시 찾기 쉬운 구조에 가깝다.
- 검색은 저장의 부속 기능이 아니라 분석기, 사전, 색인, 질의 문법을 따로 설계해야 하는 독립 계층이다.
- 이 노드는 기술 시스템 이야기이면서도, 동시에 `잊고 싶지 않다`, `다 알고 싶다`, `처리가 밀린다`는 인간의 정보 심리까지 품고 있다.

## 원문 구조를 다시 읽기
원문은 얼핏 보면 DB 제품 메모, 운영 팁, 철학적 단상, 기억 메모가 뒤섞여 있다. 하지만 실제로는 아래 다섯 층으로 읽는 편이 더 정확하다.

### 1. 저장소 기초와 설계 감각
- File, RDB, NoSQL
- ACID, CAP, N+1, 느린 쿼리
- 데이터 양, 인덱스

### 2. 대용량 조회와 메모리 제약
- paging, iterator, stream
- 한 번에 많은 요청을 보내는 경우의 제한
- 2차 캐시와 GC, 메모리 누적

### 3. 특수 저장소와 검색 계층
- Redis 자료구조와 메모리 특성
- Elasticsearch 인덱스, 매핑, analyzer, synonym, user dictionary
- 검색 UX 메모와 유의어/제외 검색

### 4. 분석계 저장소와 데이터 이동
- ClickHouse, Athena
- CDC, Kafka, Airflow
- 운영계 DB와 분석계 DB의 분리

### 5. 기록, 기억, 정보 과잉의 인간 문제
- 데이터 보존과 사료의 비유
- 노트와 기억의 관계
- 웹서핑, 인박스, 초조함, 맞춤형 서비스
- DAG, 연결 구조, 단일 진실 원천

## 원문 발췌 구조
- `Database` : RDB/NoSQL, ACID, CAP, N+1, 느린 쿼리, 인덱스 같은 저장소 기초 메모
- `대용량 작업` : paging, iterator, stream, 비동기 요청 제한, 2차 캐시와 GC 문제
- `Redis` : 자료구조 선택, rehash, ziplist, maxmemory, scan 같은 메모리/운영 메모
- `elasticsearch` : index, mapping, analyzer, synonym, user dictionary, reindexing, 검색 문법
- `클릭하우스` : Postgres와 분석계 저장소를 분리하는 복제/집계 경로
- `데이터 관리방법` 이후 : 사료 비유, 백업, 기록 욕망, 기억 훈련, 정보 과잉, 검색 UX, 단일 진실 원천 같은 넓은 사고 메모

원문 배열 자체가 중요하다. 초반에는 저장소와 조회 기술에서 출발하지만, 뒤로 갈수록 `데이터를 많이 가진 뒤에 어떻게 감당할 것인가`, 더 나아가 `인간이 정보를 너무 많이 붙들고 싶어질 때 어떤 구조가 필요한가`로 스케일이 넓어진다.

## 이 노드의 중심 재해석

### 데이터는 `어디에 넣을까`보다 `어떻게 다시 꺼낼까`의 문제다
원문은 초반에 DB 이론 키워드를 적어 두지만, 실제 무게중심은 계속 조회 비용과 운영 비용으로 이동한다. 대량 select, stream, paging, reindex, 메모리, 캐시, 검색 사전, 분석 복제 메모가 반복된다는 것은 저자가 데이터를 정적 저장물보다 **계속 읽히고 이동하고 재가공되는 흐름**으로 본다는 뜻이다.

### 저장소 선택은 제품 비교가 아니라 실패 방식 선택이다
- RDB는 관계와 일관성에는 강하지만, 조회 병목과 구조 결합 비용을 계속 의식해야 한다.
- NoSQL은 문서형·분산형 상황에 유리하지만 CAP과 모델링 부담을 함께 감당해야 한다.
- Redis는 빠르지만 메모리 한계, 자료구조 표현 방식, 운영 명령의 위험성을 늘 고려해야 한다.
- Elasticsearch는 검색 품질을 높여 주지만, 스키마 변경과 재색인 비용이 크다.
- ClickHouse, Athena는 분석에 강하지만 트랜잭션 저장소를 대체하기보다 읽기 계층을 따로 세우는 데 가깝다.

즉 이 노드의 핵심 질문은 `최고의 DB는 무엇인가`가 아니라 `내가 어떤 종류의 복잡성과 실패를 떠안을 것인가`다.

## 반복해서 보이는 판단 축

### 저장 용량 vs 조회 가능성
원문은 데이터를 많이 쌓는 문제보다, 원하는 순간 원하는 범위만 안전하게 꺼내는 문제를 더 무겁게 다룬다. 대용량 select를 피하고 paging·stream을 쓰라는 메모는 이 노드의 중심 원칙이다. 즉 `많이 저장했다`는 사실보다 `작은 단위로 다시 읽을 수 있게 설계했는가`가 더 중요하다.

### 일관성 vs 확장성
ACID와 CAP를 나란히 적어 둔 것은 이론 정리가 아니라, 어떤 시스템 성격을 택할지 계속 판단하고 있음을 보여 준다. 저자에게 저장소는 기능 목록이 아니라 **보장 조건의 선택**이다.

### 빠른 응답 vs 운영 위험
Redis, 캐시, 검색 인덱스는 모두 빠른 조회를 돕지만 동시에 메모리 압박, 재색인, 명령 오남용 같은 운영 위험을 만든다. 이 노드는 빠른 도구일수록 더 조심해야 한다는 감각을 품고 있다.

### 단일 저장소 vs 목적별 분리
운영계 DB, 검색용 인덱스, 분석계 저장소를 한 덩어리로 보지 않는다. 읽는 목적이 다르면 저장 경로를 분리해야 한다는 감각이 강하다. 원문 후반의 `단일 진실 원천` 메모도 결국 아무 곳에나 같은 책임을 얹지 말고, 진실을 고정하는 층과 읽기 최적화를 담당하는 층을 나누려는 고민으로 읽힌다.

### 정보 축적 욕망 vs 처리 지연의 불안
후반부 메모에서는 기술 문제와 심리 문제가 붙는다. 더 많이 모으고 싶지만 다 처리하지 못하고, 7번을 보고 있어도 1~20의 나머지가 신경 쓰인다. 이 긴장은 `Data`를 단순한 백엔드 노트가 아니라 정보 운영 노드로 만든다.

## 세부 해석

### 1. Database, 설계, 쿼리

#### ACID와 CAP를 함께 적어 둔 이유
관계형의 ACID와 분산 저장의 CAP를 한 노트 안에 같이 적는다는 것은, 저자가 저장소 선택을 구현 편의보다 시스템 성격의 문제로 본다는 뜻이다. 무엇을 절대 보장하고, 무엇을 일부 포기할지의 판단이 먼저 온다.

#### 인덱스와 데이터 양은 미리 생각해야 하는 운영 변수다
원문에서 `데이터 양`, `인덱스`가 짧게 적혀 있지만 무게는 크다. 스키마가 예쁜지보다 양이 커졌을 때 검색과 정렬이 어떻게 무너질지를 먼저 보는 감각이 드러난다. 이 점은 [[mapped/Deep Knowledge/Monitoring]]의 병목 감각과도 닿는다.

#### 느린 쿼리와 N+1은 사후 미세조정 이슈가 아니다
`어떻게 가장 비싼 쿼리를 찾을 것인가`, `N+1 문제`라는 메모는 데이터 설계와 애플리케이션 설계가 분리될 수 없다는 뜻이다. 질의 비용을 처음부터 관측 가능한 구조로 두려는 태도다.

### 2. 대용량 처리와 메모리

#### 많이 저장하는 것과 한 번에 많이 읽는 것은 다르다
원문은 대용량 검색에서 paging, iterator, stream을 반복한다. 핵심은 단순 팁이 아니라 다음 운영 원칙이다.
- 데이터를 모으는 것과 한 번에 꺼내는 것은 다른 문제다.
- 시스템이 버티는 단위와 사람이 다루기 쉬운 단위는 다를 수 있다.
- 조회 단위를 나누지 않으면 메모리 문제가 곧 장애가 된다.

#### 캐시는 성능 장치이면서 동시에 장애 장치다
2차 캐시 때문에 old 메모리가 잘 회수되지 않았고, 캐시를 끄자 메모리 사용량이 줄었다는 메모는 중요하다. 저자는 `성능을 위해 넣은 장치가 오히려 운영 안정성을 깨뜨릴 수 있다`는 경험을 이미 이 노드 안에 보관하고 있다.

### 3. Redis, 빠른 구조의 이면

#### Redis는 캐시 제품이 아니라 자료구조 선택 훈련장이다
set, hash, string, bitmap, ziplist, rehash, maxmemory, scan이 함께 적혀 있는 방식은, 저자가 Redis를 단순 속도 향상 도구보다 **표현 방식과 비용의 상관관계를 배우는 장소**로 보고 있음을 보여 준다.

#### Redis 운영 기준으로 남길 만한 감각
- `keys`, `flushall`보다 `scan`을 선호한다.
- 메모리 사용량과 `maxmemory`를 먼저 확인한다.
- swap이 발생하면 빠른 저장소라는 장점이 무너진다.
- 작은 자료구조 선택 하나가 메모리 표현과 성능을 크게 바꾼다.

### 4. Elasticsearch, 검색은 별도 시스템이다

#### 검색은 단순 조회의 확장이 아니다
인덱스, 문서, 매핑, tokenizer, analyzer, synonym, user dictionary, shard, routing, reindexing 메모가 촘촘하다는 것은, 저자가 검색을 DB 부가기능이 아니라 별도 설계면으로 이해하고 있다는 뜻이다.

#### 이 노드에서 중요한 검색 판단
- 인덱스 수정은 어렵고 새로 만들어 재색인해야 할 때가 많다.
- 사용자 사전과 동의어 사전은 검색 품질과 운영 복잡도를 동시에 바꾼다.
- 기본 size 같은 기본값을 모르면 `전체 검색`도 기대와 다르게 보인다.
- 검색 품질은 데이터 저장보다 분석기 설정, 용어 정규화, 질의 문법에 더 크게 좌우된다.

#### 검색 UX까지 함께 본다
후반부의 permutations, 유의어 검색, `-키워드`, `+키워드` 메모는 중요하다. 저자는 검색 엔진 내부만이 아니라 **사용자가 실제로 어떻게 찾게 될 것인가**를 함께 의식한다. 이 점에서 `Data`는 [[mapped/Deep Knowledge/Web]]과도 이어진다.

### 5. ClickHouse, Athena, CDC

#### 트랜잭션 저장소와 분석 저장소를 분리하려는 감각
Postgres에서 ClickHouse로 복제하는 흐름, 실시간과 배치 복제의 구분은 저자가 `모든 데이터를 한 저장소에 몰아넣기`보다 `쓰는 목적에 따라 읽기 경로를 나누기`를 선호한다는 신호다.

#### 여기서 드러나는 운영 문법
- 운영계 DB는 정확한 기록과 서비스 처리를 맡는다.
- 분석계 저장소는 대량 집계와 탐색을 맡는다.
- CDC, Kafka, Airflow는 저장소가 아니라 이동·연결 계층이다.
- Athena는 배치성 탐색에는 좋지만, 실시간성이 중요하면 다른 판단이 필요하다.

이 축은 [[mapped/Deep Knowledge/Monitoring]]의 지표 저장 감각, [[mapped/Deep Knowledge/Infra]]의 계층 분리 감각과 자연스럽게 이어진다.

## 데이터 판단을 묶는 운영 사다리
원문 전체를 다시 접으면 이 노드는 아래 사다리로 읽는 편이 재진입성이 높다.

1. **저장한다** — RDB, NoSQL, Redis, 파일 등 어떤 그릇을 쓸지 고른다.
2. **안전하게 꺼낸다** — 인덱스, 느린 쿼리 탐지, paging, stream, 메모리 한계를 본다.
3. **잘 찾게 만든다** — Elasticsearch, 동의어, 사용자 사전, 검색 문법을 조정한다.
4. **목적별로 분리한다** — 운영계와 분석계를 나누고 CDC·배치 복제를 설계한다.
5. **사람이 감당할 수 있게 만든다** — 기록 욕망, 처리 지연, 정보 과잉, 기억 훈련 문제까지 함께 다룬다.

이 다섯 단계 덕분에 원문의 기술 메모와 인간 메모는 분리된 잡음이 아니라, `쌓인 것을 잃지 않고 다시 쓰기 위한 한 시스템`으로 연결된다.

## 인간의 기억과 데이터 구조가 만나는 지점
이 노드 후반부는 기술 메모에서 기억, 기록, 연상력, 정보 과잉, 웹서핑, 인박스로 넘어간다. 겉보기에 주제 이탈처럼 보이지만 실제로는 같은 질문의 다른 스케일이다.

> 쌓이는 것을 어떻게 잃지 않고, 다시 찾고, 과부하 없이 다룰 것인가?

이 질문은 데이터 엔지니어링의 질문이면서 동시에 개인 지식 관리의 질문이다. 그래서 `Data` 노드는 기술 노드이면서 [[mapped/Idea/Information]]과 직접 닿는 정보 운영 노드이기도 하다.

### 반복해서 보이는 심리적 긴장
- 많이 모을수록 미래에 도움 될 것 같지만, 처리 지연은 답답함을 만든다.
- 다 기록하면 안심되지만, 기록 의존은 기억 훈련을 약하게 만들 수 있다.
- 더 많은 정보를 알고 싶지만, 모두 가지려는 욕심은 초조함을 만든다.
- 개인화 서비스는 편리하지만 감시와 편향의 문제를 함께 가져온다.

즉 저자에게 데이터는 차가운 저장 대상이 아니라, 욕망과 불안과 탐색 충동이 계속 달라붙는 대상이다.

## 이 노드에서 드러나는 저자의 선호
- 예쁜 모델링보다 나중에 다시 찾고 가공하고 운영할 수 있는 구조를 더 중시한다.
- 대용량 상황에서의 메모리, stream, paging, 캐시 부작용에 민감하다.
- 검색 품질을 위해 사전, 동의어, 쿼리 문법 같은 세부 조정을 아끼지 않는 편이다.
- 단일 시스템 만능주의보다 목적별 저장소 분리와 연결 구조를 선호한다.
- 정보 수집 욕구가 강하지만, 동시에 과잉 수집과 처리 지연의 스트레스도 크게 느끼는 편이다.

## 이번 심화에서 보강한 점
- 기존 baseline은 유지하면서 `Deep Knowledge` 안에서 이 노드가 맡는 위치를 앞쪽에 분명히 적어, 이 페이지를 DB 입문 요약보다 `데이터 운영 설계 노드`로 더 선명하게 고정했다.
- 원문 발췌 구조를 별도 구간으로 추가해 저장소 기초 → 대용량 작업 → 검색 계층 → 분석 복제 → 기억/정보 과잉 메모의 실제 배열을 빠르게 복원할 수 있게 했다.
- `저장-조회-검색-분리-감당`의 운영 사다리를 새로 세워 기술 메모와 인간 메모가 왜 한 노드 안에 함께 있어야 하는지 더 빨리 읽히게 다듬었다.
- Elasticsearch와 검색 UX 메모를 `저장 부속 기능`이 아니라 `독립 설계 계층`으로 읽는 축을 유지·강화했다.
- ClickHouse·Athena·CDC 메모를 도구 목록보다 `운영계/분석계 분리`의 판단 구조로 더 또렷하게 묶었다.
- 후반부의 기억·웹서핑·인박스 메모를 주변 잡음으로 처리하지 않고, 데이터 운영과 인간 정보 심리가 만나는 핵심 층으로 계속 전면화했다.
