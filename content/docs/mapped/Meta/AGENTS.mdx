# AGENTS

- 원본: `source/shdkej-content/Meta/AGENTS.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 메타 운영 페이지

## 한눈에 보는 요약
이 노드는 AI 에이전트를 많이 붙여 쓰자는 메모가 아니다. 더 정확히는, 리더·기획·개발·디자인·마케팅·운영 역할을 문서와 에이전트로 분해하고, 각 역할이 같은 브랜드와 같은 맥락을 공유하면서도 독립적으로 일할 수 있게 만드는 협업 운영체계를 구상한 노드다. 핵심은 에이전트 수가 아니라, 약한 고리 보강, 고객 목소리 수집, 문서 연결, 히스토리 보존, stateless 작업 방식, 그리고 문서만으로 시스템을 다시 세울 수 있는 상태다.

## 핵심 논지
원본에서 `AGENTS`는 모델 설정 메모이기 전에 "역할이 분리된 협업 시스템을 어떻게 문서 중심으로 운영할 것인가"에 대한 설계 노트다. 에이전트는 사람을 완전히 대체하는 존재라기보다, 브랜드·요구사항·기술 명세·히스토리 문서를 공유받고 각자 다른 전문 관점으로 작업하는 공동작업자에 가깝다. 따라서 이 노드의 중심은 프롬프트 기교보다 역할 분화, 공통 문서 기반, 맥락 전달, 추적 가능성, 다시 만들 수 있는 상태에 있다.

## 원본에서 먼저 잡히는 구조

### 1. 역할 분화가 먼저 나오고 도구는 나중에 나온다
원문 첫머리는 곧바로 역할 지도를 제시한다.
- 리더
- 기획자
- 개발자
- 디자이너
- 마케터
- 운영자

즉 출발점은 어떤 모델을 쓸지가 아니라, 어떤 책임을 나누고 어떤 관점으로 협업할지를 정하는 일이다. 에이전트는 이 역할 구조 위에 얹히는 실행 단위다.

### 2. 전략의 출발점은 약한 고리와 고객 목소리다
원문은 전략 파트에서 화려한 자동화보다 아래를 먼저 적는다.
- 약한 고리를 강화한다
- 고객을 찾고 목소리 들어보기
- 정보 저장 방식 세팅하기

이 순서는 중요하다. 저자는 에이전트 운영을 내부 효율화만의 문제로 보지 않고, 실제 고객 반응과 학습 루프를 붙여야 의미가 생긴다고 본다.

### 3. 문서 세트가 에이전트보다 더 근본적인 기반이다
원문 중반은 계속 유지해야 할 파일과 선행 산출물을 길게 적는다.
- 브랜드
- 디자인 시스템
- 리더 에이전트
- 공동작업자 에이전트들
- Core LLM 파일
- 인프라 문서
- BRAND 문서군
- TECH SPEC 문서군
- 결정, 가설, 캠페인, 사후 분석, 위키

즉 이 노드의 핵심 자산은 에이전트 개체보다 문서 토폴로지다.

### 4. 최종 목표는 stateless 협업이다
원문의 후반부는 계속 같은 방향으로 수렴한다.
- 각 문서가 서로 연결되고 업데이트되어야 한다.
- 문서만 보고도 시스템을 다시 만들 수 있는 상태여야 한다.
- 맥락이 문서 안에 있어야 한다.
- Code as doc.

이는 에이전트를 똑똑하게 만드는 문제보다, 맥락 의존성을 줄이고 재현 가능한 작업 구조를 만드는 문제에 더 가깝다.

## 이 노드가 실제로 다루는 질문
- 여러 역할의 에이전트를 둘 때, 무엇을 공통 기반으로 삼아야 하는가?
- 브랜드, PRD, 디자인 시스템, 기술 명세, 히스토리 문서는 어떤 순서와 관계로 놓여야 하는가?
- 한 역할이 만든 가설과 맥락을 다른 역할이 어떻게 잃지 않고 이어받을 수 있는가?
- 에이전트가 늘어나도 문서만 보면 다시 복구 가능한 상태를 어떻게 유지할 수 있는가?
- 전역 설정, 스킬, MCP, 플러그인 같은 도구층은 이 운영체계에서 어디까지가 기반이고 어디부터가 교체 가능한 부품인가?

## 원문 발췌 구조
- **역할 지도**: 리더 → 기획자 → 개발자/디자인 → 마케터 → 운영자
- **전략 메모**: 약한 고리 강화, 고객 목소리 수집, 정보 저장 방식 세팅
- **계속 관리할 공통 자산**: 브랜드, 디자인 시스템, 리더/공동작업자 에이전트, Core LLM 파일, 인프라 문서
- **선행 산출물**: BRAND 문서군, TECH SPEC 문서군
- **히스토리 문서군**: 결정 사항, 가설, 캠페인 전략, 사후 분석, 서비스 위키
- **운영 방법**: 공통 대시보드, 이슈 티켓, 문서 연결, stateless, Code as doc, PRD 모듈화
- **도구층**: Claude Code 전역 설정, skills, agents, plugins, MCP

이 구조를 그대로 보면 원문은 `역할 배치 → 외부 신호 수집 → 공통 자산 준비 → 실행 중 문서 연결 → 도구 세팅` 순서로 생각이 움직인다. 즉 도구를 먼저 고르는 노트가 아니라, 운영 골격을 먼저 세운 뒤 도구를 마지막 층에 놓는 노트다.

## 구조 재정리

### 1. 이 노드는 멀티에이전트보다 멀티역할 문서 시스템에 가깝다
리더, 기획, 개발, 디자인, 마케팅, 운영이 나뉘는 것은 단순 분업이 아니다. 서로 다른 판단 기준을 가진 관점들을 분리해 놓고, 공통 문서 기반 위에서 다시 연결하려는 시도다. 즉 역할 분리는 전문성 확보를 위한 것이고, 문서 연결은 분절 방지를 위한 것이다.

### 2. 에이전트의 품질은 선행 문서 품질에 크게 의존한다
원문은 `BRAND(O)`와 `TECH SPEC(O)`를 미리 만들어 참고하게 하려 한다. 이는 에이전트가 영리해서 잘하는 것이 아니라, 먼저 잘 정리된 기준 문서를 줘야 일관된 결과가 나온다는 판단이다.
- 브랜드 아이덴티티
- 핵심 메시지와 문제 정의
- PRD와 사용자 페르소나
- STP 전략
- 디자인 시스템
- 아키텍처 다이어그램
- ERD, API 명세, 스토리보드

즉 프롬프트보다 선행 문서 설계가 더 중요하다는 뜻이다.

### 3. 히스토리 문서는 결과보다 판단 맥락 보존 장치다
원문은 결정 사항, 가설, 캠페인 전략, 사후 분석, 서비스 위키를 따로 적는다. 이는 결과물 저장이 아니라 "왜 그렇게 했는가"를 나중에도 복원하기 위한 장치다. 특히 한 역할이 만든 가설을 다른 역할이 이어받을 때, 판단 맥락이 빠지면 협업이 금방 얕아진다.

### 4. 이 노드는 공통 문서군과 역할별 문서를 함께 상정한다
원문에서 계속 관리해야 할 파일 목록과 BRAND/TECH SPEC 산출물을 함께 보면, 저자가 상상하는 운영 구조는 대략 두 층으로 나뉜다.
- 모든 역할이 같이 보는 공통 바닥: 브랜드, 핵심 메시지, PRD, 디자인 시스템, 인프라 문서, 서비스 위키
- 특정 역할이 더 깊게 파고드는 문서: 캠페인 전략, API 명세, ERD, 스토리보드, 사후 분석, 역할별 에이전트 파일

이 구분이 중요한 이유는, 모두가 모든 세부 문서를 다 읽지 않아도 같은 방향으로 움직이게 만들 수 있기 때문이다. 공통 바닥 없이 역할 자율성만 커지면 각자 최적화가 따로 놀고, 반대로 역할별 문서가 없으면 각 전문성이 얕아진다.

### 5. PRD는 정적 문서가 아니라 모듈형 작업 루트여야 한다
원문 후반의 PRD 메모는 이 노드의 핵심 중 하나다.
- 루트는 원페이저로 둔다.
- 기능별로 잘게 쪼갠다.
- Epic 같은 상위 기능 단위를 둔다.
- 상단 5줄에 why와 비전을 적는다.
- 가설, 선택, edge case, user story, UX 흐름, 기술 제약을 녹인다.

이는 PRD를 한 번 쓰고 버리는 승인 문서가 아니라, 여러 역할이 같은 루트 아래에서 각자 필요한 수준으로 파고들 수 있는 작업 지도처럼 다룬다는 뜻이다.

### 6. 운영 방식의 핵심은 상태를 사람 머리 밖으로 끌어내는 일이다
원문은 `문서만 보고도 시스템을 다시 만들 수 있는 상태`, `stateless`, `맥락이 문서 안에 있어야 한다`는 표현을 반복한다. 이를 풀어 쓰면 다음 운영 루프로 읽힌다.
1. 리더/기획이 방향과 가설을 문서로 만든다.
2. 개발/디자인/마케팅/운영이 자기 문서 층에서 실행한다.
3. 결과와 수정 이유를 다시 결정·사후 분석·위키에 남긴다.
4. 다음 역할은 이전 사람의 기억이 아니라 남겨진 문서를 읽고 재진입한다.

즉 에이전트 시스템의 품질 기준은 개별 응답의 영리함만이 아니라, 역할이 바뀌어도 상태가 이어지는가에 있다.

### 7. 정보 저장 방식은 내부 위키만이 아니라 발행 표면까지 포함한다
원문은 정보 저장 방식 예시로 `인스타 스레드 게시물`, `사진과 프롬프트`, `블로그 글`을 바로 적어 둔다. 이 점이 중요하다. 저자가 말하는 운영 기억은 닫힌 내부 문서함만이 아니라, 고객 반응을 받는 외부 발행면까지 포함하는 더 넓은 기억 구조다. 다시 말해 이 노드의 문서 시스템은 아래 두 층을 함께 상정한다.
- 내부 재진입을 위한 기준 문서: 브랜드, PRD, 스펙, 위키, 결정 기록
- 외부 감각을 수집하는 발행 문서: 스레드, 사진+프롬프트, 블로그 글

그래서 이 노드는 단지 `정리 잘된 문서 저장소`를 원하지 않는다. 실제 사람 반응이 다시 전략과 제품 판단으로 환류되는 문서-발행 결합 구조를 원한다.

### 8. 도구 설정은 중요하지만 핵심은 아니다
`claude code 전역 세팅` 파트는 settings, skills, agents, plugins, MCP를 길게 적지만, 전체 노트에서 차지하는 의미는 분명하다. 도구는 작업 속도와 연결성을 높여주지만, 문서 구조와 역할 구조가 부실하면 설정만으로는 좋은 협업이 만들어지지 않는다. 즉 도구층은 교체 가능하지만 운영 원리는 더 오래 남는다.

## 반복되는 판단 축

### 공통 기준 vs 역할 자율성
모든 역할이 같은 브랜드와 문서 체계를 봐야 일관성이 생긴다. 하지만 각 역할은 자기 기준으로 해석하고 실행해야 한다. 너무 통일하면 각 역할의 전문성이 죽고, 너무 자율적이면 산출물이 분리된다.

### 빠른 실행 vs 맥락 보존
에이전트는 빠르게 결과를 낼 수 있지만, 맥락이 문서에 남지 않으면 다음 작업자가 다시 처음부터 추론해야 한다. 원문이 stateless를 강조하는 이유다.

### 자동 연결 vs 우아한 문서 연결 처리
원문에는 "연결되는 부분의 우아한 처리"를 고민하는 문장이 나온다. 문서가 서로 갱신되어야 하지만, 하나를 바꾸면 모두를 손으로 맞추는 구조는 곧 깨진다. 즉 연결 강도와 유지 비용 사이의 균형이 중요하다.

### 거대한 통합 문서 vs 모듈형 분해
모든 내용을 한 문서에 모으면 전체 맥락은 좋지만 수정과 위임이 어려워진다. 반대로 너무 잘게 쪼개면 큰 그림이 사라진다. 원문은 원페이저 루트 + 세부 문서 분해라는 중간 해법을 선호한다.

### 페르소나 활용 vs 맹목적 권위 의존
구루 페르소나를 베이스 에이전트에 반영하려는 메모는 흥미롭지만, 원문 전체를 보면 진짜 목적은 권위 모사가 아니라 역할 톤과 판단 관점을 빠르게 세팅하는 데 있다. 즉 페르소나는 출발점이지 근거 그 자체는 아니다.

### 설정 자산 vs 복원 가능한 운영 상태
settings, skills, plugins, MCP 목록은 많지만, 원문이 더 오래 붙잡는 것은 특정 도구 조합보다 `다시 세울 수 있는 상태`다. 따라서 설정 파일 자체보다 그 설정이 어떤 문서 흐름과 역할 구조를 위해 존재하는지 함께 남겨야 운영 기억이 보존된다.

### 내부 문서화 vs 외부 발행면
정보 저장 방식이 인스타 스레드, 사진+프롬프트, 블로그까지 뻗는다는 점을 보면, 이 노드는 지식을 안쪽에만 저장하는 시스템을 상상하지 않는다. 내부 문서는 복원성과 handoff를 맡고, 외부 발행면은 고객 반응과 시장 언어를 다시 받아오는 센서 역할을 맡는다. 둘 중 하나만 있으면 학습 루프가 닫히지 않는다.

## 오래 남길 아이디어
- 좋은 멀티에이전트 시스템은 모델 수보다 공통 문서 구조가 먼저다.
- 역할별 분업이 효과를 내려면, 브랜드와 비전, 문제 정의, 기술 제약이 공통 바닥으로 깔려 있어야 한다.
- 공통 바닥 문서와 역할별 심화 문서를 나눠야 역할 자율성과 일관성을 같이 확보할 수 있다.
- 문서만 보고도 시스템을 다시 세울 수 있어야 협업이 사람 기억에 덜 종속된다.
- PRD는 승인용 문서보다 분해 가능한 작업 루트로 설계할수록 위임과 추적이 쉬워진다.
- 결정 문서, 가설 문서, 사후 분석은 에이전트 시대에 더 중요해진다. 빠른 생성이 쉬워질수록 왜 그런 선택을 했는지 남기는 일이 더 중요해지기 때문이다.
- "Code as doc"라는 감각은 코드를 문서로 대체하자는 뜻이 아니라, 핵심 맥락이 코드와 문서 양쪽에서 복원 가능해야 한다는 요구에 가깝다.
- 설정 파일 목록은 운영 실체가 아니라 운영 실체를 지탱하는 하부 층이므로, 늘 문서 흐름과 함께 읽어야 한다.

## 추출 섹션

### 역할 지도
- 리더
- 기획자
- 개발자 / 디자인
- 마케터
- 운영자

원문은 에이전트를 기능별 봇 목록으로 적지 않고, 거의 조직도처럼 시작한다. 이 출발 방식 자체가 이 노드의 성격을 보여 준다.

### 공통으로 관리해야 할 파일
- 브랜드
- 디자인 시스템
- 리더 에이전트
- 4개의 공동작업자 에이전트
- Core LLM 파일
- 인프라 문서

이 목록은 단순 자산 체크리스트가 아니라, 어떤 파일이 역할 간 공통 기억을 형성하는지 보여 주는 운영 뼈대다.

### 정보 저장 방식의 외부 표면
- 인스타 스레드 게시물
- 사진과 프롬프트
- 블로그 글

이 짧은 목록은 쉽게 지나치기 쉽지만, 원문이 `정보 저장`을 비공개 노트 축적만으로 이해하지 않는다는 증거다. 고객 반응을 받을 수 있는 발행 표면 자체가 운영 시스템의 일부다.

### 선행 산출물과 추적 문서
- BRAND: 브랜드 아이덴티티, 핵심 메시지, PRD, 유저 페르소나, STP
- TECH SPEC: 디자인 시스템, 아키텍처 다이어그램, ERD, API 명세, UI/UX 스토리보드
- HISTORY: 결정 사항 기록, 가설 문서, 캠페인 전략, 사후 분석 보고서, 서비스 위키

세 묶음을 함께 보면, 이 노드는 생성 전 문서, 생성 중 문서, 생성 후 회고 문서를 모두 연결하려는 메타 설계 노드로 읽힌다.

## 원문에서 특히 보존해야 할 부분

### 고객 목소리를 먼저 두는 감각
에이전트 운영 노트인데도 초반에 고객 목소리 듣기가 나온다는 점이 중요하다. 이 노드는 내부 자동화 노트가 아니라 시장 반응을 포함한 운영 메모다.

### 문서 연결의 맥락 전달 문제
"기획자가 가설 제시를 한다치면 그 가설을 생각한 맥락을 문서로 뽑게 하고 개발자나 마케터가 활용할 때 맥락을 참고할 수 있게"라는 취지는 이 노드의 핵심이다. 단순 산출물 공유가 아니라, 판단 배경의 이동성이 중요하다는 뜻이다.

### stateless라는 표현
이 표현은 메모의 밀도를 높인다. 사람 머릿속에만 있는 맥락을 줄이고, 에이전트와 사람이 번갈아 작업해도 이어질 수 있게 만드는 목표가 명확히 드러난다.

## 인접 노드 연결
- [[mapped/Communication/Document]]: 문서 구조, 히스토리, 재진입 가능성을 함께 다룬다.
- [[mapped/Communication/Teamwork]]: 역할 분화와 리더십, 맥락 전달, 성장 설계를 연결해 읽을 수 있다.
- [[mapped/Deep Knowledge/Product]]: 브랜드, 사용자 가치, 문제 정의, 제품 방향 문서의 선행 조건과 맞닿아 있다.
- [[mapped/Integration/Marketing]]: 고객 유입, 메시지, STP, 브랜드 DNA를 실제 시장 언어로 연결하는 축이다.
- [[mapped/Communication/Blogging]]: 내부 위키 바깥의 발행 표면을 어떻게 운영 기억과 연결할지 읽는 데 도움이 된다.
- [[mapped/Fundamental/Architecture]]: 문서만으로 다시 만들 수 있는 상태, 설계 기준, 운영 복원성이라는 점에서 이어진다.
- [[mapped/Meta/About_Development]]: 개발의 기본, 설명 가능성, 협업 시 맥락 노출이라는 메타 감각과 연결된다.

## 선호 신호
저자는 에이전트를 마법 같은 자동 생산 장치보다, 문서 기반 협업을 더 촘촘하게 만드는 역할별 공동작업자로 보는 경향이 강하다. 또한 단순 산출물 생성보다 약한 고리 보완, 고객 목소리 반영, 맥락 문서화, 다시 만들 수 있는 상태, 그리고 여러 역할 사이의 추적 가능한 연결 구조를 더 중요하게 여긴다. 특히 설정값이나 툴 목록만 남기는 것보다, 어떤 공통 문서가 기준점이 되고 어떤 역할 문서가 파생되는지까지 드러나는 설계를 더 선호하는 신호가 강하다.

## 정리 메모
- 기존 기준선은 유지한 채 `원문 발췌 구조`를 추가해 역할 배치에서 도구층까지 이어지는 실제 배열을 더 빨리 복원할 수 있게 다듬었다.
- `정보 저장 방식` 메모를 별도 잡문이 아니라 내부 기준 문서와 외부 발행면을 함께 묶는 운영 기억 구조로 끌어올렸다.
- `stateless`를 추상 구호로 두지 않고 `방향 문서화 → 역할 실행 → 히스토리 환류 → 다음 역할 재진입` 루프로 풀어 적었다.
- `claude code 전역 세팅` 목록을 도구 수집이 아니라 복원 가능한 운영 상태의 하부 층으로 읽는 해석을 유지·보강했다.
