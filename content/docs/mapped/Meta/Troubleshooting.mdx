# Troubleshooting

- 원본: `source/shdkej-content/Meta/Troubleshooting.md`
- 성격: 만다라트 `Meta` 축의 1:1 대응 카테고리 노드
- 이번 상태: 기존 기준선을 유지한 채, 긴 사례 목록을 더 잘 재진입할 수 있는 운영형 구조로 점진 심화

## 한눈에 보는 핵심
이 노트는 잡다한 오류 메모 더미가 아니다. 범위는 Jupyter, Docker, WSL, Git, Go, AWS, Kubernetes, k3s, Linux 복구까지 매우 넓지만, 실제 중심은 꽤 선명하다.

1. 문제 해결은 영감보다 **조건 단순화, 관련 정보 연결, 근거 없는 추측 배제**로 진행해야 한다.
2. 좋은 트러블슈팅은 해결책 암기보다 **증상 분리, 상태 확인, 수정, 재검증**의 루프를 짧게 돌리는 일에 가깝다.
3. 이 노트의 자산은 명령어 자체보다 **다음 장애 때 다시 들어갈 수 있게 해 주는 운영 기억**이다.
4. 특히 Kubernetes·Docker·시스템 복구 구간에서는 화려한 우회보다 **로그, 자원, 권한, 네트워크, 초기화 절차**를 먼저 확인하는 습관이 반복된다.

## 핵심 논지
원본의 `Trouble Shooting`은 단순한 오류 사전이 아니라, 다양한 개발·운영 환경에서 실패를 다루는 개인 복구 매뉴얼에 가깝다. 첫머리의 세 문장, 즉 `조건을 단순하고 명확하게 정리`, `조건과 관련된 정보를 찾아 연결`, `근거없는 무엇이 개입되지 않아야 함`이 이 노드 전체를 지배한다. 뒤에 이어지는 수백 줄의 사례는 도구별 팁 목록이 아니라, 이 세 원칙을 여러 환경에서 반복 훈련한 흔적으로 읽는 편이 맞다.

## 원본에서 먼저 잡히는 큰 구조

### 1. 출발점은 해결책보다 문제 정의 방식이다
원문은 `Ctrl+F`로 특정 문제를 찾으라고 먼저 말한다. 즉 이 노드는 교과서형 설명보다 증상 중심의 색인형 현장 메모다. 해결법도 보편 이론보다, 지금 눈앞에 나타난 실패 신호를 어떻게 좁혀 가는지가 우선이다.

### 2. 대부분의 기록은 증상 문장으로 시작한다
`not working`, `cannot access`, `connection refuse`, `pending`, `broken`, `NotReady`, `fatal`, `404` 같은 표현이 계속 반복된다. 이것은 이 노트가 개념 정리보다 실패 신호의 포착과 재진입 속도를 중시한다는 뜻이다.

### 3. 해결은 자주 재설치·리셋·설정 보정으로 수렴한다
원문 전반에 반복되는 조치는 아래로 압축된다.
- 패키지 재설치
- 캐시 제거
- 설정 파일 수정
- 권한과 경로 확인
- 서비스 재시작
- `reset`, `delete`, `reconfigure`, `prune`
- DNS·포트·네트워크 확인
- 로그와 상태 확인

즉 작성자는 실패를 추상 교훈으로만 남기지 않고, 실제 복구 절차를 적어 다음 진입 비용을 낮추려 한다.

### 4. 후반부로 갈수록 운영 장애 비중이 커진다
초반은 로컬 개발환경, 편집기, 패키지 설치, GitHub Pages, WSL 같은 개인 작업 환경 이슈가 많다. 중반 이후는 Kubernetes, k3s, 리소스 부족, 노드 상태, 저장소, 네트워크, 재부팅 후 장애처럼 운영 이슈가 두드러진다. 즉 이 노드는 개인 개발환경 문제집이면서 동시에 소규모 인프라 운영 장애 기록이다.

### 5. 시간차가 큰 기록이 한 노드에 공존한다
원문은 오래된 개발환경 메모와 비교적 최근의 운영 메모가 함께 섞여 있다. 그래서 하나의 일관된 최신 가이드라기보다, 오랜 기간 축적된 `문제 해결 습관의 퇴적층`으로 보는 쪽이 정확하다.

## 이 노드가 실제로 다루는 질문
- 문제를 만났을 때 무엇부터 확인해야 하는가?
- 증상, 원인, 해결책을 어떻게 분리해서 기록해야 다음에도 재사용할 수 있는가?
- 재설치와 초기화는 언제 빠른 해결이고, 언제 원인 은폐가 되는가?
- 로그, 메트릭, 시스템 상태, 권한, 네트워크를 어떤 순서로 확인해야 하는가?
- 개발 환경 오류와 운영 환경 장애를 같은 메모 방식으로 다뤄도 되는가?
- 트러블슈팅 문서는 개인 기억을 어떻게 운영 자산으로 바꾸는가?

## 구조 재정리

### 1. 이 노드의 핵심은 정답 목록보다 진단 태도다
원문 첫 세 줄은 사실상 이 페이지의 헌법이다.
- 조건을 단순하고 명확하게 정리
- 조건과 관련된 정보를 찾아 연결
- 근거 없는 개입을 배제

이 세 문장을 운영 언어로 풀면 다음과 같다.
- 문제를 재현 가능한 단위로 줄인다.
- 관측된 사실과 설정 정보를 연결한다.
- 추측이 아니라 확인된 단서로 다음 행동을 정한다.

즉 이 노드는 문제 해결을 감각적 요령보다 증거 기반 정리 작업으로 본다.

### 2. 트러블슈팅은 `증상 → 상태 확인 → 수정 → 재검증`의 반복 루프다
원문 대부분의 사례는 아주 짧게 적혀 있지만 구조는 비슷하다.
- 증상: 설치 실패, 연결 거부, 포트 접근 불가, 이미지 이상, 한글 깨짐, `NotReady`, `pending`
- 상태 확인: 로그, 환경변수, 버전, 설정 파일, 리소스 상태, `describe`, `journalctl`, `syslog`
- 수정: 재설치, 설정 수정, `reset`, 서비스 재시작, 권한 변경, 경로 수정
- 재검증: 다시 접속, 다시 빌드, 다시 apply, 다시 init

이 점에서 이 노트는 해결책 모음보다 운영 루프 기록에 가깝다.

### 3. 원문은 문제의 성격을 네 가지 층위로 반복한다

#### 개발 환경 층위
Jupyter, Vimwiki, Eclipse, npm 권한, `chsh`, WSL, 키보드 설정, GitHub Pages 같은 문제는 개인 작업 환경의 마찰을 줄이는 층위다.

#### 빌드·런타임 층위
Go 테스트, gRPC 코드 생성, Gatsby SSR, React 접근, Lambda 인코딩, Alpine 바이너리 실행 문제는 코드가 실제로 구동되는 경계의 문제다.

#### 인프라·오케스트레이션 층위
Terraform, Ansible, Jenkins, AWS API Gateway, Grafana/Prometheus, Kubernetes, k3s는 서비스 구성·배포·운영 중 장애 대응의 층위다.

#### 시스템 복구 층위
리눅스 부팅 불가, Wi-Fi 하드블록, 하이퍼바이저, OOM, 노드 `NotReady`, 재부팅 후 swap 문제는 시스템 자체를 다시 살리는 복구 층위다.

이 층위를 구분해 읽으면 원문의 산만함이 줄고, 같은 종류의 실패를 더 빨리 묶어볼 수 있다.

### 4. 반복 증상 축으로 다시 읽으면 재사용성이 올라간다
원문에 흩어진 사례는 아래 반복 축으로 압축된다.

#### 설치·버전 충돌
- Jupyter 설치 실패
- TensorFlow·NumPy 버전 문제
- Helm v3 변경으로 인한 예전 명령 무효화
- 플러그인 버전 충돌

#### 권한·경로·설정 불일치
- npm global permission denied
- `kubeconfig` 위치 문제
- mount path 오류
- `go_package`, config 파일, `CNAME`, 서비스 계정 같은 경로·설정 불일치

#### 네트워크·접속 경계 문제
- 포트 접근 불가
- API endpoint 설정 오류
- DNS 누락
- ingress `pending`
- Docker/WSL/호스트 간 통신 불가

#### 자원·상태 이상
- OOM
- Prometheus 저장소 문제
- node `Unknown` / `NotReady`
- 디스크·볼륨·read-only 상태

#### 초기화·복구 필요 상태
- `kubeadm reset`
- `minikube delete`
- `terraform init -reconfigure`
- 캐시 제거, 재설치, prune

이렇게 보면 이 노트는 도구 백과사전이 아니라, 장애를 몇 개의 반복 축으로 가르는 분류 훈련장에 가깝다.

### 5. Kubernetes 메모는 이 노드의 운영 철학을 가장 잘 드러낸다
원문에서 가장 밀도 높은 구간은 Kubernetes다. 여기서 반복되는 패턴은 분명하다.
- 접속 불가면 API 서버 문제인지, 노드 문제인지, 리소스 부족인지 먼저 분리한다.
- OOM, kubelet, coredns, flannel, weave, selector, label, volume, swapoff 같은 기본 상태를 먼저 본다.
- 잘못 꼬였으면 `reset`, `delete`, `killall` 같은 초기화 절차를 과감히 사용한다.
- 다만 초기화만으로 끝내지 않고 `top`, `describe`, `journalctl`, `syslog`로 원인을 추적하려 한다.

즉 작성자는 쿠버네티스를 마법의 플랫폼보다, 상태가 꼬이면 기본 자원과 연결 상태부터 다시 살펴야 하는 복잡한 운영 시스템으로 이해한다.

### 6. 재설치는 패배가 아니라 복구 전략의 일부다
원문에는 `reset`, `delete`, `reconfigure`, `reinstall`, `prune`이 많이 나온다. 이는 원인 분석을 포기했다기보다, 시스템을 빨리 정상 상태로 되돌린 뒤 다시 관찰 가능한 상태를 확보하려는 태도에 가깝다. 이 노드에서 재설치는 무식한 마지막 수단이 아니라 복구 가능성을 확보하는 실무적 선택지다.

### 7. 많은 문제는 결국 권한·경로·버전·네트워크·자원으로 돌아온다
서로 다른 사례를 묶어 보면 실제 반복 범주는 제한적이다.
- 권한: `sudo`, 파일 소유권, `runAsUser`, npm global permission
- 경로: config 위치, mount path, `go package`, `kubeconfig`, `CNAME`
- 버전: TensorFlow-NumPy, Helm v3, 이미지 태그, 플러그인 충돌
- 네트워크: DNS, 포트포워딩, API endpoint, ingress, Docker host
- 자원: OOM, Prometheus 메모리, node 상태, disk·volume

이 노트는 결국 도구별 팁보다 `어느 축이 문제인지 먼저 가르는 습관`을 훈련시킨다.

## 반복되는 판단 축

### 추측 vs 관측
원문은 근거 없는 개입을 경계한다. 감으로 건드리기보다 현재 상태와 로그를 먼저 확인하는 쪽이 기본이다.

### 국소 수정 vs 초기화
설정 한 줄 수정으로 끝낼지, `reset`과 재설치로 복구할지 계속 판단한다. 작은 수정이 빠를 때도 있지만, 상태가 심하게 꼬였으면 초기화가 더 싸다.

### 빠른 복구 vs 근본 원인 추적
당장 환경이나 서비스를 살리는 것이 먼저일 때가 많지만, 같은 일이 반복되지 않으려면 OOM 원인, DNS 누락, selector 불일치 같은 구조적 원인을 다시 확인해야 한다.

### 도구 문제 vs 시스템 문제
애플리케이션이 안 뜨는 것처럼 보여도 실제론 포트, 볼륨, 권한, 노드 상태, swap, cgroup 설정 문제일 수 있다. 원문은 이 경계 혼동을 자주 겪었고, 그래서 시스템 층위 확인이 중요해진다.

### 단발성 팁 vs 재사용 가능한 운영 기억
각 사례는 짧은 팁처럼 보여도 축적되면 다음 장애에서 확인 순서를 빠르게 제공한다. 이 노트의 가치는 개별 해결책보다 축적된 점검 루틴에 있다.

## 원문에서 특히 보존해야 할 부분

### 첫 세 줄의 문제 해결 원칙
짧지만 전체 노트의 헌법이다. 뒤의 수백 줄은 모두 이 원칙을 도구 세계로 확장한 사례로 읽을 수 있다.

### `Ctrl+F` 전제
이 노트는 처음부터 `specific trouble`을 찾으라고 말한다. 즉 모든 문제를 매끈한 서사로 통합하기보다, 증상별 진입점을 남기는 색인형 성격을 보존해야 한다.

### Kubernetes OOM·NotReady·reset 메모
이 구간은 단순 명령어 묶음이 아니라, 관측 → 자원 확인 → 로그 확인 → killall/reset → 재기동이라는 운영 루프를 가장 잘 보여준다.

### 시스템 복구 메모
리눅스 부팅, Wi-Fi, Hypervisor, Ubuntu 22 업데이트 문제처럼 개발 도구보다 시스템 바닥면을 다시 살리는 메모가 함께 있다는 점이 중요하다. 이 노드는 앱 디버깅 노트가 아니라 작업 환경 전체의 복구 문서다.

## 오래 남길 아이디어
- 좋은 트러블슈팅은 똑똑한 추측보다 조건 분리와 관측 가능한 사실 정리에서 시작한다.
- 문제 기록은 `증상`, `확인한 상태`, `수정한 것`, `결과`가 함께 있어야 다음에 재사용된다.
- 재설치와 초기화는 무지의 신호가 아니라 상태를 다시 통제 가능한 범위로 되돌리는 전략일 수 있다.
- 서로 다른 장애도 실제로는 권한, 경로, 버전, 네트워크, 자원 다섯 축으로 자주 환원된다.
- 운영 환경의 문제는 애플리케이션 오류처럼 보여도 시스템 자원과 오케스트레이션 상태에서 시작할 때가 많다.
- 트러블슈팅 문서는 특히 `재진입 속도`를 높이는 문서다.

## 선호 신호
저자는 트러블슈팅을 멋진 한 방 해결보다, 상태를 관측하고 조건을 줄이고 다시 시도하는 복구형 작업으로 보는 경향이 강하다. 또한 장애 문서를 단순 FAQ보다 미래의 자신이 더 빨리 재진입하기 위한 현장 로그로 다루며, 특히 인프라와 쿠버네티스 문제에서는 기본 자원, 네트워크, 설정, 로그를 먼저 확인하는 쪽을 선호한다.

## 이번 패스에서 더 선명해진 점
- 긴 사례 목록을 도구별 요약으로 납작하게 만들지 않고, `반복 증상 축`으로 다시 읽어 재사용성을 높였다.
- Kubernetes 비중이 큰 이유를 단순 분량이 아니라 운영 철학이 가장 선명하게 드러나는 구간으로 해석했다.
- 개발환경 메모와 시스템 복구 메모가 함께 있는 이유를 `작업 환경 전체의 복구 문서`라는 관점으로 묶었다.
