# Tool

- 원본: `source/shdkej-content/Fundamental/Tool.md`
- 성격: 만다라트 `Fundamental` 축의 1:1 대응 카테고리 노드
- 이번 상태: 기존 기준선은 유지하되, 이 노드를 `앱/서비스를 얹기 전에 필요한 하부 운영 감각`의 축으로 더 선명하게 다시 읽었다. 특히 `환경을 직접 길들이는 법`, `꼬였을 때 원복하는 법`, `상위 도구를 떠받치는 기본 표면이 무엇인가`를 앞쪽 구조에서 더 빨리 잡히게 보강했다.

## 한눈에 보는 핵심
이 노트의 `Tool`은 앱 추천 목록이 아니다. 원문은 Linux 명령어, 한글 입력기 설정, 백업 스크립트, SSH 보안, DNS, 스토리지, Git 복구, Terraform, Ansible, Vault, gRPC, 스트리밍까지 한 문서에 쌓아 둔다. 겉으로는 산만하지만 실제 중심은 분명하다.

1. 툴은 기능 이름보다 `내 환경을 직접 만지고 다시 만들 수 있게 하는 손잡이`다.
2. 좋은 툴 메모는 사용법보다 권한, 경로, 백업, 자동화, 장애 복구를 더 오래 남긴다.
3. 로컬 생산성, 서버 운영, 배포 체인, 협업 버전관리는 한 작업 시스템의 서로 다른 면이다.
4. 이 노드는 완성된 정답집보다 `실제로 부딪히며 축적한 운영 기억`의 저장소 성격이 강하다.

## 이 노드가 실제로 답하려는 질문
- 작업 환경을 남의 기본값이 아니라 내 손으로 길들이려면 무엇을 알아야 하는가?
- 툴을 잘 쓴다는 것은 빠르게 실행하는 것인가, 다시 만들고 되돌릴 수 있게 만드는 것인가?
- 작은 쉘 스크립트, cron, Git, SSH, IaC는 어떻게 하나의 운영 감각으로 이어지는가?
- 로컬 편의 설정과 서버 보안 설정이 왜 같은 노트 안에서 함께 다뤄지는가?
- 도구 메모를 제품 카탈로그가 아니라 복구 가능한 작업 기억으로 남기려면 어떤 읽기법이 필요한가?

## 핵심 논지
원본의 `Tool`은 특정 프로그램 설명서가 아니라, 개발자가 일하고 운영하고 복구하고 확장하기 위해 몸에 붙여야 하는 조작 표면 전체를 모은 노드다. bash 조건문, tar 백업, iptables, PostgreSQL replication, DNS zone, LVM, VirtualBox 디스크 확장, Git rebase 충돌, Terraform, Serverless, Vault, Nagios, Kafka, gRPC, 동영상 스트리밍이 한 페이지에 함께 있는 이유도 여기 있다.

즉 이 노드에서 `tool`은 작은 유틸리티 이름이 아니라 다음 전체를 뜻한다.
- 환경을 원하는 상태로 바꾸는 방법
- 반복 작업을 자동화하는 최소 단위
- 실수했을 때 되돌아오는 복구 루프
- 운영 경계를 통제하는 보안 장치
- 여러 시스템을 이어 붙이는 조합 도구

## 만다라트 위치에서 읽기
이 페이지가 `Fundamental/Tool`에 놓여 있다는 점이 중요하다. 같은 `Tool`이라도 [[../Integration/Tool]]이 여러 도구층을 실제 생활·업무 흐름으로 편입하는 통합 노드라면, 여기의 `Tool`은 그보다 아래에서 손, 셸, 경로, 권한, 원복 절차를 익히게 만드는 기반 노드다.

- `Fundamental` 안의 Tool은 새 툴을 추천하는 페이지가 아니라, 어떤 툴이 오더라도 결국 부딪히게 되는 하부 조작면을 붙든다.
- 이 노드에서는 앱 자체보다 `파일이 어디에 있는가`, `권한은 왜 막히는가`, `재시작 순서는 무엇인가`, `다시 만들 수 있는가`가 더 중요하다.
- 그래서 로컬 입력기와 폰트 문제, SSH·iptables, Git 복구, Terraform·gRPC 메모가 한 페이지에 함께 있어도 산만한 이탈이 아니라 `도구를 떠받치는 기초 운영 사다리`로 읽는 편이 맞다.

결국 이 노드는 `툴을 고르는 법`보다 `툴을 감당하는 기본기`를 기록한다. 상위 노드에서 보이는 화려한 앱, 협업 서비스, 에이전트 도구, 배포 도구도 여기서 다루는 하부 감각 없이는 오래 유지되지 않는다.

## 원문을 읽을 때 먼저 보이는 큰 층위

### 1. 시작은 화려한 앱이 아니라 Linux 조작 감각이다
문서 첫머리는 bash cheatsheet, startup file, 한글 입력기, distro 이동, `wget`, `sudo`, `du`, timezone, `tar`, `crontab`, `iptables` 같은 내용으로 채워져 있다. 즉 출발점은 생산성 서비스 비교가 아니라 운영체제와 쉘을 직접 만지는 감각이다.

### 2. 그다음 층은 서버를 굴리기 위한 운영 기억이다
SSH 보안, 파일 권한, DB 설치와 백업, replication, DNS, storage, NFS, FTP, TLS, quota, WSL, code-server, cron 문제 해결 같은 항목이 이어진다. 여기서 툴은 프로그램이 아니라 `상태를 유지하는 절차`에 가깝다.

### 3. Git은 별도 대주제 수준의 복구 도구로 다뤄진다
원문 후반의 Git 구간은 add/commit/push보다 `꼬였을 때 어떻게 복구할 것인가`에 더 오래 머문다. stash, reflog, rebase, reset, merge abort, SSH config, 민감 파일 삭제가 중요하게 남는다.

### 4. ETC 구간은 잡담이 아니라 경계 확장의 기록이다
Ansible, Terraform, Serverless, Vault, Nagios, API gateway, Kafka, gRPC, 스트리밍 메모는 `tool`의 범위가 개인 환경을 넘어 서비스 운영 설계까지 확장되고 있음을 보여 준다.

## 구조 재정리

### 1. 이 노드의 중심은 기능 소개가 아니라 `환경을 손에 넣는 법`이다
원문에서 자주 반복되는 것은 멋진 기능이 아니라 다음 같은 질문이다.
- 어디 파일을 수정해야 하는가
- 권한은 어디서 막히는가
- 자동 실행은 어떻게 거는가
- 같은 세팅을 다시 만들 수 있는가
- 문제 나면 어디서부터 확인하는가

즉 툴의 핵심은 `좋다`가 아니라 `내가 제어할 수 있다`에 있다.

### 2. 쉘 메모는 작은 자동화의 문법 창고다
bash 조건문, `set -u -e`, `find -mtime`, `awk`, `sed`, `cut`, `grep`, `xargs`, `lsof | awk | xargs kill -9` 같은 조각은 사소한 치트시트처럼 보이지만 중요하다. 이 조각들이 있어야 거대한 플랫폼 없이도 반복 작업을 줄일 수 있기 때문이다.

여기서 드러나는 패턴은 다음과 같다.
- 먼저 작은 명령어 조합으로 문제를 푼다.
- 자주 하는 일은 스크립트와 cron으로 민다.
- 도구 사용은 클릭보다 텍스트 명령으로 남겨 재실행 가능하게 만든다.

### 3. 로컬 환경 최적화 메모도 운영 메모의 일부다
한글 입력기, alt key 변경, touchpad, font 깨짐, external monitor, keyboard delay, text-to-image, terminal browser, 테마, i3, WSL 같은 항목은 사소해 보이지만 실제로는 작업 지속성을 좌우한다.

이 문서는 로컬 편의 메모를 하찮게 다루지 않는다. 이유는 명확하다. 작은 마찰이 매일 반복되면 생산성 앱 몇 개보다 더 큰 손실을 만들기 때문이다. 그래서 이 노드에서 툴은 서버만이 아니라 `몸이 자주 부딪히는 인터페이스` 전체를 포함한다.

### 4. 백업과 cron은 툴의 본질을 `지속 가능성`으로 옮긴다
`tar` 백업, 날짜 포맷, 오래된 파일 삭제, DB dump cronjob, 시스템 cron과 사용자 cron 구분은 모두 같은 질문으로 이어진다. `지금 잘 돌아가느냐`보다 `내일도 다시 돌릴 수 있느냐`가 중요하다는 것이다.

이 층에서 반복되는 감각은 다음과 같다.
- 자동화는 편의성 이전에 누락 방지다.
- 백업은 저장이 아니라 원복 가능성 확보다.
- cron은 사소한 예약 기능이 아니라 생활 운영의 최소 플랫폼이다.

### 5. 보안 메모는 툴을 `경계 관리`의 문제로 확장한다
SSH 포트 변경, root 접속 제한, fail2ban, sudoers, 파일 권한, iptables, TLS, github without password, ssh-keygen 같은 내용은 빠른 작업법보다 누가 어떻게 접근하는지를 통제하는 감각을 보여 준다.

즉 이 노드에서 툴 숙련은 다음을 함께 뜻한다.
- 열어야 할 곳만 연다.
- 키와 권한을 분리한다.
- 편의가 신뢰 경계 붕괴로 이어지지 않게 한다.
- 접속 성공 경험보다 안전한 기본 상태를 더 중시한다.

### 6. 데이터베이스, DNS, 스토리지 메모는 `도구`와 `인프라`의 경계를 흐린다
PostgreSQL, Oracle DB, DNS zone, NFS, quota, LVM, VirtualBox HDD 확장 메모는 전형적인 인프라 운영 항목이다. 그런데 이 노드 안에 들어 있다는 것은 저자가 툴을 편집기나 앱보다 훨씬 넓게 본다는 뜻이다.

여기서 툴은 다음처럼 읽는 편이 더 정확하다.
- 시스템을 구성하는 부품
- 상태를 저장하고 전달하는 기반
- 장애 시 직접 해부해야 하는 표면
- 서비스 운영 능력으로 이어지는 실습 장치

### 7. Git 구간의 중심은 협업보다 `작업 복원력`이다
원문 Git 메모의 핵심은 add/commit/push 자체가 아니다. 진짜 중요한 부분은 꼬인 브랜치, 충돌 뒤 복구, 잃어버린 커밋, stash 활용, remote URL, SSH key, hook, fast-forward 문제 같은 운영 기억이다.

이 구간은 다음을 보여 준다.
- Git은 저장소가 아니라 시간축 복구 장치다.
- 브랜치는 분기 전략이면서 안전장치다.
- rebase와 merge는 스타일 문제가 아니라 히스토리 해석과 복원 전략의 문제다.
- 실수를 지우는 능력보다 실수 후 되돌아오는 능력이 더 중요하다.

### 8. IaC와 운영 도구 메모는 `도구를 이어 시스템을 만드는 단계`다
Terraform, Ansible, Serverless, Vault, Nagios, Lambda cronjob, API gateway, gRPC, Kafka, streaming 구간은 더 이상 단일 도구 사용법이 아니다. 여기서는 어떤 도구가 어느 책임을 맡고 어떻게 서로 이어질지를 고민한다.

이 층에서 반복되는 질문은 다음과 같다.
- 수동 명령과 선언형 관리 중 어디에 둘 것인가?
- 빠르게 띄우는 편의와 장기 유지 가능 구조 중 무엇을 우선할 것인가?
- 비밀정보, 배포, 모니터링, 이벤트, 통신을 어떤 레이어에서 분리할 것인가?
- 작은 서비스에도 운영 체계를 얼마나 들여와야 하는가?

### 9. 원문은 `기초 명령어 모음`에서 끝나지 않고 `하부 감각 → 상부 설계`로 올라간다
이 페이지를 다시 읽으면 초반 Linux 메모와 후반 gRPC·API gateway·streaming 메모가 단절된 주제가 아니라는 점이 더 선명하다. 앞쪽에서는 파일, 권한, 경로, 프로세스, cron, 백업, 네트워크를 손으로 다루는 감각을 쌓고, 뒤쪽에서는 그 감각 위에 배포 구조·통신 구조·비밀관리·운영 자동화를 올리려 한다.

즉 이 노드는 단순히 `많이 아는 도구 목록`이 아니라 아래 사다리를 밟는다.
1. OS와 셸을 직접 만질 수 있어야 한다.
2. 그 위에서 로컬/서버 환경을 다시 만들 수 있어야 한다.
3. 그 다음에야 Git, 백업, 보안, 모니터링으로 복원력을 붙인다.
4. 마지막에 Terraform, Serverless, Vault, gRPC처럼 시스템 책임 배치를 고민한다.

이 사다리를 읽어 두면 원문 후반의 넓은 기술 메모가 수집벽처럼 보이지 않고, `기초 운영 감각을 더 큰 구조 판단으로 밀어 올리는 과정`으로 보인다.

## 원문 발췌 구조

### A. Linux 초반부는 `손으로 바로 만지는 조작면`이다
- bash 조건문, `set -u -e`, `find -mtime`, `echo >>`, 날짜 포맷
- startup files, dotfiles 심볼릭 링크
- 한글 입력기, alt key, distro 이동, touchpad·font·monitor 문제
- `du`, `wget`, timezone, `sudo`

원문은 개념 설명보다 `어디를 고치고 어떤 명령을 치는가`를 먼저 남긴다.

### B. 중반부는 `운영 상태를 오래 유지하는 법`으로 넘어간다
- `tar` 백업과 crontab
- iptables와 SSH 보안
- PostgreSQL 설치·replication·dump cronjob
- DNS zone, storage, quota, NFS, FTP

여기서는 도구 이름보다 `서비스가 계속 살아 있게 만드는 절차`가 핵심이다.

### C. 로컬 보조 메모도 생산성 부록이 아니라 `몸의 작업면 튜닝`이다
- OBS, 배터리 상태, i3, w3m, code-server, WSL
- scrcpy, vim lag, xrandr, iPad 보조 모니터, numpad mouse keys
- font cache, keyboard repeat, touchpad gesture, clipboard alias

저자에게 로컬 마찰 제거는 서버 운영보다 덜 중요한 일이 아니라, 같은 생산성 기반층이다.

### D. Git 파트는 사용법보다 `꼬인 히스토리 복구법`에 오래 머문다
- global user 설정, branch 이동, push without login
- sensitive file 삭제, hooks, ignore
- fetch/rebase/merge/reset이 꼬였을 때의 상황 메모
- stash branch, ssh config, fast-forward abort, merge abort

즉 Git도 협업 예절보다 `작업 복원 장치`로 먼저 읽힌다.

### E. ETC 파트는 `서비스 조합 설계`의 실험장이다
- Ansible vs shell, Terraform vs Serverless
- Vault와 대체 비밀관리 도구
- Nagios, Lambda cronjob, cloud infra, Kafka
- gRPC, API gateway, streaming

후반부는 도구 단품 사용보다 `어느 레이어에 어떤 책임을 둘지`를 계속 시험한다.

## 이 원문에서 특히 보존해야 하는 결

### 치트시트의 조각남 자체
원문은 매끈한 교재가 아니다. 이 산발성은 나쁜 품질의 징후만이 아니라 실제 현장에서 필요한 조각을 빠르게 붙잡아 두는 방식이기도 하다. mapped에서는 읽기 쉬운 구조를 만들되, 원문이 가진 `탐색 흔적`을 완전히 지워서는 안 된다.

### `왜`보다 먼저 `어디를 건드렸는가`가 나오는 서술
파일 경로, 설정 파일명, 명령어 옵션, 서비스 재시작 순서가 많이 남아 있다. 이는 개념 요약보다 현장 재진입을 우선하는 기록 습관을 보여 준다.

### 미완성 질문과 망설임
- Terraform만 쓸지 Serverless를 같이 쓸지
- DevOps 역할을 어디까지 분리할지
- API gateway를 운영 환경과 어떻게 연결할지
- gRPC 응답 구조를 어떻게 설계할지
- 영상 스트리밍 서버를 어떤 방식으로 둘지

이런 질문은 단순 미정 상태가 아니다. 이 노드가 도구를 소비가 아니라 설계 판단의 대상으로 다룬다는 증거다.

## 반복되는 판단 축

### 편의성 vs 재현성
쉽게 한 번 되는 것보다 나중에 다시 세팅 가능한가가 더 중요하게 남는다.

### GUI vs CLI
GUI는 빠르지만 휘발되고, CLI는 번거롭지만 기록과 자동화가 가능하다. 원문은 지속적으로 CLI 쪽으로 기운다.

### 단발 해결 vs 운영 기억
문제 하나를 지금 푸는 것보다 다음에도 같은 문제를 덜 겪게 만드는 쪽으로 메모가 축적된다.

### 로컬 작업성 vs 서버 운영성
키보드와 폰트 문제, DB replication과 DNS 설정이 함께 있는 것은 이상한 혼합이 아니라 실제 개발자의 작업면이 분리되지 않기 때문이다.

### 빠른 실행 vs 안전한 복구
배포와 자동화 욕구는 강하지만, 동시에 백업, 롤백, 키 관리, 버전관리, 재빌드를 계속 붙잡는다.

### 상위 도구의 편리함 vs 하부 표면의 직접 통제
원문 후반에는 Terraform, Serverless, Vault, gRPC 같은 상위 도구가 나오지만, 메모의 무게중심은 여전히 셸, 파일, 권한, 네트워크, Git 복구에 남아 있다. 즉 저자는 추상화를 좋아하더라도, 문제가 나면 결국 하부 표면을 다시 열어야 한다는 사실을 놓치지 않는다.

## 원문 층위별 압축

### A. 손에 익혀야 하는 최소 조작
- shell 문법
- 파일/프로세스/권한 조작
- text processing
- cron과 간단한 스크립트

### B. 오래 일하기 위한 환경 세팅
- 입력기, 폰트, 키보드, 터치패드
- 모니터, 배터리, 테마, 터미널
- WSL, code-server 같은 작업 진입 환경

### C. 서버를 유지하기 위한 운영 기억
- SSH, sudo, 방화벽, TLS
- 백업, dump, timezone, storage
- DB, DNS, NFS, FTP, quota

### D. 작업 이력을 지키는 복구 도구
- Git 브랜치 전략
- 충돌 처리와 복귀
- 원격 인증과 훅
- 큰 파일과 민감 정보 정리

### E. 시스템을 확장하는 조합 도구
- Terraform, Ansible, Serverless
- Vault, Nagios, Lambda cronjob
- API gateway, gRPC, Kafka, streaming

## 오래 남길 아이디어
- 툴은 앱 이름보다 상태를 제어하고 다시 만들 수 있게 해 주는 손잡이다.
- 좋은 도구 메모는 기능 소개보다 설치, 권한, 경로, 자동화, 복구 순서를 남긴다.
- 작은 쉘 스크립트와 cron은 거대한 플랫폼 이전에 갖춰야 할 가장 현실적인 자동화 단위다.
- 로컬 환경 마찰 제거도 서버 운영만큼 중요한 툴 작업이다.
- Git은 버전 관리보다 작업 복원력 확보라는 측면에서 읽어야 이 노드가 살아난다.
- IaC와 운영 도구 메모는 취미 수집이 아니라 `시스템을 조합하는 능력`의 초기 형태다.
- 이 노드에서는 상위 도구를 더 붙이기 전에 `권한/경로/원복`의 하부 표면을 먼저 붙잡는 태도가 핵심 자산이다.
- 이 노드의 좋은 읽기 순서는 `명령어 암기 → 파일 경로 기억 → 복구 루프 이해 → 레이어 책임 배치`가 아니라, 그 역순에 가깝다. 먼저 왜 복구와 책임 배치가 필요한지 보이고, 그 다음 명령어 조각이 의미를 얻는다.

## 선호 신호
저자는 툴을 화려한 최신 앱보다 `마찰을 줄이고, 원하는 상태를 만들고, 다시 복원할 수 있게 하는 운영 표면`으로 보는 경향이 강하다. 또한 도구를 기록할 때도 기능 설명보다 파일 경로, 권한, 실행 순서, 재시작, 백업, 자동화처럼 실패를 줄이는 정보를 더 오래 붙잡는다. 즉 툴을 소비하는 사람이라기보다, 툴을 조합해 자기 작업 환경을 길들이는 사람에 가깝다.

## 이번 갱신 메모
- 기존 baseline은 유지한 채, 이 노드를 만다라트 `Fundamental` 위치에서 다시 읽는 해석을 추가해 `상위 도구를 떠받치는 하부 운영 감각`이라는 역할이 더 빨리 드러나게 했다.
- `상위 도구의 편리함 vs 하부 표면의 직접 통제` 축을 보강해, 후반 Terraform·gRPC·Vault 메모도 결국 셸·권한·경로·복구 감각 위에서 읽어야 한다는 점을 더 선명하게 적었다.
- `권한/경로/원복`을 먼저 붙잡는 태도를 오래 남길 아이디어에 명시해, 이 페이지가 기능 모음보다 운영 기본기 노드라는 성격을 더 단단히 고정했다.
