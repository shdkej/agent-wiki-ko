# Architecture

- 원본: `source/shdkej-content/Fundamental/Architecture.md`
- 상태: 점진 심화
- 역할: 원본 만다라트 노드와 1:1 대응하는 기반 카테고리 페이지

## 한눈에 보는 요약
이 노드는 특정 아키텍처 스타일을 주장하는 문서라기보다, 변화가 많은 현실에서 무엇을 먼저 고정하고 무엇을 늦게 결정해야 하는지에 대한 운영 메모에 가깝다. 핵심은 화려한 구조 선택보다 변경 용이성, 일관성, 피드백 루프, 실패 대응, 자동화의 신중한 도입, 그리고 사람과 팀이 감당할 수 있는 복잡도 관리다.

## 핵심 논지
원본에서 아키텍처는 정답 설계도가 아니라, 세부사항 결정을 가능한 한 늦추면서도 시스템이 안정적으로 진화할 수 있게 만드는 기준선이다. 즉 좋은 아키텍처는 처음부터 완성된 구조를 강요하는 것이 아니라, 변경 비용을 낮추고 일관성을 높이며, 실패와 운영 부담을 견딜 수 있는 방향으로 정책과 경계를 잡는 일이다.

## 왜 이 노드가 중요한가
- `Fundamental` 축에서 코딩, 소프트웨어, 인프라, 네트워크 같은 하위 판단을 묶어주는 상위 기준 노드다.
- 원본의 관심사가 단순한 패턴 암기가 아니라 운영 가능한 구조, 팀 적합성, 자동화의 조건, 분산 시스템의 경계 설정에 있음을 보여준다.
- 이후 다른 노드를 읽을 때도, "무엇이 더 강력한가"보다 "무엇이 더 늦게 결정 가능하고, 더 쉽게 바꿀 수 있고, 더 일관되게 유지되는가"라는 판단 기준을 제공한다.

## 원본에서 먼저 잡히는 구조

### 1. 아키텍처의 목적을 속도보다 진화 가능성으로 둔다
도입부는 인프라 구축과 배포를 쉽고 빠르게 하겠다고 시작하지만, 바로 이어서 강조되는 것은 다음이다.
- 세부사항 결정을 가능한 한 늦춘다.
- 변하는 것과 변하지 않을 것을 구분한다.
- 변경이 한 부분에 국한되도록 만든다.
- 피드백을 받고 점진적으로 발전한다.

즉 빠름 자체보다, 빠른 수정과 지속 가능한 발전이 가능한 구조가 목표다.

### 2. 좋은 구조보다 나쁘지 않은 구조를 더 중시한다
원본은 이상적인 아키텍처를 찾기보다 아래 원칙을 반복한다.
- 완벽보다는 안정
- 모든 것은 트레이드오프
- 환경에 맞추는 것이 강력한 것보다 낫다
- 나쁘면 망한다

이 때문에 이 노드는 특정 방법론 예찬보다, 안티패턴을 줄이고 운영 가능한 기준점을 유지하려는 방어적이면서도 실용적인 성향이 강하다.

### 3. 구조 논의가 반드시 팀과 운영 현실로 이어진다
원본은 레이어, MSA, DDD, Clean Architecture, 육각형 아키텍처 같은 이름들을 적지만, 관심은 명칭보다 아래 질문에 있다.
- 우리 팀 구조와 맞는가
- 관리 포인트가 늘어나지 않는가
- 문서화와 피드백 루프가 있는가
- 자동화가 오히려 불안정성을 키우지 않는가
- 한 사람이 감당 가능한 범위를 넘기지 않는가

즉 아키텍처는 코드 구조와 조직 구조를 함께 다루는 현실 문제로 읽힌다.

## 원문 발췌 구조
- 설계 목표: 빠른 구축보다 변경 용이성, 피드백, 안정성을 우선한다.
- 체크포인트 질문군: 배포 주기, 강제성/자율성, 코드·설정 분리, 레포/디렉터리 경계, 팀 구조 적합성을 사전 질문으로 둔다.
- 설계/구성 요소: 서비스 유형, 개인정보, 데이터 양, IaC, 최소 컴포넌트와 추가 컴포넌트를 함께 적어 둔다.
- 일반 아키텍처 메모: 일관성, 중심만 남기기, 인프라-데이터-서비스 3요소, 대표적 구조 패턴, 코드 레이아웃, 자동화 태도가 한 덩어리로 이어진다.
- 보안 층: SSH, TLS, OAuth, 세션 관리 메모가 `구조를 안전하게 운영하는 법`을 배우는 질문집처럼 붙어 있다.
- 분산 시스템 층: MSA, 서비스 메시, CQRS, 이벤트 드리븐, 섀시가 `분리 이후 어떻게 다시 관리할 것인가`라는 후속 문제로 모인다.

## 반복되는 판단 축

### 1. 변경 용이성
원본에서 가장 자주 되돌아오는 축이다.
- 변하는 것과 변하지 않는 것을 나눈다.
- 코드와 설정을 분리한다.
- 기능 추가 시 기존 코드를 크게 흔들지 않게 한다.
- 배포 이미지는 덮어쓰지 않고 처음부터 다시 빌드한다.
- 플러그인, 어댑터, 피처 플래그처럼 변경을 국소화하는 장치를 선호한다.

여기서 아키텍처는 현재 구조를 설명하는 도면보다, 미래 변경 비용을 낮추는 장치에 가깝다.

### 2. 일관성
원본은 "어떤 아키텍처인가"보다 "일관성이 있는가"를 더 앞세운다.
- 한 부분을 익히면 다른 부분도 비슷하게 이해되어야 한다.
- 역할과 책임이 반복 가능한 규칙으로 나뉘어야 한다.
- 코드 레이아웃과 에러 처리, 인터페이스, 배포 방식이 제각각이면 학습 비용이 커진다.

즉 일관성은 미학이 아니라 온보딩, 유지보수, 가독성을 위한 운영 자산이다.

### 3. 피드백과 관측 가능성
원본은 구조를 만들었다고 끝내지 않는다.
- 무엇을 보면 잘 진행되는지 확인할 수 있어야 한다.
- 변화의 기록을 남겨야 한다.
- 테스트 문서, 모니터링 문서, API 문서를 자동화 대상으로 본다.
- 로그, 텔레메트리, 헬스체크, 분산 추적을 반복해서 언급한다.

즉 좋은 구조는 "잘 만들었다고 믿는 구조"가 아니라 "계속 상태를 확인할 수 있는 구조"다.

### 4. 자동화의 양면성
자동화는 선호되지만 맹신되지 않는다.
- 자동화는 오류를 일으키기 쉽다.
- 한 번에 전체를 바꾸는 스크립트는 자주 실패한다.
- 기존 업무 환경과 누락된 예외 때문에 자동화가 꼬인다.
- AI 자동화는 결과 일관성이 낮을 수 있다.

그래서 원본은 자동화를 목표가 아니라, 신중히 도입해야 하는 증폭기로 본다.

### 5. 복잡도와 관리점 통제
원본은 기능을 늘리면서도 관리 포인트는 줄이고 싶어 한다.
- 서비스는 많아질 수 있지만 복잡도는 낮추고 싶다.
- 외부 서비스와 자체 구축의 균형을 고민한다.
- 쿠버네티스 기능을 많이 쓰면 다른 도구는 줄지만 종속성이 커진다.
- MSA는 분명한 장점이 있지만 작은 규모에서는 과할 수 있다.

이 축은 기술 선택의 핵심을 "더 세련된 구조"가 아니라 "감당 가능한 운영 범위"에 두고 있음을 보여준다.

## 원본을 관통하는 주요 설계 태도

### 정책과 세부사항을 분리하려는 태도
클린 아키텍처 인용이 단순 레퍼런스가 아닌 이유는, 원본 전반이 계속 같은 감각을 반복하기 때문이다.
- 도메인이 디자인을 만든다.
- 정책과 가이드를 구분한다.
- 세부 구현은 뒤로 미룬다.
- 설정 가능성과 플러그인 방식을 선호한다.

즉 저자는 구조를 미리 꽉 채우기보다, 중심 정책만 남기고 세부는 교체 가능하게 두려 한다.

### 구조보다 경계 설정을 중시하는 태도
원본의 질문들은 대부분 경계 문제다.
- 코드와 설정의 경계
- 자주 변하는 것과 덜 변하는 것의 경계
- 서비스와 인프라의 경계
- 동기와 비동기의 경계
- 사용자 자율성과 시스템 강제성의 경계
- 모놀리스와 마이크로서비스의 경계

이 노드에서 아키텍처 역량은 패턴 이름을 아는 것보다, 어디를 나누고 어디를 묶을지 판단하는 능력에 더 가깝다.

### 운영 현실을 끝까지 끌고 가는 태도
원본은 추상 논의만 하지 않는다.
- 백업 전략
- 오토스케일링과 오토리커버리
- 보안 키 관리
- 배포 단위와 롤백 위험
- 세션 관리와 SPOF
- 대용량 데이터 처리와 캐시 규칙

즉 설계 원칙은 반드시 운영 디테일과 연결되어야 의미가 있다고 본다.

## 세부 영역별 해석

### 체크포인트 메모의 의미
`Check Point` 섹션은 아키텍처를 사전 점검 질문 집합으로 다룬다는 점에서 중요하다.
- 수시 배포 vs 정기 배포
- 강제성 vs 자율성
- 계산 중심 구조 vs 데이터 중심 구조
- 기능별 분리 vs 역할별 분리
- monorepo vs microrepo
- 깊이 vs 넓이

이 질문들은 정답을 제시하지 않는다. 대신 설계 전에 반드시 드러내야 할 트레이드오프를 목록화한다. 즉 원본은 아키텍처를 답안집보다 질문 프레임으로 이해한다.

### 최소 컴포넌트에서 다음 패러다임으로 넘어가는 방식
원본 후반의 `필요한 컴포넌트` 목록은 단순 기술 스택 메모 이상이다. 먼저 `db / backend / frontend / file / image / domain / dns / MQ` 같은 최소 구성을 잡고, 이후 필요가 생길 때 오토스케일링, ingress, CDN, secret 관리, 분석 저장소, 오류 추적, 대기열, failover를 붙인다. 이 흐름은 중요한 선호를 드러낸다.
- 처음부터 완성형 플랫폼을 세우기보다 최소 동작면을 먼저 확보한다.
- 규모가 커질 때는 다음 패러다임을 준비하되, 새 도구는 관리점 축소와 관측 가능성 증가를 동시에 줘야 한다.
- `서비스를 많이 만들고 싶지만 복잡도는 낮추고 싶다`는 메모는 바로 이 증설 사다리의 긴장을 설명한다.

즉 이 노드에서 아키텍처는 거대한 청사진보다, `최소 기반 → 운영 관측 → 병목 확인 → 필요한 층만 추가`의 순차적 증설 절차에 가깝다.

### 마이크로서비스 메모의 의미
원본은 MSA를 단순히 선진 구조로 이상화하지 않는다.
- 관리 포인트가 늘어난다.
- 서비스 간 커뮤니케이션이 필요하다.
- API gateway, 서비스 메시, 서비스 디스커버리의 역할 차이를 봐야 한다.
- 데이터 일관성과 조인 문제를 따로 해결해야 한다.
- 큰 팀을 작은 팀으로 나누면 정보 장벽도 커진다.

즉 마이크로서비스는 기술 구조인 동시에 조직 구조 재편이기도 하며, 분산은 자유만이 아니라 새로운 복잡성도 만든다.

### 보안 메모의 의미
보안 파트는 완성된 설명보다 학습 과정이 많이 남아 있다. 그런데 바로 그 점이 중요하다.
- SSH, TLS, OAuth, 세션 관리 등 기본 메커니즘을 이해하려는 흔적이 보인다.
- 정확히 모르는 지점을 질문 형태로 남긴다.
- 인증, 신원 확인, 비밀키 관리, 세션 저장 위치처럼 운영적 핵심에 관심이 모인다.
- `ssh key를 어디에 두는가`, `CI가 비밀키를 갖는 상황을 어떻게 이해할 것인가` 같은 메모는 추상 보안 원리보다 실제 운영 배치 문제를 먼저 붙든다.

즉 이 노드는 완성된 지식 노트이면서 동시에 "아키텍처를 배우는 방식"도 보여준다. 모호한 부분을 숨기지 않고 질문으로 남겨 경계를 확인한다.

### 자동화와 AI 메모의 의미
원본은 자동화가 인간 오류를 줄일 수 있다는 점을 인정하면서도, 자동화 방식마다 다른 실패 양상을 구분한다.
- 사람은 누락한다.
- 스크립트는 정해진 폼 밖을 처리하지 못한다.
- AI는 결과가 매번 달라질 수 있다.

이 구분은 중요하다. 자동화는 하나의 축이 아니라, 결정성, 유연성, 누락 가능성, 검증 가능성을 어떻게 조합할지의 문제라는 뜻이기 때문이다.

## 이 노드에서 특히 보존해야 할 긴장
- 변경 용이성을 높일수록 추상화와 구성 요소 수가 늘어날 수 있다는 긴장
- 자동화를 늘릴수록 실패 반경과 불투명성이 커질 수 있다는 긴장
- MSA가 자율성을 주지만 관리 포인트도 폭증시킨다는 긴장
- 외부 서비스를 쓰면 편하지만 통제력은 줄어든다는 긴장
- 일관성을 높일수록 자유도가 줄 수 있다는 긴장
- 세부 결정을 늦출수록 초기 속도는 좋아지지만, 기준선이 약하면 나중에 혼란이 커질 수 있다는 긴장

원본을 납작하게 만들지 않으려면 이 긴장들을 해소해버리기보다 계속 남겨두는 편이 맞다.

## 원본에서 드러나는 저자의 선호

### 1. 완결형 설계보다 기준선 설계를 선호한다
처음부터 모든 것을 정하기보다, 좋은 기준점을 만들고 계속 개선하는 방식을 선호한다.

### 2. 강한 프레임워크보다 붙이는 구조를 선호한다
어댑터, 플러그인, 외부화, 설정 가능성 같은 표현이 반복된다. 즉 전체를 덮는 구조보다 교체 가능한 연결 방식을 더 편하게 느낀다.

### 3. 사람이 이해할 수 있는 구조를 중시한다
일관성, 쉽게 설명 가능함, 팀 구조 적합성, 사용자가 관리 가능한 서비스, 문서화 같은 요소가 반복된다. 이는 구조의 우아함보다 인간의 이해 가능성을 더 중시한다는 뜻이다.

### 4. 실패를 감추지 않고 빠르게 드러내는 구조를 선호한다
헬스체크, 피드백 루프, 로그, 테스트, 에러 처리, 실패 상황 고려, 버퍼, 백업 전략이 모두 같은 방향을 가리킨다.
