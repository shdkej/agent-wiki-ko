# Coding

- 원본: `source/shdkej-content/Fundamental/Coding.md`
- 성격: 만다라트 `Fundamental` 축의 1:1 대응 카테고리 노드
- 이번 상태: 기존 baseline을 유지하면서 `설계 원칙 → 실패 처리 → 검증 → 언어별 실행 모델`의 사다리로 점진 심화

## 한눈에 보는 핵심
이 노트에서 말하는 `코딩`은 문법 암기나 스타일 취향이 아니라, **변경을 견디는 구조를 만들고, 예외 상황을 드러내고, 테스트와 문서화로 동작을 고정하는 일**에 가깝다. 원문은 SOLID, OOP, DI, 테스트, 에러 처리, 코딩 컨벤션, Python/Go/JS/Kotlin/함수형 메모까지 매우 넓게 퍼져 있지만, 실제로는 같은 질문을 반복한다.

1. 좋은 코드는 처음부터 완벽한 코드가 아니라 **나중에 바꾸기 쉬운 코드**다.
2. 추상화는 멋이 아니라 **교체 가능성·테스트 가능성·변경 반경 축소**를 위한 장치다.
3. 해피 패스보다 **경계 조건·실패 경로·운영 중 복구 방식**이 더 중요한 설계 대상이다.
4. 언어 차이는 크지만, 결국 **읽기 쉬움·명시성·단순함·유지보수성**으로 수렴한다.

## 만다라트 위치에서 읽기
이 노드는 `Fundamental` 안에 있기 때문에 최신 프레임워크 정답표보다 더 밑바닥의 판단 문법을 다룬다. 즉 `코딩 잘하기` 팁 모음이 아니라,

- 무엇을 추상화해야 하는가
- 실패는 어디서 처리해야 하는가
- 어떤 테스트가 어떤 변경을 허용하는가
- 언어별로 어디까지 상태와 복잡성을 허용할 것인가

를 반복 학습하는 기초 카테고리 노드로 읽는 편이 맞다.

## 원문을 다시 읽고 보이는 큰 구조
원문은 아래 층위가 길게 겹쳐 있다.

- 설계 원칙: SOLID, OOP, 상속 vs 구성, 결합도와 응집도
- 제어 구조: DI, IoC, AOP, 인터페이스, 추상화
- 품질 확보: Clean Code, TDD, 리팩토링, 각종 테스트
- 운영 안전성: 에러 처리, 빠른 리턴, 복구 가능성, 로깅
- 팀 규칙: 코딩 컨벤션, lint, 파일명/공백 스타일
- 언어 메모: Python, Golang, Dart, Javascript, Kotlin, 함수형 프로그래밍

즉 이 노트는 특정 언어 튜토리얼이 아니라, **코드를 어떻게 짜야 오래 버티는가**를 여러 언어와 사례로 시험하는 기준 노트다.

## 원문 발췌 구조
- **설계 원칙 층**: SOLID, OOP, DI/IoC, Clean Code, 패턴
- **검증 층**: TDD, 유닛/시스템/E2E/인수/부하/스트레스 테스트
- **실패 처리 층**: 에러 전파, 복구 가능성, 로깅, 호출자 책임
- **구현 레이아웃 층**: `main`에는 흐름만 두기, 큰 util 경계, handler/logic 분리
- **언어 철학 층**: Python의 명시성, Go의 단순함, JS/Dart의 비동기 실행 모델, Kotlin/함수형의 상태 제한 감각
- **세부 실험 층**: 포인터/레퍼런스, 복사, context, defer, HTTP handler, WASM, 내장 함수, 문법 실험

이 구조를 보면 원문은 `코드 테크닉 모음`이 아니라 **변경 가능한 프로그램을 만들기 위한 판단 훈련장**에 가깝다.

## 코딩을 바라보는 중심 질문
### 1) 변경이 생겨도 수정 범위가 국소적인가
SOLID, 추상화, 인터페이스, 패키지 목적 단일화가 계속 반복되는 이유는 모두 여기에 연결된다. 좋은 구조는 멋진 구조가 아니라 **변경 이유가 퍼지지 않는 구조**다.

### 2) 실패가 생겨도 어디서 멈추고 어디서 복구할지 분명한가
에러 처리 메모가 유난히 길다는 것은 작성자가 기능 구현보다 **실패 처리 책임 배치**를 더 중요한 설계 문제로 본다는 뜻이다.

### 3) 테스트와 문서화로 동작을 다시 고정할 수 있는가
테스트는 품질 장식이 아니라, 리팩토링과 배포를 가능하게 하는 안전장치로 읽힌다. 특히 `외부 API가 아니라 내 구현을 테스트한다`는 메모가 이 시선을 잘 보여준다.

### 4) 사용하는 언어의 철학을 거슬러 복잡성을 늘리고 있지 않은가
Python, Go, JS, Dart, Kotlin, 함수형 메모가 다 흩어져 있어도 공통 메시지는 같다. **그 언어가 기본으로 유도하는 안전한 문법을 따라가라**는 것이다.

## 설계 원칙 파트의 재구성
### SOLID는 교과서가 아니라 변경 비용 관리법
원문은 SOLID를 짧게 적어두지만 실제 강조점은 두 가지다.

- 한 클래스, 한 모듈, 한 패키지가 감당하는 이유를 좁힌다.
- 구체 구현보다 추상 기준에 의존해 테스트와 교체를 쉽게 만든다.

특히 DIP에 대한 관심은 외부 라이브러리 예찬보다 **테스트 더블을 세우고 경계를 뚜렷하게 하려는 욕구**에 가깝다. 추상화는 미학이 아니라 검증과 교체의 도구다.

### OOP는 상속 찬양보다 역할 분리 감각에 가깝다
원문은 캡슐화, 폴리모피즘, 상속, 위임을 모두 언급하지만 실제 결론은 상속을 넓게 쓰기보다 **구성·위임·인터페이스** 쪽에 더 기울어 있다.

- 상속은 공통 속성을 빠르게 묶는다.
- 그러나 요구가 갈라지면 상속 트리가 쉽게 경직된다.
- 무엇을 고정하고 무엇을 바꿀지 아직 모호하다면 구성이 더 안전하다.

이 흐름은 [[./Architecture]]에서 반복되는 `정책과 세부사항 분리`, `교체 반경 축소`와 같은 감각과 직접 닿아 있다.

### 결합도와 응집도는 결국 변경 반경을 설명하는 말
원문은 교과서 정의를 길게 풀지 않는다. 대신 실전 감각으로 남긴다.

- 응집도: 한 단위 내부의 데이터와 동작이 실제로 서로 붙어 있는가
- 결합도: 다른 단위와의 연결이 느슨해 교체와 수정이 쉬운가

즉 좋은 코딩은 우아한 문체보다 **변경이 퍼지지 않도록 경계를 긋는 기술**에 가깝다.

## 구현 레이아웃 파트의 재구성
### `main`은 실행 순서를 설명하는 표면이다
원문 중간의 Go 예시는 단순 문법 노트가 아니라, 작성자가 프로그램의 상층을 어떻게 보고 싶은지 보여준다.

- `main`에는 전체 흐름만 둔다.
- 초기화, 헬스체크, 로깅, HTTP 연결, 비즈니스 로직은 분리한다.
- 엔트리포인트만 읽어도 어떤 순서로 부팅되고 어디서 실패할지 보여야 한다.

즉 좋은 코드는 함수 하나가 예쁜 것보다 **프로그램 전체 흐름이 설명 가능한 상태**를 우선한다.

### 큰 util 파일을 경계하는 이유
`common.js`나 `util.js`를 비판하는 메모는 중복 제거 반대가 아니다. 핵심은 **맥락 없는 공용화가 실제 재사용성을 해친다**는 것이다. 공용 함수가 많다는 사실보다,

- 왜 이 경계에 있어야 하는지
- 누구의 호출 규칙을 대신 묶는지
- 다른 모듈이 이 파일에 기대면서 결합이 커지지 않는지

가 더 중요하다.

### 빠른 리턴은 문체가 아니라 실패 노출 방식
원문은 Go식 빠른 리턴과 명시적 에러 반환을 자주 붙잡는다. 이는 취향보다,

- 실패 경로를 빨리 드러내고
- 중첩을 줄이며
- 호출자가 선택권을 갖게 하고
- 어디서 멈췄는지 한눈에 보이게 하려는

운영 친화적 레이아웃 감각으로 읽는 편이 맞다.

## 테스트 파트의 재구성
### 테스트는 리팩토링 허가서다
원문은 TDD, 유닛 테스트, 시스템 테스트, E2E, 인수 테스트, 부하 테스트를 모두 훑는다. 하지만 핵심은 한 줄이다.

> 테스트는 코드를 더 구체적으로 만들고, 나중의 변경을 덜 무섭게 만든다.

그래서 이 노트에서 테스트는 예절이 아니라 **변경 가능한 코드의 전제조건**이다.

### 테스트 층위를 구분하려는 의도가 강하다
원문은 계속 이런 질문을 던진다.

- 단일 모듈을 검증하는가, 전체 요구사항을 검증하는가
- 기능 전반이 바뀌면 A/B 테스트나 블루그린 배포가 필요한가
- 성능 테스트, 부하 테스트, 스트레스 테스트는 어떻게 다른가
- 인수 테스트를 기준으로 유닛 테스트를 재배치할 수 있는가

즉 작성자는 테스트를 하나의 기술보다 **변화 크기와 위험도에 맞게 다른 검증 표면을 조합하는 체계**로 이해하려 한다.

### 경계 조건 감각이 핵심이다
원문 전반에 `boundary check`, `해피 패스만 생각하지 않기`, `최악의 상황`이 반복된다. 이는 코딩을 알고리즘 풀이보다 **예외 상황에서 무너지지 않는 시스템 설계**로 보는 시선이다.

## 에러 처리 파트의 재구성
### 실패를 어디서 처리할지 정하는 것이 설계다
원문에서 가장 오래 붙드는 질문 중 하나는 이것이다.

- 호출자가 처리해야 하는가
- 하위 유닛이 자체 처리해야 하는가
- 복구 가능한가, 중단해야 하는가
- 로그만 남길 것인가, 상위로 전파할 것인가

중요한 건 정답 하나가 아니라, **에러 처리 책임을 흐리지 않는 것**이다.

### Go식 에러 처리에 끌리는 이유
원문은 Go의 `error` 반환, `defer`, `panic/recover`, `Fatal`의 위치를 자주 언급한다. 이는 문법 선호라기보다,

- 실패가 숨지 않고
- 호출자가 선택권을 가지며
- 복구와 중단의 경계가 비교적 명시되고
- 운영 중 관측 가능성이 높아지기 때문

으로 읽힌다.

### 복구 가능성 중심 시선이 강하다
원문에는 `정지 없이 회복해서 진행`, `호스트가 결정`, `에러 종류 구분`, `어디서 났는지 알아야 한다`는 생각이 반복된다. 이 노트는 예외를 단순히 잡는 법보다 **복구 가능성을 얼마나 늦게까지 보존할 수 있는가**를 더 중요하게 본다.

## 컨벤션 파트의 재구성
### 컨벤션은 취향 통일이 아니라 인지 비용 절감 장치
원문은 `-` vs `_`, `tab` vs `space`, camelCase vs underscore 등을 적어 놓지만, 결론은 단순하다.

- 회사나 팀의 컨벤션을 따른다.
- lint 같은 자동 기준을 둔다.
- 읽는 사람의 비용을 줄인다.

즉 컨벤션은 미적 논쟁보다 **협업 재진입성을 위한 운영 규칙**이다.

### 코드만으로 설명되게 하려는 욕구와 문서 필요성이 함께 있다
원문은 `코드만으로 얘기할 수 있도록`, `주석 대신 프로그램만 읽어서 이해되도록`을 말하면서도 `문서화는 중요하다`, `주석은 필요하다`도 함께 적는다. 이는 모순이라기보다,

- 1차 목표는 코드 자체의 자명성
- 2차 보조는 문서와 주석

이라는 계층적 태도로 읽는 편이 자연스럽다.

## 언어별 메모를 하나의 사다리로 읽기
원문 후반부의 방대한 언어 메모는 사실 수집이 아니라, 같은 코딩 원칙이 각 언어에서 어떻게 구현되는지 시험하는 재료층이다.

### Python — 명시성과 객체 감각
- Beautiful / Explicit / Simple에 대한 선호
- 객체 모델, `id()`, 복사, 참조, 내장 함수, 메모리 관리 호기심
- 실용성이 순수성보다 앞선다는 태도

### Golang — 실패를 숨기지 않는 운영 문법
- 명시적 에러 처리, 빠른 리턴, 패키지 목적 단일화
- `context`, `defer`, `handler`, `goroutine` 같은 운영형 관심사
- HTTP 서버, 모듈 관리, WASM, 배포 실험까지 이어지는 실제 제작 메모

### Dart / Javascript — 비동기 실행 모델 이해
- callback, Promise, event loop, Future, isolate, queue 차이 확인
- 문법 암기보다 런타임에서 언제 실행되고 무엇이 블로킹되는지 파악하려는 성향

### Kotlin / 함수형 — 상태·상속·부작용 제한 감각
- `final` 기본값, `open`, `abstract`, `interface`의 경계 확인
- 불변성, 순수함수, 커링, 고계함수로 추론 가능성 높이기
- 상속/가변 상태/부작용을 어디까지 허용할지에 대한 경계 감각

### 언어 메모를 묶어서 보면 보이는 공통 질문
- Python에서는 `명시적이되 과하게 복잡하지 않게` 쓸 수 있는가
- Go에서는 `실패와 동시성을 숨기지 않고` 드러낼 수 있는가
- Javascript/Dart에서는 `비동기 실행 모델을 착각 없이` 다룰 수 있는가
- Kotlin/함수형에서는 `상속·상태 변경·부작용`을 얼마나 조심스럽게 제한할 수 있는가

즉 언어 메모의 공통 주제는 기능 비교가 아니라, **복잡성을 어디서 허용하고 어디서 차단할지에 대한 운영 판단**이다.

## 이 노드가 실제로 선호하는 코딩 루프
원문을 실천 흐름으로 압축하면 아래 순서가 반복된다.

1. **경계를 먼저 세운다** — 역할, 변경 이유, 패키지 목적을 나눈다.
2. **상층 흐름을 보이게 둔다** — `main`에는 순서만 남기고 세부는 하위로 내린다.
3. **실패를 숨기지 않는다** — 에러 전파와 복구 책임을 분명히 둔다.
4. **테스트로 동작을 고정한다** — 리팩토링과 배포가 가능할 만큼 검증 장치를 둔다.
5. **컨벤션과 문서로 재진입성을 높인다** — 개인 취향보다 팀의 읽기 비용을 줄인다.
6. **언어 철학에 맞춰 다듬는다** — Python/Go/JS/Kotlin을 같은 방식으로 밀어붙이지 않는다.

이렇게 보면 이 노드의 핵심은 `예쁜 코드`보다 **다시 읽고, 고치고, 넘겨받을 수 있는 코드 상태**를 만드는 데 있다.

## 이 노드에서 반복되는 숨은 패턴
### 1. 교과서 개념을 실전 질문으로 번역한다
SOLID를 외우는 데서 멈추지 않고 테스트 용이성, 변경 이유 분리, 교체 가능성으로 다시 읽는다.

### 2. 구현보다 운영 시나리오를 먼저 걱정한다
에러 처리, 부하 테스트, 블루그린 배포, 경계 조건, 최악의 상황 대비가 계속 나온다.

### 3. 언어 기능보다 읽는 사람의 비용을 더 중시한다
명시적 에러 처리, 명확한 컨벤션, util 남용 경계, `main`에는 흐름만 두려는 태도가 반복된다.

### 4. 코드 상층에서 실행 흐름이 보여야 한다
엔트리포인트는 순서를 설명하고, 세부 구현은 하위 레이어로 내려보내며, 실패 지점은 숨기지 않는다.

### 5. 추상화의 목적을 검증 가능성과 교체 가능성에 둔다
DIP, 인터페이스, 구성, 패키지 API 고정, 테스트 더블 메모가 이 축으로 연결된다.

### 6. 언어 지식도 결국 운영 판단으로 끌고 온다
Python 복사/참조, Go `context/defer`, JS Promise, Kotlin `final` 기본값 모두 `실수 줄이기`와 `재진입성 높이기`의 문제로 수렴한다.

## 지금 기준으로 다시 묶은 실천 체크리스트
- 이 코드는 나중에 바꿀 때 수정 범위가 국소적인가?
- 추상화가 멋 부리기가 아니라 테스트 가능성과 교체 가능성을 높이는가?
- 해피 패스 외의 경계 조건과 실패 시나리오를 생각했는가?
- 에러를 어디서 처리하고 어디서 전파할지 책임이 분명한가?
- 테스트가 리팩토링과 배포를 허용할 정도로 동작을 고정해주는가?
- 상층 엔트리포인트만 읽어도 프로그램 흐름과 실패 지점이 보이는가?
- 공용화가 진짜 재사용성인지, 아니면 맥락 없는 잡동사니 축적인지 점검했는가?
- 팀 컨벤션과 자동 검사가 읽기 비용을 줄여주는가?
- 사용하는 언어의 기본 철학을 거슬러 복잡성을 억지로 늘리고 있지 않은가?

## 후속 심화 포인트
- 원문 후반의 Python/Go/JS/Kotlin 세부 메모를 `실행 모델`, `상태 관리`, `에러 노출`, `패키지 경계`의 4축으로 한 번 더 압축할 수 있다.
- 테스트 파트를 `개발 중 검증`, `배포 전 검증`, `운영 중 검증`으로 더 명확히 나눌 수 있다.
- `main / handler / logic / adapter` 감각을 다른 노드와 연결해 공통 설계 패턴으로 보강할 수 있다.
- util/common 경계 문제를 [[./Architecture]]나 [[./Software]]와 더 촘촘히 엮어 재사용성 판단 기준으로 심화할 수 있다.

## 정리 메모
- 기존 baseline을 유지한 채, 원문의 넓은 언어 메모를 `설계 원칙의 언어별 구현 실험`으로 다시 묶었다.
- `main` 레이아웃, util 경계, 빠른 리턴을 별도 축으로 끌어올려 코딩을 함수 미학보다 프로그램 상층 설명력의 문제로 보강했다.
- 테스트와 에러 처리를 각각 따로 떼지 않고 `변경 허용`과 `복구 가능성`이라는 두 운영 질문으로 다시 연결했다.
- 이 노드는 결국 코딩 팁 모음이 아니라, **변경 가능한 구조를 만들기 위한 기초 판단 문법 노드**라는 점을 더 선명하게 고정했다.
