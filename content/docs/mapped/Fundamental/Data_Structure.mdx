# Data_Structure

- 원본: `source/shdkej-content/Fundamental/Data_Structure.md`
- 상태: 심화 재구성 완료
- 역할: 만다라트 `Fundamental` 축에서 자료구조 선택 기준, 대표 구조, 알고리즘 감각을 함께 묶는 대응 노트

## 핵심 논지
이 노트의 중심은 개별 자료구조를 외우는 데 있지 않다. 저자는 계속해서 **"어떤 입력과 작업 패턴에서 무엇을 선택해야 하는가"**를 묻는다. 즉 자료구조를 이름 목록으로 배우기보다, 접근 방식, 메모리 배치, 삽입/삭제 비용, 탐색 방식, 순서성, 중복 허용 여부, 데이터 분포, 입출력 빈도에 따라 판단하는 감각을 만들려는 노트다.

## 만다라트 위치에서 읽기
이 페이지가 `Fundamental/Data_Structure`에 놓여 있다는 점이 중요하다. 여기서 자료구조는 특정 언어의 컬렉션 API 설명서가 아니라, 이후 [[Coding]], [[Computer_Architecture]], [[Tool]], [[Network]] 같은 노드를 읽을 때 계속 바닥에 깔리는 `연산 선택의 기본 문법`에 가깝다.

- `Fundamental` 안의 자료구조는 구현 트릭보다 `무슨 연산을 빠르게 만들고 싶은가`를 먼저 묻는다.
- 같은 배열·해시·트리라도 여기서는 프레임워크 사용법보다 `메모리 배치`, `탐색 비용`, `삽입/삭제 반경`, `입력 형태`가 더 중요하다.
- 그래서 후반의 BFS/DFS, DB 인덱스, trie, 검색 엔진 메모도 주제 이탈이 아니라 `구조를 고르면 어떤 탐색이 가능해지는가`를 보여 주는 확장 사례로 읽는 편이 맞다.

결국 이 노드는 자료구조 사전이 아니라, **문제를 보면 먼저 어떤 연산 표면을 떠올릴지 훈련하는 기초 노드**다.

## 이 노드가 다루는 문제
- 배열, 연결 리스트, 해시 테이블, 트리, 그래프, 스택, 큐, 힙을 언제 구분해서 써야 하는가
- 자료구조와 알고리즘이 실제로 어디서 만나는가
- 메모리 관점의 스택/힙과 자료구조 관점의 스택/힙은 어떻게 다른가
- 탐색, 정렬, 최단경로, 인덱스 같은 실전 문제에서 어떤 구조를 떠올려야 하는가
- "문제를 보면 어떤 구조가 숨어 있는지"를 어떻게 감지할 수 있는가

## 선택 기준
원문 초반의 체크리스트는 이 노트 전체를 관통하는 판단 축이다.

- 접근 방식: 임의 접근이 중요한가, 순차 탐색이 허용되는가
- 메모리 공간: 연속 메모리가 유리한가, 포인터 기반 연결이 허용되는가
- 순서성: 입력 순서나 정렬 상태를 보존해야 하는가
- 동시성: 여러 작업이 동시에 접근할 때 구조적 제약이 있는가
- 중복 허용: 같은 값을 여러 번 담아야 하는가
- 변경 가능성: 값이 자주 바뀌는가, 구조 변경이 잦은가
- 읽기/쓰기 빈도: 조회 중심인가, 삽입/삭제 중심인가
- 크기와 분포: 데이터 크기, 희소성, 키 분포가 어떤가

이 기준을 보면 저자는 자료구조를 "정답 목록"이 아니라 **연산 비용과 데이터 성질의 조합 문제**로 본다.

## 원문 발췌 구조
- 선택 기준 층: 접근 방식, 메모리 공간, 순서성, 중복, 변경 빈도처럼 `무엇을 빨리 하고 싶은가`를 먼저 적는다.
- 기본 구조 층: 배열, 연결 리스트, 해시 테이블, 트리, 그래프, 스택/큐/힙으로 내려가며 각 구조의 대표 연산 차이를 붙든다.
- 알고리즘 접속 층: BFS/DFS, 정렬, DP, 분할 정복, 다익스트라로 이동하며 구조와 풀이를 분리하지 않는다.
- 탐색 가속 층: 우선순위 큐, trie, DB 인덱스, 검색 엔진 메모로 넘어가 `찾기 위해 미리 조직하는 법`을 확장한다.
- 언어/시스템 층: Golang, Python, Redis 자료형과 메모리 스택/힙 구분을 덧대며 추상 개념이 실제 구현에서 어떻게 바뀌는지 확인한다.

이 배열을 보면 원문은 교과서 챕터 순서대로 정리된 문서가 아니라, **기본 구조 → 문제 풀이 → 검색 최적화 → 실제 구현 층**으로 넓어지는 학습 노드다.

## 기본 구조 해석

### 1. 배열, 가장 단순하지만 강력한 기준선
배열은 연속 메모리를 바탕으로 인덱스 접근이 빠르다. 그래서 탐색 자체가 아니라 **위치를 이미 알고 있을 때의 접근 속도**가 핵심 장점이다.

- 장점: 인덱스 접근이 빠름, 구조가 단순함
- 약점: 중간 삽입/삭제 시 이동 비용이 큼
- 연결되는 감각: "조회가 많고 위치 계산이 가능하면 배열 쪽이 기본값"

원문은 Python list까지 같이 언급하면서, 언어가 추상화를 제공해도 내부에서는 여전히 메모리 배치와 재배치 비용이 중요하다는 점을 암시한다.

### 2. 연결 리스트, 변경 비용을 줄이기 위한 구조
연결 리스트는 배열의 삽입/삭제 부담을 줄이기 위한 발상으로 등장한다. 대신 원하는 위치를 찾는 탐색이 느리다.

- 장점: 노드 연결만 바꾸면 삽입/삭제가 쉬움
- 약점: 탐색이 느림, n번째 접근이 비효율적임
- 핵심 판단: "구조 변경은 자주 일어나지만 임의 접근은 적은가"

원문이 연결 리스트를 설명하면서 곧바로 트리, 그래프, BFS/DFS로 이동하는 흐름은 중요하다. 저자에게 연결 리스트는 단독 구조라기보다 **노드와 연결이라는 사고방식의 입문**이다.

### 3. 해시 테이블, 검색 속도를 얻기 위한 공간-분포 전략
해시 테이블은 원문에서 반복적으로 궁금증을 불러오는 구조다. 저자는 단순히 "검색 O(1)"을 외우지 않고 왜 그런지, 배열과 뭐가 다른지, 충돌이 생기면 왜 느려지는지, 공간 효율과 어떤 관계가 있는지를 캐묻는다.

- 핵심 장점: 키 기반 탐색이 빠름
- 핵심 변수: 해시 함수 품질, 충돌 처리 방식
- 저자의 질문: "딕셔너리와 해시 테이블은 어떻게 연결되는가", "왜 굳이 해시화하는가"

이 노트에서 해시 테이블은 단순 설명 대상이 아니라, **속도와 공간의 교환**, **주소 계산을 통한 접근**, **현실 구현의 타협**을 이해하는 출발점이다.

### 4. 트리, 그래프, 힙, 우선순위 큐
원문은 각 구조를 엄밀히 분리해 교과서식으로 정리하기보다, 문제 풀이 경험 속에서 서로 연결해 이해한다.

- 트리: 계층 구조와 탐색 구조의 기반
- 그래프: 경로, 연결, 네트워크 문제의 기본 표현
- 힙: 완전 이진 트리 성질을 이용한 우선순위 관리 구조
- 우선순위 큐: "정렬된 상태를 계속 유지해야 할 때"의 실용 구조

특히 우선순위 큐에 대해 "미리 다 받아 한번 정렬하는 것과 언제 다른가"를 묻는 부분이 좋다. 이 질문은 정적 일괄 처리와 동적 삽입 환경을 구분하는 감각으로 이어진다.

## 이 노드를 묶는 연산 사다리
원문 전체를 다시 읽으면 자료구조 이름보다 아래 네 질문이 반복된다.

### 1. 바로 가야 하는가, 순서대로 훑어도 되는가
배열과 해시는 `특정 위치/키로 곧바로 가는 능력`을 대표하고, 연결 리스트는 그 대신 구조 변경을 쉽게 만든다. 즉 첫 갈림길은 늘 `임의 접근 vs 순차 이동`이다.

### 2. 지금 바꿔야 하는가, 나중에 한 번에 정리해도 되는가
중간 삽입/삭제가 많으면 연결 구조나 힙 쪽이 유리해지고, 입력을 다 받은 뒤 정렬해도 되면 배열 기반 일괄 처리도 가능하다. 원문이 우선순위 큐를 궁금해하는 이유도 여기 있다.

### 3. 연결을 따라가야 하는가, 키를 계산해 바로 가야 하는가
트리·그래프는 관계 자체가 문제의 본체일 때 등장하고, 해시·인덱스·trie는 `찾기`를 빠르게 만들기 위해 미리 구조를 심어 두는 방식이다.

### 4. 메모리/실행 층을 같이 봐야 하는가
스택/힙, 포인터, 배열 주소, 인덱스 메모가 같이 적혀 있는 이유는 자료구조가 순수 추상 개념이 아니라 실제 메모리 배치와 실행 비용을 끌고 다니기 때문이다.

이 사다리를 잡으면 원문 후반의 알고리즘 메모도 `어떤 구조 위에서 어떤 연산을 빠르게 만들 것인가`라는 같은 질문 아래 다시 붙는다.

## 자료구조와 알고리즘의 접점
이 노트는 자료구조만 말하다가 자연스럽게 정렬, 그래프 탐색, 다익스트라, DP, 분할 정복으로 넘어간다. 이는 저자가 둘을 별개 과목으로 보지 않는다는 뜻이다.

### 탐색과 순회
- BFS: 큐가 필요하다
- DFS: 스택적 사고 또는 재귀와 연결된다
- 그래프/트리 문제는 방문 여부, 현재 위치, 방향성 관리가 핵심이다

원문에 있는 질문들, 예를 들면 "어떻게 현재 위치를 기억하지?", "이미 들렸던 곳을 어떻게 체크하지?"는 알고리즘 공식보다 **상태 표현 방식**을 먼저 고민하는 태도를 보여준다.

### 정렬
- merge sort: 분할 후 병합, 안정적인 사고 훈련의 중심
- quick sort: 피벗 선택이 성능에 중요
- bubble sort, insertion sort: 단순하지만 성능 한계가 뚜렷

여기서 중요한 포인트는 정렬 알고리즘 암기보다 **왜 분할이 잘못되면 성능이 무너지는가**를 감각적으로 붙잡으려는 흐름이다.

### 동적 계획법과 분할 정복
원문은 둘을 비슷해 보이지만 다른 방식으로 구분한다.

- 분할 정복: 문제를 나누고, 각 조각을 독립적으로 푼 뒤 합친다
- 동적 계획법: 중복되는 부분문제를 허용하고 저장해 재사용한다
- 재귀: 문제 축소의 표현 방식이지, 그 자체가 항상 전략은 아니다

이 구분은 훗날 알고리즘 문제 풀이뿐 아니라 시스템 설계에서도 재사용 가능한 사고 틀이다.

### 우선순위 큐를 따로 오래 붙드는 이유
원문에서 우선순위 큐는 단순히 `힙의 응용`으로 끝나지 않는다. 저자는 `리스트로도 만들 수 있는데 왜 굳이 힙을 쓰지?`, `다 정렬해 두는 것과 언제 다르지?`를 계속 묻는다.

이 질문을 원문 맥락대로 압축하면 다음과 같다.

- 모든 데이터를 다 받은 뒤 한 번 정렬하면 되는가
- 아니면 데이터가 계속 들어오는 동안 `가장 급한 것`만 계속 뽑아야 하는가
- 삽입 때마다 전체를 다시 훑는 비용을 줄여야 하는가

즉 우선순위 큐는 추상 정의보다, **정렬을 한 번 끝내는 문제와 정렬 상태를 계속 유지해야 하는 문제를 가르는 구조**로 읽는 편이 맞다. 이 대목이 살아 있어야 다익스트라, 작업 스케줄링, 이벤트 처리 같은 후속 문제와도 연결된다.

## 메모리 관점과 자료구조 관점의 겹침
원문은 [[Computer_Architecture]]를 링크하며 스택과 힙의 의미가 맥락에 따라 달라진다고 적는다. 이 대목은 아주 중요하다.

- 컴퓨터 구조 관점의 스택/힙: 프로그램 메모리 영역과 할당 방식의 이야기
- 자료구조 관점의 스택/큐/힙: 데이터 접근 규칙과 연산 특성의 이야기

같은 단어라도 층위가 다르다는 점을 분리해 이해해야 혼동이 줄어든다. 이 노트는 자료구조 학습이 결국 메모리 모델, 실행 모델과 완전히 분리되지 않는다는 사실을 보여준다.

## 실전 감각으로 번역한 핵심 질문
원문 전반에 흩어진 질문을 실전형 판단으로 압축하면 다음과 같다.

1. 조회가 많은가, 변경이 많은가
2. 순서가 중요한가, 키 기반 탐색이 중요한가
3. 모든 데이터를 한 번에 다 아는가, 중간중간 들어오는가
4. 최단경로/연결성/네트워크 문제인가
5. 방문 여부와 상태 저장이 핵심인가
6. 인덱스로 바로 가는 구조가 필요한가
7. 메모리 사용량과 구현 복잡도를 얼마나 감수할 수 있는가

이 질문들은 자료구조를 외우는 대신 문제를 읽는 체크리스트로 쓸 수 있다.

## 원문에서 반복되는 숨은 패턴

### 1. "이론 이름"보다 "왜 그 구조를 떠올려야 하는가"
저자는 B+트리, trie, priority queue, 인덱스 같은 용어를 적어두지만, 관심은 정의보다 선택 근거에 있다.

### 2. 구현보다 먼저 감지 능력
"이 문제가 트리/그래프 구조라는 걸 어떻게 알아차릴까"라는 질문은 매우 핵심적이다. 이는 코딩 전에 문제를 구조화하는 능력을 중시한다는 뜻이다.

### 3. 자료구조와 문제 풀이를 분리하지 않음
배열, 해시, 트리, 그래프를 설명하다가 바로 BFS, 다익스트라, DP, 인덱스로 연결한다. 즉 자료구조는 문제를 푸는 렌즈이지, 독립된 암기 목록이 아니다.

### 4. 언어별 구현 차이에 대한 감각
Python, Go, Redis, DB 인덱스까지 함께 적어두는 방식은 "추상 개념이 언어와 시스템에서 어떻게 구체화되는가"를 신경 쓴다는 신호다.

## 문제를 보고 구조를 알아차리는 신호
원문에서 가장 자주 되돌아오는 질문은 결국 이것이다. `정의는 알겠는데, 문제를 보면 언제 그 구조를 떠올려야 하지?` 이 감각을 원문 메모 기준으로 압축하면 아래처럼 읽을 수 있다.

### 배열/리스트를 먼저 떠올릴 때
- 위치 접근이 자주 나오고 인덱스로 상태를 관리하기 쉽다.
- 전체를 한 번 순회하거나, 정렬 후 양끝/인접 원소를 비교하는 흐름이 보인다.
- 입력이 연속 구간, 순서 보존, 고정 크기 버퍼처럼 읽힌다.

### 해시 테이블/딕셔너리를 먼저 떠올릴 때
- `빠르게 찾기`, `존재 여부`, `중복 체크`, `카운팅`이 핵심이다.
- 값 자체보다 키로 곧바로 접근해야 한다.
- 순서보다 membership과 빈도 관리가 중요하다.

### 연결 리스트, 트리, 그래프를 먼저 떠올릴 때
- `현재 위치`, `다음 노드`, `연결`, `경로`, `방문 여부` 같은 표현이 나온다.
- 입력이 그대로 2차원 배열이 아니어도, 관계를 노드-간선 구조로 바꿔 읽는 편이 자연스럽다.
- 상하 관계, 포함 관계, 네트워크, 추천/링크처럼 연결 자체가 문제의 본체다.

### 큐/스택/우선순위 큐를 먼저 떠올릴 때
- 처리 순서가 중요하다.
- 먼저 들어온 것을 먼저 처리하거나, 나중 것을 되돌리거나, 가장 급한 것을 먼저 꺼내야 한다.
- 데이터 전체를 다 모아 정렬하는 것이 아니라, 들어오는 동안 계속 꺼내 써야 한다.

이 문단의 핵심은 정답표를 외우는 것이 아니라, **문제 문장 속 동사를 보고 필요한 연산을 먼저 감지하는 습관**이다. 원문이 계속 `어떻게 알아차리지?`를 묻는 이유도 여기에 있다.

## 탐색 최적화라는 하나의 묶음
원문 후반의 trie, DB 인덱스, 검색 엔진 메모는 따로 놀지 않는다. 모두 `원하는 것을 빨리 찾기 위해 데이터를 미리 어떻게 조직할 것인가`라는 한 질문 아래 묶인다.

### 해시 테이블
- 키를 해시로 바꿔 바로 접근한다.
- 평균적으로 빠르지만 충돌과 공간 전략이 중요하다.

### B트리 / B+트리 인덱스
- 디스크/페이지 단위 접근을 줄이도록 정렬된 탐색 구조를 만든다.
- 해시보다 범위 검색과 순차 접근에 강하다.

### trie
- 문자열 공통 접두어를 구조 자체에 녹여 검색한다.
- 자동완성, prefix search, 링크 후보 탐색 같은 장면과 잘 맞는다.

### 검색 엔진의 역인덱스
- 문서를 단어 기준으로 다시 배열해 검색 시간을 줄인다.
- 단순 포함 여부를 넘어서 위치, 거리, 랭킹까지 함께 계산한다.

이렇게 보면 원문은 자료구조를 교과서 챕터별로 배우기보다, **검색을 빠르게 만드는 여러 설계 방식**을 느슨하게 연결해 이해하려 한다. 이 축은 [[Tool]]의 검색성 메모나 [[Network]]의 라우팅/경로 감각과도 이어질 수 있다.

## 언어와 시스템 층에서 다시 읽기
원문 후반에 Golang, Python, Redis 타입 메모가 붙고, 중간에는 스택/힙과 DB 인덱스가 섞인다. 이는 저자가 자료구조를 `언어 내장 타입 설명`으로 한정하지 않는다는 뜻이다.

- 언어 층: list, dict, tuple, set처럼 언어가 어떤 기본 컨테이너를 주는가
- 저장소 층: Redis string/list/set/hash/sorted set처럼 시스템이 어떤 접근 패턴을 기본값으로 제공하는가
- DB 층: B트리/B+트리 인덱스처럼 디스크 환경에서 무엇이 탐색을 빠르게 만드는가

즉 이 노드는 `배열/트리/해시 정의`만 배우는 페이지가 아니라, **같은 선택 기준이 언어 런타임, 저장소, 검색 시스템까지 어떻게 반복되는지 보는 기본 감각 노드**다.

## 보존해야 할 긴장
- 단순한 구조가 이해와 구현에 유리하지만, 성능 요구가 커지면 더 복잡한 구조가 필요하다.
- 이론적 시간복잡도는 중요하지만, 실제로는 메모리 배치, 충돌, 입력 형태, 데이터 유입 방식이 함께 작동한다.
- 자료구조를 배우면 알고리즘이 쉬워지지만, 반대로 알고리즘 문제를 많이 풀어야 자료구조 선택 감각도 생긴다.
- 언어가 자료구조를 감춰 주더라도 내부 원리를 모르면 성능과 동작을 오해하기 쉽다.
