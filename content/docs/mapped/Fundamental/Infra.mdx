# Infra

- 원본: `source/shdkej-content/Fundamental/Infra.md`
- 상태: 기준선 수준으로 재구성한 1:1 대응 페이지
- 역할: 만다라트 `Fundamental > Infra` 카테고리 노드의 구조적 읽기 기준

## 한눈에 보는 요약
이 노트는 AWS 서비스 암기장이나 서버 구축 체크리스트가 아니다. 원문 전체를 다시 읽으면, 저자는 인프라를 `무엇을 어디에 올릴 것인가`보다 `어떤 성격의 시스템에 어떤 운영 부담을 허용할 것인가`의 문제로 다룬다. 정적 페이지, CDN, S3, Lambda 같은 가벼운 조합에서 시작해 데이터베이스, 캐시, 오토스케일링, 메시지 큐, VPC, IAM, EKS, 관측, 비용 최적화까지 점점 층을 늘려 가지만, 중심 질문은 일관된다. **서비스 성격, 실시간성, 비용, 통제력, 복구 가능성에 맞춰 어느 정도의 인프라를 선택할 것인가**가 이 노드의 핵심이다.

## 핵심 논지
원본에서 인프라는 클라우드 제품 모음이 아니라, 워크로드의 특성에 따라 점진적으로 두꺼워지는 운영 구조다. 가장 단순한 서비스는 정적 자산과 CDN, 스토리지, API Gateway 같은 얇은 레이어로도 충분하지만, 상태 관리, 트래픽 변화, 실시간성, 장애 대응, 조직 운영이 커질수록 데이터베이스, 캐시, 큐, 권한, 네트워크 경계, 모니터링, 배포 자동화가 필요해진다. 다만 저자는 무조건 복잡한 구조로 가는 대신, 가능하면 서버리스와 매니지드 서비스를 먼저 검토하고, 한계가 분명해질 때만 더 무거운 구성을 받아들이려는 태도를 보인다.

## 이 노드의 중심 질문
- 현재 서비스는 정말 서버가 필요한가, 아니면 정적 배포와 서버리스로도 충분한가?
- 실시간성, 지속 연결, 트랜잭션, 상태 관리는 어느 순간 서버리스의 한계를 드러내는가?
- 클라우드는 관리 포인트를 줄여 주지만 어떤 통제력을 가져가며, 그 대가를 어디서 치르게 되는가?
- AWS의 네트워크, 권한, 스토리지, 로드밸런서, 쿠버네티스 구성은 각각 어떤 운영 부담을 새로 만든는가?
- 비용은 인스턴스 가격표보다 egress, NAT, 공인 IP, 잘못된 기본 설정에서 더 크게 새지 않는가?
- 좋은 인프라 이해는 서비스 목록 암기보다 `증상, 경계, 비용, 복구 루프`를 함께 보는 능력이 아닌가?

## 원문 구조 재정리

### 1. 인프라는 규모에 따라 층이 늘어난다
도입부의 `최소한의 인프라 → 안정적 운영을 위한 추가 인프라 → 강력하게 탄탄한 인프라` 구분은 이 노드 전체의 압축판이다.
- 최소한의 인프라: 정적 페이지, CDN, S3, Lambda, API Gateway
- 운영 안정성 보강: 데이터베이스, 캐시, 오토스케일러
- 더 탄탄한 구조: 메시지 큐

즉 저자에게 인프라는 처음부터 거대한 플랫폼이 아니라, **서비스 성격과 장애 허용도에 따라 점층적으로 추가되는 운영 장치**다.

### 2. 클라우드는 편하지만, 편리함만큼 추상화 비용이 있다
원문은 클라우드의 장점을 인정한다.
- 하드웨어 효율 고민을 덜 해도 된다.
- 관리 포인트가 줄어든다.
- 다양한 서비스 연결이 쉽다.
- 서버리스 조합이 가볍고 빠르다.

동시에 단점도 놓치지 않는다.
- 내부를 완전히 통제할 수 없다.
- 하드웨어 수준 최적화는 감춰진다.
- 벤더 종속과 멀티 클라우드 복잡성 사이에서 판단이 필요하다.

즉 이 노드는 클라우드를 만능 해답보다 **관리 포인트 감소와 통제력 감소를 교환하는 추상화 선택**으로 본다.

### 3. 서버리스는 기본 선호지만 만능은 아니다
원문에서 가장 자주 드러나는 태도는 `가능하면 서버리스부터`다.
- 관리 포인트 감소
- 모듈화와 서비스 연결 용이성
- 가벼움
- 작업 실행 보장

하지만 곧바로 한계도 적는다.
- 지속 연결이 필요하면 Lambda가 맞지 않는다.
- 빠른 응답이 중요하면 콜드스타트가 치명적일 수 있다.
- 사용자가 많아지면 요청당 비용과 상태 관리 비용이 커진다.
- 채팅처럼 웹소켓, 브로드캐스팅, 낮은 지연이 중요한 서비스는 Fargate나 Redis pub/sub 같은 대안이 더 낫다.

즉 저자는 서버리스를 신념으로 밀어붙이지 않고, **실시간성, 지속성, 상태성**이 강해지는 순간 서버 구조로 넘어가야 한다는 감각을 분명히 갖고 있다.

### 4. 비용은 겉가격보다 경계 비용에서 터진다
원문은 비용을 단순히 인스턴스 타입 비교로만 보지 않는다.
특히 반복해서 나오는 것은 다음이다.
- 공인 고정 IP 비용
- bandwidth, egress 비용
- NAT Gateway 비용
- S3와의 통신량
- EKS egress 확인
- DynamoDB의 과금 방식

이 흐름은 중요하다. 저자는 인프라 비용을 `서버 한 대 가격`보다 **경계 통신과 기본 설정이 만드는 숨은 누수**로 읽는다. NAT Gateway와 VPC Endpoint 메모가 반복되는 것도 같은 이유다.

## 반복해서 드러나는 판단 축

### 1. 정적 vs 상태성
원문 첫머리의 `스테이트리스 한건 정적으로, 스테이트풀한건 서버를 사용해야 한다`는 문장은 이 노드 전체의 출발점이다.
- 정적 자산은 최대한 CDN과 정적 배포로 밀어낸다.
- 상태 관리가 필요한 순간 서버, DB, 캐시, 세션 관리, 연결 유지 문제가 나타난다.

이 축은 이후 서버리스 판단, 채팅 아키텍처, 캐시 전략, 데이터 저장소 선택까지 계속 이어진다.

### 2. 편의성 vs 통제력
클라우드는 운영 편의와 추상화를 제공하지만, 내부 자원 배치와 세밀한 통제는 감춰진다. 이 때문에 저자는 클라우드를 쓰더라도 아래를 계속 궁금해한다.
- 내가 실제로 제어 가능한 경계는 어디인가?
- 관리 책임은 AWS와 사용자 중 어디에 남는가?
- 매니지드 서비스가 편한 대신 나중에 어떤 제약을 만드는가?

### 3. 간단함 vs 실시간성
서버리스는 간단하지만, 실시간 서비스는 간단함을 깨뜨린다.
- 채팅, 웹소켓, 빠른 응답
- Redis 기반 브로드캐스팅
- 지속 연결과 세션 상태
- 콜드스타트 회피를 위한 상시 실행 비용

즉 인프라 선택은 단순함을 끝까지 유지할 것인지, 실시간 요구를 위해 더 무거운 구조를 받아들일 것인지의 조정이다.

### 4. 서비스 구성 vs 운영 복구
원문에는 구성 요소 설명만 있는 것이 아니라, 실제 장애와 수정 흐름이 많이 남아 있다.
- Beanstalk 접속 문제
- NAT/IGW/VPC Endpoint 고민
- EKS 서브넷, 로드밸런서, ingress 문제
- OS 패치와 도커 재시작 위험
- 인증서, DNS, Redis, RDS 접근 문제

따라서 이 노드는 이상적인 구성도보다 **문제가 났을 때 어디를 확인하고 어떻게 되돌릴지**까지 포함한 운영 메모다.

## 세부 영역별 해석

### 서버리스 판단 메모
원문에서 서버리스는 단순히 비용 절감 수단이 아니다.
- 먼저 시도해 볼 기본 선택지
- 작은 시스템을 빠르게 올리는 방식
- 관리 포인트를 줄이는 방법

그러나 실시간 채팅 사례를 붙이면서 저자는 중요한 보정도 한다.
- Lambda는 요청당 비용 구조라 사용자 수가 늘면 부담이 커진다.
- 콜드스타트는 대화형 서비스 체감 품질을 깨뜨린다.
- 웹소켓 상태 관리가 어렵다.

그래서 `Lambda + 큐` 조합에 대한 호기심을 남기면서도, 실제 서비스에서는 Fargate, Redis pub/sub, DynamoDB, RDS의 조합으로 넘어가는 판단을 적어 둔다. 이 점에서 이 노드는 기술 예찬보다 **워크로드 적합성 평가 노트**에 가깝다.

### AWS 네트워크와 경계 감각
VPC, subnet, route table, IGW, NAT Gateway, VPC Endpoint, ENI, Route53, CloudFront 메모는 모두 다른 주제처럼 보이지만, 실제로는 한 질문으로 묶인다.

**외부와 내부를 어떤 경계로 나누고, 어느 지점에서 연결을 허용할 것인가?**

- 퍼블릭/프라이빗 서브넷 구분
- IGW와 NAT의 차이
- S3/DynamoDB Endpoint 활용
- CloudFront와 ALB를 통한 우회
- 프런트는 퍼블릭, 백엔드는 프라이빗에 두는 방식

즉 저자는 네트워크를 단순 연결이 아니라, **노출면과 보호면을 어떻게 분리할지**의 문제로 보고 있다.

### IAM과 책임 분리
IAM 부분은 짧지 않지만 핵심은 명확하다.
- 최소 권한 원칙
- root 계정 사용 제한
- 그룹 기반 권한 관리
- 사용자 책임과 AWS 책임 구분
- MFA, 키 회전, 접근 분석

이 부분은 AWS 자격증 학습 메모처럼 보이지만, 실제로는 운영 감각을 드러낸다. 저자는 인프라를 서버와 네트워크만이 아니라 **누가 무엇을 할 수 있는가**까지 포함한 시스템으로 이해하려 한다.

### 쿠버네티스와 운영 복잡성
후반의 EKS, ingress, NLB/ALB, CNI, warm IP, ArgoCD, Helm, lifecycle policy, external-dns 메모는 이 노드가 어디서 두꺼워지는지 보여준다.
- 쿠버네티스는 강력하지만, 자동으로 복잡성이 따라온다.
- 네트워크, 보안그룹, 서브넷, 인증서, ingress controller, IP 할당까지 함께 이해해야 한다.
- 버전 업데이트나 인프라 코드 변경은 순식간에 운영 리스크가 된다.

이 구간은 `Infra` 노드가 단순 AWS 학습을 넘어 실제 운영 장애와 플랫폼 관리 경험으로 연결되는 부분이다.

### 관측과 장애 대응
원문은 모니터링을 별도 노드로 다루지만, 여기서도 인프라 운영의 기본 감각이 반복된다.
- 에러가 뜨면 바로 반응 와야 한다.
- `/var/log/messages` 같은 기본 로그를 본다.
- 인프라 모니터링과 APM을 함께 본다.
- 장애는 서비스가 의도한 대로 동작하지 않는 모든 것이다.

즉 인프라는 구성 후 방치하는 기반이 아니라, **이상 징후를 빨리 감지하고 복구 루프에 진입하기 위한 관측 표면**까지 포함한다.

## 이 노드에서 특히 보존해야 할 긴장
- 서버리스가 관리 포인트를 줄여 주지만, 실시간성과 지속 연결은 서버 구조를 다시 요구한다.
- 클라우드는 빠른 시작을 돕지만, 숨은 비용과 통제력 상실을 함께 가져온다.
- 쿠버네티스는 공통 플랫폼을 만들지만, 작은 팀에는 과한 관리 복잡성을 줄 수 있다.
- 프라이빗 네트워크는 안전하지만, NAT, ALB, Endpoint, DNS, 인증서 구성이 더 어려워진다.
- 매니지드 서비스를 많이 쓸수록 구축은 쉬워지지만, 내부 작동 원리를 이해하지 못하면 장애 대응이 느려진다.

이 긴장을 제거하지 않고 남겨 두는 편이 원문 의도에 더 맞다.

- [[Deep Knowledge/Devops]]: 배포, 자동화, 복구 가능성, 플랫폼 팀 역할과 연결된다.

## 선호 신호
저자는 인프라를 최신 스택 과시보다 워크로드 적합성, 숨은 비용, 네트워크 경계, 권한 분리, 관측 가능성, 복구 가능성의 문제로 다루는 경향이 강하다. 또한 가능하면 서버리스와 매니지드 서비스를 먼저 검토하지만, 그 한계가 드러나는 순간 실시간성, 지속 연결, 운영 복잡성을 기준으로 더 무거운 구성을 받아들이는 실용적 태도를 보인다.
