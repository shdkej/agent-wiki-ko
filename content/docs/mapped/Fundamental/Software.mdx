# Software

- 원본: `source/shdkej-content/Fundamental/Software.md`
- 상태: baseline 이후 점진 심화
- 역할: 만다라트 `Fundamental` 축에서 코딩, 아키텍처, 제품, 운영을 잇는 상위 카테고리 노드

## 한눈에 보는 요약
이 노드에서 `소프트웨어`는 언어·패턴·프레임워크를 모아 놓은 기술 노트가 아니다. 사람의 흐름을 돕고, 바뀌는 현실을 버티고, 실패해도 복구 가능하게 만들며, 결국 운영·문서·수익화까지 감당해야 하는 **현대의 메타도구**에 가깝다. 원문은 동시성, 객체지향, 테스트, 캐시, 12 factor, 마이크로서비스, 언어별 메모, 무료/유료 고민까지 넓게 흩어져 있지만, 실제 중심 질문은 하나다. **사람이 감당할 수 있으면서도 계속 수정 가능한 소프트웨어를 어떻게 만들 것인가.**

## 이번 패스에서 다시 선명해진 핵심
기존 mapped 노트의 중심은 유지하되, 원문을 다시 읽어 보면 이 페이지의 핵심은 단순 `변경 용이성`만이 아니다. 실제로는 아래 다섯 축이 함께 움직인다.

1. **사람 중심성**: 소프트웨어는 `사람 - 소프트웨어 - 사람` 사이의 매개물이다.
2. **변경 가능성**: 종이와 달리 수정 가능하다는 점이 본질적 장점이다.
3. **복구 가능성**: 앱 재시작, 멱등성, 에러 처리, 로그, 테스트가 계속 반복된다.
4. **운영 확장성**: 캐시, 대용량 조회, API 명세, 12 factor, 마이크로서비스가 모두 뒤따른다.
5. **가치 교환 구조**: 무료/유료 경계, 개인 맞춤형 서비스, 잠자는 앱 운영 지원까지 시야에 들어온다.

즉 이 노드는 `좋은 코드`보다 넓고, `아키텍처`보다 생활 세계와 가깝다.

## 이 노드가 만다라트에서 맡는 자리
- [[./Coding]]이 구현 습관과 코드 단위의 변경 반경을 다룬다면, 이 노드는 **왜 그런 구현 습관이 필요한지**를 설명하는 상위 기준 노드다.
- [[./Architecture]]가 정책·경계·컴포넌트 분리를 다룬다면, 이 노드는 그보다 더 넓게 **소프트웨어라는 매체 전체의 성질**을 다룬다.
- [[../Deep Knowledge/Product]], [[../Deep Knowledge/Devops]], [[../Integration/Business]]로 넘어가기 직전의 **기술-운영-가치 연결면** 역할을 한다.

## 원문 발췌 구조
원문은 아래 흐름으로 읽는 편이 정확하다.

1. 소프트웨어 일반론: 사람, 도구, 변경, 컨셉, 무료/유료, 맞춤형 서비스
2. 안정성과 실무 기본기: 200ms, 앱 안정성, UTC 저장
3. 개발 방법론: Agile, concurrency, pointer/reference, design pattern
4. 설계 원칙: abstraction, declarative, encapsulation, low coupling, pure function
5. 운영 기술: 직렬화, 람다 아키텍처, JPA, 정적/동적 콘텐츠, 12 factor, 서버리스
6. 서비스 구조: 문서화, 캐시, 버저닝, 대용량 데이터, 필터/정렬
7. 언어별 메모: Python, Go, Dart, JS, Kotlin, 함수형 프로그래밍

이 배열은 `언어 사전`이 아니라 **소프트웨어라는 큰 질문 아래 여러 층을 붙여 보는 공부 방식**에 가깝다.

## 소프트웨어를 읽는 4개 층위

### 1. 매체 층: 소프트웨어는 사람 사이의 도구다
원문 초반의 `사람 - 소프트웨어 - 사람`, `제작자 - 소프트웨어 - 유저`는 이 페이지의 출발점이다. 소프트웨어는 코드 그 자체보다 **사람의 행위와 판단을 매개하는 구조물**이다.

그래서 `좋은 소프트웨어는 기존의 흐름을 바꾸는 게 아니라 길을 잡아주고 확장된 길을 열어준다`는 문장이 중요하다. 이 노드에서 좋은 소프트웨어는 사용자를 다시 훈련시키는 폐쇄적 체계보다, 이미 하던 일을 더 넓고 덜 고통스럽게 하게 만드는 도구다.

### 2. 구현 층: 변경은 본질이고, 안정성은 기본이다
원문은 소프트웨어의 가장 큰 장점으로 `변경 가능성`을 잡는다. 기준을 세우고 그 기준 안에서는 빠르게 선택하는 태도도 여기에서 나온다. 다만 변경 가능성은 추상 미덕이 아니라 곧바로 안정성 문제와 붙는다.

- 앱은 계속 껐다 켜도 문제 없어야 한다.
- API는 멱등성을 가져야 한다.
- 에러 핸들링을 먼저 세팅한다.
- 어떤 동작을 기대하는지 테스트 코드로 남긴다.
- 날짜는 UTC로 통일 저장한다.

즉 `쉽게 바뀌는 구조`와 `쉽게 망가지지 않는 구조`를 동시에 추구한다.

### 3. 운영 층: 코드만 돌아가면 끝이 아니다
원문 후반의 12 factor, OpenAPI, 캐시 규칙, 대용량 데이터 조회 제한, 버저닝, EOL 안내는 모두 같은 방향을 가리킨다. 소프트웨어는 파일 몇 개가 아니라 **배포되고, 문서화되고, 관측되고, 오래 유지되는 서비스**다.

여기서 중요한 반복 신호는 다음과 같다.
- 환경과 설정을 분리한다.
- 무상태성을 선호한다.
- 로그를 파일보다 스트림처럼 다룬다.
- 캐시는 명확한 룰로만 허용한다.
- 큰 조회는 시간 범위와 배치로 나눈다.
- 버전과 종료 시점을 안내해야 한다.

### 4. 가치 층: 소프트웨어는 상품이자 관계 유지 장치다
무료/유료 경계, 유튜브 광고 정책, 개인 맞춤형 서비스, 잠자는 앱 운영 지원 메모는 주변부가 아니다. 저자는 코드 작성 이후에도 `누가 왜 돈을 내는가`, `운영 지원이 가치가 되는가`, `개인 대상 서비스는 어떤 구조여야 하는가`를 함께 묻는다. 이 점에서 이 노드는 [[../Integration/Business]]와 직접 닿는다.

## 원문을 다시 묶은 운영 사다리

### 1. 가벼운 기본형을 먼저 세운다
원문은 무거운 기본 기능이 주렁주렁 달린 도구보다, 단순하지만 필요할 때 붙일 수 있는 구조를 선호한다. `Convenience <-> Control`, no dependency, low coupling, lightweight가 계속 반복된다.

이 선호는 도구 취향이 아니라 운영 철학이다. 기본형이 가벼워야:
- 시스템이 무엇을 하는지 직접 이해하기 쉽고
- 실패 지점을 찾기 쉽고
- 나중에 다른 방식으로 갈아타기도 쉽다.

### 2. 확장은 기존 코드를 덜 건드리는 방식이어야 한다
원문에서 Strategy, Command, Adapter, Proxy를 보는 시선은 패턴 외우기가 아니다.
- Strategy: 교체를 국소화한다.
- Command: 기능 단위를 분리한다.
- Adapter: 외부 세계와 내 구조의 마찰을 줄인다.
- Proxy: 무거운 작업을 흐름상 뒤로 미룬다.

또한 `외부 모듈 사용 시 호출 함수와 실제 동작 함수를 분리`한다는 메모는, 외부 의존을 무조건 피하겠다는 뜻이 아니라 **붙일 때도 경계를 세워 붙이겠다**는 뜻에 가깝다.

### 3. 실패를 먼저 상상한다
원문은 happy path보다 edge case를 더 자주 떠올린다. 앱 안정성, 멱등성, 에러 처리, 빠른 리턴, boundary check, load/stress test, failover, 캐시 안전조건이 모두 이 축으로 묶인다.

즉 여기서 소프트웨어는 `정상 시연`보다 **이상 상황에서도 계속 버티는 상태**를 향해 설계된다.

### 4. 커질수록 데이터와 흐름을 다시 나눠 본다
람다 아키텍처, 대용량 목록 조회 제한, 정렬/필터, 캐시 정책, 동시성·비동기, MSA 메모를 함께 읽으면, 저자는 규모가 커질수록 기능보다 **흐름 제어와 조회 비용 관리**가 중요해진다고 본다.

- 실시간과 배치를 나누고
- 읽기와 쓰기를 나누고
- 동기와 비동기를 구분하고
- 한 번에 다 보여주기보다 필터와 시간 범위를 둔다.

이는 [[./Architecture]]의 `최소 기반 → 관측 → 필요한 층 추가` 감각과도 닿는다.

## 언어 메모를 이 노드 안에서 읽는 법
이 페이지의 언어별 메모는 문법 공부 부록이 아니다. 각각이 어떤 운영 스타일을 선호하는지 보여주는 재료다.

### Python
객체, 레퍼런스, copy, generator, requests, 비동기, pickling 등을 넓게 모아 둔다. 핵심은 `쉽게 만들되 내부 동작을 알고 싶다`는 태도다.

### Golang
명시적 에러 처리, 단순함, 패키지 목적성, goroutine, context, HTTP handler 구조가 반복된다. 이는 이 노드 전반의 `투명성`, `호출 흐름 가시성`, `운영 친화성` 선호와 잘 맞는다.

### Javascript / Dart / Kotlin / FP
비동기, 실행 컨텍스트, 함수형, 상속 제한, immutable 같은 메모들은 단순 비교보다 **상태와 흐름을 어떻게 통제할 것인가**라는 공통 질문으로 읽는 편이 정확하다.

즉 이 노드에서 언어는 `좋아하는 언어 목록`이 아니라, 소프트웨어를 어떤 철학으로 다룰지 시험하는 실험장이다.

## 이 페이지에서 특히 보존해야 할 긴장

### 편의 vs 통제
편의 기능은 좋지만, 기본 기능이 많아질수록 내가 모르는 동작과 종속성이 늘어난다.

### 추상화 vs 투명성
추상화는 변경을 쉽게 하지만, 지나치면 실제 동작과 실패 지점을 가린다.

### 빠른 개발 vs 운영 복구성
라이브러리와 자동화는 속도를 높이지만, 문제가 나면 이해와 복구가 어려워질 수 있다.

### 개인 맞춤 서비스의 가능성 vs 개인의 낮은 지불 의사
개인 대상 서비스는 매력적이지만, 시장 구조상 무료 기대가 강하다는 현실이 붙어 있다.

### 마이크로서비스의 분리 이점 vs 관리 포인트 증가
분리는 의존성을 줄이지만, 통신·문서화·데이터 조합·운영 복잡성을 키운다.

이 긴장을 지우면 원문은 평범한 개발 개론처럼 평평해진다.

## 저자의 선호 신호

### 1. 소프트웨어를 `작동하는 기계`보다 `관리 가능한 구조`로 본다
기능 구현보다 기준, 테스트, 멱등성, 버전, 로그, 문서화가 계속 붙는다.

### 2. 외부 도구를 쓰더라도 내부 경계를 알고 싶어 한다
무조건 직접 구현만 고집한다기보다, 외부 의존이 어디서 시작되고 끝나는지 보이게 두고 싶어 한다.

### 3. 코드와 서비스 운영을 분리하지 않는다
API 명세, 캐시, 대용량 데이터, 서버리스, 마이크로서비스, 유료화가 같은 노트 안에 있는 것이 그 증거다.

### 4. 소프트웨어를 결국 사람의 생활을 조정하는 도구로 본다
200ms 반응 시간, 개인 맞춤 서비스, 잠자는 앱 운영 지원, 디지털 도구 선호가 모두 사람의 실제 사용감과 연결된다.

## 재사용 가능한 질문 세트
이 노드의 내용을 실무 판단 질문으로 압축하면 다음과 같다.

- 이 소프트웨어는 사람의 기존 흐름을 돕는가, 불필요하게 깨는가?
- 바뀔 때 수정 범위를 국소화할 수 있는가?
- 재시작, 중복 요청, 네트워크 불안정, 데이터 증가를 견디는가?
- 의존성과 자동화는 얻는 이득에 비해 과하지 않은가?
- 운영자가 상태를 이해하고 문서화하고 복구할 수 있는가?
- 목록·검색·정렬·필터가 사용자 재진입성을 높이는가?
- 기능 제공을 넘어 가치 교환 구조까지 설계되어 있는가?

## 이번 런에서 남기는 정리
이 페이지를 다시 읽으면 `Software`는 단순히 `Coding보다 큰 노트`가 아니다. 오히려 코딩, 아키텍처, 운영, 제품, 가치화를 한 칸 위에서 묶어 보는 **기술 생활세계의 루트 노드**에 가깝다. 그래서 이 노드는 개별 기술 항목을 더 예쁘게 정리하는 것보다, `사람 중심 → 변경 가능성 → 복구 가능성 → 운영 확장 → 가치 교환`의 흐름을 계속 선명하게 유지하는 편이 드리프트를 줄인다.
