# Network

- 원본: `source/shdkej-content/Fundamental/Network.md`
- 상태: 심화 정리 완료, 점진 개선 대상
- 역할: 만다라트 `Fundamental > Network` 카테고리 노드의 1:1 대응 페이지

## 핵심 요약
이 노트는 네트워크를 단순한 프로토콜 암기 목록이 아니라, **연결을 만들고, 전달을 보장하고, 병목을 줄이고, 역할을 분리하는 방법**으로 이해하려는 사고의 흔적이다. 원문은 TCP/IP, HTTP, DNS, 로드밸런서, CDN, 네트워크 장비를 넓게 훑지만, 반복해서 드러나는 관심사는 따로 있다. 무엇이 어느 계층에서 처리되어야 하는지, 성능 저하는 어디서 생기는지, 웹 요청 하나가 실제로 어떤 경로를 지나가는지, 그리고 분산된 장치들이 어떻게 전체 서비스 경험을 만든다는 것인지에 대한 감각이다.

특히 이 페이지는 만다라트의 `Fundamental` 칸에 놓여 있다는 점이 중요하다. 여기서 네트워크는 특정 클라우드 서비스 사용법이나 운영 도구 선택보다 앞서, **요청이 길을 찾고, 전달되고, 분산되고, 보호되는 기본 사다리**를 붙잡는 기초 노드로 읽는 편이 맞다.

## 이 노드의 중심 질문
- 브라우저에서 서버까지의 요청은 실제로 어떤 층을 통과하며 완성되는가?
- TCP, HTTP, DNS, TLS, 로드밸런서, CDN은 각각 무엇을 책임지는가?
- 성능과 안정성 문제는 네트워크의 어느 지점에서 발생하고 완화되는가?
- 인프라 구성 요소를 늘리는 것은 단순한 장비 추가가 아니라 어떤 역할 분리를 뜻하는가?

## 원문 구조 재정리

### 원문 발췌 구조
- `Internet / WWW`: 인터넷과 웹의 기원을 짧게 붙잡는다.
- `TCP, UDP`: 전송 보장, BDP, RTT, 혼잡 제어, keep-alive를 이해하려 한다.
- `Browser - Server`: 브라우저 요청 하나가 DNS, TCP, TLS, HTTP를 어떻게 통과하는지 순서로 적어 둔다.
- `HTTP`: 버전 차이, 요청/응답 구조, REST API, 멱등성까지 묶어 본다.
- `프록시 서버 / 로드밸런서 / 인프라`: 어디서 어떤 책임을 맡길지 역할 분리를 고민한다.
- `네트워크 과부하 / Load Balance / CDN`: 병목, 분산, 캐시, 대기열, 해외 접근 문제를 함께 다룬다.
- `API / Network Machine / ETC`: WebSocket, gRPC, GraphQL, 계층 장비, NAT, VLAN, 서브넷 같은 후속 학습 질문을 넓게 남긴다.

### 1. 인터넷과 웹의 기원
- 인터넷은 단순한 선 연결이 아니라 데이터 네트워크 위에 문서 공유 체계가 올라간 구조다.
- WWW는 URI, HTTP, HTML을 통해 문서 연결을 가능하게 만든 계층으로 읽힌다.
- 즉 네트워크 자체와 웹은 같은 말이 아니라, **전송 기반 위에 문서 시스템이 얹힌 관계**다.

### 2. 전송 계층 감각, TCP와 UDP
- TCP는 순서, 전달, 재전송, 흐름 제어, 혼잡 제어 같은 "안전한 전달"의 성격이 강하다.
- UDP는 이런 보장을 줄이는 대신 단순하고 가볍다.
- 원문은 TCP를 통해 데이터가 직렬화되고 목적지를 확인하며 전달된다고 이해하고 있고, 네트워크 지연과 윈도우 크기까지 함께 보려 한다.

### 3. 웹 요청의 실제 흐름
- URI 입력
- DNS 조회
- IP 확인
- TCP 연결
- TLS/SSL 암호화
- HTTP 요청 전송
- 서버 응답 수신
- 브라우저 렌더링

원문이 중요하게 보는 것은 각 기술을 따로 외우는 것이 아니라, **요청 하나가 여러 프로토콜의 협업 결과**라는 점이다.

### 4. HTTP의 진화
- HTTP/0.9: 매우 단순한 문서 전달
- HTTP/1.0, 1.1: 메서드, 헤더, 인증, 지속 연결, 부분 요청
- HTTP/2: 멀티플렉싱, 헤더 압축, 프레임 기반 전송, 서버 푸시
- HTTP/3: 보다 현대적인 전송 개선의 방향

여기서 핵심은 버전별 기능 암기가 아니라, **연결 재사용과 병렬성, 헤더 처리 효율, 지연 감소**라는 개선 축이다.

### 5. 네트워크 장비와 역할 분리
- L2, L3, L4, L7 장비 구분
- 로드밸런서, 방화벽, CDN, 프록시의 책임 분배
- 어디까지를 웹서버가 처리하고, 어디부터 인프라 계층이 처리하는가

원문은 이 구분을 특히 중요하게 여긴다. 서비스 품질은 애플리케이션 코드만이 아니라, **앞단 장치에서 무엇을 대신 처리하게 하느냐**에 따라 크게 달라지기 때문이다.

## 네트워크를 읽는 4개 사다리

### 1. 이름을 길로 바꾸는 사다리
URI, DNS, IP 같은 항목은 사용자가 입력한 이름을 실제 목적지로 바꾸는 단계다. 이 사다리가 흔들리면 요청은 아예 출발하지 못한다.

### 2. 길 위에 데이터를 안전하게 올리는 사다리
TCP, UDP, RTT, BDP, 혼잡 제어, keep-alive는 데이터를 어떤 방식으로 흘릴지 정하는 층이다. 여기서는 `빠르냐`보다 **얼마나 보장할지, 얼마나 오래 연결을 재사용할지**가 더 중요하다.

### 3. 도착지와 약속을 맞추는 사다리
HTTP 버전, 메서드, 상태 코드, REST, 멱등성, gRPC, WebSocket, GraphQL 메모는 모두 `도착한 뒤 어떤 문법으로 상호작용할 것인가`를 묻는다. 즉 네트워크가 단지 전송이 아니라 **서비스 인터페이스의 외피**이기도 하다는 뜻이다.

### 4. 병목과 경계를 바깥층에서 조절하는 사다리
프록시, 로드밸런서, CDN, NAT, VLAN, 방화벽, 서브넷 메모는 요청을 그냥 보내는 데서 끝나지 않고, 어디서 걸러 내고, 나눠 보내고, 보호하고, 가까이 둘지를 다룬다. 이 층이 추가되면서 네트워크는 통신 기술에서 **운영 설계 기술**로 바뀐다.

## 핵심 논지

### 네트워크는 "연결"보다 "역할 분리"의 시스템이다
원문에는 DNS, TCP, HTTP, TLS, 프록시, 로드밸런서, CDN이 한 노트에 함께 들어 있다. 이는 주제가 산만해서가 아니라, 실제 서비스 환경에서는 이 요소들이 한 요청 경로 위에서 연쇄적으로 작동하기 때문이다. 저자는 네트워크를 단순한 통신 규약이 아니라, **주소 해석, 연결 성립, 안전한 전송, 요청 표현, 부하 분산, 캐싱, 필터링**이 계층적으로 분담된 구조로 보고 있다.

### 좋은 네트워크 이해는 "웹 요청 추적 능력"으로 드러난다
브라우저에 주소를 입력한 뒤 무슨 일이 일어나는지 단계별로 설명하려는 대목이 길게 등장한다. 이는 네트워크를 시험용 지식이 아니라, 장애 분석과 성능 개선에 직접 쓰이는 실행 감각으로 익히려는 방향을 보여준다. 결국 이 노드의 중심에는 "패킷이 어떻게 가는가"뿐 아니라, **문제가 생기면 어느 층부터 의심해야 하는가**라는 질문이 숨어 있다.

### 성능 문제는 대역폭 하나로 설명되지 않는다
BDP, RTT, 혼잡 제어, keep-alive, 멀티플렉싱, CDN, 캐싱, 로드밸런서가 한 노트 안에 묶여 있는 이유는 분명하다. 저자는 속도를 단순 스펙이 아니라 **지연, 동시성, 경로 길이, 연결 재사용, 캐시 적중, 병목 분산**의 합성 결과로 이해하려고 한다. 즉 "빠른 네트워크"는 회선 숫자보다 구조 설계의 문제다.

### 네트워크는 주소와 경계를 관리하는 기술이기도 하다
원문 후반의 DNS, NAT, VLAN, 서브넷, 브로드캐스트 도메인, 라우터 메모는 앞부분의 HTTP/TCP 설명과 별개가 아니다. 요청이 어디로 가야 하는지, 누가 외부와 내부를 가르는지, 어느 범위까지 같은 네트워크로 볼 것인지를 정하지 않으면 그 위의 전송 최적화도 의미가 약해진다. 이 노드에서 네트워크는 결국 **전달 기술 + 경계 설계**의 결합으로 읽는 편이 맞다.

## 반복해서 나타나는 판단 축

### 1. 어느 계층에서 해결할 것인가
- 애플리케이션에서 처리할 일인가
- 웹서버에서 처리할 일인가
- 로드밸런서나 CDN에서 흡수할 일인가
- 방화벽이나 네트워크 장비가 맡을 수 있는가

이 축은 이후 [[../../mapped/Deep Knowledge/Infra]]나 [[../../mapped/Deep Knowledge/Monitoring]] 같은 노드와도 강하게 연결될 수 있다.

### 2. 무엇을 보장하고 무엇을 포기할 것인가
- TCP는 전달 보장을 더 많이 한다.
- UDP는 가벼움을 더 많이 남긴다.
- HTTP 버전도 연결 관리와 지연 최적화 방식이 다르다.
- 로드밸런서와 CDN도 기능이 많아질수록 제어 범위와 복잡도도 커진다.

즉 네트워크 선택은 언제나 기능 추가가 아니라 **보장과 단순성 사이의 조정**이다.

### 3. 가까움과 반복 사용이 성능을 만든다
CDN 설명에서 저자는 "왜 캐시가 빠른가"를 물으며 물리적 거리와 네트워크 혼잡의 문제를 감각적으로 이해하려 한다. 이는 CPU 캐시 비유로 이어지는데, 결국 공통된 생각은 **자주 쓰는 것을 더 가까운 곳에 두면 전체 비용이 줄어든다**는 것이다.

### 4. 주소 공간과 경계는 초반에 대충 넘길 수 없는 기반이다
- DNS는 이름을 목적지로 번역한다.
- NAT는 제한된 공인 주소를 여러 내부 자원과 연결한다.
- VLAN, 서브넷, 브로드캐스트 도메인은 네트워크 범위를 어떻게 자를지 결정한다.
- 라우터와 기본 경로 설정은 결국 "어디로 나가야 인터넷이 되는가"의 문제다.

원문이 이 항목들을 ETC처럼 흩어 적어도, 실제 관심은 분명하다. **주소 체계와 경계 설계가 어긋나면 그 위의 애플리케이션 통신도 함께 흔들린다.**

## 앞단 장치 3종을 가르는 최소 지도

### 프록시
- 요청을 대신 받아 넘기거나, 중간에서 내용을 바꾸거나, 필터링 규칙을 적용하는 지점이다.
- 원문에서 TRACE 메서드 메모가 붙어 있는 것도, 프록시를 단순 통과 장치보다 `중간 변화 추적면`으로 보고 있기 때문이다.

### 로드밸런서
- 여러 서버 앞에서 연결을 분산하고, 포트/프로토콜 수준 혹은 HTTP 수준 정책을 적용한다.
- SSL 종료, 접속량 조절, 타겟 그룹 분리, 정적 응답 일부 처리 같은 메모는 `요청을 어느 백엔드로 어떻게 보낼지`를 다루는 층으로 묶인다.

### CDN
- 사용자와 더 가까운 지점에서 캐시된 응답을 내주거나, 원서버 앞에서 트래픽과 공격을 흡수한다.
- 원문이 반복해서 묻는 `왜 더 빠른가`는 결국 서버 성능보다 `거리 단축과 재사용`의 효과를 이해하려는 질문이다.

이 셋은 모두 앞단에 놓일 수 있지만 관심 초점이 다르다.
- 프록시: **중간 정책과 변형**
- 로드밸런서: **분산과 연결 제어**
- CDN: **가까움과 캐시 재사용**

원문이 이 장치들을 따로따로 적지 않고 섞어 두는 이유도 여기에 있다. 실제 운영에서는 셋이 경쟁 개념이 아니라, **서버 앞에서 서로 다른 종류의 부담을 덜어 주는 연속 계층**이기 때문이다.

## 주소·경계 메모를 다시 읽는 3단계

### 1. 이름을 푼다
DNS, URI, 포트 메모는 `사용자가 부르는 이름`을 `실제 목적지`로 푸는 층이다.

### 2. 안과 밖을 나눈다
NAT, private instance, 방화벽, 라우터 기본 경로 메모는 `누가 외부와 직접 통신하고 누가 숨을지`를 정하는 층이다.

### 3. 내부 반경을 자른다
VLAN, 서브넷, 브로드캐스트 도메인 메모는 내부를 한 덩어리로 두지 않고, 장애와 소음이 번지는 범위를 조절하는 층이다.

이렇게 읽으면 원문 후반의 ETC 메모는 잡다한 꼬리표가 아니라, HTTP/TCP 앞쪽 설명이 실제 네트워크 설계로 내려오는 손잡이로 바뀐다.

## 세부 해석

### TCP, BDP, 혼잡 제어
원문은 BDP를 이해하려다 중간에 질문을 남긴다. 이 흔적은 중요하다. 저자는 정의 암기보다, 실제로 "회선 위에 동시에 얼마나 많은 데이터가 떠 있을 수 있는가"를 감각적으로 파악하려 한다. 혼잡 제어 역시 단순 알고리즘 이름보다, 네트워크 상태를 모르니 먼저 조심스럽게 탐색하고 이후 천천히 늘린다는 운영 철학으로 읽는 편이 더 맞다.

### HTTP는 문서 전송 규약이면서 애플리케이션 인터페이스다
원문은 요청/응답 구조, 메서드, 상태 코드, REST API, 멱등성까지 함께 적고 있다. 이는 HTTP를 페이지 불러오기용 기술에 한정하지 않고, **서비스 동작을 노출하는 공용 인터페이스**로 보고 있음을 보여준다. 특히 DELETE의 멱등성에 대한 고민은 네트워크 지식이 서버 내부 설계로 바로 이어진다는 점을 드러낸다.

또한 HTTP/1.1의 지속 연결, HTTP/2의 멀티플렉싱, HTTP/3에 대한 짧은 메모가 한자리에 있는 것은, 저자가 프로토콜 버전을 `최신이 무엇인가`보다 **요청이 덜 기다리게 만드는 방식의 진화**로 읽고 있음을 보여 준다.

### 프록시, 로드밸런서, CDN은 단순히 "앞단"이 아니라 정책 계층이다
- 프록시는 요청 변형과 필터링 지점이 된다.
- 로드밸런서는 트래픽 분산, SSL 종료, 정적 응답, 접속 제어를 맡을 수 있다.
- CDN은 캐싱과 지리적 분산, 원서버 보호, 트래픽 흡수 역할을 한다.

즉 이 장치들은 서버 앞에 있는 부속품이 아니라, **서비스 정책을 외부화하는 계층**이다.

### 장비 계층 구분은 추상화 감각을 훈련한다
L1부터 L7까지의 설명은 엄밀한 교과서식 정리보다 실무적 감각에 가깝다. 어떤 장비가 어디까지의 정보를 보고 판단하는지 이해하면, "왜 어떤 장비는 포트까지만 보고, 어떤 장비는 HTTP 헤더나 페이로드까지 보는가"가 자연스럽게 이어진다. 이 노트는 정확한 정의보다 **추상화 수준별 책임 차이**를 몸에 익히려는 노트다.

### API 메모는 전송 방식 비교로 읽는 편이 맞다
원문 말미의 WebSocket, gRPC, GraphQL 메모는 별도 주제로 튀어나온 것처럼 보이지만, 사실 이 노드의 범위를 넓혀 준다. 저자는 네트워크를 브라우저-HTML 요청만이 아니라 **실시간 연결, 스트리밍, 질의 유연성**까지 포함한 인터페이스 운반층으로 보고 있다. 그래서 HTTP 요청/응답, 양방향 연결, 스트리밍 RPC를 한 시야에서 비교하려는 흔적이 남는다.

GraphQL에 `server-side cache?` 같은 메모가 붙어 있는 점도 흥미롭다. 이 노트는 단지 API 스타일을 비교하는 게 아니라, **표현력이 늘어날수록 캐시와 운영이 어떻게 달라지는가**까지 이미 같이 묻고 있다.

### 미해결 질문을 남기는 방식 자체가 학습 구조다
원문에는 `BDP가 크다는 게 무슨 뜻인지`, `CDN 캐시가 왜 빠른지`, `VLAN은 왜 필요한지` 같은 질문형 메모가 그대로 남아 있다. 이 노트의 가치는 정리 완료된 지식만이 아니라, **이해가 멈춘 지점과 다음 탐색 포인트를 노출하는 방식**에도 있다. 따라서 mapped에서도 모든 공백을 매끈하게 지우기보다, 어떤 축이 아직 탐구 중인지 보이게 두는 편이 원문 성격에 더 맞다.

## 숨은 패턴
- 웹 기술과 네트워크 기술을 분리하지 않고 하나의 흐름으로 본다.
- 이론을 적다가도 바로 실무 질문으로 내려온다.
- 장비 이름이나 프로토콜 이름보다 "어디서 병목을 줄이는가"를 더 궁금해한다.
- 개념 정리 중간에 스스로의 이해 공백을 드러내며 다음 학습 질문을 남긴다.
- `Fundamental` 노드답게 특정 벤더보다 보편 계층과 책임 경계를 먼저 세우려 한다.

이 패턴은 저자가 네트워크를 자격증형 지식보다 **서비스 운영 관점의 시스템 사고**로 흡수하고 있음을 보여준다.

## 보존해야 할 긴장
- 단순한 구조가 좋은가, 기능이 많은 앞단 계층이 좋은가
- 전달 보장을 늘릴수록 성능 손해는 어디까지 감수할 수 있는가
- 서버 내부 로직으로 처리할 것과 인프라 계층으로 밀어낼 것을 어떻게 구분할 것인가
- 표준 프로토콜 사용과 서비스별 예외 처리를 어떻게 균형 잡을 것인가

이 긴장을 남겨두는 편이 원문 의도에 더 충실하다. 이 노트는 하나의 정답을 선언하기보다, 네트워크 설계가 계속되는 경계 조정임을 보여준다.

## 이번 런에서 보강한 포인트
- 기존 baseline은 유지한 채, 프록시·로드밸런서·CDN을 한 번에 구분하는 `앞단 장치 3종을 가르는 최소 지도`를 추가해 원문 후반의 운영 장치 메모가 더 빨리 분리되도록 다듬었다.
- DNS·NAT·VLAN·서브넷·기본 경로 메모를 `이름 풀기 → 안과 밖 나누기 → 내부 반경 자르기`의 3단계로 다시 묶어, ETC 구간이 잡항이 아니라 실제 경계 설계 손잡이로 읽히게 보강했다.
- 기존의 `이름 → 전달 → 인터페이스 → 경계/분산` 4개 사다리 해석은 유지하되, 그중 마지막 사다리를 실제 장치와 주소 체계 수준까지 더 세분화했다.

## 다음 개선 포인트
- HTTP/3, QUIC, TLS 종료 지점 같은 현대적 설명을 원문 경계 안에서 더 명확히 보강할 수 있다.
- 프록시, 로드밸런서, CDN의 차이를 비교표 없이도 더 선명하게 보이도록 예시를 보강할 수 있다.
- NAT, 서브넷, VLAN, 브로드캐스트 도메인을 `외부 연결`, `내부 분리`, `장애 반경 통제` 관점에서 한 번 더 정리할 수 있다.

## 한 줄 결론
이 노트에서 네트워크는 선과 패킷의 문제가 아니라, **요청이 통과하는 여러 경계에서 책임을 어떻게 나누고 병목을 어떻게 줄일지 판단하는 시스템 설계 감각**이다.
